## 一、现状评估

### 1.1 现有实现概览

当前 IPS 模块（`scp-ips-api` + `scp-ips-sdk`）基于 **Flowable 引擎**实现了一套初版审批流框架，主要包含：

| 已实现能力 | 评估 |
|-----------|------|
| 待办/待领/已办任务查询 | 基础能力，已有 v1/v2 两套接口，整体可用 |
| 任务完成（同意） | 已实现，兼容委派后处理逻辑 |
| 前加签/后加签 | 已实现，但注释中标注「多次加签只能显示第一次前加签的处理人」，存在已知 Bug |
| 委派任务 | 已实现 |
| 任务认领 | 已实现（候选人组模式） |
| 已读标记 | 已实现（以流程变量存储，机制较简陋） |
| 审批意见记录 | 已实现，意见类型含 AUDIT/REJECT/DELEGATE/ASSIGN_BEFORE/ASSIGN_AFTER |
| WebSocket 实时通知 | 已有基础架构 |
| 流程监听（全局） | GlobalProcessListener/GlobalTaskBeforeListener 已有骨架 |
| 业务模块接入样例 | 需求提报版本审批（DFP）已集成，含 Delegate 和 Listener |

### 1.2 关键缺口

| 缺口项 | 严重程度 | 说明 |
|--------|---------|------|
| **驳回（退回）功能未实现** | 高 | `BackTaskVO` 已定义但 Service/Controller 均未实现；驳回是审批流基本操作 |
| **无管理端配置能力** | 高 | 流程模板完全通过 BPMN XML 文件部署，业务方无法自主配置审批规则 |
| **无审批规则引擎** | 高 | 审批人硬编码在 BPMN 流程图中，条件分支需修改流程定义，无法动态配置 |
| **通知集成不完整** | 中 | 钉钉待办发送有 `AbstractDelegate.sendDingDingTodo()` 方法骨架，但实现注释为 TODO |
| **审批进度追踪 API 缺失** | 中 | 无面向申请人的「我的申请」列表及可视化进度接口 |
| **无撤回功能** | 中 | 申请人无法撤回进行中的审批 |
| **加签多次存在已知 Bug** | 中 | 代码注释已标注问题，多次加签场景不稳定 |
| **已读标记机制脆弱** | 低 | 以流程变量 `taskDefinitionKey#staffCode` 存储，无法支持批量/时间维度查询 |
| **无超时处理机制** | 低 | 无节点超时自动处理（自动通过/升级/提醒） |
| **无委托代理设置** | 低 | 委派仅为逐任务委派，不支持按时间段全局委托 |

### 1.3 现有代码质量问题（需配套修复）

| 问题 | 位置 | 影响 |
|------|------|------|
| 动态拼接 tableName 的 SQL | `BpmTaskServiceImpl` 约第 124 行 | 潜在注入风险（虽来源为框架 API，仍需排查）|
| Service 层缺乏显式事务控制 | `BpmTaskServiceImpl.complete()` | 审批意见已保存但任务完成失败时数据不一致 |
| 意见内容超 3900 字符被截断 | `AddHisCommentCmd` | fullMessage 与 message 字段分裂，查询不一致 |
| `complete()` 新旧两套接口并存 | Controller 层 | 旧接口 `/bpmTask/{taskId}/complete` 与新接口 `/bpmTask/complete` 功能重叠 |

## 三、分阶段实现计划

### 阶段一：已有实现缺陷修复（P0）

**目标：** 修复现有已实现功能中的已知缺陷，使当前框架稳定可用。P0 仅包含**已有代码实现**的功能，不新增未开始的能力。

#### 3.1.1 修复加签多次 Bug

| 项 | 内容 |
|----|------|
| **实现程度** | 前加签、后加签核心逻辑已实现，接口已上线（`/bpmDynamicTask/beforeAddSignTask`、`/bpmDynamicTask/afterAddSignTask`） |
| **缺陷描述** | 代码注释明确标注「如果多次加签只能显示第一次前加签的处理人来处理任务」，多次加签场景下处理人展示与流转均不正确 |
| **缺陷原因** | `BpmDynamicTaskServiceImpl.addSignTask()` 中通过 `parentTaskId` 判断是否为重复加签，若任务已有 parentTaskId 则跳过重新设置 owner/assignee 步骤，导致第二次加签后的子任务链路断裂，原处理人无法正确恢复为待处理状态 |

**修复方向：**
- 检查 `parentTaskId` 判断逻辑，确保多轮加签时 owner/assignee 链路正确重建
- 补充专项测试用例：一次前加签 → 被加签人完成 → 再次前加签 → 验证原处理人能正常收到任务

#### 3.1.2 补齐审批事务控制

| 项 | 内容 |
|----|------|
| **实现程度** | `BpmTaskServiceImpl.complete()` 业务逻辑已完整实现，覆盖委派处理、加签后续、审批意见记录 |
| **缺陷描述** | `complete()` 方法缺乏显式事务控制（未见 `@Transactional` 注解） |
| **缺陷原因** | 当 Flowable 任务完成操作抛出异常时，之前已执行的 `addComment()` 写入不会回滚，导致审批意见记录存在但任务实际未完成，数据不一致 |

**修复方向：**
- `BpmTaskServiceImpl.complete()` 增加 `@Transactional` 注解
- 确认 `addComment()` 调用位置在 Flowable 任务完成之后（保证事务语义正确）

#### 3.1.3 修复审批意见截断问题

| 项 | 内容 |
|----|------|
| **实现程度** | 审批意见记录已实现（`AddHisCommentCmd`），支持 AUDIT/REJECT/DELEGATE/ASSIGN_BEFORE/ASSIGN_AFTER 五种类型 |
| **缺陷描述** | 意见内容超过 3900 字符时被截断并存入 `message` 字段，完整内容存入 `fullMessage` 字段，但常规查询接口返回的是截断版 `message`，导致审批人填写的长意见丢失 |
| **缺陷原因** | `AddHisCommentCmd.execute()` 中对内容长度做了硬截断（3900 字符），设计上做了 message/fullMessage 双字段区分，但上层查询接口未统一使用 fullMessage |

**修复方向：**
- 查询接口统一返回 `fullMessage`；若字段有存储限制，评估是否放宽或改为 TEXT 类型
- 或去掉截断逻辑，仅保留单一字段

#### 3.1.4 优化已读标记机制

| 项 | 内容 |
|----|------|
| **实现程度** | 已读标记功能已实现，接口 `/bpmTask/{taskId}/read` 可用 |
| **缺陷描述** | 已读状态以 `taskDefinitionKey#staffCode` 为 key 存储在流程变量中，无法支持批量标记已读、按时间查询、跨任务统计等场景 |
| **缺陷原因** | 利用 Flowable 流程变量存储属于"借用"而非正式存储，流程变量本身为 key-value 结构，不适合集合/历史类数据；当任务量大时，流程变量膨胀也影响性能 |

**修复方向：** 新增独立数据表 `ips_bpm_task_read_record`：

```sql
CREATE TABLE ips_bpm_task_read_record (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    task_id     VARCHAR(64) NOT NULL,
    staff_code  VARCHAR(64) NOT NULL,
    read_time   DATETIME    NOT NULL,
    tenant_id   VARCHAR(64),
    INDEX idx_task_staff (task_id, staff_code)
);
```

### 阶段二：补全未开始的核心操作（P1，原 P0 移入）

> 以下条目在原规划中归属 P0，但代码调研确认**主体实现尚未开始**（仅有 VO 定义或完全空白），因此移入 P1 统一规划。

#### 3.2.1 驳回（退回）功能

**现状：** 仅有 `BackTaskVO` 数据结构定义（含 `distFlowElementId` 字段），`BpmDynamicTaskService` 接口和实现类中均**未见 backTask 方法**，Controller 层也无对应接口，属于「已预留接口定义，主体未实现」。

**实现要点：**
- 实现 `BpmDynamicTaskService.backTask(BackTaskVO)` 方法
- 支持两种退回模式：退回至发起节点（申请人重新填写） / 退回至指定历史审批节点（利用已预留的 `distFlowElementId`）
- `CommentTypeEnum` 新增 `BACK("退回")` 类型
- 退回后流程实例保持进行中状态，申请人重新提交后流程从退回节点重新推进
- 新增接口：`POST /bpmDynamicTask/backTask`
- **注意：** 退回至指定历史节点需使用 Flowable `changeActivityState` API，涉及活动迁移，实现复杂度较高；可先只支持退回至发起节点

#### 3.2.2 撤回功能

**现状：** 完全未开始，代码中无任何撤回相关实现。

**实现要点：**
- 新增 `BpmInstanceService.cancelInstance(processInstanceId, operatorCode, reason)` 方法
- 调用 `runtimeService.deleteProcessInstance()` 终止流程
- 利用已有 `ProcessStatusEnum.MANUAL_ABORT` 更新流程变量状态
- 新增接口：`POST /bpmTask/cancel`
- **权限约束：** 仅申请人本人可撤回，且流程须处于审批中状态

#### 3.2.3 申请人「我的申请」接口

**现状：** 完全未开始，当前仅有审批人侧的待办/已办接口，无申请人视角的查询能力。

**实现要点：**

| 接口 | 参数 | 返回 | 说明 |
|------|------|------|------|
| `GET /bpmInstance/myApply/page` | staffCode, status, pageNum, pageSize | PageResult\<ProcessInstanceVO\> | 分页查询本人发起的流程，支持按状态筛选 |
| `GET /bpmInstance/{processInstanceId}/detail` | processInstanceId | ProcessInstanceDetailVO | 流程详情（申请内容+审批进度+历史记录） |

`ProcessInstanceVO` 补充字段：`currentNodeName`（当前节点）、`currentAssignee`（当前负责人）。

#### 3.2.4 审批进度可视化数据结构

**现状：** 完全未开始，`ProcessInstanceVO` 中无节点维度的进度字段。

**实现要点：** 新增 `ApprovalNodeVO`，作为 `ProcessInstanceDetailVO` 的子结构：

```java
class ApprovalNodeVO {
    String nodeId;              // 节点 ID（flowElementId）
    String nodeName;            // 节点名称
    String nodeType;            // APPROVAL / GATEWAY_AND / GATEWAY_OR / COPY / START / END
    NodeStatus status;          // PENDING / IN_PROGRESS / COMPLETED / SKIPPED
    List<AssigneeVO> assignees; // 本节点审批人列表（并行场景含多人）
    Date startTime;
    Date endTime;
    String comment;             // 审批意见
    CommentTypeEnum action;     // 操作类型
}
```

### 阶段三：构建管理端配置能力（P1，自助配置）

**目标：** 业务管理员可通过管理端自主配置审批规则，无需开发介入部署新 BPMN。

> ⚠️ 这是本方案中技术复杂度最高的阶段，建议在阶段一、二完成并上线后再启动。

#### 3.3.1 审批模板数据模型设计

新增以下核心表：

**模板主表 `ips_bpm_template`：**

```sql
CREATE TABLE ips_bpm_template (
    id                  BIGINT PRIMARY KEY AUTO_INCREMENT,
    template_code       VARCHAR(64)  NOT NULL UNIQUE,   -- 模板唯一标识（对应 processDefinitionKey）
    template_name       VARCHAR(128) NOT NULL,           -- 模板名称
    module_code         VARCHAR(32)  NOT NULL,           -- 所属模块（DFP/SOP/MPS...）
    description         VARCHAR(512),
    status              VARCHAR(16)  NOT NULL DEFAULT 'DRAFT',  -- DRAFT/PUBLISHED/DISABLED
    allow_back          TINYINT(1)   DEFAULT 1,          -- 是否允许退回
    allow_transfer      TINYINT(1)   DEFAULT 1,          -- 是否允许转交
    allow_add_sign      TINYINT(1)   DEFAULT 1,          -- 是否允许加签
    allow_cancel        TINYINT(1)   DEFAULT 1,          -- 是否允许撤回
    callback_url        VARCHAR(256),                    -- 审批完成后的业务回调地址
    tenant_id           VARCHAR(64),
    create_time         DATETIME,
    update_time         DATETIME
);
```

**审批节点表 `ips_bpm_template_node`：**

```sql
CREATE TABLE ips_bpm_template_node (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    template_id     BIGINT       NOT NULL,
    node_order      INT          NOT NULL,    -- 节点顺序
    node_name       VARCHAR(128) NOT NULL,
    node_type       VARCHAR(32)  NOT NULL,    -- APPROVAL/COPY/FILL
    multi_mode      VARCHAR(32),             -- SEQUENTIAL（顺签）/AND（会签）/OR（或签）
    approver_type   VARCHAR(32)  NOT NULL,   -- FIXED/DIRECT_MANAGER/DEPT_HEAD/ROLE/SELF_SELECT
    approver_value  VARCHAR(512),            -- 固定人员列表/角色 code（JSON 格式）
    fallback_rule   VARCHAR(32),             -- 找不到审批人时：AUTO_APPROVE/TRANSFER_TO_ADMIN/TRANSFER_TO_MANAGER
    fallback_value  VARCHAR(128),
    timeout_hours   INT,                     -- 超时小时数（null 表示不限）
    timeout_action  VARCHAR(32),             -- 超时动作：AUTO_APPROVE/AUTO_REJECT/ESCALATE
    condition_expr  TEXT,                    -- 节点进入条件表达式（JSON 格式条件组）
    parent_node_id  BIGINT,                  -- 条件分支时的父节点
    INDEX idx_template (template_id, node_order)
);
```

#### 3.3.2 规则配置与 BPMN 生成策略

**方案 A（推荐）：动态解析，不生成 BPMN XML**

- 管理端配置存储在 `ips_bpm_template*` 表中
- 运行时由 **规则引擎服务** 直接读取配置，动态决定：
  - 下一个任务分配给谁
  - 当前节点是否满足完成条件（会签场景）
  - 是否需要跳转至其他分支
- Flowable 仅用作任务存储与历史记录引擎，不使用其条件网关等高级特性
- 流程定义使用**通用模板 BPMN**（只有开始节点、动态任务节点、结束节点），业务逻辑全部由 Java 代码驱动

**方案 B（不推荐）：每次配置发布时自动生成 BPMN XML 并部署**

- 将管理端配置转译为 BPMN XML，调用 Flowable Deployment API 部署
- 缺点：复杂的条件分支和并行网关转译难度高，版本管理复杂

#### 3.3.3 管理端 API 清单

| 接口 | 方法 | 功能 |
|------|------|------|
| `/bpmTemplate/page` | GET | 模板列表分页查询 |
| `/bpmTemplate/save` | POST | 新建/更新模板（含节点配置） |
| `/bpmTemplate/{id}/publish` | POST | 发布模板（校验后生效） |
| `/bpmTemplate/{id}/disable` | POST | 禁用模板 |
| `/bpmTemplate/{id}/simulate` | POST | 模拟测试（输入表单值，预览流程走向） |
| `/bpmTemplate/{id}/nodeList` | GET | 查询模板节点列表 |

### 阶段四：通知与集成增强（P1）

#### 3.4.1 完善钉钉/IM 消息通知

**现状：** `AbstractDelegate.sendDingDingTodo()` 有骨架但未完整实现。

**实现要点：**
- 封装统一通知接口 `BpmNotificationService`，支持多渠道：系统内推送（WebSocket）/ 钉钉 / 企业微信
- 通知渠道通过配置决定，不硬编码
- 消息卡片内容模板（Markdown/JSON）须可配置
- 关键通知场景须覆盖（参见设计大纲第 4.6 节）
- 失败重试机制：通知发送失败时写入重试队列，不阻塞审批主流程

#### 3.4.2 业务回调机制

新增 `BpmCallbackService`，在流程终结时异步调用业务方 Callback URL：

```java
// 回调参数
class BpmCallbackPayload {
    String processInstanceId;
    String businessKey;
    String templateCode;
    String result;          // APPROVED / REJECTED / CANCELED
    String operator;        // 最终操作人
    Date finishTime;
    String finalComment;    // 最终审批意见
}
```

- 使用异步消息队列（MQ）解耦，保证至少一次投递
- 回调失败时记录告警，支持手动重试

## 四、核心技术决策

### 4.1 是否继续使用 Flowable 作为底层引擎

**建议：保留 Flowable，但以「轻度使用」模式**

| 使用 Flowable 的点 | 说明 |
|-------------------|------|
| 任务生命周期管理 | 待办/已办/历史任务的存储与查询 |
| 流程实例管理 | 发起、终止、历史记录 |
| 审批意见（Comments）| 利用 ACT_HI_COMMENT 表 |
| 事件监听 | 全局流程开始/结束事件监听 |

| 不使用 Flowable 的点 | 替代方案 |
|--------------------|---------|
| 复杂 BPMN 条件网关 | 由 Java 规则引擎驱动，通用 BPMN 模板 |
| BPMN 可视化设计器 | 管理端自建表单式配置 |
| Flowable 原生表单 | 自建表单模型 |

### 4.2 新旧接口共存策略

- 旧接口（`/bpmTask/{taskId}/complete`）不删除，但标记为 `@Deprecated`
- 新业务模块接入一律使用新接口（`/bpmTask/complete` 基于 CompleteTaskVO）
- 制定时间窗口，在新接口稳定后，统一推动旧接入方迁移

### 4.3 多租户隔离

- 所有新增表均包含 `tenant_id` 字段
- 查询接口默认按当前租户过滤
- 流程变量中已有 `tenant` 变量，保持一致

## 五、关键风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 驳回功能实现复杂（涉及 Flowable 活动迁移） | 中 | 高 | 先只实现「退回至发起节点」，退回至任意历史节点作为后续迭代 |
| 加签 Bug 修复影响现有 DFP 集成 | 中 | 高 | 修复前回归 DFP 需求提报版本审批流完整测试用例 |
| 阶段三管理端规则引擎开发量大 | 高 | 中 | 阶段三前先收集业务方实际审批规则，评估复杂度再决策 |
| 已读标记表切换引起行为差异 | 低 | 低 | 历史任务已归档，仅新任务走新逻辑，无需迁移历史数据 |
| 通知集成（钉钉/IM）依赖外部平台配置 | 中 | 中 | 通知渠道设计为插件化，确保主流程不因通知失败而阻塞 |

## 六、阶段划分总结

| 阶段 | 核心工作项 | 优先级 | 说明 |
|------|-----------|--------|------|
| **阶段一** | 加签多次 Bug、事务控制、意见截断、已读标记优化 | P0 | 均为已有实现的缺陷修复 |
| **阶段二** | 驳回、撤回、我的申请接口、审批进度可视化 | P1 | 原 P0 中未开始的部分，移入 P1 |
| **阶段三** | 管理端模板配置、规则引擎、动态审批人计算 | P1 | 架构级新增，技术复杂度最高 |
| **阶段四** | IM 通知、业务回调机制 | P1 | 通知与集成增强 |
| **阶段五** | 委托代理、超时处理、批量审批、监控大盘 | P2 | 后期迭代 |

## 七、接入新业务模块的标准步骤（对外 SDK 视角）

面向未来新业务模块接入审批流，标准接入步骤：

1. **在管理端创建审批模板**：配置节点、审批人规则、表单字段
2. **在业务代码中调用发起接口**：
   ```java
   BpmStartInstanceDTO dto = new BpmStartInstanceDTO();
   dto.setTemplateCode("SOP_BUDGET_APPROVAL");  // 模板标识
   dto.setBusinessKey("SOP:BUDGET:20260101001"); // 业务键
   dto.setTitle("SOP 预算申请 - 2026 Q1");
   dto.setInitiator(currentUser.getStaffCode());
   dto.setVariables(Map.of("amount", 150000, "deptCode", "SUPPLY"));
   bpmInstanceService.startInstance(dto);
   ```
3. **配置业务回调接口**：实现 `POST /your-module/bpm/callback` 接收审批结果
4. **更新业务单据状态**：在回调中根据 `result`（APPROVED/REJECTED/CANCELED）更新本模块状态机
