# IPS 审批流设计大纲

> 版本：v0.1（初稿，头脑风暴阶段）
> 日期：2026-03-26
> 作者：AI 辅助调研整理
> 参考：SAP S/4HANA Fiori 审批体系、飞书审批中心、IPS 现有 Flowable 实现

---

## 一、背景与目标

### 1.1 背景

IPS（集成平台）已基于 Flowable 引擎实现了一套初版审批流框架，覆盖任务管理、加签、委派、审批意见等核心能力，并已在「需求提报版本审批」等场景落地。但当前框架属于技术驱动的低级封装，缺乏完整的管理端能力（流程模板配置、规则管理）、缺乏标准化的用户端体验（待办中心、进度追踪），整体属于「能用但不好用」的阶段。

### 1.2 目标

构建一套**面向供应链计划系统各业务模块通用**的审批流平台能力，使：
- **业务方** 能以低代码/零代码方式配置审批规则，无需开发介入
- **申请人** 能流畅发起审批、追踪进度、收到及时通知
- **审批人** 能高效处理待办、灵活加签/转交/退回
- **管理员** 能监控流程状态、处理异常任务
- **开发方** 能以标准 SDK 将各业务模块快速接入审批流，不重复造轮子

---

## 二、设计原则

| 原则 | 说明 |
|------|------|
| **通用优先** | 框架层与业务层解耦，审批流引擎不感知具体业务逻辑 |
| **配置驱动** | 审批规则（审批人、条件、层级）通过管理端配置，不写死代码 |
| **通知前置** | 审批通知须及时到达审批人，避免任务积压无人知晓 |
| **进度可见** | 申请人始终能看到当前审批停在哪、由谁负责 |
| **操作可逆** | 在流程未终态前，申请人可撤回、审批人可退回（不可逆操作需二次确认） |
| **与 IM 集成** | 通知消息须能在企业 IM（如钉钉/企业微信）完成快捷审批，减少跳转 |
| **渐进增强** | 核心功能先落地，高级功能（AI 推荐、超时自动处理）后期迭代 |

## 三、用户角色

| 角色 | 描述 | 核心诉求 |
|------|------|----------|
| **系统管理员** | 负责租户级审批平台配置 | 流程模板管理、全局参数、异常任务介入 |
| **业务管理员** | 各模块的流程规则配置者 | 低代码配置审批规则、条件、审批人 |
| **申请人** | 发起审批请求的业务用户 | 快速发起、实时追踪、按需催办/撤回 |
| **审批人** | 处理审批任务的用户 | 待办聚合、信息清晰、高效处理（批量/快捷） |
| **抄送人** | 仅需知晓审批结果的观察者 | 收到通知、可查看详情 |
| **代理人** | 在审批人不在岗期间临时代为处理 | 委托期内接收并处理转来的任务 |

## 四、核心功能模块

### 4.1 流程模板管理（管理员）

**目标：** 管理员可通过可视化界面配置审批流程，无需开发介入。

**功能清单：**

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| 模板列表 | 展示所有审批流程模板，支持启用/禁用、搜索过滤 | P0 |
| 模板基本信息 | 模板名称、模板图标、所属模块、描述、是否启用 | P0 |
| 表单字段设计 | 配置申请表单字段（文本/数字/日期/选择/人员/附件等） | P0 |
| 流程节点设计 | 可视化配置审批节点（串行/并行/条件分支） | P0 |
| 审批人规则配置 | 支持固定人员、直属上级、部门负责人、发起人自选、按角色 | P0 |
| 多人节点模式 | 会签（全员同意）/ 或签（任一人）/ 依次审批 | P0 |
| 条件分支配置 | 按表单字段值（金额/类型/组织等）自动路由到不同审批链 | P1 |
| 抄送节点 | 流程中任意位置插入抄送人（自动通知但无需操作） | P1 |
| 超时规则 | 审批节点超时后自动通过/自动拒绝/升级至上级 | P1 |
| 撤回权限配置 | 是否允许申请人撤回、在哪些状态下允许 | P1 |
| 加签/转交权限 | 是否允许审批人加签、转交 | P1 |
| 版本管理 | 模板发布后不可直接修改，需新建版本；旧流程实例沿用旧版本 | P2 |
| 模拟测试 | 发布前可模拟输入参数，预览流程走向 | P2 |

### 4.2 审批待办中心（审批人）

**目标：** 审批人能在一个入口管理所有待处理任务。

**功能清单：**

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| 待办列表 | 展示所有分配给当前用户的待处理任务，含标题、申请人、发起时间、到期时间 | P0 |
| 已办列表 | 展示当前用户已处理完的历史任务 | P0 |
| 待领任务 | 展示候选人池中的未认领任务（组级任务） | P1 |
| 已读/未读标记 | 新任务默认未读，查看后标为已读 | P0 |
| 筛选与排序 | 按业务模块、发起时间、到期时间、优先级等筛选 | P1 |
| 批量审批 | 选中多条同类任务一键批量同意（适合大量同质任务场景） | P1 |
| 任务搜索 | 按标题、申请人、业务单据号等搜索 | P2 |

### 4.3 审批任务处理（审批人）

**目标：** 审批人能清晰了解申请内容并高效完成审批操作。

**功能清单：**

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| 任务详情 | 展示申请人、发起时间、业务摘要、表单内容、附件 | P0 |
| 同意 | 流程流转至下一节点 | P0 |
| 拒绝 | 流程终止（需填写原因，不可逆操作二次确认） | P0 |
| 退回 | 退回至指定前置节点（申请人节点/指定审批节点）重新提交 | P0 |
| 转交 | 将任务转给其他人处理，原审批人退出此节点 | P1 |
| 前加签 | 在当前审批人之前插入新审批人 | P1 |
| 后加签 | 在当前审批人之后插入新审批人 | P1 |
| 批注/审批意见 | 填写文字意见，留存于审批记录，不改变流程走向 | P0 |
| 委托代理设置 | 设置有效期内代理人，期间内任务自动转交 | P1 |

### 4.4 审批发起（申请人）

**目标：** 申请人能快速发起审批，清楚了解将经过哪些审批节点。

**功能清单：**

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| 从业务模块触发 | 在业务单据（如计划版本、SOP 预算等）上提交时自动触发审批流 | P0 |
| 申请表单 | 展示已配置的表单字段，支持条件显示/隐藏 | P0 |
| 附件上传 | 支持多文件上传，限制格式与大小 | P1 |
| 预览审批链 | 发起时展示本次将经过的审批节点与审批人 | P1 |
| 临时抄送 | 发起时手动添加抄送人 | P2 |
| 草稿保存 | 未完成的申请可暂存草稿 | P2 |

### 4.5 审批进度追踪（申请人）

**目标：** 申请人能随时查看审批进度，及时知晓结果。

**功能清单：**

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| 我的申请列表 | 展示本人发起的所有审批，含状态（审批中/已通过/已拒绝/已撤回） | P0 |
| 审批进度详情 | 可视化展示审批链，当前停留节点高亮，已完成节点显示操作人与时间戳 | P0 |
| 审批历史记录 | 每个节点的操作人、操作时间、审批意见完整记录 | P0 |
| 撤回 | 审批未终结前，申请人可主动撤回 | P1 |
| 催办 | 向当前待处理审批人发送催办消息 | P1 |
| 状态变更通知 | 审批通过/拒绝/退回时，主动通知申请人 | P0 |

### 4.6 消息通知

**目标：** 相关人员能及时收到与自己相关的审批事件通知。

| 通知场景 | 接收方 | 通知渠道 |
|----------|--------|----------|
| 新任务到达 | 审批人 | 系统内推送 + 钉钉/企业 IM 消息卡片 |
| 流程通过 | 申请人 | 系统内推送 + IM 通知 |
| 流程拒绝/退回 | 申请人 | 系统内推送 + IM 通知（含拒绝原因） |
| 抄送通知 | 抄送人 | 系统内推送 + IM 通知 |
| 催办 | 审批人 | IM 消息提醒 |
| 超时提醒 | 审批人 | IM 消息提醒 |
| 委托接管 | 代理人 | 系统内推送 |

**IM 消息卡片设计要点：**
- 卡片展示：审批标题、申请人、业务摘要、当前状态
- 快捷操作按钮（可配置）：同意 / 拒绝（点击后跳转详情页填写原因）
- 卡片状态实时更新（已处理后卡片内容变更为终态）

### 4.7 异常任务管理（系统管理员）

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| 任务列表 | 展示所有流程实例，支持按状态/模块/时间筛选 | P1 |
| 人工干预 | 管理员强制推进/终止/撤回某流程 | P1 |
| 转派任务 | 管理员将某审批人的任务批量转给其他人（适用于离职场景） | P1 |
| 流程监控大盘 | 待办数量、超时率、平均处理时长等统计指标 | P2 |

## 五、核心审批节点类型

| 类型 | 含义 | 说明 |
|------|------|------|
| 普通审批节点 | 指定一个或多个审批人依次处理 | 多人时为顺序审批 |
| 会签节点 | 多个审批人同时收到，全部同意才通过 | 一人拒绝则整体拒绝 |
| 或签节点 | 多个审批人同时收到，任意一人操作即生效 | 效率优先场景 |
| 条件分支 | 根据表单字段值自动路由至不同子链路 | 实现多级金额分档 |
| 抄送节点 | 仅通知，无需操作 | 不阻塞流程 |
| 填写节点 | 指定人员在流程中补充填写信息 | 非审批，用于信息补全 |
| 结束节点 | 流程终态 | 触发业务回调 |

## 六、审批人规则体系

| 规则类型 | 说明 | 典型场景 |
|----------|------|----------|
| 固定人员 | 指定具体人员，不随申请人变化 | 总经理审批 |
| 直属上级 | 自动取申请人的直属上级（取自 HR 组织架构） | 大多数请审场景 |
| 多级上级 | 依次从 1 级上级向上，可设最大层级 | 大金额逐级审批 |
| 部门负责人 | 申请人所在部门的负责人 | 跨部门资源申请 |
| 指定角色 | 取某一系统角色下的全部成员 | 财务审核角色 |
| 发起人自选 | 发起时由申请人手动选择 | 灵活场景 |
| 发起人本人 | 审批人即申请人本人（自批自） | 知会/备案模板 |
| 找不到时兜底 | 自动通过 / 转交指定人 / 转交直属上级 | 兜底策略必须配置 |

## 七、审批状态机

```
草稿(DRAFT)
   ↓ 提交
审批中(IN_PROGRESS)
   ├─ 全部通过 → 已通过(APPROVED)
   ├─ 任一节点拒绝 → 已拒绝(REJECTED)
   ├─ 退回至申请人 → 已退回(RETURNED)，可重新提交 → 审批中
   └─ 申请人主动撤回 → 已撤回(CANCELED)
已通过/已拒绝/已撤回 → 终态（不可再操作）
已退回 → 可重新提交（重新进入审批中）
```

## 八、与业务模块集成设计

IPS 审批流定位为**平台级通用能力**，业务模块接入方式：

### 8.1 接入契约

业务模块接入时须提供：
- **业务键（businessKey）**：唯一标识业务单据（如 `DFP:VERSION:20260101001`）
- **模板标识（processDefinitionKey）**：对应在管理端配置的审批模板
- **流程标题**：展示在待办列表中
- **业务回调 URL**：审批通过/拒绝后，IPS 主动调用业务方接口更新状态
- **申请人（initiator）**：发起人员工号

### 8.2 业务回调机制

审批流终结（通过/拒绝/撤回）时，IPS 向业务方回调并携带：
- 审批结果（APPROVED/REJECTED/CANCELED）
- 流程实例 ID
- 终结时间
- 最终审批意见摘要

### 8.3 已有集成案例（参考）

| 业务模块 | 审批场景 | 集成状态 |
|----------|----------|----------|
| DFP（需求计划） | 需求提报版本审批 | 已集成（初版） |

## 九、对标竞品关键能力总结

### SAP 关键启示

| 能力 | 对我们的启示 |
|------|-------------|
| 主从布局待办中心（My Inbox） | 待办列表与任务详情同屏展示，减少点击层级 |
| 跨系统聚合（Task Center） | 长期目标：统一收件箱汇聚多模块待办 |
| 拒绝操作必须填写原因 | 不可逆操作需二次确认 + 原因字段必填 |
| 批量审批（Mass Approval） | 同质任务场景必须支持批量操作 |
| 可视化流程状态（Approval Control + Workflow Pattern） | 审批进度用节点图展示，当前节点高亮 |
| BRFplus 规则引擎 | 复杂条件路由须有独立规则配置层，不写死代码 |

### 飞书关键启示

| 能力 | 对我们的启示 |
|------|-------------|
| 消息卡片内嵌快捷操作 | 审批通知卡片直接嵌入「同意/拒绝」按钮，减少跳转 |
| 31 种表单控件 | 表单字段类型要丰富，尤其是金额、人员、附件、明细表格 |
| 条件显示字段 | 表单字段根据其他字段值动态显示/隐藏 |
| 发起时预览审批链 | 申请人提交前能看到将经过谁的审批 |
| 委托代理（按时间段） | 不在岗时按时间范围委托，到期自动恢复 |
| 自动化节点（审批通过后触发动作） | 审批终结后自动触发业务回调，减少人工操作 |

## 十、关键待决策点

以下问题在推进到 PRD 之前需明确：

| 序号 | 问题 | 背景 | 影响范围 |
|------|------|------|----------|
| 1 | **流程模板是否做可视化拖拽设计器，还是以固定配置表单为主？** | 可视化设计器开发成本高，但易用性好；配置表单开发快但灵活性差。可参考飞书「模板配置表单」而非 BPMN 可视化设计器 | 管理端核心功能决策 |
| 2 | **是否保留 Flowable BPMN 作为底层引擎，还是做更轻量的自研状态机？** | 当前已用 Flowable，功能强大但学习成本高，部分高级特性（如条件分支、并行网关）需要 BPMN 知识；自研状态机更可控但需自行实现更多功能 | 技术架构方向 |
| 3 | **与 IM（钉钉/企业微信）的集成深度？** | 现有代码有钉钉待办发送的 TODO 注释，但未完整实现。需确认企业当前使用哪种 IM，及消息卡片快捷审批是否必须实现 | 通知模块方向 |
| 4 | **审批流是否支持跨租户共享模板，还是租户独立配置？** | 影响多租户模型设计 | 数据模型与权限 |
| 5 | **审批结果回调机制是同步还是异步（消息队列）？** | 同步回调简单但有耦合和超时风险；异步可靠性更好 | 集成架构 |
| 6 | **退回时是「退回到发起人」还是「可退回到任意前置审批节点」？** | 飞书和 SAP 都支持退回到指定节点，当前代码 BackTaskVO 已预留 `distFlowElementId` 字段 | 操作体验与实现复杂度 |
