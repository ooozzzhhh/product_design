# 问题分析报告：物料供应能力汇总表三大缺陷

- **档案编号**：SDS_BUG_2026-03-06_01
- **功能路径**：供需管理 > 供应管理 > 物料供应能力汇总表
- **报告日期**：2026-03-06
- **知识库来源**：`产品设计/产品知识库/SDS/代码/SDS_supplySummary_代码实现.md`
- **报告性质**：技术专项，含代码细节，面向研发团队

---

## 一、现状（现象与背景）

### 1.1 功能定位

物料供应能力汇总表（`supplySummary`）是 SDS 模块的核心供应视图，负责将来自多个供应来源的数据聚合展示为「维度×时段」的矩阵表格。当前覆盖 5 类供应来源：采购订单（`sds_ord_purchase_order`）、调拨订单（`sds_ord_transport_order`）、生产订单（`sds_ord_work_order`）、期初库存（`sds_sup_boh_stock`）、采购预测库存（`sds_sup_purchase_forecast_stock`）。用户可按供应分类（计划供应 / 非计划供应 / 全部）筛选，并从维度下拉列表中选取聚合维度进行分组展示。

### 1.2 三个问题的表现形式

#### 问题1：可扩展性差
当需要新增一张供应来源表（如通用供应表 `sds_supply_universal`）时，开发人员需要同时修改 ServiceImpl、DAO XML（6 条 SQL）、SupplyTypeEnum 三处，且 DAO XML 中每条 SQL 均需追加一段 UNION ALL 分支。遗漏任何一处将导致新供应类型数据在某个查询场景下静默缺失，且目前没有自动化手段发现此类遗漏。

**影响范围**：每次项目上线新供应类型时，改动量大、出错风险高；未来若供应来源数量增加，维护成本将以线性速度上升。

#### 问题2：聚合维度列顺序错乱
用户在维度下拉列表中选择聚合维度后，表格中所选维度列不出现在最前几列，而是与时间列混排，顺序不可控。用户期望的效果是：**用户所选聚合维度按选择顺序排列在表格左侧，时段列紧随其后**。

**影响范围**：所有使用聚合维度功能的查询场景均受影响，直接影响表格可读性。

#### 问题3：前端直接暴露 Java 异常堆栈与 SQL 内容
当用户在「供应类型有值、聚合维度没值」等非法参数组合下发起查询，页面弹出「发生未知错误」弹窗，弹窗内容包含完整的 `BadSqlGrammarException` 堆栈信息与实际执行的 SQL 语句（含表名、字段名等内部结构）。

**影响范围**：所有用户均可能触发。暴露了内部数据库结构，存在安全隐患；对业务用户而言，无法从错误信息判断应如何操作，体验极差。

## 二、问题（根因分析）

### 2.1 问题1 根因：可扩展性差

**根因定位**：整个供应汇总功能对「供应类型 → 数据库表」的映射采用全量硬编码，分散在三处独立代码单元，彼此没有联动机制。

**具体硬编码点**：

**（1）`SupplySummaryServiceImpl.getTableNames()`**

```java
// 伪代码示意（实际为 if/else 分支）
if (supplyCategory == PLAN_SUPPLY) {
    return [purchase_order, work_order, transport_order];
} else if (supplyCategory == NON_PLAN_SUPPLY) {
    return [transport_order, work_order, boh_stock, purchase_forecast_stock];
} else {
    return [purchase_order, work_order, transport_order, boh_stock, purchase_forecast_stock];
}
```

新增供应表必须在三个分支中分别判断是否应纳入，逻辑判断无法复用。

**（2）`SupplySummaryBasicDao.xml`**

6 条 SQL（3 分类 × 2 用途：汇总+明细），每条均为多段 UNION ALL 硬连接：

```xml
<!-- 以 selectPlanSupplyByCondition 为例（伪代码示意） -->
SELECT ... FROM sds_ord_purchase_order WHERE order_flag='NO'
UNION ALL
SELECT ... FROM sds_ord_work_order WHERE order_flag='NO'
UNION ALL
SELECT ... FROM sds_ord_transport_order WHERE order_flag='NO'
GROUP BY ${groupByParam}
```

新增供应表 = 在 6 条 SQL 中各追加一段 UNION ALL，且每段的字段映射、JOIN 逻辑需手动对齐。6 条 SQL 之间没有共享片段，全量重复。

**（3）`SupplyTypeEnum`**

当前枚举固定 5 个值：`PURCHASE_ORDER`、`TRANSPORT_ORDER`、`WORK_ORDER`、`BOH_STOCK`、`PURCHASE_FORECAST_STOCK`。新增供应类型必须新增枚举值，且在 `translate()` 的 ImmutableMap 中同步追加映射，否则翻译阶段静默失效。

**根本原因**：三处硬编码之间没有单一事实来源（Single Source of Truth），扩展新供应类型时必须在三处手动同步，任意遗漏都将导致数据不一致。

---

### 2.2 问题2 根因：维度列无法排在时间列之前

**根因定位**：分页接口响应体 `PageInfo<SupplySummaryVO>` 中不含列元数据，前端只能依赖 VO 字段的定义顺序渲染表格，无法感知「哪些列是维度列」「哪些列是时间列」「维度列的用户指定顺序」。

**具体分析**：

`SupplySummaryVO` 中，维度字段（`itemCode`、`itemName` 等）与时间数据（`supplyQuantityMap`）混合存放于同一对象中，没有任何元数据标注字段的类型（维度 vs 时间）或渲染顺序。

前端在接收到响应后，只能按照 VO 的固定字段顺序渲染列，而这个顺序与用户在维度下拉列表中的选择顺序无关。

`groupByParam` 参数虽然携带了用户选择的维度顺序（逗号分隔的字段名序列），但该信息仅用于后端 GROUP BY 子句，没有被回传给前端作为列渲染的依据。

**结论**：这是接口协议设计问题，不是前端框架能力不足的问题。当前接口未将「列结构意图」传递给前端。

---

### 2.3 问题3 根因：前端暴露技术异常堆栈

**主因（SQL 注入路径与异常透传）**：

`SupplySummaryBasicDao.xml` 中所有 6 条 SQL 的 GROUP BY 子句使用 MyBatis `${}` 插值：

```xml
GROUP BY ${groupByParam}
```

`${}` 是字符串直接拼接，不做任何转义或参数化处理。若 `groupByParam` 为空字符串（用户未选聚合维度）或包含非法字符（如分号、引号），拼接后 SQL 语法直接报错，MySQL 抛出 `BadSqlGrammarException`。

`SupplySummaryServiceImpl.page()` 中未对该异常做捕获处理，异常携带完整的 SQL 原文沿调用栈向上冒泡，最终被全局异常处理器以 `e.getMessage()` 形式序列化返回前端，将表名、字段名、SQL 结构全部暴露。

**同样问题存在于** `${queryCriteriaParam}` 的处理：WHERE 条件也是字符串插值，非法值将导致同类型异常。

**次因（三处异常规范违规）**：

Service 层现有三处直接用字符串文案构造异常，违反项目强制规范（须使用 `BusinessErrorEnum` + `getI18nErrordesc()`）：

| 位置 | 违规代码 | 违规说明 |
|------|----------|----------|
| 时段获取分支 | `throw new BusinessException("请先在时段设置维护时段")` | 字符串硬编码文案，未走 BusinessErrorEnum |
| 供应类型校验（两处） | `throw new BusinessException("供应类型错误")` | 同上；文案过于简单，不符合「对象 + 问题 + 下一步指引」结构 |

**附带 Bug（同类问题，需一并修复）**：

**Bug A（`dimensionDropdown` 中 `orderFlag` 重复维度）**：

`existFields` 存储的是从 `information_schema.COLUMNS` 查询到的原始字段名（下划线格式，如 `order_flag`），但判断语句为：

```java
if (existFields.contains("orderFlag")) { ... }  // camelCase，永远 false
```

因格式不一致，条件永远为 `false`，导致「订单标识（`orderFlag`）」维度无条件追加到列表末尾。若 `information_schema` 查询已将 `order_flag` 包含在公共字段中，则下拉列表出现**重复的「订单标识」维度**。

**Bug B（`translate()` 中 `targetCommonEnums.get(0)` 越界）**：

```java
// 伪代码示意
List<CommonEnum> targetCommonEnums = allEnums.stream()
    .filter(e -> e.getCode().equals(fieldValue))
    .collect(toList());
return targetCommonEnums.get(0).getLabel();  // 若 targetCommonEnums 为空 → IndexOutOfBoundsException
```

当某字段的枚举 code 在枚举列表中不存在匹配项时（如旧数据的枚举值、新枚举上线后的兼容期），`targetCommonEnums` 为空列表，`get(0)` 直接抛出 `IndexOutOfBoundsException`，同样会导致整个翻译阶段崩溃，且异常未被捕获。

## 三、解决方案（修复建议与验证思路）

### 3.1 问题1：通用大表方案与产品纳入策略

#### 3.1.1 后端已定方向

后端计划新增通用供应表 `sds_supply_universal`，未来项目上所有新增供应类型统一写入此表，以此替代「每新增供应类型就改 6 条 SQL」的模式。

#### 3.1.2 产品侧建表字段建议

`sds_supply_universal` 需要能容纳所有供应类型的共性字段，同时保留灵活扩展空间。建议最低字段集如下：

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `id` | BIGINT PK | 主键 |
| `supply_version_id` | BIGINT | 关联供应版本，与现有表保持一致 |
| `supply_type` | VARCHAR | 供应子类型标识，对应 `SupplyTypeEnum`（如 `CUSTOM_SOURCE_01`） |
| `item_code` | VARCHAR | 物品代码，与 `mds_pro_product_base` JOIN 用 |
| `stock_point_code` | VARCHAR | 库存点代码，与 `mds_pro_stock_point` JOIN 用 |
| `supply_date` | DATE | 供应日期，用于时段归档 |
| `quantity` | DECIMAL | 供应数量 |
| `order_flag` | VARCHAR | 计划/非计划标识（`YES`/`NO`），与现有供应表保持语义一致 |
| `remark` | VARCHAR | 备注，供业务侧标注来源或场景说明 |
| `created_at` / `updated_at` | DATETIME | 审计字段 |

**扩展约定**：若项目上某供应类型有特殊字段（如自定义属性），暂不建议加宽通用表字段，应通过额外的扩展表或 JSON 字段补充，避免通用表列数膨胀。

#### 3.1.3 产品侧接入现有汇总逻辑的策略

核心思路：**让 `sds_supply_universal` 作为第 6 条固定 UNION ALL 分支参与所有 SQL 查询，不再为每种新供应类型单独追加分支**。具体接入方式：

**SQL 层面**：在 6 条 SQL 中各追加一段固定的 `sds_supply_universal` UNION ALL 分支，该分支始终参与查询（`supply_type IS NOT NULL` 作为兜底条件），计划/非计划过滤继续依赖 `order_flag` 字段。这是本次改造中仅有的一次侵入式 SQL 修改，之后新增供应类型只需向 `sds_supply_universal` 写入数据，无需再改 SQL。

**Service 层面**：`getTableNames()` 中将 `sds_supply_universal` 作为三个分类均包含的固定表；`SupplyTypeEnum` 可为通用表新增一个通配枚举值（如 `UNIVERSAL`），或直接通过 `supply_type` 字段区分不同子来源。

**`translate()` 层面**：`sds_supply_universal` 的 `supply_type` 字段值直接来自 `SupplyTypeEnum`，现有翻译逻辑可复用；若引入自定义 `supply_type` 字符串，需在枚举中为通用类型预留一个「透传原始值」的 fallback 分支。

#### 3.1.4 兼容现有供应类型的策略

- 现有 5 张表的 UNION ALL 分支**保持不变**，不做合并或迁移，避免引入数据迁移风险。
- `sds_supply_universal` 以**追加**形式并入，现有供应类型数据不从原表迁移到通用表。
- 项目上若有新供应来源，直接写入 `sds_supply_universal`，历史数据不受影响。

#### 3.1.5 改动范围

| 文件/类 | 改动内容 |
|---------|----------|
| `SupplySummaryBasicDao.xml` | 6 条 SQL 各追加一段 `sds_supply_universal` UNION ALL 分支（一次性改造） |
| `SupplySummaryServiceImpl.getTableNames()` | 将 `sds_supply_universal` 加入三个分类的表名列表 |
| `SupplyTypeEnum` | 新增通用来源枚举值（具体命名由研发确认） |
| `sds_supply_universal`（DDL） | 新建表，按 3.1.2 字段集执行 |

#### 3.1.6 验证思路

1. 在 `sds_supply_universal` 中插入若干测试行（覆盖 `order_flag='YES'`、`order_flag='NO'` 两种），分别以「计划供应」「非计划供应」「全部」三种 `supplyCategory` 查询汇总表，验证通用表数据是否正确出现在各分类结果中；
2. 验证原有 5 类供应数据在改造后的查询结果不变（回归测试）；
3. 验证新供应类型在维度下拉列表、时段归档、翻译逻辑中均能正常展示。

### 3.2 问题2：列元数据方案（响应结构改造）

#### 3.2.1 核心思路

在分页接口的响应中，**额外返回一份「列定义列表」**，让前端知道：哪些列是维度列（及其用户指定顺序）、哪些列是时间列（及其时段键顺序）。前端依据此列表驱动表格渲染，而非依赖 VO 字段的定义顺序。

#### 3.2.2 建议的响应结构改造

当前响应：
```json
{
  "data": {
    "list": [...],   // PageInfo<SupplySummaryVO>
    "total": 100
  }
}
```

建议改为：
```json
{
  "data": {
    "columns": [
      { "key": "itemCode",         "label": "物品代码",   "type": "DIMENSION", "order": 0 },
      { "key": "stockPointCode",   "label": "库存点代码", "type": "DIMENSION", "order": 1 },
      { "key": "2026-01-01",       "label": "2026-01-01", "type": "TIME",      "order": 2 },
      { "key": "2026-02-01",       "label": "2026-02-01", "type": "TIME",      "order": 3 }
    ],
    "list": [...],
    "total": 100
  }
}
```

字段说明：

| 字段 | 说明 |
|------|------|
| `key` | 对应 VO 中的字段名（维度列）或 `supplyQuantityMap` 中的 key（时间列） |
| `label` | 前端表头展示文案（维度列来自字段的中文名，时间列来自 periodStart） |
| `type` | `DIMENSION`（维度列）或 `TIME`（时间列） |
| `order` | 列的渲染顺序，前端按此字段升序排列表格列 |

#### 3.2.3 后端生成 `columns` 的逻辑

`columns` 列表由 `SupplySummaryServiceImpl.page()` 在返回结果前组装：

1. **维度列**：按用户传入的 `groupByParam`（逗号分隔）顺序，将每个字段名转换为对应 `label`（可复用现有 `initDimensionList()` 中的字段名→中文名映射）；`order` 值从 0 开始递增。
2. **时间列**：按 MDS feign 返回的时段列表顺序，将每个 `periodStart` 作为 `key` 和 `label`；`order` 值紧接维度列序号继续递增。

#### 3.2.4 改动范围

| 文件/类 | 改动内容 |
|---------|----------|
| `SupplySummaryVO` 或新建 `ColumnDefinition` VO | 新增 `ColumnDefinition` 对象（key、label、type、order） |
| Controller 响应包装类 | 新增 `columns` 字段，与 `list`、`total` 并列返回 |
| `SupplySummaryServiceImpl.page()` | 在返回前，按 groupByParam + 时段列表组装 columns |
| 前端表格渲染逻辑 | 读取 `columns` 列表，按 `order` 排序后驱动表格列渲染（前端改动，由前端研发实现） |

#### 3.2.5 验证思路

1. 选择 2 个聚合维度（如「物品代码」+「库存点代码」），验证响应中 `columns` 的前两项对应这两个维度，且顺序与用户选择顺序一致；
2. 验证维度列在表格中排在所有时间列之前；
3. 切换不同维度组合，验证 `columns` 随之动态变化；
4. 验证原有 `list` 数据不受影响（回归测试）。

### 3.3 问题3：异常处理全面改造

#### 3.3.1 白名单校验（主因修复）

在 `SupplySummaryServiceImpl.page()` 中，进入 DAO 查询之前，对 `groupByParam` 进行白名单校验：

**校验逻辑**：将 `groupByParam` 按逗号拆分，逐一校验每个字段名是否存在于 `dimensionDropdown()` 返回的合法字段列表中。若存在不合法字段名，直接抛出 `BusinessException`，**不进入 DAO 层**。

```java
// 伪代码示意
List<String> allowedFields = getAllowedFieldNames(); // 取 dimensionDropdown 的合法字段集
List<String> requestedFields = Arrays.asList(groupByParam.split(","));
if (!allowedFields.containsAll(requestedFields)) {
    throw new BusinessException(BusinessErrorEnum.SDS_GROUP_BY_PARAM_INVALID.getI18nErrordesc());
}
```

对 `queryCriteriaParam` 同样建议引入白名单或格式校验，防止非法 WHERE 条件注入。

#### 3.3.2 SQL 层防护（深度防御）

在条件允许时，建议将 `SupplySummaryBasicDao.xml` 中 `${groupByParam}` 改为基于白名单的动态 `<foreach>` 或动态 SQL 构建，避免字符串直接插值。如果 MyBatis 的动态 SQL 无法优雅实现，至少在白名单校验通过后再进入 DAO，将 SQL 注入风险降至最低。

#### 3.3.3 新增 BusinessErrorEnum 错误码

需在 `BusinessErrorEnum` 中新增以下 SDS 相关错误码（具体编码需与现有 SDS 错误码对齐，避免重复）：

| 错误码（建议） | 场景 | 中文文案（messages_zh_CN.properties） | 英文文案（messages_en_US.properties） |
|-------------|------|--------------------------------------|--------------------------------------|
| `SDS001` | 时段未配置 | 尚未配置「时段」，请前往「时段设置」完成配置后重试。 | No period configured. Please go to "Period Settings" to complete the configuration and try again. |
| `SDS002` | 供应分类参数非法 | 「供应类型」参数不合法，请刷新页面后重新选择。 | Invalid supply category. Please refresh the page and select again. |
| `SDS003` | 聚合维度参数非法 | 「聚合维度」包含不合法字段，请重新选择聚合维度后查询。 | Invalid grouping dimension. Please reselect the dimension and try again. |

> 注：具体编码（SDS001 等）需研发确认当前 SDS 已使用的编码范围，新增时从未使用的编号开始，保证全局唯一。

同步在 `messages_zh_CN.properties` 和 `messages_en_US.properties` 中维护上述 key-value。

#### 3.3.4 三处违规异常替换

将 Service 层三处硬编码文案改为 BusinessErrorEnum：

| 原代码 | 替换为 |
|--------|--------|
| `throw new BusinessException("请先在时段设置维护时段")` | `throw new BusinessException(BusinessErrorEnum.SDS001.getI18nErrordesc())` |
| `throw new BusinessException("供应类型错误")`（第1处） | `throw new BusinessException(BusinessErrorEnum.SDS002.getI18nErrordesc())` |
| `throw new BusinessException("供应类型错误")`（第2处） | `throw new BusinessException(BusinessErrorEnum.SDS002.getI18nErrordesc())` |

#### 3.3.5 Bug A 修复：`dimensionDropdown` 中 `orderFlag` 重复维度

将判断条件从 camelCase 改为下划线格式，与 `existFields` 的实际存储格式一致：

```java
// 修复前
if (existFields.contains("orderFlag")) { ... }

// 修复后
if (existFields.contains("order_flag")) { ... }
```

**效果**：当 `information_schema` 查询结果中已包含 `order_flag`（即它是所有涉及表的公共字段），则不再重复追加「订单标识」维度；若不包含，则正常追加。

#### 3.3.6 Bug B 修复：`translate()` 中 `IndexOutOfBoundsException`

在调用 `get(0)` 前增加空列表判断：
