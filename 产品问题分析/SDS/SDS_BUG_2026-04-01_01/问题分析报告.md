# 问题分析报告：动态列列顺序紊乱

- **档案编号**：SDS_BUG_2026-04-01_01
- **功能路径**：
  - 供需管理 > 供应管理 > 物料供应能力汇总表
  - 供需管理 > 需求优先级编码方案 > 需求打分结果表
- **报告日期**：2026-04-01
- **代码依据**：`UHA/scp-foundation`（`scp-sds-sdk`、`scp-sds-front`、`scp-sds-api`）
- **报告性质**：技术专项，含代码细节，面向研发团队

## 一、现状（现象与背景）

### 1.1 功能定位

物料供应能力汇总表和需求打分结果表均采用**动态列矩阵**展示模式：左侧为固定属性列（如物品代码、库存点），右侧为动态数据列（如时段序列、优先级因子明细）。用户可通过选择聚合维度来控制属性列的展示内容。

### 1.2 两个问题的表现形式

#### 问题1：物料供应能力汇总表 - 属性列与数据列混排

在物料供应能力汇总表中，当用户选择聚合维度（如物品代码、库存点）后，表格列的显示顺序存在以下问题：

- **属性列与数据列相对位置不固定**：用户选择的聚合维度（属性列）未能固定展示在表格左侧，而是与时段序列（数据列）混排
- **保存视图后顺序紊乱**：用户调整列顺序并保存视图后，重新加载视图时列顺序可能与预期不符
- **切换供应类型后顺序变化**：在「计划供应」「非计划供应」「全部」之间切换时，列顺序可能发生变化

**影响范围**：所有使用聚合维度功能的查询场景均受影响，直接影响表格可读性与用户体验。

#### 问题2：需求打分结果表 - 加权求和模式列顺序混乱

需求打分结果表在加权求和（WEIGHT_SUM）计算方式下存在类似问题：

- **优先级因子明细列位置不固定**：`priorityFactorDetailMap` 展开后的动态列与聚合维度列、总分数列的相对位置难以控制
- **保存视图后列顺序紊乱**：与物料供应能力汇总表类似，保存视图后重新加载时列顺序可能不正确

**影响范围**：使用加权求和计算方式的优先级方案，在配置评分组后查看打分结果时均受影响。

## 二、问题（根因分析）

### 2.1 前端列构建逻辑缺陷 - 缺乏后端列元数据

**根因定位**：两个分页接口的响应体中均不含「列定义元数据」，前端只能依赖硬编码逻辑拼接列数组，无法感知「哪些列是属性列」「哪些列是数据列」「列的展示顺序意图」。

**物料供应能力汇总表前端代码分析**：

```javascript
// table.vue (第650-653行)
// 问题：列顺序完全由前端数组拼接顺序决定
this.tableColumns = [
    ...dynamicCondition,        // 聚合维度列（属性列）
    ...timeSeriesSequenceData,  // 时段序列（数据列）
];

// 生成逻辑（第235-274行）
this.tableColumnsData = res.data.map((item, index) => {
    return {
        label: item.label,
        prop: item.value,
        sortBy: index + 1,  // 仅基于遍历顺序
        // 缺少 columnType 标识
    };
});
```

**需求打分结果表前端代码分析**：

```javascript
// index.vue (第625-650行)
generateTableColumns(dataList) {
    let columns = [];
    const isWeightSum = this.eData.calculateMode === "WEIGHT_SUM";
    const hasGroupBy = this.groupByParam.length > 0;

    if (hasGroupBy) {
        // 1. 先生成分组列（属性列）
        columns = this.generateGroupColumns();
        if (isWeightSum) {
            // 2. 再添加加权求和列（数据列）
            columns = this.addWeightSumColumns(columns, dataList[0]);
        }
    } else {
        columns = this.deepCopy(this.tableColumnsData);
        if (isWeightSum) {
            columns = this.addWeightSumColumns(columns, dataList[0]);
        }
    }
    // 3. 最后添加总分数列（汇总列）
    return this.addTotalScoreColumn(columns);
}
```

### 2.2 后端响应数据结构缺陷

**当前响应结构**：

```json
{
  "success": true,
  "data": {
    "list": [...],   // 行数据，列信息隐藏在VO字段中
    "total": 100
  }
}
```

**问题**：
1. `list` 中的 VO 对象混合存放属性字段（`itemCode`、`itemName`）与动态数据（`supplyQuantityMap`），没有任何元数据标注字段的类型（属性 vs 数据）或渲染顺序
2. `groupByParam` 参数虽然携带了用户选择的维度顺序，但该信息仅用于后端 GROUP BY 子句，没有被回传给前端作为列渲染的依据

### 2.3 保存视图时的列类型信息丢失

**根因**：
- 前端保存视图时，仅保存了列的 `prop`、`label`、`width` 等展示属性
- **未保存** `columnType`（属性列/数据列/汇总列）信息
- 重新加载视图时，前端无法判断各列的类型，无法按类型进行排序，导致顺序紊乱

### 2.4 两个表格各自实现列控制 - 代码重复

| 表格 | 属性列来源 | 数据列来源 | 控制方式 | 问题 |
|------|-----------|-----------|---------|------|
| 物料供应能力汇总表 | `tableColumnsData` (聚合维度) | `timeSeriesSequence` (时段序列) | 数组拼接 | 硬编码拼接顺序 |
| 需求打分结果表 | `aggregationDimensionOption` (聚合维度) | `priorityFactorDetailMap` (优先级因子) | 分步生成 | 分步方法硬编码 |

**结论**：两个表格各自实现了一套列顺序控制逻辑，没有复用机制，维护成本高。

## 三、解决方案（修复建议与验证思路）

### 3.1 核心思路：后端标识 + 前端按标识排序

**方案概述**：
1. **后端改造**：在分页接口响应中增加 `columns` 字段，包含列的元数据（`prop`、`label`、`columnType`、`sortOrder`）
2. **前端改造**：前端接收 `columns` 后，按 `columnType` 优先级 + `sortOrder` 排序后渲染表格
3. **视图保存**：保存视图时保留 `columnType` 信息，确保还原时顺序正确

### 3.2 列类型定义

| 列类型 | 说明 | 排序优先级 | 示例 |
|--------|------|-----------|------|
| `ATTRIBUTE` | 属性列 - 固定维度字段，来自聚合维度选择 | 1 | 物品代码、物品名称、供应子类型、库存点代码 |
| `DYNAMIC_DATA` | 数据列 - 动态生成的数据，来自时段或因子明细 | 2 | 时段序列（2025-06-24~2025-06-30）、优先级因子明细 |
| `SUMMARY` | 汇总列 - 汇总计算结果 | 3 | 总分数、合计 |

### 3.6 前端改造

#### 3.6.1 公共工具函数

在 `tableUtils.js` 中新增：

```javascript
/**
 * 按 columnType 和 sortOrder 排序表格列
 * @param {Array} columns - 列配置数组
 * @returns {Array} 排序后的列数组
 */
export function sortTableColumns(columns) {
  const typePriority = {
    'ATTRIBUTE': 1,
    'DYNAMIC_DATA': 2,
    'SUMMARY': 3
  };
  
  return [...columns].sort((a, b) => {
    // 1. 按列类型优先级排序
    const priorityA = typePriority[a.columnType] || 99;
    const priorityB = typePriority[b.columnType] || 99;
    if (priorityA !== priorityB) {
      return priorityA - priorityB;
    }
    
    // 2. 同类型内按 sortOrder 排序
    return (a.sortOrder || 0) - (b.sortOrder || 0);
  });
}
```

#### 3.6.2 物料供应能力汇总表改造

移除硬编码拼接逻辑，使用后端返回的 `columns`：

### 3.7 渐进式改造方案（可选）

如果一次性改造范围过大，可采用渐进式方案：

**Phase 1**：后端新增 `columns` 字段，同时保持现有 `list` 结构不变（向后兼容）
**Phase 2**：前端优先使用 `columns`，如果不存在则回退到现有硬编码逻辑
**Phase 3**：验证稳定后，移除前端硬编码逻辑，仅保留 `columns` 方式

### 3.8 极简方案（隐藏列设置）- 推荐短期落地

考虑到动态列报表的特殊性（列结构随查询条件动态变化），可以采用**极简方案**快速解决问题：

#### 3.8.1 核心思路

- **完全以后端返回顺序为准**
- **隐藏列设置按钮**，用户无法调整列顺序
- 通过「聚合维度选择」给用户提供列控制感

#### 3.8.2 实现方式

**前端配置**：

```javascript
// table.vue 中配置 yhl-table，禁用列设置
data() {
  return {
    // 禁用列设置按钮
    showColumnSetting: false,
    
    // yhl-table 组件属性
    tableConfig: {
      columnSetting: false,  // 隐藏列设置
      customColumnSave: null, // 不触发列配置保存
    }
  };
}
```

**后端优化**（提供更合理的默认顺序）：

```java
// SupplySummaryServiceImpl.java

// 1. 属性列按业务重要程度排序
List<String> attributeFieldOrder = Arrays.asList(
    "productCode",      // 物品代码 - 最核心
    "productName",      // 物品名称
    "stockPointCode",   // 库存点代码
    "stockPointName",   // 库存点名称
    "supplyType",       // 供应子类型
    "productSeriesCode", // 物品系列代码
    "orderFlag"         // 订单标识
);

// 2. 数据列按时段时间顺序（自然就是有序的）
// 3. 汇总列固定在最后（如有）
```

#### 3.8.3 方案特点

| 特点 | 说明 |
|------|------|
| 实现简洁 | 无需处理用户自定义顺序与后端顺序的冲突，逻辑清晰 |
| 体验一致 | 所有用户看到统一的列顺序，符合业务逻辑规范 |
| 兼容性好 | 无需考虑旧版本保存的视图配置迁移问题 |
| 快速落地 | 开发周期短，可快速验证效果 |
| 维护简单 | 列顺序统一由后端控制，便于后续统一调整优化 |

#### 3.8.4 交互说明

在表格上方添加引导说明，帮助用户理解列展示逻辑：

> 「当前展示顺序：属性列（物品、库存点等）→ 时段数据列。可通过调整「聚合维度」来选择需要展示的属性字段。」

### 3.9 验证思路

| 场景 | 验证要点 |
|------|----------|
| 列顺序正确性 | 选择多个聚合维度，验证属性列始终排在数据列之前 |
