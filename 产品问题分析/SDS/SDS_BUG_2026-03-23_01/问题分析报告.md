# 问题分析报告：需求优先级编码方案 · 加权求和配置弹窗

- **档案编号**：SDS_BUG_2026-03-23_01
- **功能路径**：供需管理 > 需求优先级编码方案 > 优先级因子配置（计算方式 = 加权求和）> 优先级评分规则配置弹窗
- **报告日期**：2026-03-23
- **代码依据**：`UHA/scp-foundation`（`scp-sds-sdk`、`scp-sds-front`）
- **数据库依据**：测试环境 MySQL 库 **`scp_mps`**（`10.7.60.157:3306`）。§3.3 字段建议以该库表结构为准。
- **报告性质**：技术专项，含代码与表设计注释，面向产品与研发团队

---

## 一、现状（现象与背景）

### 1.1 功能定位

「需求优先级编码方案」支持多种计算方式；当方案为 **加权求和（WEIGHT_SUM）** 时，因子配置通过 **优先级评分规则配置** 弹窗完成。弹窗分为：

- **左侧**：因子来源（单选）+ 与需求数据的 **关联字段**（多选）。
- **右侧**：**参与计算的因子**（多选，用于规则行匹配打分）。

前端实现：`scp-front/scp-sds-front/src/views/demandPriorityCodingScheme/priorityFactorConfiguration/priorityScoringRuleConfigurationDialog.vue`。  
数据来自接口：`sourceTableEnumDropdown`、`relatedFieldsDropdown`、`sourceTableFieldsDropdown`（`PriorityFactorConfigController`）。

### 1.2 测试环境现象

| 现象 | 用户侧理解 |
|------|------------|
| 左侧因子来源列表中 **没有销售部门（或销售组织/销售片区等）相关主数据表** | 无法按销售维度配置评分来源。 |
| 右侧「参与计算的因子」 **几乎全部为 `id` / `xxxId` 字段**，看不到客户代码、物料代码等可读业务字段 | 配置规则时需手工维护 ID，无法按业务代码维护，可用性差。 |

### 1.3 影响范围

- 所有使用 **加权求和** 的优先级方案，在配置评分组时均受同一套后端下拉逻辑约束。
- 与「按销售维度、按客户/物料代码区分优先级」类业务诉求存在明显差距。

---

## 二、问题（根因分析）

### 2.1 左侧缺少销售部门相关表——枚举与 Provider 未覆盖

**根因**：因子来源下拉 **并非** 扫描数据库动态生成，而是由 `PriorityFactorConfigServiceImpl.getSourceTableEnumDropdown()` **遍历 `FactorSourceTableEnum`** 固定输出。

当前枚举 **仅三项**：

| 别名 | 含义（描述） | 代码中注释的物理对象 |
|------|--------------|----------------------|
| `PRODUCT_STOCK_POINT` | 库存点物品基础信息 | `mds_pro_product_stock_point_base` |
| `DEMAND_SUMMARY` | 需求汇总数据指表 | 注释为预测/客户订单 union 视图相关 |
| `CUSTOMER` | 客户表 | `dps_ord_customer` |

对应 Spring Bean：`ProductStockBaseTableProvider`、`DemandSummaryTableProvider`、`CustomerTableProvider`（均实现 `FactorSourceTableProvider`）。

**结论**：未在枚举中声明、且未实现对应 `FactorSourceTableProvider` 的 **销售部门/销售片区/销售组织** 等主数据，**不可能**出现在左侧来源列表。这与「框架已预留接口、业务表未接全」的判断一致。

**业务数据侧线索**：订单域服务中对客户订单 VO 存在 `salesSegmentId` 等字段的批量处理（如 `CustomerOrderServiceImpl`），说明需求链路上 **可能已有销售维度 ID**，但优先级因子来源 **未挂接** 对应主数据表。

### 2.2 右侧因子全是 ID——后端接口显式过滤所致

**根因**：`getSourceTableFieldsDropdown(String tableCode)` 在遍历来源表 VO 字段时，**仅将字段名等于 `id` 或以 `Id` 结尾的字段** 加入下拉列表；其余字段（如客户代码、物料编码等字符串/业务键）**被代码直接排除**。

设计意图（代码注释）：与 `relatedFieldsDropdown` 中的关联字段建立映射，强调外键型 `*Id` 关联。

**结论**：测试环境「因子全是 ID」是 **当前实现的确定行为**，不是数据缺失或前端展示错误；产品期望的「客户代码类字段」**在当前接口规范下不会被返回**。

### 2.3 关联字段下拉同样以 `*Id` 为中心

`getRelatedFieldsDropdown` 对需求侧 VO 仅收集 **以 `Id` 结尾** 的字段，再与来源表允许的关联字段集合求交（含多需求类型时的交集逻辑）。整条链路的设计假设是：**用外键型 ID 关联来源表**，而非业务代码字段作为关联键。

### 2.4 小结

| 表象 | 根因类型 | 说明 |
|------|----------|------|
| 无销售部门表 | 功能未覆盖 | 枚举 + Provider 未扩展。 |
| 因子仅 ID | 实现策略与产品预期不一致 | 下拉接口硬编码只返回 `id` / `*Id`。 |

## 三、解决方案（修复建议与验证思路）

### 3.1 产品侧（优先）

1. **明确「销售部门」对象**：对应主数据是销售组织、销售片区（如 `salesSegment`）、还是其他档案；与需求行上的关联字段名（如是否已有 `salesSegmentId`）对齐。
2. **明确因子取值语义**：规则匹配是以 **主数据 ID** 为准，还是必须以 **业务代码** 配置（影响是否改接口与运行时取值路径）。

### 3.2 研发侧

1. **扩展因子来源**  
   - 在 `FactorSourceTableEnum` 增加新来源项（别名、中文描述、表/视图注释）。  
   - 新增 `FactorSourceTableProvider`：`getVOClass()`、`loadSourceRecords`、关联键（`getDemandRelatedFieldNames` / `getSourceTableKeyField` 等）与批量查询实现一致。  
   - 保证与 `PriorityCalculatorAssembledDomainServiceImpl` 中 `loadSourceRecords` 使用方式兼容（当前实现会取 **关联字段配置中的第一个** 字段从需求 VO 取值，多键场景需一并评估）。

2. **支持非 ID 类「参与计算的因子」（若产品确认需要）**  
   - 调整 `getSourceTableFieldsDropdown` 的过滤规则（白名单/类型策略/排除敏感字段），使客户代码、物料代码等可出现在右侧列表。  
   - **必须同步**：加权求和运行时从来源 VO 读取选中字段、匹配模式（精确/模糊）与存库规则，避免仅下拉放开而计算仍按 ID 处理导致配置无效或结果错误。

3. **若仅增加销售主数据表、因子仍用 ID**  
   - 可先只做「来源表 + Provider + 关联字段」闭环，右侧仍只选 ID 字段，作为阶段性交付；产品与测试需接受配置仍为 ID 文案。

### 3.3 业务可读字段建议（基于 **`scp_mps`** 实际列，供下拉白名单 / 产品拍板）

下列字段名为 **Java VO 惯用 camelCase**（由库表 snake_case 对应）；**P0** 为优先建议出现在「参与计算的因子」中的业务可读项；**P1** 为次要或规则类；**跨表** 表示当前因子来源表本身无该列，需在 VO 上关联填充或改 Provider 查询范围。

**不包含**：是否启用、备注、显示顺序、创建人/修改人/时间等 **功能性或非业务语义** 字段，不作为评分因子建议。

#### （1）客户 `dps_ord_customer`（对应 `CUSTOMER`）

| 优先级 | 字段 | 库列注释 / 说明 |
|--------|------|-----------------|
| P0 | `customerCode` | 客户编码 |
| P0 | `customerName` | 客户名称 |
| P1 | `priority` | 客户优先级（数值，可用于分档规则） |
| P1 | `contact`、`telephone` | 联系人、电话（一般较少用于优先级，可按项目需要隐藏） |
| 慎选 | `serviceLevel`、`proportionalPenalty`、`fixedPenalty` | 数值/罚金类，若开放需约定匹配方式（通常不适合字符串精确匹配） |

#### （2）库存点物品基础信息 `mds_pro_product_stock_point_base`（对应 `PRODUCT_STOCK_POINT`）

**说明**：该表 **仅有** `product_id`、`stock_point_id` 等关联键，**不包含** `product_code`、`stock_point_code`。若产品强依赖「物品代码 / 库存点代码」参与打分，需在 **`ProductStockPointVO` 侧已有关联字段** 的前提下纳入下拉，或 **扩展 Provider 查询**（联表 `mds_pro_product_base`、`mds_pro_stock_point`）。

| 优先级 | 字段 | 库列注释 / 说明 |
|--------|------|-----------------|
| P0（跨表） | `productCode`、`productName` | 见 `mds_pro_product_base.product_code` / `product_name` |
| P0（跨表） | `stockPointCode`、`stockPointName` | 见 `mds_pro_stock_point.stock_point_code` / `stock_point_name` |
| P1（本表） | `productType` | 物品类型 |
| P1（本表） | `allocationType`、`replenishmentType`、`calculationCategory` | 分配方式、补充方式、计算类别 |
| P1（本表） | `participationCalculated`、`keyMaterial`、`longCycleMaterial`、`kitComputation` | 是否类标志位 |
| P1（本表） | `priority`、`lowLevelCode` | 优先级、低阶码 |
| P1（本表） | `expandRatio`、`expandLevel` | 展开比例、是否展开层 |

#### （3）建议新增来源：销售部门（销售片区）`mds_sal_sales_segment`

与需求行 `salesSegmentId` 等关联时，建议在枚举中增加独立来源（需新 Provider），**参与计算的因子**可优先开放：

| 优先级 | 字段 | 库列注释 / 说明 |
|--------|------|-----------------|
| P0 | `salesSegmentCode` | 销售部门代码 |
| P0 | `salesSegmentName` | 销售部门名称 |
| P1 | `shortName` | 销售部门简称 |
| P1 | `allocationRatio` | 需求分摊比例 |

#### （4）需求汇总来源 `DEMAND_SUMMARY`

代码侧当前对下拉 **固定仅 `demandType`**。若产品确认需与需求汇总界面一致，可对照前端需求汇总列与 `dps_ord_customer_order` / `dps_for_demand_forecast` 及视图字段 **另开清单**（本报告未对两订单表做全量结构展开；需要时可在对应库中核对列清单后附录）。

### 3.4 验证建议

| 场景 | 验证要点 |
|------|----------|
| 来源表扩展 | 新来源出现在 `sourceTableEnumDropdown`；选后可拉取关联字段；保存后重开弹窗回显正确。 |
| 关联字段（多需求类型方案） | 交集逻辑下关联字段是否仍包含新维度；空交集时是否有明确提示。 |
| 计算结果 | 修改评分组后执行优先级计算，抽样核对来源表预加载条数与得分是否符合规则。 |
| 非 ID 因子（若实施） | 规则行录入业务代码与来源表数据一致时得分正确；大小写/空格与 `MatchModeEnum` 行为一致。 |
