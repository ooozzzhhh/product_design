## 1. 现状（现象与背景）

### 1.1 当前实现方式

> **注意**：产品代码已有重大更新（截至 2026-04-02），以下描述基于最新代码实际情况。

SDS 需求优先级编码方案的「加权求和」模式，其因子来源表与参与计算因子字段当前通过 **Provider 策略模式 + 硬编码定义** 方式实现：

1. **因子来源表枚举**：`FactorSourceTableEnum` 当前包含 3 类来源：
   - `PRODUCT`（物品，表 `mds_pro_product_base`）— **新增**
   - `PRODUCT_STOCK_POINT`（库存点物品，表 `mds_pro_product_stock_point_base`）
   - `CUSTOMER`（客户，表 `dps_ord_customer`）
   - ~~`DEMAND_SUMMARY`~~ — **已移除**（Provider 实现类 `DemandSummaryTableProvider` 已被完全注释）

2. **字段白名单机制（已重构）**：
   - **不再**通过 `getSourceTableFieldsDropdown()` 内的 `id/*Id` 硬编码过滤
   - 改为每个 Provider 实现 `getSourceTableDefinition()` 方法，返回 `PrioritySourceTableDefinitionDO` 对象，其中包含：
     - `demandJoinFields`：需求端关联字段列表（如 `["customerId"]`）
     - `sourceJoinFields`：来源端关联字段列表（如 `["id"]`）
     - `configurableFields`：可配置因子字段列表（含 code + label）
   - `getSourceTableFieldsDropdown()` 直接从 `configurableFields` 读取

3. **各 Provider 当前配置的 configurableFields**：

   | Provider | 来源 | configurableFields |
   |----------|------|-------------------|
   | `ProductTableProvider`（优先级 15） | PRODUCT | `product_code`、`product_name`、`product_type`、`marketing_model`、`allocation_type`、`key_material`、`long_period_material`（共 7 个） |
   | `ProductStockBaseTableProvider`（优先级 10） | PRODUCT_STOCK_POINT | `key_material`、`long_cycle_material`、`product_type`、`allocation_type`、`replenishment_type`、`calculation_category`（共 6 个） |
   | `CustomerTableProvider`（优先级 30） | CUSTOMER | `customer_code`、`customer_name`（共 2 个） |

4. **关联字段机制（已重构）**：
   - **不再**通过反射取需求 VO 的 `*Id` 字段与来源 VO 求交
   - 改为 Provider 在 `getSourceTableDefinition()` 中精确声明 `demandJoinFields` 和 `sourceJoinFields`
   - `getRelatedFieldsDropdown()` 直接从 `demandJoinFields` 读取

5. **复合键支持（已实现）**：
   - 新增 `PrioritySourceJoinKey` 类，支持多字段复合关联键
   - 新增 `loadSourceRecordsByKeys(Set<PrioritySourceJoinKey>)` 方法
   - `ProductStockBaseTableProvider` 已使用双键关联：`demandJoinFields = ["stockPointId", "productId"]`，`sourceJoinFields = ["stockPointId", "productId"]`
   - 运行时 `loadSourceRecords()` 通过反射从需求 VO 提取关联字段值，构建 `PrioritySourceJoinKey`，再调用 `loadSourceRecordsByKeys()` 批量查询

### 1.2 项目实施中的问题场景

虽然代码已从「Service 层硬编码过滤」重构为「Provider 策略模式 + `PrioritySourceTableDefinitionDO` 定义」，但**字段白名单仍然写在 Java 代码中**。在项目实施时，客户可能需要：

- 新增因子来源表（如销售部门、销售片区等）
- 扩展某个来源表的参与计算因子字段（如库存点物品增加 `expand_ratio`、客户增加 `priority` 等）
- 调整关联字段的可选范围

**按当前实现方式**，实施顾问必须修改 Java 代码：
1. 新增来源：补充 `FactorSourceTableEnum` 枚举 + 新建 Provider 实现类
2. 扩展字段：修改对应 Provider 的 `getSourceTableDefinition()` 方法中的 `configurableFields` 列表
3. 调整关联：修改 `demandJoinFields` / `sourceJoinFields`
4. 重新编译、打包、部署

**这违反了公司「项目可通过 SDK 升级产品」的规则**，导致：
- 项目代码与产品代码分叉，后续产品升级困难
- 实施成本高，每个项目都需要定制开发
- 虽然已收敛到 Provider 的 `getSourceTableDefinition()` 一处，但仍需改代码

## 2. 问题（根因分析）

### 2.1 架构层面的根本原因

**字段白名单与业务配置未分离**：当前架构虽已通过 `PrioritySourceTableDefinitionDO` 将字段定义收敛到 Provider 的 `getSourceTableDefinition()` 一处，但这仍然是**代码级定义**，不是数据库配置。实施顾问无法通过界面调整，必须修改 Java 代码。

### 2.2 具体技术原因

| 问题点 | 当前实现 | 导致的问题 |
|--------|----------|-----------|
| 因子来源表扩展 | `FactorSourceTableEnum` 硬编码枚举 + 对应 Provider 实现类 | 新增来源表必须修改代码（补枚举 + 写 Provider） |
| 字段白名单维护 | `PrioritySourceTableDefinitionDO.configurableFields` 在 Provider 中硬编码 | 扩展字段必须修改 Provider 代码 |
| 关联字段配置 | `PrioritySourceTableDefinitionDO.demandJoinFields / sourceJoinFields` 在 Provider 中硬编码 | 调整关联字段必须修改 Provider 代码 |
| 字段中文名 | 在 Provider 的 `configurableFields` 构造时写死 label | 字段中文名与代码强耦合 |
| 运行时字段取值 | `loadSourceRecordsByKeys()` 返回 VO 对象，通过反射取字段值 | 反射取值已通用化，**此处改造量较小** |

### 2.3 影响范围

- **实施顾问**：无法通过配置界面灵活调整字段白名单，必须依赖开发团队修改代码
- **项目交付**：每个项目的定制需求都会导致代码分叉，后续产品升级困难
- **产品团队**：维护成本高，字段白名单散落在多处代码中，难以统一管理

---

## 3. 解决方案（修复建议与验证思路）

### 3.1 总体设计思路

**将字段白名单从代码层移至数据库配置层**，通过「系统全局参数 > 需求优先级编码方案设置」TAB 页（仅实施顾问可见）实现配置化管理。

### 3.2 配置表设计

#### 3.2.1 因子来源配置表

```sql
CREATE TABLE `dps_priority_factor_source_config` (

  `is_enabled` tinyint(1) NOT NULL DEFAULT 1 COMMENT '是否启用（0=否，1=是）',
  `sort_order` int NOT NULL DEFAULT 0 COMMENT '排序号',

  `create_time` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',

```

#### 3.2.2 因子关联关系配置表

关联字段定义的是需求表与来源表之间的 **JOIN 关系**，即 SQL 中的 `LEFT JOIN source ON demand.xxxId = source.yyyId`。因此关联关系是一对「需求端字段 ↔ 来源端字段」的映射，而不是来源表字段的单独属性。

```sql
CREATE TABLE `dps_priority_factor_relation_config` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '主键',
  `source_config_id` bigint NOT NULL COMMENT '因子来源配置 ID（外键关联 dps_priority_factor_source_config.id）',
  `demand_field_code` varchar(100) NOT NULL COMMENT '需求端关联字段编码（如 customerId、productId）',
  `demand_field_name` varchar(100) NOT NULL COMMENT '需求端关联字段中文名（如 客户ID、物料ID）',
  `source_field_code` varchar(100) NOT NULL COMMENT '来源端关联字段编码（如 id、productId）',
  `source_field_name` varchar(100) NOT NULL COMMENT '来源端关联字段中文名（如 主键、物料ID）',
  `is_enabled` tinyint(1) NOT NULL DEFAULT 1 COMMENT '是否启用（0=否，1=是）',
  `creator` varchar(50) DEFAULT NULL COMMENT '创建人',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `modifier` varchar(50) DEFAULT NULL COMMENT '修改人',
  `modify_time` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '修改时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_source_demand_field` (`source_config_id`, `demand_field_code`),
  KEY `idx_source_config_id` (`source_config_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='需求优先级因子关联关系配置表';
```

**设计说明**：
- 一个来源可配置多条关联关系（如 `PRODUCT_STOCK_POINT` 需要 `productId` + `stockPointId` 双键关联）
- `demand_field_code` 对应需求 VO 上的 `*Id` 字段
- `source_field_code` 对应来源表/VO 上的字段（通常是 `id` 或同名 `*Id`）
- 运行时按此配置拼接 JOIN 条件或过滤条件

#### 3.2.3 因子字段配置表

```sql
CREATE TABLE `dps_priority_factor_field_config` (
  `id` bigint NOT NULL AUTO_INCREMENT COMMENT '主键',
  `source_config_id` bigint NOT NULL COMMENT '因子来源配置 ID（外键关联 dps_priority_factor_source_config.id）',
  `field_code` varchar(100) NOT NULL COMMENT '字段编码（VO 字段名，如 productCode、customerName）',
  `field_name` varchar(100) NOT NULL COMMENT '字段名称（中文）',
  `is_factor_field` tinyint(1) NOT NULL DEFAULT 0 COMMENT '是否可作为参与计算因子（0=否，1=是）',
  `is_enabled` tinyint(1) NOT NULL DEFAULT 1 COMMENT '是否启用（0=否，1=是）',
  `creator` varchar(50) DEFAULT NULL COMMENT '创建人',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `modifier` varchar(50) DEFAULT NULL COMMENT '修改人',
  `modify_time` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '修改时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_source_field` (`source_config_id`, `field_code`),
  KEY `idx_source_config_id` (`source_config_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='需求优先级因子字段配置表';
```

**与旧方案的区别**：
- 移除了 `is_related_field`（关联关系由独立的 `dps_priority_factor_relation_config` 表管理）
- 移除了 `field_type`（不需要数据类型，实施顾问应自行了解）
- 移除了 `sort_order`（按数据库列顺序返回即可）

#### 3.3.2 改造现有下拉接口

**改造 `PriorityFactorConfigServiceImpl`**（当前返回类型为 `List<LabelValue<String>>`）：

```java
@Service
public class PriorityFactorConfigServiceImpl implements PriorityFactorConfigService {

    @Autowired
    private PriorityFactorSourceConfigService factorSourceConfigService;

    /**
     * 获取因子来源表下拉
     * 当前实现：遍历 FactorSourceTableEnum.values()
     * 改造后：优先从配置表读取；配置表无数据时回退到枚举
     */
    @Override
    public List<LabelValue<String>> getSourceTableEnumDropdown() {
        List<FactorSourceConfigVO> sources = factorSourceConfigService.listEnabledFactorSources();
        if (CollectionUtils.isEmpty(sources)) {
            // 降级：回退到原有枚举逻辑
            return Arrays.stream(FactorSourceTableEnum.values())
                .map(e -> new LabelValue<>(e.getDesc(), e.getAlias()))
                .collect(Collectors.toList());
        }
        return sources.stream()
            .map(s -> new LabelValue<>(s.getSourceName(), s.getSourceCode()))
            .collect(Collectors.toList());
    }

    /**
     * 获取关联字段下拉
     * 当前实现：从 Provider.getSourceTableDefinition().demandJoinFields 读取
     * 改造后：优先从关联关系配置表读取
     *
     * 关联字段是需求表与来源表之间的 JOIN 映射关系：
     * 例如 demand.customerId = customer.id
     * 下拉返回的是需求端的 *Id 字段列表
     */
    @Override
    public List<LabelValue<String>> getRelatedFieldsDropdown(String priorityPlanId, String tableCode) {
        List<FactorRelationConfigVO> relations = factorSourceConfigService
            .listRelationsBySourceCode(tableCode);
        if (CollectionUtils.isEmpty(relations)) {
            // 降级：回退到原有 Provider 定义逻辑
            return getFixedRelatedFieldsDropdown(tableCode);
        }
        return relations.stream()
            .map(r -> new LabelValue<>(buildJoinFieldLabel(r.getDemandFieldCode()), r.getDemandFieldCode()))
            .collect(Collectors.toList());
    }

    /**
     * 获取参与计算因子下拉
     * 当前实现：从 Provider.getSourceTableDefinition().configurableFields 读取
     * 改造后：优先从字段配置表读取
     */
    @Override
    public List<LabelValue<String>> getSourceTableFieldsDropdown(String tableCode) {
        List<FactorFieldConfigVO> factorFields = factorSourceConfigService
            .listFactorFieldsBySourceCode(tableCode);
        if (CollectionUtils.isEmpty(factorFields)) {
            // 降级：回退到原有 Provider 定义逻辑
            return getFixedSourceFieldsDropdown(tableCode);
        }
        return factorFields.stream()
            .filter(FactorFieldConfigVO::getIsFactorField)
            .map(f -> new LabelValue<>(f.getFieldName(), f.getFieldCode()))
            .collect(Collectors.toList());
    }
}
```

#### 3.3.3 改造 Provider 的 getSourceTableDefinition()

当前每个 Provider 在 `getSourceTableDefinition()` 中硬编码了 `demandJoinFields`、`sourceJoinFields`、`configurableFields`。改造后，`getSourceTableDefinition()` 应优先从配置表读取，配置表无数据时降级到原有硬编码定义。

**改造思路**：

```java
public interface FactorSourceTableProvider {

    // 现有方法保持不变
    String supportedTableCode();
    String getTableName();
    Class<?> getVOClass();
    default String getSourceTableKeyField() { return "id"; }
    default Map<PrioritySourceJoinKey, Object> loadSourceRecordsByKeys(Set<PrioritySourceJoinKey> joinKeys) { ... }

    /**
     * 获取来源表定义
     * 改造后：优先从配置表读取，配置表无数据时降级到原有硬编码定义
     */
    PrioritySourceTableDefinitionDO getSourceTableDefinition();

    /**
     * 原有的硬编码定义（降级用）
     * 改造后新增此方法，将原来 getSourceTableDefinition() 中的硬编码逻辑移到此处
     */
    default PrioritySourceTableDefinitionDO getDefaultSourceTableDefinition() {
        return getSourceTableDefinition(); // 默认实现，子类可覆盖
    }
}
```

**运行时字段取值**：当前已通过 `ReflectUtils.getFieldValueByName()` 反射取值，天然支持任意字段名，**无需额外改造**。只要字段存在于 VO 中，配置表新增的字段就能自动被取到。

#### 3.3.4 改造运行时计算器

**当前 `PriorityCalculatorAssembledDomainServiceImpl.loadSourceRecords()` 已支持复合键**：
- 通过 `PrioritySourceTableDefinitionDO.demandJoinFields` 获取关联字段列表
- 通过反射从需求 VO 提取各关联字段值，构建 `PrioritySourceJoinKey`
- 调用 `provider.loadSourceRecordsByKeys(joinKeys)` 批量查询

**改造点**：将 `demandJoinFields` / `sourceJoinFields` 的读取来源从 Provider 硬编码改为优先读配置表。

```java
@Service
public class PriorityCalculatorAssembledDomainServiceImpl {

    @Autowired
    private PriorityFactorSourceConfigService factorSourceConfigService;

    /**
     * 预加载来源表记录
     * 改造后：关联字段从配置表读取，构建 PrioritySourceJoinKey
     */
    private Map<PrioritySourceJoinKey, Object> loadSourceRecords(
            String factorSource,
            List<PriorityDemandDO> demandDOList,
            Map<String, FactorSourceTableProvider> source2Provider) {

        FactorSourceTableProvider provider = source2Provider.get(factorSource);
        if (provider == null) return Collections.emptyMap();

        // 改造点：优先从配置表获取关联关系
        List<FactorRelationConfigVO> relations = factorSourceConfigService
            .listRelationsBySourceCode(factorSource);

        List<String> demandJoinFields;
        if (CollectionUtils.isNotEmpty(relations)) {
            // 从配置表读取
            demandJoinFields = relations.stream()
                .map(FactorRelationConfigVO::getDemandFieldCode)
                .collect(Collectors.toList());
        } else {
            // 降级：从 Provider 定义读取
            PrioritySourceTableDefinitionDO def = provider.getSourceTableDefinition();
            demandJoinFields = def.getDemandJoinFields();
        }

        // 后续逻辑不变：构建 PrioritySourceJoinKey → 调用 loadSourceRecordsByKeys()
        Set<PrioritySourceJoinKey> joinKeys = demandDOList.stream()
            .map(demand -> buildDemandJoinKey(demand.getRealDemandVO(), demandJoinFields))
            .filter(PrioritySourceJoinKey::isComplete)
            .collect(Collectors.toSet());

        return provider.loadSourceRecordsByKeys(joinKeys);
    }
}
```

### 3.4 前端配置页面设计

#### 3.4.1 页面位置

**路径**：系统全局参数 > 需求优先级编码方案设置（新增 TAB 页）

**权限控制**：仅实施顾问角色可见

#### 3.4.2 页面结构

**左侧**：因子来源列表
- 显示所有已配置的因子来源（如物品库存点、客户、需求汇总等）
- 支持启用/禁用

**右侧**：分为两个配置区域

1. **关联关系配置区**：
   - 定义需求表与来源表的 JOIN 关系
   - 每行一个映射对：需求端字段（下拉）= 来源端字段（下拉）
   - 支持新增/删除关联行
   - 示例：`demand.customerId = customer.id`
   - 支持多键关联：如 `PRODUCT_STOCK_POINT` 配置 `productId` + `stockPointId` 两行

2. **参与计算因子配置区**：
   - 列出来源表的业务字段（非 ID 字段）
   - 列：字段编码、字段中文名、是否作为参与计算因子（复选框）
   - 支持批量勾选、批量保存

#### 3.4.3 操作流程

1. 实施顾问选择某个因子来源（如「客户」）
2. **配置关联关系**：在关联关系区设置 `demand.customerId = customer.id`
3. **配置参与计算因子**：勾选 `customerCode`、`customerName` 等字段
4. 点击保存，配置立即生效
5. 业务用户在「优先级评分规则配置」弹窗中：
   - 关联字段下拉只显示已配置的关联关系中的需求端字段
   - 因子下拉只显示已勾选的参与计算因子字段

#### 3.5.2 兼容性处理

**降级策略**：若配置表中无数据，回退到原有 Provider 硬编码逻辑（即 `PrioritySourceTableDefinitionDO` 中的定义）

降级逻辑已体现在 §3.3.2 的三个下拉接口中（每个接口都有 `if (CollectionUtils.isEmpty(...))` 降级分支），此处不再重复。

### 3.6 实施流程

#### 3.6.1 产品侧改造（一次性）

1. 创建配置表（`dps_priority_factor_source_config`、`dps_priority_factor_relation_config`、`dps_priority_factor_field_config`）
2. 改造下拉接口（关联字段从关联关系配置表读取，因子字段从字段配置表读取）
3. 扩展 Provider 接口（新增 `getFieldValue` 方法）
4. 改造运行时计算器（支持配置化字段）
5. 开发前端配置页面（系统全局参数 > 需求优先级编码方案设置）
6. 编写数据初始化脚本（将当前硬编码字段迁移到配置表）

#### 3.6.2 项目实施侧操作（每个项目）

1. 执行数据初始化脚本（若是新项目）
2. 实施顾问登录系统，进入「系统全局参数 > 需求优先级编码方案设置」
3. 根据客户需求，配置关联关系与参与计算因子：
   - 选择因子来源（如「客户」）
   - 配置关联关系（如 `demand.customerId = customer.id`）
   - 勾选参与计算因子（如 `customerCode`、`customerName`）
4. 保存配置，立即生效
5. 业务用户在「优先级评分规则配置」弹窗中，只能看到实施顾问配置好的字段

**关键优势**：项目实施时**无需修改代码**，通过配置界面即可完成字段白名单调整，符合「项目可通过 SDK 升级产品」的规则。

本问题分析报告详细阐述了 SDS 需求优先级编码方案字段配置化改造的现状、问题根因与解决方案。核心改造思路是**将字段白名单从代码层移至数据库配置层**，通过前端配置页面实现配置化管理，使项目实施时无需修改代码，符合公司「项目可通过 SDK 升级产品」的规则。
