# 动态透视宽表框架设计 — 问题分析报告

**报告日期**：2026-04-02
**问题类型**：架构设计（新框架）
**涉及模块**：SDS（供应计划），可扩展至全模块

---

## 1. 现状（现象与背景）

### 1.1 SDS 中两类典型透视宽表

当前 SDS 模块存在两种结构高度相似的宽表报表，均以「时段为列、某种行定义为行、数量为单元格」的形态向前端输出。

**供应汇总（supplySummary）**

- 数据来源：6 类供应表（采购订单、制造订单、运输订单、期初库存、采购在途、通用供应），通过 UNION ALL 归一化
- 行定义：**用户动态选择的维度组合**（如物品×库存点，或仅物品），GROUP BY 后每组一行
- 列定义：MDS Feign 获取的时段序列
- 单元格值：`SUM(quantity)`，简单聚合
- 结果结构：`PageInfo<SupplySummaryVO>`，每行含维度字段 + `supplyQuantityMap<periodStart, qty>`

**供需明细汇总（supplyAndDemandSummary）**

- 数据来源：`DemandService` + `SupplyService`（Service 层调用，非直接查表）
- 行定义：**固定指标树**（期初库存 → 毛需求 → 非计划供应[子项] → 净需求 → 计划供应[子项] → 期末库存 → 缺料量）
- 列定义：场景时段组（非 AMS）或按天拆分（AMS）
- 单元格值：各指标的计算值，**含跨行派生公式和跨期滚动**
  - 净需求 = `max(0, 毛需求 - (期初 + 非计划供应合计))`
  - 缺料量 = `max(0, 毛需求 - 期初 - 非计划供应 - 计划供应)`
  - 期初（本期）= 期末（上一期）——跨期滚动
- 分页维度：以物品+库存点分页，每页内含完整指标树
- 结果结构：`PageInfo<SupplyAndDemandSummaryVO>`，每条含 `timeHead` + `dataList: List<IndicatorRowVO>`

### 1.2 两类报表的公共模式

```
步骤 1：时间轴获取     → 拿到有序时段列表（列头）
步骤 2：数据加载       → 按时间范围拉取原始数据
步骤 3：行定义展开     → 确定结果有哪些行
步骤 4：单元格归档     → 将数据按时段 + 行定位，填入 quantityMap
步骤 5：结果渲染       → 封装列元数据 + 行数据，分页返回
```

两者在**步骤 1、5**上逻辑完全相同；**步骤 2 差异中等**；**步骤 3、4 差异最大**（行定义方式 + 填值算法不同）。

### 1.3 当前实现的硬编码现状

## 2. 问题（根因分析）

### 2.1 架构层面根本原因

**同一报表范式，每次新增都全量重新实现，公共骨架没有被抽象出来。**

两类报表 60% 以上的逻辑（时间轴获取、分页、quantityMap 填值、结果封装、权限控制）是公共的，但代码完全独立，形成两套平行实现。每当需要新的「时段×维度/指标」宽表时，都要从头实现，导致代码膨胀。

### 2.2 对"项目可升级 SDK"目标的影响

| 场景 | 当前做法 | 后果 |
|------|---------|------|
| 新增供应来源表（项目定制） | 改 `getTableNames()`、改 DAO XML × 6 处、加枚举 | 项目代码分叉，后续产品升级困难 |
| 新增指标行（如新需求类型） | 改 `convertDataStruct()` 和 `calSupplyAndDemand()` | 同上 |
| 新模块需要类似宽表 | 完整重新实现一遍 | 代码膨胀，重复劳动 |
| 调整维度下拉 | 改 `initDimensionList()` 硬编码 | 同上 |

### 2.3 具体技术问题

| 问题点 | 严重程度 | 说明 |
|--------|---------|------|
| 新增供应来源必须改多处代码 | 高 | 6 条 SQL × N 个 UNION 分支，极易遗漏 |
| SQL 注入风险 | 高 | `${}` 插值的 `groupByParam` 和 `queryCriteriaParam` |
| 无框架复用，代码膨胀 | 高 | 两类报表完全平行实现 |
| 列元数据缺失 | 中 | 前端无法区分维度列与时间列，渲染顺序混乱 |
| orderFlag 维度 Bug | 中 | `existFields.contains("orderFlag")` 用 camelCase 比对下划线列名，永不匹配 |
| 异常处理违规 3 处 | 中 | 硬编码中文字符串，未用 BusinessErrorEnum |
| translate() 无匹配残留 | 低 | 旧 code 以原始值展示（展示一致性问题） |

## 3. 解决方案

### 3.1 总体框架设计

引入「动态透视宽表框架」，核心思想参考 Excel 数据透视表：将报表定义拆分为**数据源、行区域、列区域（时间轴）、值区域（度量）、筛选区域**五个可独立配置的部分，并通过两种引擎分别覆盖「简单聚合类」与「复杂公式类」宽表。

```
┌───────────────────────────────────────────────────────────────┐
│                   公共骨架层 ReportFramework                   │
│  时间轴获取 | 分页 | 权限控制 | quantityMap渲染 | 列元数据输出  │
└───────────────────────────────────────────────────────────────┘
                    ↓ 两种 ReportEngine 实现

┌────────────────────────────┐  ┌────────────────────────────────┐
│   聚合引擎 AggregationEngine│  │  指标树引擎 IndicatorTreeEngine │
│                            │  │                                │
│  配置驱动 SQL 生成           │  │  行定义可配置（标签/顺序）        │
│  UNION ALL 自动拼装          │  │  数据加载：SPI DataProvider      │
│  GROUP BY 白名单校验         │  │  公式计算：SPI FormulaRegistry   │
│  ✅ 完全可配，零代码新增       │  │  公式在 SDK 中，项目不分叉       │
└────────────────────────────┘  └────────────────────────────────┘
         ↑                                    ↑
  supplySummary / 需求汇总              supplyAndDemandSummary / 供需平衡
```

**权限分层**：

| 角色 | 能做什么 |
|------|---------|
| 实施顾问 | 通过配置界面创建/修改报表模板、配置数据段、字段池、时间轴 |
| 业务用户 | 调用查询 API，在字段池范围内选择维度，填写筛选条件 |
| 产品研发 | 实现 SPI（DataProvider、Formula），不暴露给实施顾问 |

### 3.3 核心 Java 抽象

#### 3.3.1 框架骨架接口

```java
/**
 * 报表引擎接口，两种实现分别注册为 Spring Bean
 */
public interface ReportEngine {
    /** 对应 rpt_template.engine_type 值 */
    String supportedEngineType();
    /** 执行报表查询，返回分页结果 */
    ReportResult execute(ReportTemplate template, ReportQueryParam param);
}

/**
 * 框架入口 Service（骨架层）
 * 负责：加载模板 → 鉴权 → 获取时间轴 → 路由到对应 Engine → 封装列元数据
 */
@Service
public class ReportFrameworkService {

    @Autowired private ReportTemplateRepository templateRepo;
    @Autowired private ColumnAxisResolver columnAxisResolver; // 时间轴解析
    @Autowired private List<ReportEngine> engines;            // 所有引擎
    @Autowired private ReportPermissionChecker permissionChecker;

    public ReportPageResult query(String templateCode, ReportQueryParam param) {
        ReportTemplate template = templateRepo.loadByCode(templateCode);
        permissionChecker.checkQueryPermission(template, param.getCurrentUser());

        // 1. 解析时间轴（公共骨架逻辑）
        List<TimePeriodVO> periods = columnAxisResolver.resolve(template.getColumnAxisDef(), param);
        param.setPeriods(periods);

        // 2. 路由到对应引擎
        ReportEngine engine = engines.stream()
            .filter(e -> e.supportedEngineType().equals(template.getEngineType()))
            .findFirst()
            .orElseThrow(() -> new BusinessException(
                BusinessErrorEnum.RPT_ENGINE_NOT_FOUND.getI18nErrordesc(template.getEngineType())));

        ReportResult result = engine.execute(template, param);

        // 3. 组装列元数据（公共骨架逻辑）
        List<ColumnMeta> columnMetas = buildColumnMetas(template, periods, param.getGroupByFields());

        return ReportPageResult.of(result.getPageInfo(), columnMetas);
    }
}
```

### 3.4 聚合引擎实现思路（SQL 生成逻辑闭环）

#### 3.4.1 完整 SQL 生成流程

```
ReportQueryParam（用户请求）
  ↓ 1. 加载模板配置（rpt_data_segment + rpt_dimension_field + rpt_measure_field）
  ↓ 2. 校验 groupByFields ∈ 维度字段池白名单（安全门）
  ↓ 3. 为每个启用且参与当前 supplyCategory 的 DataSegment 生成 SQL 段
  ↓ 4. UNION ALL 拼装
  ↓ 5. 外层 GROUP BY + 聚合
  ↓ 6. 执行（filterValues 全部用 #{} 参数化）
  ↓ 7. 结果集按时段归档，填入 quantityMap
```

#### 3.4.2 单段 SQL 生成（伪代码）

```java
private String buildSegmentSQL(DataSegmentDef seg, String supplyCategory,
                                List<String> groupByFields) {
    // ① SELECT 子句：虚拟列 → 物理列映射
    List<String> selectClauses = new ArrayList<>();
    for (String virtualCol : groupByFields) {
        String physicalCol = seg.getFieldMappings().getOrDefault(virtualCol, virtualCol);
        // 列名来自配置，保存时已校验合法性；此处再次校验格式（仅允许字母数字下划线）
        validateIdentifier(physicalCol);
        selectClauses.add(physicalCol + " AS " + virtualCol);
    }
    // 度量列
    for (MeasureFieldDef measure : template.getMeasureFields()) {
        String physicalCol = seg.getFieldMappings().getOrDefault(measure.getFieldCode(), measure.getFieldCode());
        validateIdentifier(physicalCol);
        selectClauses.add(physicalCol + " AS " + measure.getFieldCode());
    }
    // 时间列
    String supplyDateCol = seg.getFieldMappings().getOrDefault("supply_date", "supply_date");
    validateIdentifier(supplyDateCol);
    selectClauses.add(supplyDateCol + " AS supply_date");
    // 来源标签常量
    selectClauses.add("'" + escapeStringLiteral(seg.getSegmentCode()) + "' AS supply_type");

    // ② FROM + JOIN 子句
    validateTableName(seg.getTableName()); // 从 rpt_table_registry 读取白名单校验
    StringBuilder sql = new StringBuilder("SELECT ")
        .append(String.join(", ", selectClauses))
        .append(" FROM ").append(seg.getTableName()).append(" s");

    for (JoinConfig join : seg.getJoinConfigs()) {
        validateTableName(join.getTable());
        // ON 条件来自配置，保存时已解析为结构化对象，此处拼装
        sql.append(" LEFT JOIN ").append(join.getTable()).append(" ").append(join.getAlias())
           .append(" ON ").append(buildOnClause(join.getOnCondition()));
    }

    // ③ WHERE 子句：分类条件（来自配置，非用户输入）
    List<ConditionDef> conditions = seg.getCategoryConditions().get(supplyCategory);
    if (conditions != null && !conditions.isEmpty()) {
        sql.append(" WHERE ").append(buildWhereClause(conditions));
        // buildWhereClause 输出如 "s.order_flag = 'NO'"
        // 值来自配置，配置保存时已做枚举白名单校验
    }

    return sql.toString();
}
```

#### 3.4.3 外层 UNION ALL + GROUP BY

```java
private String buildFullSQL(List<DataSegmentDef> enabledSegs, String supplyCategory,
                             List<String> groupByFields, List<MeasureFieldDef> measures) {
    // 各段均校验过，这里直接拼
    String unionSQL = enabledSegs.stream()
        .map(seg -> buildSegmentSQL(seg, supplyCategory, groupByFields))
        .collect(Collectors.joining(" UNION ALL "));

    // GROUP BY 字段来自白名单校验后的用户选择
    String groupByClause = groupByFields.stream()
        .map(f -> { validateIdentifier(f); return f; })
        .collect(Collectors.joining(", "));

    String aggClauses = measures.stream()
        .map(m -> m.getAggFunction() + "(" + m.getFieldCode() + ") AS " + m.getFieldCode())
        .collect(Collectors.joining(", "));

    return "SELECT " + groupByClause + ", " + aggClauses + ", supply_date, supply_type"
         + " FROM (" + unionSQL + ") t"
         + " GROUP BY " + groupByClause + ", supply_date, supply_type";
}
```

#### 3.4.4 用户筛选值参数化（安全闭环）

用户传入的 `filterValues`（如 `{"itemCode": "A001"}`）**不拼入 SQL**，而是作为 MyBatis 的命名参数：

```java
// DAO 层使用 MyBatis 的 @Param 机制
Map<String, Object> paramMap = new HashMap<>();
paramMap.put("_dynamicSQL", buildFullSQL(...));  // 框架生成的 SQL
paramMap.put("_filterValues", param.getFilterValues()); // 用户输入的筛选值

// XML 中：
// <where>
//   <if test="_filterValues.itemCode != null">AND item_code = #{_filterValues.itemCode}</if>
// </where>
// 筛选字段白名单在 rpt_filter_field 中定义，XML 中只写已知安全的条件块
```

> **安全性总结**：所有表名来自 `rpt_table_registry` 白名单（产品预置 + 项目注册），配置保存和 SQL 执行前双重校验；列名来自配置（保存时校验格式）；GROUP BY 字段来自维度池白名单；用户输入的筛选值全部通过 `#{}` 参数化，完全消除 SQL 注入风险。

#### 3.5.2 指标树引擎执行流程

```java
@Service
public class IndicatorTreeEngine implements ReportEngine {

    @Autowired private List<IndicatorDataProvider> dataProviders;
    @Autowired private ApplicationContext ctx;

    @Override
    public ReportResult execute(ReportTemplate template, ReportQueryParam param) {
        // 1. 加载指标定义（已排序，含父子关系）
        List<IndicatorDef> indicators = template.getIndicatorDefs();

        // 2. 找到对应的 DataProvider
        IndicatorDataProvider provider = dataProviders.stream()
            .filter(p -> p.supportedTemplateCode().equals(template.getTemplateCode()))
            .findFirst()
            .orElseThrow(() -> new BusinessException(
                BusinessErrorEnum.RPT_PROVIDER_NOT_FOUND.getI18nErrordesc(template.getTemplateCode())));

        // 3. 加载原始数据（分页）
        PageInfo<IndicatorDataSet> dataSets = provider.loadData(param, param.getPeriods());

        // 4. 对每个分页单元（如每个物品+库存点）计算指标树
        List<ReportRowVO> pageItems = dataSets.getList().stream()
            .map(dataSet -> computeIndicatorTree(indicators, dataSet, param.getPeriods()))
            .collect(Collectors.toList());

        return ReportResult.of(pageItems, dataSets.getTotal());
    }

    private ReportRowVO computeIndicatorTree(List<IndicatorDef> indicators,
                                              IndicatorDataSet dataSet,
                                              List<TimePeriodVO> periods) {
        // 指标值矩阵：indicatorCode × periodKey → value
        Map<String, Map<String, BigDecimal>> valueMatrix = new LinkedHashMap<>();

        // 按指标定义顺序逐个计算（公式可能依赖前面已算指标）
        for (IndicatorDef indicator : indicators) {
            IndicatorFormula formula = ctx.getBean(indicator.getFormulaBean(), IndicatorFormula.class);
            Map<String, BigDecimal> periodValues = new LinkedHashMap<>();

            for (TimePeriodVO period : periods) {
                IndicatorComputeContext computeCtx = new DefaultIndicatorComputeContext(
                    period, valueMatrix, dataSet, periods);
                BigDecimal value = formula.compute(computeCtx);
                periodValues.put(period.getPeriodStart(), value);
            }
            valueMatrix.put(indicator.getIndicatorCode(), periodValues);
        }

        // 构建树形行结构
        return buildTreeRows(indicators, valueMatrix);
    }
}
```

#### 3.5.3 公式实现示例（SDK 中，不分叉）

```java
/** 净需求公式 */
@Component("netDemandFormula")
public class NetDemandFormula implements IndicatorFormula {
    @Override
    public String formulaBeanName() { return "netDemandFormula"; }

    @Override
    public BigDecimal compute(IndicatorComputeContext ctx) {
        BigDecimal grossDemand    = ctx.getIndicatorValue("GROSS_DEMAND");
        BigDecimal beginningStock = ctx.getIndicatorValue("BEGINNING_STOCK");
        BigDecimal nonPlanSupply  = ctx.getIndicatorValue("NON_PLAN_SUPPLY");
        BigDecimal available = beginningStock.add(nonPlanSupply);
        return grossDemand.subtract(available).max(BigDecimal.ZERO);
    }
}

/** 期初滚动公式（依赖上一期期末） */
@Component("beginningStockFormula")
public class BeginningStockFormula implements IndicatorFormula {
    @Override
    public BigDecimal compute(IndicatorComputeContext ctx) {
        // 上一期期末
        BigDecimal prevEnding = ctx.getPreviousPeriodValue("ENDING_STOCK");
        if (prevEnding == null) {
            // 第一期：从原始数据取 BOH 库存值
            return ctx.rawData().getBohStock();
        }
        // 本期的库存类供应需加入期初（STOCK_SUPPLY 类型）
        BigDecimal stockSupply = ctx.rawData().getSupplyInPeriod(
            ctx.currentPeriod(), "STOCK_SUPPLY");
        return prevEnding.add(stockSupply);
    }
}
```

> **SPI 闭环说明**：`IndicatorFormula` 在 SDK 中实现并注册为 Spring Bean。`rpt_indicator_def.formula_bean` 存储 Bean 名称（字符串），引擎通过 `ApplicationContext.getBean()` 动态获取。项目升级 SDK 后，公式在 SDK 中更新；项目的数据库配置（指标定义的 `indicator_code`、`formula_bean`）保持不变，实现升级兼容。

## 5. 附录：核心概念通俗解释

> 本节用业务语言解释报告中涉及的技术概念，供非开发背景的读者理解方案设计意图。

### 5.1 两种引擎的本质区别

#### 聚合引擎 —— "数据决定行"

类比 Excel 数据透视表：把原始明细数据拖进去，选"按什么分组"，Excel 自动汇总出结果。有多少种组合，就出多少行。

```
原始数据（6张供应表合并后）：
┌──────────┬──────────┬────────────┬──────┐
│ 物品编码  │ 库存点    │ 供应日期    │ 数量  │
├──────────┼──────────┼────────────┼──────┤
│ A001     │ WH-01    │ 2026-04-01 │ 100  │
│ A001     │ WH-01    │ 2026-04-01 │ 50   │
│ A001     │ WH-02    │ 2026-04-01 │ 200  │
└──────────┴──────────┴────────────┴──────┘

用户选择维度 = [物品编码, 库存点]，度量 = SUM(数量)

→ 按 物品编码 + 库存点 + 时段 分组汇总后：
┌──────────┬──────────┬──────────┬──────┐
│ 物品编码  │ 库存点    │ 4月第1周  │ 数量  │
├──────────┼──────────┼──────────┼──────┤
│ A001     │ WH-01    │ 4月第1周  │ 150  │ ← 100+50 合并了
│ A001     │ WH-02    │ 4月第1周  │ 200  │
└──────────┴──────────┴──────────┴──────┘
```

- **维度字段**（行区域）：决定"按什么分组"——用户从字段池里勾选，不同选择出不同的行
- **度量字段**（值区域）：决定"对哪个数值列做什么运算"——通常是 `SUM(数量)`，也可以是 `COUNT(订单数)` 等
- **时间轴**（列区域）：决定列头是哪些时段
- **单元格** = 该维度组合 + 该时段内，所有数量的求和结果

#### 指标树引擎 —— "行是固定的，行与行之间有公式"

供需明细汇总的行**不是**数据分组出来的，而是预先定义好的一棵指标树。不管有没有数据，这些行永远存在，且**行与行之间存在计算依赖**：

```
┌──────────────┬─────────┬─────────┬─────────┐
│ 指标          │ 4月第1周 │ 4月第2周 │ 4月第3周 │
├──────────────┼─────────┼─────────┼─────────┤
│ 期初库存      │ 1000    │  800    │  500    │ ← 第2周期初 = 第1周期末（跨列滚动）
│ 毛需求        │  500    │  400    │  300    │
│  ├ 客户订单   │  300    │  200    │  200    │ ← 子项，展开显示
│  └ 预测需求   │  200    │  200    │  100    │ ← 子项，展开显示
│ 非计划供应    │  200    │   50    │    0    │
│ 净需求        │    0    │  150    │  300    │ ← = max(0, 毛需求 - 期初 - 非计划)
│ 计划供应      │  300    │  350    │  200    │
│ 期末库存      │  800    │  500    │  200    │ ← = 期初 - 毛需求 + 非计划 + 计划
│ 缺料量        │    0    │    0    │  100    │ ← = max(0, 毛需求 - 期初 - 非计划 - 计划)
└──────────────┴─────────┴─────────┴─────────┘
```

**为什么不能用聚合引擎？** 因为聚合引擎只会"按分组求和"，做不了以下三件事：

| 能力 | 聚合引擎 | 指标树引擎 |
|------|---------|-----------|
| 跨行公式（净需求依赖毛需求、期初、非计划三行的值） | 做不到 | 支持 |
| 跨列滚动（本周期初 = 上周期末） | 做不到 | 支持 |
| 固定行 + 父子层级（毛需求下挂客户订单和预测需求） | 做不到 | 支持 |

#### "指标定义"是什么

就是上面表格中**每一行的定义**。配置表 `rpt_indicator_def` 里存的是：

- 这个指标叫什么名字（如"净需求"）
- 排在第几行
- 父指标是谁（如"客户订单"的父指标是"毛需求"）
- 用哪个公式来计算

实施顾问可以改名字、调顺序、启用/禁用，但**不能改公式逻辑**——公式由产品研发在 SDK 代码中实现。

### 5.2 分页维度：为什么聚合引擎没有，指标树引擎必须有

**聚合引擎**的结果是扁平列表（每行独立），直接按常规方式分页即可（第1页20行、第2页20行……）：

```
第1页：
│ A001 │ WH-01 │ 100 │ 200 │ 150 │
│ A001 │ WH-02 │ 50  │ 80  │ 60  │
│ A002 │ WH-01 │ 300 │ 400 │ 350 │
...每行独立，随便切页
```

**指标树引擎**的每一"页"不是一行，而是**一棵完整的指标树**（7~15行）。"分页维度"告诉系统"每棵树对应什么"：

```
配置：分页维度 = 物品 + 库存点

第1页：A001 × WH-01 的完整指标树
┌──────────────┬─────────┬─────────┐
│ 期初库存      │ 1000    │  800    │
│ 毛需求        │  500    │  400    │
│ ...          │  ...    │  ...    │
│ 缺料量        │    0    │    0    │
└──────────────┴─────────┴─────────┘

第2页：A001 × WH-02 的完整指标树
（同样的7~15行，但数据不同）

第3页：A002 × WH-01 的完整指标树
...
```

如果不配分页维度，系统不知道按什么拆分——是每个物品一棵树，还是每个物品×库存点一棵树？所以指标树引擎**必须**指定分页维度。

### 5.3 Provider Bean（数据取数器）

指标树引擎的工作分两步：
1. **取原始数据**——从数据库拉出需求明细、供应明细、库存数据
2. **按公式计算**——用取到的数据逐行逐列填值

Provider Bean 就是负责第 1 步的程序组件。它封装了"从哪些表查、怎么查"的逻辑。

**类比**：聚合引擎是"框架自己按配置拼 SQL 取数"（完全自动化），指标树引擎是"交给一个专门的取数程序去取数，框架只管算公式"（取数逻辑更复杂，无法纯靠配置驱动）。

**实施顾问不需要关心 Provider Bean**——它由产品研发在 SDK 里写好，配置页面上只展示名字，标为只读不可编辑。

### 5.4 MDS Feign（时间轴来源）

- **MDS**（Master Data Service）= 主数据服务，是系统中管理「物品、库存点、时段组」等基础数据的模块
- **Feign** = 微服务之间远程调用的方式。SDS（供应计划）和 MDS（主数据）是两个独立的微服务，SDS 通过 Feign 调用向 MDS 请求数据

**时间轴的 MDS Feign** 意思就是：报表的列头（时段序列）从 MDS 服务获取。

```
报表需要知道列头是哪些时段
      ↓
通过 Feign 调用 MDS 主数据服务的接口
      ↓
MDS 返回：[4月第1周, 4月第2周, 4月第3周, ...]
      ↓
这些就成为报表的列头
```

框架支持三种时间轴来源，适用不同场景：

| 类型 | 通俗解释 | 适用场景 |
|------|---------|---------|
| MDS Feign 接口 | 从主数据服务拿已配置的时段组 | SDS 当前使用的方式，时段由 MDS 统一管理 |
| 自然月 | 不依赖 MDS，直接按自然月生成列头 | 不需要精细时段划分的简单报表 |
| 数据库时段表 | 从一张自定义表里读时段定义 | 项目有特殊的时段结构，不走 MDS 管理 |

### 5.5 scp-extension 与表注册机制

#### 系统的双仓库架构

系统代码分为两个仓库：

| 仓库 | 定位 | 谁维护 |
|------|------|--------|
| **scp-foundation** | 产品 SDK，包含核心业务逻辑和框架代码 | 产品研发团队 |
| **scp-extension** | 项目扩展层，用于项目定制（加字段、加表） | 各项目实施团队 |

#### extension 的 "extend" 模式（加字段）

extension 采用**类继承**模式——不是创建独立的扩展表，而是在**同一张物理表**上扩展：

```
产品 SDK 定义的基础表 sds_ord_purchase_order：
┌──────────┬──────────┬──────────┬──────┐
│ item_code│ stock_pt │ sup_date │ qty  │  ← SDK 预定义的标准列
└──────────┴──────────┴──────────┴──────┘

项目通过 extension 加了自定义字段后，同一张表变为：
┌──────────┬──────────┬──────────┬──────┬────────────┬───────────┐
│ item_code│ stock_pt │ sup_date │ qty  │ custom_flag│ batch_no  │  ← 新增的列
└──────────┴──────────┴──────────┴──────┴────────────┴───────────┘
```

**对本框架的影响**：项目加字段后，配置页面的「探测列」功能（读取 `information_schema.COLUMNS`）会自动发现新列，实施顾问在数据段的字段映射中补上即可，**无需改代码**。

#### 表注册机制（加表）

当项目需要新增一张全新的供应/需求表时（不是给已有表加字段），面临的问题是：配置页面不知道数据库里新建了什么表。

解决方案是引入**表注册表**（`rpt_table_registry`）：

```
项目 DBA 建表 sds_supply_custom
       ↓
实施顾问打开配置页面 → 数据段 Tab → 新增数据段
       ↓
点击「扫描新表」按钮
       ↓
系统查 information_schema.TABLES，列出匹配 sds_* 前缀的所有表
       ↓
找到未注册的 sds_supply_custom → 勾选 → 填写中文名 → 注册
       ↓
新表出现在来源表下拉列表中 → 选择 → 探测列 → 配置映射 → 保存
       ↓
全程零代码，产品 SDK 不需要任何改动
```

这张注册表同时充当 **SQL 安全白名单**：只有注册过的表才能被配置为数据段来源，未注册的表名会被框架拒绝，防止被恶意利用。

| 场景 | 操作方式 | 是否需要改代码 |
|------|---------|--------------|
| 项目给已有表加字段（extend） | 探测列自动发现 → 补字段映射 | 否 |
| 项目新增一张表 | 扫描注册 → 配置数据段 | 否 |
| 产品新增供应类型表 | SDK 升级时 flyway 脚本自动写入 registry | 否（数据脚本） |
