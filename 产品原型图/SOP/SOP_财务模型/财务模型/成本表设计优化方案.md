## 成本表设计优化方案（sop_pla_finance_cost）

本文基于当前 UI 设计（生产成本、采购成本、运输成本、库存成本、违约罚金）对现有成本明细表 `sop_pla_finance_cost` 进行评估与改造建议。

---

### 一、现有表结构问题概览

当前表结构：

```sql
CREATE TABLE `sop_pla_finance_cost` (
  `id` varchar(64) NOT NULL COMMENT '主键id',
  `version_id` varchar(64) DEFAULT NULL COMMENT '版本id',
  `stock_point_id` varchar(64) DEFAULT NULL COMMENT '库存点id',
  `product_id` varchar(64) DEFAULT NULL COMMENT '物品id',
  `product_stock_point_id` varchar(64) DEFAULT NULL COMMENT '库存点物品id',
  `cost_group` varchar(64) DEFAULT NULL COMMENT '成本组',
  `cost_item` varchar(64) DEFAULT NULL COMMENT '成本项',
  `cost_object` varchar(64) DEFAULT NULL COMMENT '成本关联对象',
  `cost_object_id` varchar(64) DEFAULT NULL COMMENT '成本关联对象id',
  `cost_type` varchar(64) DEFAULT NULL COMMENT '成本类型',
  `cost_value` decimal(14,4) DEFAULT 0 COMMENT '成本值',
  `time_cost_uom` varchar(64) DEFAULT NULL COMMENT '时间单位',
  `currency_unit_id` varchar(64) DEFAULT NULL COMMENT '货币单位id',
  `start_time` datetime DEFAULT NULL COMMENT '开始时间',
  `end_time` datetime DEFAULT NULL COMMENT '结束时间',
  `remark` varchar(255) DEFAULT NULL COMMENT '备注',
  `creator` varchar(64) DEFAULT NULL COMMENT '创建人',
  `create_time` datetime DEFAULT NULL COMMENT '创建时间',
  `modifier` varchar(64) DEFAULT NULL COMMENT '修改人',
  `modify_time` datetime DEFAULT NULL COMMENT '修改时间',
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='成本明细';
```

**主要问题：**

- **语义模糊**  
  - `cost_group / cost_item / cost_type / cost_object / cost_object_id` 都是宽泛的 `varchar`，含义重叠且无约束。  
  - 很难从表结构上区分：
    - 生产 / 采购 / 运输 / 库存 / 违约罚金是哪一类？
    - 单位成本 vs 固定成本 vs 时间成本？
    - 关联的是生产资源、采购路径、运输路径、客户还是其他对象？
- **维度不结构化**  
  - UI 层已经有清晰维度：生产资源代码、采购路径代码、运输路径代码、库存点物品、客户等。  
  - 表中没有对应字段，只能通过 `cost_object(_id)` 间接表示，查询和约束都比较困难。
- **缺少去重约束**  
  - 没有唯一键来保证「同一维度+同一度量」只有一条记录，容易出现重复行。

---

### 二、推荐方案：头-行两层结构 + 强类型字段

#### 2.1 头表：公共维度（版本 + 库存点 + 物品 + 时间段）

```sql
CREATE TABLE sop_pla_finance_cost_header (
  id                   varchar(64)  PRIMARY KEY,
  version_id           varchar(64)  NOT NULL COMMENT '财务版本id',
  stock_point_id       varchar(64)  NOT NULL COMMENT '库存点id',
  product_id           varchar(64)  NOT NULL COMMENT '物品id',
  product_stock_point_id varchar(64) DEFAULT NULL COMMENT '库存点物品id',
  start_time           datetime     NOT NULL COMMENT '开始时间',
  end_time             datetime     NOT NULL COMMENT '结束时间',
  currency_unit_id     varchar(64)  NOT NULL COMMENT '货币单位id',
  enabled              char(1)      DEFAULT '1' COMMENT '是否启用',
  remark               varchar(255) DEFAULT NULL,
  creator              varchar(64),
  create_time          datetime,
  modifier             varchar(64),
  modify_time          datetime,
  UNIQUE KEY uk_cost_hdr (version_id, stock_point_id, product_id, start_time, end_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='成本明细头';
```

**设计要点：**

- 与前端分组维度完全对齐：**版本 + 库存点 + 物品 + 时间段 + 货币**。  
- `enabled` 放在头上，统一控制该维度下所有成本行是否有效。  
- 通过唯一键 `uk_cost_hdr` 保证一个版本+库存点+物品+时间段只有一个头记录。

#### 2.2 行表：成本类型 + 具体对象维度

```sql
CREATE TABLE sop_pla_finance_cost_line (
  id              varchar(64) PRIMARY KEY,
  header_id       varchar(64) NOT NULL COMMENT '头表id，FK->cost_header.id',

  cost_category   varchar(32) NOT NULL COMMENT '成本大类: PRODUCTION/PURCHASE/TRANSPORT/INVENTORY/PENALTY',
  metric_type     varchar(32) NOT NULL COMMENT '度量类型: UNIT/FIXED/VIOLATION/HOLDING/LATE_PENALTY/NON_DELIVERY',

  -- 明确业务维度（可空）
  standard_resource_id   varchar(64) DEFAULT NULL COMMENT '生产资源id',
  purchase_routing_id    varchar(64) DEFAULT NULL COMMENT '采购路径id',
  transport_routing_id   varchar(64) DEFAULT NULL COMMENT '运输路径id',
  source_stock_point_id  varchar(64) DEFAULT NULL COMMENT '运输来源库存点id',
  customer_id            varchar(64) DEFAULT NULL COMMENT '客户id',

  cost_value      decimal(14,4) NOT NULL DEFAULT 0 COMMENT '成本值',
  time_cost_uom   varchar(32)   DEFAULT NULL COMMENT '时间成本单位(天/小时等)',
  remark          varchar(255)  DEFAULT NULL,

  creator         varchar(64),
  create_time     datetime,
  modifier        varchar(64),
  modify_time     datetime,

  CONSTRAINT fk_cost_line_header
    FOREIGN KEY (header_id) REFERENCES sop_pla_finance_cost_header(id),

  -- 保证同一维度下一个度量仅一条记录
  UNIQUE KEY uk_cost_dim (
    header_id,
    cost_category,
    metric_type,
    COALESCE(standard_resource_id, ''),
    COALESCE(purchase_routing_id, ''),
    COALESCE(transport_routing_id, ''),
    COALESCE(source_stock_point_id, ''),
    COALESCE(customer_id, '')
  )
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='成本明细行';
```

**映射到业务场景：**

- **生产成本**  
  - 行：`cost_category='PRODUCTION'`  
    - 单位生产成本：`metric_type='UNIT'` + `standard_resource_id`  
    - 固定生产成本：`metric_type='FIXED'` + `standard_resource_id`
- **采购成本**  
  - 行：`cost_category='PURCHASE'`  
    - 单位/固定采购成本：`metric_type='UNIT'/'FIXED'` + `purchase_routing_id`
- **运输成本**  
  - 行：`cost_category='TRANSPORT'`  
    - 单位/固定运输成本：`metric_type='UNIT'/'FIXED'` + `transport_routing_id` + `source_stock_point_id`
- **库存成本**  
  - 行：`cost_category='INVENTORY'`  
    - 单位最大库存违反成本：`metric_type='VIOLATION'`  
    - 库存持有成本：`metric_type='HOLDING'`
- **违约罚金**  
  - 行：`cost_category='PENALTY'`  
    - 单位晚交罚金：`metric_type='LATE_PENALTY'` + `customer_id`  
    - 单位欠交罚金：`metric_type='NON_DELIVERY'` + `customer_id`

> 通过 `cost_category + metric_type + 具体对象维度`，可以精确定位一条成本记录，无需再依赖语义不清晰的 `cost_group/cost_item/cost_object`。

---

### 三、小结

- 现有 `sop_pla_finance_cost` 表能工作，但语义较松散，不利于长期维护与扩展。  
- **推荐改造路线**：引入「头+行」两张表，把公共维度与具体成本项分离，字段强类型化，直接贴合当前前端五个成本 Tab 的业务模型。

不论选择哪条方案，目标都是：  
在不牺牲灵活性的前提下，让成本明细「一眼能看懂」「易于约束」「易于查询统计」，并能稳定支撑后续库存成本、违约罚金等进一步扩展。
