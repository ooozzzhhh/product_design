## 1. 功能说明

### 1.1 功能定位

提供数据库层面的「自动归档任务管理」原型，用于展示如何在前端配置与查看数据库归档任务。

### 1.2 核心能力

- **管理归档任务列表**：任务名称、归档对象（表）、触发条件、状态与操作；
- **配置新建/编辑归档任务**：
  - 基础信息：任务名称、存储格式、存储路径、文件命名规则；
  - 触发规则：定时触发（每周/每月/每季度/每年）或数据量触发；
  - 归档对象与范围：支持「按业务对象归档」和「按数据表归档」两种模式；
  - 归档后操作：自动清理或手动配置清理规则。
- **查看归档任务执行日志**：归档文件名、执行时间、状态、文件路径等。

### 1.3 术语统一

- 界面上所有中文用语统一使用「归档」，不再使用「备份」一词；
- 仅在文件名、路径或英文标识中保留 `backup` 作为历史命名（如 `backup_20241218_020000.sql`、`/data/backup/` 等）。

### 3.2 新建/编辑归档任务弹窗

#### 3.2.1 弹窗结构

弹窗分为三部分：
1. **顶部标题栏**：标题 + 关闭按钮（✕）

#### 3.2.2 配置区域 1：基础信息配置

**字段**：

- 存储格式：下拉框，选项"SQL 文件"、"NB3 格式"
- 存储路径（必填）：文本输入框 + 「校验权限」按钮
- 文件命名规则：单选按钮，两个选项：
  - `{task_name}_{timestamp}.{ext}` (推荐)
  - `{db_name}_{timestamp}.{ext}`

**交互**：
- 点击「校验权限」：验证路径是否可写（原型中暂未实现，实际开发时需对接后端接口）

#### 3.2.3 配置区域 2：触发规则

**字段**：
- 触发类型：单选按钮，「定时触发」或「数据量触发」

**定时触发**：
- 时间颗粒度：下拉框，选项"每周"、"每月"、"每季度"、"每年"
- 具体时间：根据颗粒度动态展示（原型中简化为"星期一 + 时间选择器"）

**数据量触发**：
- 总行数阈值：数字输入框 + "行"文字

**交互**：
- 切换触发类型时，显示/隐藏对应的配置区域

#### 3.2.4 配置区域 3：归档对象选择与范围配置（核心）

##### 3.2.4.1 归档模式切换

**UI 元素**：
- 模式切换区域：浅灰背景 `#f9f9f9`，圆角 `6px`
- 两个单选按钮：
  - 📦 按业务对象归档（推荐）
  - 📋 按数据表归档（高级）
- 提示文字："两种模式数据独立，切换模式不会保留已配置的数据"

**交互行为**：
- 切换模式时，显示/隐藏对应的配置区域
- **数据隔离**：两种模式的数据独立存储（`objectModeData` 和 `tableModeData`），切换时不会丢失已配置数据，但会显示对应模式的数据

##### 3.2.4.2 按业务对象归档模式

**选择区域**：
- 业务对象分类下拉框：「业务对象类」「系统与日志类」「其他表」
- 业务对象下拉框：根据分类动态加载对应的业务对象列表
- 「添加」按钮：将选中的业务对象添加到列表

**业务对象列表**（已选对象）：

每个业务对象以卡片形式展示，包含：

1. **卡片头部**：
   - 左侧：对象名称 + 类型徽章（如"业务对象 · 3张表"）
   - 右侧：「展开配置」「删除」按钮

2. **对象级配置**（默认显示）：
   - 归档方式下拉框 + 「应用到所有表」按钮
   - 基准字段输入框 + 「应用到所有表」按钮
   - 归档范围配置（数字输入框 + 单位下拉框 + "之前的数据"文字）

3. **子表配置区域**（点击「展开配置」后显示）：
   - 提示文字："该对象包含以下表，可单独配置归档规则："
   - 每个子表以白色卡片展示，包含：
     - 表名（带 ✓ 标记）
     - 归档方式下拉框
     - 基准字段输入框

2. **对象级配置**：
   - 修改归档方式/基准字段/归档范围时，只更新对象级别的显示值
   - 点击「应用到所有表」按钮，才会将配置批量应用到该对象下的所有子表

3. **展开/收起配置**：
   - 点击「展开配置」按钮，显示子表配置区域，按钮文字变为「收起配置」
   - 再次点击，隐藏子表配置区域，按钮文字变为「展开配置」

4. **子表单独配置**：
   - 在展开的子表配置区域，可以修改每个表的归档方式、基准字段、归档范围
   - 修改后立即生效，并在界面上显示差异化标记

5. **删除对象**：
   - 点击「删除」按钮，弹出确认对话框"确定要删除该业务对象吗？"
   - 确认后，从列表中移除该对象

**空状态**：
- 未添加任何对象时，显示空状态提示：
  - 图标：📦
  - 文字："暂无已选业务对象，请从上方添加"

## 4. 数据管理

### 4.1 数据隔离机制

**核心原则**：两种归档模式的数据完全隔离，互不影响。

**实现方式**：
- 使用两个独立的数组存储数据：
  - `objectModeData[]`：按业务对象模式的数据
  - `tableModeData[]`：按数据表模式的数据
- 切换模式时，调用 `switchArchiveMode()` 函数：
  - 显示/隐藏对应的 DOM 元素
  - 渲染对应模式的数据列表
- 保存时，根据当前选中的模式，只保存对应的数据数组

**用户体验**：
- 用户在"按业务对象"模式下添加了 3 个对象
- 切换到"按数据表"模式，看到的是空列表（或之前在该模式下添加的表）
- 再切换回"按业务对象"模式，之前添加的 3 个对象仍然存在
- 保存时，只保存当前模式的数据

### 4.2 业务对象定义

**数据结构**：
```javascript
const businessObjects = {
  business: [
    { id: 'mrp_run', name: 'MRP 运行记录', tables: ['mrp_run_header', 'mrp_run_detail', 'mrp_run_log'] },
    { id: 'mps_order', name: '主计划订单', tables: ['mps_order', 'mps_order_item'] },
    // ...
  ],
  system: [
    { id: 'operation_log', name: '操作日志', tables: ['sys_operation_log'] },
    // ...
  ],
  other: [
    { id: 'temp_table', name: '临时表', tables: ['temp_data'] },
    // ...
  ]
};
```

**说明**：
- 业务对象定义在前端硬编码（原型阶段）
- 实际开发时，应从后端接口动态加载
- 每个业务对象包含：id（唯一标识）、name（显示名称）、tables（关联的表名数组）

### 4.3 可搜索表列表

**数据结构**：
```javascript
const allTables = [
  { name: 't_user_logs', desc: '用户操作日志表' },
```

## 5. 实现注意事项

### 5.1 术语一致性

- 实现时前后端、配置与文档中，应统一将此能力称为「自动归档」或「归档任务」
- 避免与传统数据库"备份/恢复"混淆

### 5.2 数据安全与风险控制

- 归档后清理属于高风险操作，正式实现时建议增加权限与审批控制
- 建议在后端层面对删除操作做分批执行与事务控制，避免对在线业务产生明显影响

### 5.3 前后端对接

#### 5.3.1 业务对象管理

- 后端需提供业务对象列表接口：`GET /api/archive/business-objects`
- 返回数据格式：
  ```json
  {
    "business": [
      {
        "id": "mrp_run",
        "name": "MRP 运行记录",
        "tables": ["mrp_run_header", "mrp_run_detail", "mrp_run_log"]
      }
    ],
    "system": [...],
    "other": [...]
  }
  ```

#### 5.3.2 表搜索接口

- 后端需提供表搜索接口：`GET /api/archive/tables?keyword=xxx`
- 返回数据格式：
  ```json
  [
    { "name": "t_user_logs", "desc": "用户操作日志表" },
    { "name": "t_orders", "desc": "订单表" }
  ]
  ```

#### 5.3.3 保存任务接口

- 后端需提供保存任务接口：`POST /api/archive/tasks`
- 请求数据格式：
  ```json
  {
    "taskName": "Daily_MRP_Archive",
    "storageFormat": "SQL",
    "storagePath": "/data/backup/",
    "namingRule": "{task_name}_{timestamp}.{ext}",
    "triggerType": "time",
    "triggerConfig": { ... },
    "archiveMode": "byObject",
    "archiveData": [ ... ],
    "cleanConfig": { ... }
  }
  ```

### 5.4 性能优化

- 业务对象列表和表列表应做缓存，避免频繁请求
- 大量表的搜索应做防抖处理，避免频繁触发搜索
- 子表配置区域的展开/收起应做动画过渡，提升用户体验

### 5.5 可访问性

- 所有交互元素应支持键盘操作
- 表单控件应有明确的 label 关联
- 错误提示应使用 aria-live 区域，便于屏幕阅读器识别

### 5.6 响应式设计

- 原型按 1400px 宽度设计，实际开发时应考虑不同屏幕尺寸的适配
- 弹窗宽度设为 90%，最大宽度 1200px，确保在小屏幕上也能正常显示
- 配置行应支持换行，避免在小屏幕上内容溢出

## 6. 与旧版原型的差异

| 项目 | 旧版 | v2 版 |
|------|------|-------|
| 数据隔离 | 无，两种模式共用数据 | 有，两种模式数据完全隔离 |
| 添加业务对象 | 只弹提示框，不真实添加 | 真实添加到列表，可配置 |
| 子表配置 | 展开后不可操作 | 展开后可单独配置每个表 |
| 差异化标记 | 无 | 有，红色文字标记 |
| 空状态提示 | 无 | 有，友好的空状态提示 |
| UI 结构 | 表格式布局 | 卡片式布局，层次更清晰 |
| 视觉区分 | 无 | 不同类型对象用不同颜色左边框 |
| 交互反馈 | 简单 | 完善，所有操作都有反馈 |
