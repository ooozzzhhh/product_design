## 二、新增功能点

### 2.1 并发控制规范 (PRD 2.1.6)

**实现方式**:
- 在功能配置中声明并发模式(`concurrencyMode`)与并发数限制(`maxConcurrency`)
- 前端展示:
  - EXCLUSIVE模式:若检测到锁已被占用,在运行配置弹窗提交时展示错误提示并阻止提交;功能卡片上该功能名称右侧展示加载动画
  - QUEUEABLE模式:若锁已被占用,提交后展示排队提示并关闭弹窗;执行历史列表中该任务显示为"排队中"状态
  - LIMITED_PARALLEL模式:若拒绝模式,返回错误提示;若排队模式,同QUEUEABLE模式

### 2.2 实时推送机制 (PRD 2.1.7)

**实现方式**:
- 当前实现:前端定时轮询
- 推荐实现:WebSocket实时推送(后续迭代)
- 降级策略:WebSocket不可用时自动降级到轮询模式

### 2.3 终止与异常跳过 (PRD 2.1.8)

**实现方式**:
- 终止接口规范:
  - 接口路径:`POST /progressListener/process/interruptTask`
  - 前端交互:点击"终止"按钮 → 二次确认对话框 → 发送终止请求 → 关闭进度弹窗,刷新执行历史列表
- 节点级可中断控制:
  - 在节点枚举中定义`isInterruptible()`属性
  - 可中断节点:数据加载、校验、计算类步骤
  - 不可中断节点:结果写回、事务提交类步骤
- 异常跳过框架:
  - 在节点枚举中定义`isSkippableOnError()`属性
  - 可跳过节点:失败时记录日志但继续执行下一节点
  - 不可跳过节点:失败时终止整个任务

### 2.4 性能监控 (PRD 2.1.9)

**实现方式**:
- 节点级业务指标:
  - 记录格式:`[指标名称]: [指标值] [单位]`
  - 示例:"处理物料数: 10,000 / 50,000"、"数据加载耗时: 5 秒"
- 性能监控接口(按需触发):
  - 接口路径:`GET /progressListener/process/getPerformanceMetrics`
  - 响应数据:JVM内存、线程、数据库连接池、节点执行情况
- 前端展示:
  - 进度弹窗中增加"性能详情"按钮
  - 点击后弹出性能详情对话框,展示JVM、线程、数据库连接池等指标

## 三、技术实现说明

### 3.1 并发控制

**前端职责**:
- 在运行配置弹窗提交前,根据功能的并发配置检查是否允许提交
- 若不允许,展示错误提示(如"上一次MRP正在运行中,请稍后再试!")
- 若允许排队,提交后展示排队提示(如"当前排队位置:第3位")

**后端职责**:
- 使用Redis分布式锁实现并发控制
- 锁key格式:`concurrency:{functionCode}:{scenario}:lock`
- 队列key格式:`concurrency:{functionCode}:{scenario}:queue`
- 锁超时时间:15分钟(可配置)

### 3.2 实时推送

**当前实现**:
- 前端定时轮询接口:`GET /progressListener/process/getProgressByTaskId?taskId={任务ID}`
- 轮询间隔:1～2秒
- 轮询生命周期:进度弹窗打开时启动,弹窗关闭或任务结束时停止

**推荐实现**(后续迭代):
- WebSocket端点:`/ws/progress/{taskId}`
- 消息格式:JSON,包含type、taskId、timestamp、data字段
- 消息类型:PROGRESS_UPDATE、STATUS_CHANGE、NODE_START、NODE_COMPLETE、ERROR、HEARTBEAT
- 断线重连:最多5次,间隔3秒;重连失败后降级到轮询模式

### 3.3 终止与异常跳过

**终止流程**:
1. 用户点击"终止"按钮
2. 前端弹出二次确认对话框
3. 用户确认后,前端发送终止请求
4. 后端收到请求后:
   - 检查当前节点是否可中断
   - 若可中断,通过`Future.cancel(true)`中断线程,将任务状态置为"已终止"
   - 若不可中断,返回错误提示(如"当前节点'建议计划生成与报告汇总'不可中断")
5. 前端收到响应后:
   - 成功:关闭进度弹窗,刷新执行历史列表,该任务显示为"已终止"
   - 失败:展示错误提示,保持任务为运行中状态

**异常跳过流程**:
1. 后端在节点执行过程中捕获异常
2. 根据节点的`isSkippableOnError()`属性决定是否继续执行下一节点
3. 若可跳过:记录失败但继续执行下一节点,在运行日志中记录"节点X失败但已跳过"
4. 若不可跳过:终止整个任务,将任务状态置为"失败"
5. 前端在执行节点详情弹窗中:
   - 失败但已跳过的节点显示为黄色警告图标
   - 节点说明显示:"节点执行失败但已跳过:部分BOM数据缺失(10条)"

### 3.4 性能监控

**前端实现**:
- 在进度弹窗右上角添加"性能详情"按钮(蓝色,仪表盘图标)
- 点击后弹出性能详情对话框
- 对话框内容:
  - JVM内存使用情况(堆内存、非堆内存、GC次数与耗时)
  - 线程情况(线程数、峰值线程数、守护线程数)
  - 数据库连接池情况(活跃连接数、空闲连接数、最大连接数)
  - 节点执行情况(各节点耗时、处理数量、进度百分比)
- 底部按钮:刷新、关闭

**后端实现**:
- 提供性能监控接口:`GET /progressListener/process/getPerformanceMetrics`
- 请求参数:taskId(必填)、metricsType(可选,默认ALL)
- 响应数据:包含JVM、线程、数据库连接池、节点执行情况等指标

```typescript
// 根据 func.concurrencyConfig.mode 渲染对应标签
const getConcurrencyLabel = () => {
  if (!func.concurrencyConfig) return null;
  const { mode, maxConcurrency } = func.concurrencyConfig;
  switch (mode) {
    case 'PARALLEL': return { text: '可并行', color: 'bg-emerald-50 text-emerald-600', icon: 'fa-arrows-split-up-and-left' };
    case 'EXCLUSIVE': return { text: '不可并行', color: 'bg-rose-50 text-rose-600', icon: 'fa-lock' };
    case 'QUEUEABLE': return { text: '可排队', color: 'bg-amber-50 text-amber-600', icon: 'fa-clock' };
    case 'LIMITED_PARALLEL': return { text: `限${maxConcurrency}并发`, color: 'bg-indigo-50 text-indigo-600', icon: 'fa-gauge-high' };
    default: return null;
  }
};
```

## 八、2026-03-11 模拟数据与卡片设计优化

**优化时间**: 2026-03-11

**优化目的**: 让模拟数据更贴合实际并发场景，重设计功能卡片为正方形布局

### 8.1 模拟数据优化

#### 8.1.1 优化策略

根据不同并发模式生成相应的模拟数据，让数据能充分展示各种并发场景的特点：

| 并发模式 | 数据场景设计 | 记录数 | 状态分布 |
|----------|-------------|--------|----------|
| **PARALLEL**（可并行） | 展示多个任务同时运行 | 9 条 | 成功、运行中×3、成功×3、失败、成功 |
| **EXCLUSIVE**（不可并行） | 展示运行中时其他请求被阻止 | 7 条 | 成功×2、运行中、被阻止×2、成功、失败 |
| **QUEUEABLE**（可排队） | 展示排队机制 | 8 条 | 成功×2、运行中、排队×2、成功、失败、成功 |
| **LIMITED_PARALLEL**（限N并发） | 展示达到并发上限 | N+5 条 | 成功×2、运行中×N、排队×2、成功、失败、成功 |

#### 8.1.2 新增状态类型

- **blocked**（被阻止）：用于 EXCLUSIVE 模式，表示因功能不可并行而被阻止的请求
- 在实际存储时记为 `failed`，但在日志和摘要中特殊处理，显示阻止原因

#### 8.1.3 数据字段增强

```typescript
// 新增字段
record.blockReason = `功能「${f.name}」不可并行，当前有任务正在运行`;
record.summary = `因功能「${f.name}」不可并行，当前有任务运行中，本次请求已被阻止。`;
record.algorithmLogs = [
  `[时间] 检测到功能「${f.name}」不可并行`,
  `[时间] 当前有任务 XXX 正在运行`,
  `[时间] 本次请求已被阻止，请等待运行中任务完成后重试`,
];
```

#### 8.1.4 优化效果

- **可排队模式**：用户可看到排队位置和预计等待时间
- **不可并行模式**：用户可看到被阻止的原因和当前运行的任务ID
- **限N并发模式**：用户可看到当前并发数和排队情况

## 九、2026-03-11 严重问题修复

**修复时间**: 2026-03-11

**问题严重性**: 高（逻辑漏洞 + 用户体验问题）

### 9.1 问题描述

#### 9.1.1 问题1：正方形卡片中间空白过大
- 卡片中间有大片空白区域，布局不合理
- 信息密度不足，视觉效果不佳
- 用户反馈卡片看起来"空荡荡的"

#### 9.1.2 问题2：模拟数据逻辑漏洞（严重）

**核心问题**：模拟数据违反了并发模式的基本规则

1. **EXCLUSIVE（不可并行）模式逻辑错误**：
   - 错误：生成了多个 running 状态的任务
   - 违反规则：同一时间只能有1个运行中的任务
   - 用户困惑：为什么"不可并行"却有多个任务在运行？

2. **排队任务不可见**：
   - 用户反馈：没看到排队的任务
   - 原因：排队数据生成逻辑有问题，或排队位置计算错误

3. **所有功能数据未检查**：
   - 只关注了个别功能，未系统性检查所有功能
   - 可能存在更多隐藏的逻辑问题

### 9.2 修复方案

#### 9.2.1 卡片布局修复

**新布局结构**：卡片分为5个区域，从上到下依次为：顶部（图标+模块标签）、中上部（功能名称+描述）、中部（并发模式标签）、中下部（运行状态+时间）、底部（统计信息，使用mt-auto推到最下方）

**关键改进**：
- 恢复功能描述（2行截断），填充中上部空白
- 并发模式标签独立一行，带边框更醒目
- 运行状态和时间分两行显示，信息更清晰
- 统计信息始终显示（即使为0），避免空白
- 使用 mt-auto 将底部统计信息推到最下方

#### 9.2.2 模拟数据彻底重做

**重做原则**：严格按照并发模式定义生成数据

| 并发模式 | 运行中任务数 | 排队任务数 | 历史记录 | 总记录数 |
|----------|-------------|-----------|---------|---------|
| PARALLEL | 3个 | 0个 | 6个（成功+失败） | 9条 |
| EXCLUSIVE | 1个 | 0个 | 6个（成功+失败） | 7条 |
| QUEUEABLE | 1个 | 3个 | 4个（成功+失败） | 8条 |
| LIMITED_PARALLEL | N个 | 2个 | 4个（成功+失败） | N+6条 |

**数据生成逻辑**：
1. 定义记录配置（状态 + 时间偏移）
2. 根据并发模式生成配置
3. 遍历配置生成记录
4. 添加排队详情（如果是 queued 状态）

**排队任务处理**：排队任务的 queueStatus 包含排队位置（从1开始）和预计等待时间

**时间计算逻辑**：
- 使用当前时间作为基准
- dayOffset: 0=今天，1=昨天，2=前天
- hourOffset: 负数表示过去的小时数
- 排队任务使用当前时间（hourOffset: 0）
- 历史记录使用过去时间（hourOffset: -2, -4 等）
