## 二、并发控制机制（评审问题1）

### 2.1 现有实现

**实现方式**：Redis分布式锁（`setIfAbsent`）

**参考代码**（MRP模块）：
```java
// MrpController.java
private static final String MRP_RUN_LOCK_SUFFIX = "_mrpRun";

if (!redisUtil.setIfAbsent(dataSource + MRP_RUN_LOCK_SUFFIX,
                           UUID.randomUUID().toString(),
                           15, TimeUnit.MINUTES)) {
    redisUtil.delete(dataSource + MRP_RUN_LOCK_SUFFIX);
    return BaseResponse.error("上一次mrp正在运行中，请稍后再试！");
}
```

**锁的关键属性**：
- **锁key格式**：`{scenario}_{功能标识}_RUN_LOCK`（如 `scp-test_mrpRun`）
- **超时时间**：15分钟（防止异常情况下锁未释放）
- **并发粒度**：按功能 + 场景（scenario）维度限制
- **锁释放**：任务执行完成（成功/失败/已终止）后在 `finally` 块中释放

**前端交互**：
- 若检测到锁被占用，后端返回明确提示（如"上一次 MRP 正在运行中，请稍后再试！"）
- 前端在运行配置弹窗提交时展示该提示并阻止提交
- 功能卡片上该功能名称右侧展示加载动画，提示用户该功能当前有运行中任务

### 2.2 现有实现的局限性

**不支持任务排队机制**：
- 当前实现为简单的互斥锁，第二个请求会直接被拒绝
- 无法支持"不可并行但可排队"的场景
- 无法支持"可并行N个任务"的场景

**缺少并发模式配置**：
- 未定义统一的并发模式枚举（如：可并行、不可并行、可排队、限制并发数等）
- 每个功能的并发控制逻辑分散在各自的Controller中，缺乏统一框架

## 三、进度推送机制（评审问题2）

### 3.1 现有实现：前端定时轮询

**轮询接口**：`GET /progressListener/process/getProgressByTaskId?taskId={任务ID}`

**返回数据结构**（`ProcessProgressInfoVO`）：
```java
{
    "taskId": "任务ID",
    "processType": "流程类型",
    "currentNodeName": "当前节点名称",
    "status": "任务状态（PROCESSING/SUCCESS/FAIL/INTERRUPT）",
    "overallProgress": 75,  // 整体进度（0-100）
    "allNodes": {"1": "节点1", "2": "节点2", ...},  // 所有节点映射
    "nodeListByTaskId": [节点进度信息列表],
    "taskStartTime": "任务开始时间",
    "taskEndTime": "任务结束时间",
    "remark": {...}  // 备注信息（JSON格式）
}
```

**轮询间隔**：推荐 1～2 秒（MRP前端实现为1秒）

**轮询生命周期**：
- 进度弹窗打开时启动轮询
- 弹窗关闭或任务结束（成功/失败/已终止）时停止轮询
- 组件销毁时（`beforeDestroy`）必须清理定时器，避免内存泄漏

**前端参考代码**（MrpProgressDialog.vue）：
```javascript
// 启动轮询
startPolling() {
    this.pollingTimer = setInterval(() => {
        getProgressByTaskId(this.taskId).then(res => {
            // 更新进度数据
            if (res.data.status === 'SUCCESS' || res.data.status === 'FAIL') {
                this.stopPolling();  // 任务结束，停止轮询
            }
        });
    }, 1000);  // 1秒轮询一次
}

// 停止轮询
stopPolling() {
    if (this.pollingTimer) {
        clearInterval(this.pollingTimer);
        this.pollingTimer = null;
    }
}

// 组件销毁时清理
beforeDestroy() {
    this.stopPolling();
}
```

### 3.2 数据存储机制

**双存储策略**：Redis（缓存） + MySQL（持久化）

**Redis存储**：
- **整体进度key**：`ope:task:overall:{taskId}`
- **节点列表key**：`ope:task:nodes:{taskId}`
- **过期时间**：7天（7 * 24 * 60 * 60秒）
- **数据格式**：JSON字符串

**MySQL持久化**（可选）：
- **控制方式**：通过 `ProgressNodeEnum.isPersistentToDatabase()` 控制
- **持久化表**：
  - `ope_task_progress`：任务整体进度
  - `ope_task_node_log`：节点进度日志
  - `ope_task_error`：任务错误信息
- **持久化时机**：节点进度更新时，若 `isPersistentToDatabase() == true`，则同步写入MySQL

### 3.3 WebSocket推送（未实现）

**现状**：
- 现有代码未实现WebSocket推送
- PRD中提到"若后续需要更实时的推送能力（如秒级更新、多用户协同监控），可考虑采用WebSocket推送"

- Spring Boot支持WebSocket（通过 `spring-boot-starter-websocket`）

## 四、终止接口与异常跳过（评审问题3）

### 4.1 终止接口实现

**终止接口**：`POST /progressListener/process/interruptTask?taskId={任务ID}`

**核心逻辑**（`ProcessProgressManager.interruptTaskById()`）：
```java
public void interruptTaskById(String taskId) {
    // 1. 获取当前任务进度信息
    ProcessProgressInfo progressInfo = getProgress(taskId);

    // 2. 获取当前节点信息
    ProgressNodeEnum currentNode = processNodeLoaderService.getNode(processType, currentNodeCode);

    // 3. 检查节点是否可中断
    if (!currentNode.isInterruptible()) {
        throw new BusinessException("当前节点 '" + currentNode.getNodeName() + "' 不可中断");
    }

    // 4. 执行中断操作（通过Future.cancel(true)）
    boolean interruptTask = interruptTask(taskId);

    // 5. 更新节点状态为INTERRUPT
    if (interruptTask) {
        updateNodeProgress(taskId, currentNode, ProgressStatusEnum.INTERRUPT,
                          null, "人工中断", null, new Date(), null);
    }
}
```

**中断实现方式**：
```java
public boolean interruptTask(String taskId) {
    Future<?> future = taskFutureMap.get(taskId);
    boolean cancelled = true;
    if (future != null && !future.isDone()) {
        cancelled = future.cancel(true);  // true表示允许中断正在执行的任务
    }
    taskFutureMap.remove(taskId);
    return cancelled;
}
```

### 4.2 节点级可中断控制

**可中断节点示例**（MRP）：
- 物料分层排序（`isInterruptible = true`）
- 基础数据获取与校验（`isInterruptible = true`）
- 执行供需平衡计算（`isInterruptible = true`）

**不可中断节点示例**（MRP）：
- 建议计划生成与报告汇总（`isInterruptible = false`）
  - 原因：数据持久化、事务提交类步骤，若正在执行则不应立即终止，以免数据不一致

**前端交互**：
- 进度弹窗在"查看运行中任务"时，右上角展示"终止"按钮（黄色）
- 用户点击后弹出二次确认对话框（如"确认终止当前运行？终止后已产生的中间结果可能丢失。"）
- 用户确认后发送终止请求
- 若后端返回"无法终止"或"终止失败"，前端展示明确提示并保持任务为运行中状态

### 4.3 异常跳过机制（未实现）

**现状**：
- 现有框架未实现异常跳过机制
- PRD中提到"若某节点执行失败，任务状态应置为「失败」"
- PRD中提到"若业务上确有「部分节点失败但允许继续」的需求，应在后端实现时通过配置或策略控制"

**技术可行性**：
- 可在 `ProgressNodeEnum` 中增加 `isSkippableOnError()` 属性
- 在 `ProcessProgressManager` 中捕获节点异常时，根据该属性决定是否继续执行下一节点
- 需在运行日志中明确记录"节点 X 失败但已跳过"类信息

## 五、性能监控接口（评审问题4）

### 5.1 现有实现：节点级业务指标

**节点进度信息**（`NodeProgressInfo`）：
```java
{
    "taskId": "任务ID",
    "nodeCode": "节点编码",
    "nodeName": "节点名称",
    "status": "节点状态",
    "logContent": "日志内容（可包含业务指标）",
    "startTime": "开始时间",
    "endTime": "结束时间",
    "errorFilePath": "错误文件路径"
}
```

**业务指标记录方式**：
- 后端在节点进度信息的 `logContent` 字段中记录关键指标
- 示例：`"当前处理 10,000 / 总 50,000"`、`"数据加载耗时 5 秒"`、`"生成计划订单 1,200 条"`
- 前端在进度弹窗的步骤状态中展示这些信息

**前端展示示例**（MrpProgressDialog.vue）：
```vue
<div class="step-status" v-if="step.status === 'processing'">
    {{ step.statusText }}
    <span v-if="step.showProgress" class="step-detail">
        (当前处理 {{ processedCount.toLocaleString() }} / 总 {{ totalCount.toLocaleString() }})
    </span>
</div>
```

### 5.2 系统级性能监控（未实现）

**现状**：
- 现有框架不包含系统级性能监控（CPU、内存、线程、数据库连接池等）
- PRD中明确提到"本 PRD 主要聚焦于业务层面信息，暂不包含系统级性能监控"

**PRD建议**：
- 若需要更细粒度的性能监控与告警（如实时CPU/内存占用、慢查询分析、数据卡点定位等）
- 应作为**独立的性能监控模块**在后续迭代中规划
- 而非在本运行监控中心内直接展示技术指标

**技术可行性**：
- 可通过JMX（Java Management Extensions）获取JVM指标（堆内存、GC、线程数等）
- 可通过Spring Boot Actuator暴露应用指标
- 可集成Prometheus + Grafana实现监控与可视化
- 但需要额外的架构设计与开发工作量

## 七、调研结论与建议

### 7.1 现有能力总结

**已实现**：
1. 基于Redis分布式锁的并发控制（简单互斥）
2. 前端定时轮询的进度推送机制（1-2秒间隔）
3. 节点级可中断控制（通过 `isInterruptible` 属性）
4. 节点级业务指标记录（通过 `logContent` 字段）
5. Redis + MySQL双存储机制
6. 任务Future管理与中断实现

**未实现**：
1. 任务排队机制
2. 多种并发模式配置（可并行、不可并行、可排队、限制并发数等）
3. WebSocket实时推送
4. 异常跳过框架
5. 系统级性能监控（CPU、内存、线程等）

### 7.2 评审问题的技术规范建议

**问题1（并发处理模式）**：
- 需定义统一的并发模式枚举（如：`PARALLEL`、`EXCLUSIVE`、`QUEUEABLE`、`LIMITED_PARALLEL`）
- 需实现任务排队机制（基于Redis队列或消息队列）
- 需在功能配置中声明并发模式与并发数限制

**问题2（WebSocket推送）**：
- 需实现WebSocket端点（如 `/ws/progress/{taskId}`）
- 需实现消息推送逻辑（任务进度更新时主动推送）
- 需实现前端WebSocket客户端（替代轮询）
- 需考虑消息可靠性（断线重连、消息补发等）

**问题3（终止接口与异常跳过）**：
- 终止接口已实现，需补充规范文档
- 异常跳过需在 `ProgressNodeEnum` 中增加 `isSkippableOnError()` 属性
- 需在 `ProcessProgressManager` 中实现异常跳过逻辑
- 需在运行日志中明确记录跳过信息

**问题4（性能监控接口）**：
- 节点级业务指标已支持，需补充规范文档
- 系统级性能监控建议作为独立模块规划
- 可考虑在进度弹窗中增加"性能详情"按钮，按需触发性能数据采集
