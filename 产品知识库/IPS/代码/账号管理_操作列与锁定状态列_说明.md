# 账号管理：「操作」列与「锁定状态」列说明

- **调研日期**：2026-03-31  
- **适用页面**：系统设置 → 数据权限管理 → **账号管理**（含左侧部门树 + 右侧用户列表）。  
- **文档目的**：帮助实施、运维、管理员理解**两列分别管什么、日常怎么操作、和登录/安全策略的关系**；文末附简要实现索引供开发联调。

---

## 一、先说清楚：账号和「能不能登录」相关的几件事

在实际业务里，用户「登不进去」常见有三类原因，在系统里对应**不同能力**，不要混为一谈：

| 概念（口语） | 在账号管理里主要由谁管 | 典型意图 |
|-------------|------------------------|---------|
| **账号被停掉、不用了** | **操作列里的开关**（启用 / 禁用） | 离职、外包结束、长期不用、合规要求先停权等 |
| **账号暂时或永久不能登录（安全/风控）** | **锁定状态**列 + 密码策略里的失败锁定等 | 暴力破解防护、管理员紧急冻结、策略自动锁等 |
| **忘记密码或需要统一改初始密码** | **操作列里的锁形图标**（实际是**重置密码**） | 用户找回不了密码、批量开户后通知用户用初始密码首次登录等 |

**重要**：「禁用」和「锁定」在界面上是**两列、两套状态**，可以出现「禁用但显示未锁定」或相反组合；**重置密码不会改变**是否启用、是否锁定。

## 二、「操作」列：给管理员用的操作说明

### 2.1 这一列里有什么

- **一个开关**：控制该用户在本系统中是否为**启用**状态（可理解为「账号是否有效」）。  
- **一个锁形小图标**：点击后确认，用于**把该用户的登录密码重置为系统配置的默认初始密码**（**不是**把账号锁定）。

### 2.2 开关怎么用（启用 / 禁用）

**建议谁操作**：租户管理员或有账号管理权限的实施/运维。

**操作步骤**：

1. 在左侧树选中部门（或 appropriate 范围），右侧列表找到用户。  
2. 看「操作」列开关：  
   - **绿色 / 打开**：当前为**启用**，用户若未被锁定且密码正确，可以正常登录。  
   - **红色 / 关闭**：当前为**禁用**，用户**无法登录**（一般会提示账号失效一类文案，具体以系统提示为准）。  
3. 需要**停用账号**时：将开关从打开拨到关闭，保存后列表会刷新。  
4. 需要**恢复使用**时：将开关从关闭拨到打开。

**业务上什么时候用禁用**：

- 员工离职、账号回收，但希望**保留历史数据与审计关联**时，优先用「禁用」而不是物理删除用户（若系统是否提供删除以权限为准）。  
- 临时收回系统访问权（例如项目结束、账号交接前冻结登录）。  
- 与「锁定」相比：禁用更偏向**行政/生命周期**上的「这张账号不用了」；锁定更偏向**安全状态**（见第三节）。

**注意**：

- 禁用后，即使用户记得密码也**不能登录**；需管理员重新**启用**后才能再走登录流程。  
- 禁用**不会自动**帮用户改密码；若担心泄露，可再在锁图标处做**重置密码**。

### 2.3 锁形图标怎么用（重置密码）

**真实含义**：**重置登录密码**，与右侧「锁定状态」**无关**。图标容易让人以为是「上锁」，属于已知体验问题（见第六节）。

**操作步骤**：

1. 鼠标移到锁图标上，应能看到与「重置密码」相关的提示（具体文案以环境语言包为准）。  
2. 点击图标，在二次确认框中确认。  
3. 成功后，该用户密码会被改为**密码策略中配置的默认初始密码**（各环境可能不同，实施需与《密码策略》或运维文档对齐）。  
4. **务必通过安全渠道**将新密码告知用户，并建议用户首次登录后修改密码（若系统支持）。

**业务上什么时候用**：

- 用户忘记密码，且当前流程走管理员代为重置。  
- 新员工或批量导入账号后，统一使用初始密码首次登录。  
- 安全事件怀疑泄露，需要强制改密（重置后仍需用户尽快改密）。

**注意**：

- 重置密码**不会**把账号设为「锁定」，也**不会**自动「禁用」。  
- 若用户因多次输错密码被**策略锁定**，仅重置密码**不一定**解除锁定状态；是否还需管理员在「锁定状态」列解锁，取决于当前策略与实现（一般**手动解锁**在「锁定状态」列操作更直接）。

## 三、「锁定状态」列：业务含义与操作说明

### 3.1 这一列管什么

表示账号在安全意义上的 **正常 / 锁定**（能否因「锁定」被拒绝登录）。  

锁定可能来自：

- **管理员**在本列将用户**手动切到锁定**；  
- **密码策略**（如连续登录失败次数达到阈值）由系统自动把用户标为锁定（与 `LoginFailCountService` 等逻辑相关）；  

解锁时，除把状态改回正常外，系统还会**清除该用户名的登录失败计数**（便于用户重新尝试登录），具体以环境行为为准。

### 3.2 开关怎么读、怎么用

**读数建议**（结合界面 tooltip）：

- 鼠标悬停在开关上，应看到 **「已锁定」/「未锁定」** 一类说明。  
- 当前实现里，**开关拨到表示「锁定」的一侧时，往往用红色强调**；**未锁定**一侧偏绿色。若与「操作」列「绿=启用」混在一起，**务必以 tooltip 文案为准**，不要仅凭颜色猜。

**操作步骤**：

1. 找到目标用户的「锁定状态」列。  
2. 若用户应恢复登录：将状态切换为**未锁定**（并确认操作成功提示）。  
3. 若需紧急禁止登录且希望走「锁定」语义：切换为**已锁定**。

**业务上什么时候用手动锁定**：

- 发现异常登录、账号疑似被盗，需**立即禁止登录**，稍后再排查。  
- 与「禁用」对比：锁定更强调**安全事件/风控**；禁用更强调**账号生命周期停权**。实际工作中可以**同时**禁用 + 锁定，以双保险（取决于管理制度）。

**注意**：

- 用户若处于**禁用**状态，即使「锁定状态」显示未锁定，**仍然无法登录**（登录前就会按禁用拦截）。  
- 列表上若存在「预计解锁时间」类逻辑，可能与自动解锁策略有关；展示与库表是否即时一致，以环境为准。

## 四、对照总表（给培训与答疑用）

| 能力 | 界面位置 | 用户侧效果 | 典型使用场景 |
|------|----------|------------|--------------|
| 启用 / 禁用 | 操作列 · 开关 | 禁用后无法登录（与密码、锁定无关） | 离职停权、长期不用、回收访问权 |
| 重置密码 | 操作列 · 锁图标 | 密码变为默认初始密码 | 忘密、开户初始密码、怀疑泄露后强制改密 |
| 锁定 / 解锁 | 锁定状态列 · 开关 | 锁定后无法登录（策略或人工） | 风控冻结、失败次数锁、管理员解锁 |

**组合示例**：

- **禁用 + 未锁定**：账号行政上停用，但「安全状态」未标锁定——界面上可能出现「操作红、锁定绿」。  
- **启用 + 锁定**：账号仍「有效」，但因锁定无法登录，需解锁或等待策略自动解锁。

## 五、常见问题（FAQ）

**Q1：操作列的锁图标是不是「锁定账号」？**  
**A**：**不是。** 它是**重置密码**入口；真正的锁定在「锁定状态」列。

**Q2：禁用和锁定有什么区别？**  
**A**：都是「不让登录」，但**管理语义不同**：禁用偏**账号是否还有效**；锁定偏**安全状态**。恢复时分别在两列操作。

**Q3：为什么有人操作列是红的，锁定状态还是绿的？**  
**A**：说明账号被**禁用**了，但**没有**处在「锁定」状态——两列独立，属于正常现象。

**Q4：只重置密码，用户能马上登录吗？**  
**A**：若账号被**禁用**或仍处于**锁定**，仍不能登录；需对应列恢复为可登录状态。

**Q5：解锁锁定状态会做什么？**  
**A**：除恢复为正常状态外，一般会**清掉该用户名的登录失败次数**，避免用户一解锁仍因历史次数再次被封（具体以环境为准）。

## 六、改进方案：列名建议与配套调整

### 6.1 列名怎么改（直接采用）

| 当前列名 | **建议列名（首选）** | 命名理由 |
|----------|----------------------|----------|
| **操作** | **启用与重置密码** | 列内实际是「启用/禁用开关 + 重置密码」两件事；避免泛称「操作」，也与下一列「锁定」划清界限。 |
| **锁定状态** | **登录锁定** | 强调影响的是**能否登录**，与「账号是否启用（行政停权）」区分；比「锁定状态」更具体，减少与「密码锁图标」的联想。 |

**备选（列宽很紧时）**：

- 第一列：**启停与密码**（略简写，实施培训时需口头补全「含重置密码」）。  
- 第二列：**安全锁定**（偏风控语义；若希望弱化「安全」二字，仍推荐 **登录锁定**）。

### 6.2 列内控件与列名的配套（建议一起做）

1. **第一列**：在列头或表头说明中固定出现「启用」「重置密码」两个词；将锁图标改为 **文字链「重置密码」** 或 **钥匙类图标 + 悬停说明**，与第二列「登录锁定」不再共用「锁」的视觉符号。  
2. **第二列**：开关的 tooltip 固定为 **「已锁定，无法登录」/「未锁定」**（或「允许登录」），与第一列「启用/禁用」tooltip 文案避免混用「失效」「锁定」等交叉表述。  
3. **可选（信息架构更清）**：将第一列拆成两列——**「账号启用」**（仅开关）、**「重置密码」**（仅操作入口）；第二列仍用 **「登录锁定」**。列变多时注意表格横向滚动与低代码列配置。

### 6.3 不建议使用的列名

| 不建议 | 原因 |
|--------|------|
| 单列仍叫 **「操作」** | 无法表达「启停 + 密码」双能力，用户会继续猜。 |
| 第一列叫 **「账号状态」** | 易与锁定、启用混为一谈，语义仍模糊。 |
| 第二列叫 **「锁定」** 单字 | 过短，仍易与密码、权限「锁定」混淆；**「登录锁定」**更准确。 |

## 附录 A：实现与字段摘要（供开发联调）

- **操作列**在实现上列 `prop` 为历史字段名 `creator`，**不表示创建人**；开关对应 `enabled`（列表中映射为 `valid`）；锁图标调用重置密码接口。  
- **锁定状态**对应 `user_status`：`NORMAL` / `LOCKED`；切换走 `toggleLock`。  
- 登录前会校验 **启用** 与 **非锁定**（及密码策略等），详见认证模块登录实现。
