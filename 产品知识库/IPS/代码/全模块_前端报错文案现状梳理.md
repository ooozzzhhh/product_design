## 3. 总体观察

### 3.1 前端报错文案的共性问题

1. **空泛提示占比高**
   - 大量"操作失败"、"查询失败"、"删除失败"、"保存失败"、"获取失败"等通用文案，不指明具体原因或下一步操作
   - 示例：
     - `this.$message.error("操作失败！")`（IPS）
     - `this.$message.error('查询失败!')`（IPS）
     - `this.$message.error('保存失败')`（IPS）
     - `this.$message.error('导出失败')`（DFP）

2. **参数校验文案不够业务化**
   - 多为"XXX不能为空"、"请选择XXX"，但未说明该参数的业务含义或在哪里填写
   - 示例：
     - `this.$message.error('菜单名不可为空！')`（IPS）
     - `this.$message.error("请选择部门！")`（IPS）
     - `this.$message.error("请选择版本")`（SOP）
     - `this.$message.error("请选择类型")`（SOP）
     - `this.$message.error("请选择优先级因子")`（SDS）

3. **技术细节暴露**
   - 部分错误提示直接暴露技术实现细节或内部字段名
   - 示例：
     - `this.$message.error('调用悠悠分析失败: ' + error.message)`（IPS）
     - `this.$message.error('复制失败: ' + err)`（AMS）
     - `this.$message.error('导出失败：服务器返回了非文件格式的响应')`（DFP）
     - `this.$message.error('导出失败：无法解析服务器响应')`（DFP）

4. **缺少错误码**
   - 前端错误提示几乎不包含错误码，用户反馈问题时难以定位
   - 与后端报错规范（要求包含错误码）不一致

5. **规则类报错缺少上下文**
   - 排程、配置类模块中存在大量规则校验，但只说"不允许"，不说明原因或如何修正
   - 示例：
     - `this.$message.error("请选择要取消计划任务！")`（AMS）
     - `this.$message.error("回退失败，已达回退次数上限！")`（AMS）
     - `this.$message.error('只能进行同组场景之间比较！')`（SOP）
     - `this.$message.error('结束时间应大于开始时间')`（SOP）

6. **数值范围校验文案冗长**
   - MDS、AMS 等模块中存在大量"只能输入0到100的数字"类提示，可统一为"请输入有效范围（0-100）"
   - 示例：
     - `this.$message.error("制造效率只能输入0到100的数字")`（MDS、AMS）
     - `this.$message.error("最大负载不能小于最小负载")`（MDS）
     - `this.$message.error("预警阈值上限不能小于预警阈值下限")`（MDS）

### 3.2 与后端报错的对比

| 维度 | 后端报错 | 前端报错 |
|------|----------|----------|
| 总数 | 208 个（按后端梳理文档） | 5,141 个（本次统计） |
| 错误码 | 部分有（通过 BusinessErrorEnum） | 几乎没有 |
| 文案规范性 | 部分遵循规范，部分空泛 | 大量空泛提示 |

| 国际化支持 | 部分支持（通过 i18n） | 未明确（需进一步调研） |

### 3.3 重复文案现象

- **"操作失败"**：在 IPS、AMS、SOP、MDS 等多个模块中重复出现
- **"查询失败"**：在 IPS、SOP、DFP 等模块中重复出现
- **"删除失败"**：在 IPS、MDS、SDS 等模块中重复出现

## 5. 改进建议

### 5.1 纳入"报错规范化优化"改造范围的建议

基于本次调研，建议将以下前端报错纳入改造范围：

1. **空泛提示类**（优先级：高）
   - 所有"操作失败"、"查询失败"、"删除失败"、"保存失败"、"获取失败"、"导出失败"等通用文案
   - 改造方向：明确指出失败原因或下一步操作

2. **技术细节暴露类**（优先级：高）
   - 所有直接拼接 `error.message`、`err` 等技术异常信息的提示
   - 改造方向：转换为业务语言，隐藏技术细节

3. **缺少错误码类**（优先级：中）
   - 所有前端错误提示
   - 改造方向：为前端错误提示增加错误码，便于问题定位

4. **参数校验类**（优先级：中）
   - 所有"XXX不能为空"、"请选择XXX"类提示
   - 改造方向：补充业务含义和操作指引

5. **数值范围校验类**（优先级：低）
   - 所有"只能输入0到100的数字"类提示
   - 改造方向：统一为"请输入有效范围（X-Y）"

### 5.2 与后端报错规范的协同

1. **错误码体系统一**
   - 前端错误码与后端错误码使用同一套编码规则
   - 前端独有的错误（如"请选择XXX"）可使用独立的错误码段（如 FE001-FE999）

2. **文案规范统一**
   - 前端报错文案遵循后端报错规范：「对象/参数 + 问题 + 下一步指引」
   - 前端报错文案必须包含错误码

3. **国际化支持**
   - 前端报错文案统一使用 i18n 机制
   - 与后端 `messages_zh_CN.properties`、`messages_en_US.properties` 保持一致

### 5.3 改造优先级建议

| 优先级 | 改造范围 | 预计工作量 | 影响范围 |
|--------|----------|-----------|----------|
| P0 | 空泛提示类（"操作失败"等） | 大（涉及 1000+ 处） | 全模块 |
| P1 | 技术细节暴露类 | 中（涉及 100+ 处） | IPS、DFP、AMS |
| P2 | 缺少错误码类 | 大（涉及 5000+ 处） | 全模块 |
| P3 | 参数校验类 | 中（涉及 500+ 处） | 全模块 |
| P4 | 数值范围校验类 | 小（涉及 50+ 处） | MDS、AMS |
