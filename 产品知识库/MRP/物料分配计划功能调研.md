# 物料分配计划功能调研

> 更新时间：2026-04-01
> 调研范围：后端 `scp-mrp-sdk` / 前端 `scp-sds-front`

---

## 一、功能概述

物料分配计划（materialAllocation）是 AMS/MPS/MRP 模块下「监控调整」区域的核心功能，用于对已有供应（库存、采购单、制造订单等）与需求（客户订单、需求预测、制造订单需求）之间建立「分配关系（Fulfillment）」，实现供需匹配可视化管理。

**与完整 MRP 运行的本质区别**：物料分配计划不生成新供应、不做 BOM 展开，只是对**现有供应**重新分配给现有需求。

## 四、后端接口

### 自动分配

```
POST /mrp/runByCustomizedProcess
Body: MrpCustomizedRunProcessDTO（前端传入 orderIds、orderType 等参数）
```

Controller 固定追加：
```java
mrpCustomizedRunProcessDTO.setMrpProcessEnumCodes(Arrays.asList(
    MrpProcessEnum.CLEAR_OLD_FULFILLMENT.getCode(),
    MrpProcessEnum.MATERIAL_ALLOCATION.getCode()
));
```
注意：**同步执行**，不开异步线程，不用进度管理器。

## 五、后端执行流程

### 5.1 应用层（`MrpRunServiceImpl.materialAllocate()`）

```
1. 获取默认 MRP 策略（非按 strategyId，而是取默认）
2. 若 processEnumCodes 包含 CAL_NEXT_LEVEL_DEMAND 则校验低阶码
   → 物料分配不包含该枚举，此步跳过
3. 获取计划周期：planStartTime ~ planEndTime
   （注意：与完整MRP不同，这里用 planStartTime 而非 historyRetrospectStartTime）
4. 组装 MrpContextDO（getMrpStrategyDataDomainServiceImpl.getNewMrpContextData）
5. 设置 customizedProcessEnumCodes 到 contextDO
6. 调用 materialAllocateDomainService.run(mrpContextDO)
7. 调用 handleResult() 持久化
8. 后置处理 postProcessing()：
   AMS/MPS 模块 → 更新制造订单工序齐套状态（sdsFeign.kitStatusUpdate）
```

### 5.2 Domain 层（`MaterialAllocateDomainServiceImpl.run()`）

继承 `AbstractMrpRunDomainService`，走模板方法骨架，各方法实现如下：

| 方法 | 实现 | 说明 |
|------|------|------|
| `clearOldFulfillment` | 自定义 | 见 5.3 节 |
| `getDemandListForCalNetDemand` | 独立需求 + **旧非独立需求**（非本次新生成的） | 关键：物料分配不展开BOM，不产生新非独立需求，只用库里已有的 |
| `getSupplyListForCalNetDemand` | 委托 `IncrementalMrpRunDomainServiceImpl` | 计划供应 + 非计划供应全量参与 |
| `calNetDemand` | 委托 `getRealMrpServiceImpl()` | MRP模块→Full；AMS/MPS→FullMpsSch |
| `groupDemandList` | 委托 `getRealMrpServiceImpl()` | 按规格+生产组织+BOM版本分组 |
| `newSupplyMatch` | **返回空列表** | 不生成任何新供应，纯分配 |
| `calNextLevelDemand` | **返回空列表** | 不做BOM展开 |
| `handleRedundantSupply` | **空实现** | 不删除冗余供应 |
| `handleRedundantDependentDemand` | **空实现** | 不删除冗余需求 |

`getRealMrpServiceImpl()` 逻辑：
- `SystemHolder.getModuleCode() == "MRP"` → `FullMrpRunDomainServiceImpl`
- 其他模块 → `FullMrpRunDomainServiceMpsSchImpl`

### 5.3 清除旧分配关系（`clearOldFulfillment`）

```
若 mrpContextDO.isMaintainFulfillment() == true
  → 直接返回，不清除任何分配（用于固定分配关系保留场景）

否则：
  1. 取 MRP 策略物料范围内所有 productStockPointIds
  2. 取所有需求 IDs（独立 + 非独立）
  3. 调 supplyClient.getFulfillmentListByParams(productStockPointIds, demandIds, fixed=NO)
     → 查非固定分配关系
  4. 调 doClearOldFulfillment() 清除（更新 supply/demand 的 unfulfilledQuantity，删除 fulfillment 记录）
```

**关键**：清除范围比全量 MRP 小——全量 MRP 按需求清，物料分配是按「策略物料范围 + 需求 IDs」交集清；且**已固定的分配关系不会被清除**（fixed=NO 过滤）。

## 六、自动分配 vs 完整 MRP 运行对比

| 维度 | 完整 MRP 运行 | 物料分配计划（自动分配） |
|------|--------------|------------------------|
| 触发接口 | `/mrp/mrpRun` | `/mrp/runByCustomizedProcess` |
| 执行方式 | 异步 + 进度管理 | **同步** |
| 低阶码校验 | 必须 | 不需要 |
| 计划周期起点 | historyRetrospectStartTime | planStartTime |
| 生成新供应 | 是（制造订单、采购订单） | **否** |
| BOM 展开 | 是 | **否** |
| 冗余供应删除 | 是 | 否 |
| 供应来源 | 全量：仅非计划供应 | 计划 + 非计划供应都参与 |
| 固定分配保留 | 不保留 | 固定分配不被清除 |

## 七、手动分配流程

手动分配**绕过 MRP 引擎**，直接操作 Fulfillment 表：

```
用户选中需求 + 选择供应 → 点击「手动分配」
  ↓
前端调 POST /fulfillment/createFulfillment
  ↓
直接在 SDS 中创建 Fulfillment 记录（供应ID、需求ID、分配数量）
  ↓
更新 Supply.unfulfilledQuantity
更新 Demand.unfulfilledQuantity / fulfillmentStatus
```

---

## 八、固定分配关系

`Fulfillment.fixed = YES` 标志决定该分配关系在自动分配时是否保留：

- **固定**：自动分配时 `clearOldFulfillment` 通过 `fixed=NO` 过滤，不清除此分配关系，`newSupplyMatch` 也不会覆盖它
- **取消固定**：恢复为普通分配，下次自动分配会重新计算

> 注意：`maintainFulfillment` 标志（在 `MrpCustomizedRunProcessDTO` 中）是更粗粒度的控制——为 true 时整个清除步骤都跳过，不只是保留固定分配；`fixed=NO` 过滤则是细粒度保留。

## 九、物料分配 vs 库存分配

| 维度 | 物料分配（materialAllocate） | 库存分配（stockAllocate） |
|------|------------------------------|--------------------------|
| 接口 | `POST /mrp/runByCustomizedProcess` | `POST /mrp/stockAllocate` |
| 触发场景 | 物料分配计划页面-自动分配按钮 | — |
| 供应范围 | 策略物料范围内所有供应 | 仅库存（currentStock） |
| Domain 实现 | MaterialAllocateDomainServiceImpl | MaterialAllocateDomainServiceImpl（另一路径）|
| 固定分配控制 | fixed 标志 | `ifMainFulfillment` 参数 |
