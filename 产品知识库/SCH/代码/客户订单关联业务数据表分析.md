# 客户订单删除影响面：关联数据分级（归档调研）

## 文档定位（2026-03-25 修订）

| 项 | 说明 |
|----|------|
| **与旧版差异** | 旧版按「与客户订单多跳可追溯」**尽量全收**，易混入配置表、参数表、统计表及库中遗留对象。本版改为：**以「删除客户订单」为触发点**，只讨论「是否应随删单清理关联行」，并单独标注**疑似废表**。 |
| **分级含义** | **P0**：从数据一致性出发，删除该客户订单（及对应需求）后，**通常应**清理的链路数据（或必须通过 Pegging/订单域服务按单清理）。**P1**：与订单可能有关，但是否删行取决于**删除粒度、计划版本、是否多单一并维护**等，需业务/实现确认。**P2**：**不应**因删除单条客户订单而删除（全局配置、主数据、排程 UI 配置等）；删单后最多触发**重算**，不删配置行。 |
| **废表判定** | 在 **`scp-foundation`** 仓库内对**表名/视图名字符串**做全文检索（含 Java / Vue / XML / md，**2026-03-25**）。**未检出**者标为**疑似废表**（仍可能存在于其它仓库、存储过程、历史作业或对象元数据命名不一致）；**检出**仅代表当前代码路径有引用，不代表一定随单删除。 |
| **不在本文** | `mds_*` 主数据域；明显测试对象（如 `sds_ord_test_order_vacancy`）。 |

## 一、P0 — 删除客户订单后通常应清理（订单 / 需求 / Pegging / 下游订单及子表）

**前提**：客户订单在需求侧一般对应 **`dps_ord_customer_order`**；与供需的关系仍按 **`demand_order_id`、`demand_type`** 等与 **`sds_peg_demand`**（及后续 supply/fulfillment）衔接（字段以 DDL 为准）。

| 域 | 典型对象（基表名；多有 `v_*` 视图） | 说明 |
|----|-------------------------------------|------|
| 客户订单 | `dps_ord_customer_order` | 删单本体。 |
| Pegging / 净需求 | `sds_peg_demand`、`sds_peg_supply`、`sds_peg_fulfillment`、`sds_peg_net_demand` | 需求行指向客户订单时，清理 Pegging 链，避免孤儿分配与净需求。代码侧对象类型常为 `v_prs_peg_*` / `v_sds_peg_net_demand`。 |
| 制造订单及子表 | `sds_ord_work_order`、`sds_ord_work_order_bom`、`sds_ord_work_order_routing_step`、`sds_ord_plan_unit`、`sds_ord_operation` | 由该客户需求 Peg 出的制造侧，删单或删需求后应按域服务级联删除子对象（工序、计划单元、BOM 等）。 |
| **工序投入 / 产出 / 资源** | **基表**：`sds_ord_operation_input`、`sds_ord_operation_output`、`sds_ord_operation_resource`。**逻辑视图（元数据 objectType）**：`v_prs_ord_operation_input`、`v_prs_ord_operation_output`、`v_prs_ord_operation_resource`。 | 与 **`operation_id`**（及工艺 `routing_step_id`、库存点物品等）挂钩；**投入**侧可与需求（`operation_input_id`）关联，**产出**侧可与供应（`operation_output_id`）关联（以 DDL 为准）。删除工序时应随 **`OperationInputService` / `OperationOutputService` / `OperationResourceService`** 及 Pegging 域规则一并清理；MRP/算法回写会通过 `SdsAlgorithmFeignController` 清理旧 `OperationInput`/`OperationOutput` 数据。 |
| 工序扩展与任务 | `sds_ord_operation_extend`、`sds_ord_operation_task`、`sds_ord_operation_sub_task`、`sds_ord_preferred_resource` | 代码中有 `OperationTaskServiceImpl`、`OperationSubTaskServiceImpl`、`PreferredResourceServiceImpl` 等；`operation_extend` 在前端 SQL 片段中有引用。 |
| 工序下达 | 物理表以库为准（如 **`ams_ord_operation_issued`** / **`ams_ord_operation_issued_version`**）；逻辑视图 **`v_sds_ord_operation_issued`**、**`v_sds_ord_operation_issued_version`** | 与 `operation_id`、`order_id`（或等价键）挂钩，删工序/订单链时应一并处理。 |
| 采购 / 运输订单 | `sds_ord_purchase_order`、`sds_ord_transport_order` | 作为供应类型进入 Pegging 时，是否删「整单」取决于是否仅服务该客户需求；**至少**应解除与该需求相关的 Pegging/分配。 |
| 通用供应扩展 | `sds_supply_universal` | 供应汇总/其它供应场景中有引用；若行级可定位到该订单链供应，应随链清理。 |
| 履行反馈 | `sds_fee_feedback_operation`、`sds_fee_feedback_plan_unit`、`sds_fee_feedback_production`、`sds_fee_feedback_picking`、`sds_fee_feedback_stocking` | 代码与前端大量以 `v_sds_fee_feedback_*` 引用；挂 `operation_id` / `plan_unit_id` 等，随工序或计划单元删除而应清理。 |

**说明**：P0 强调「链路一致」；具体删除顺序与接口应以 **SDS/AMS 订单域、Pegging 服务** 的实现为准（避免手工删表破坏约束）。

## 二、P1 — 不确定：是否随删单清理需业务或实现确认

| 产销平衡 / 计划结果 | `sds_pla_master_production_plan`、`sds_pla_product_supply_demand`、`sds_pla_product_supply_demand_balance`、`sds_order_main_plan`、`sds_order_main_version` | 多为计划版本或平衡结果；与单客户订单的对应关系需看 DDL 与生成任务，**全文检索几乎未命中**业务代码（平衡类仅见前端 copy 文件引用），属高不确定。 |

## 三、P2 — 不应因删除客户订单而删除（配置 / 参数 / 主数据 / 排程 UI 配置）

删除客户订单**不应**删除下列类型数据（除非独立做「配置清理」类需求）：

| 类别 | 示例（原「多跳清单」中已剔除） |
|------|--------------------------------|
| 客户主数据 | `dps_ord_customer`（订单上 `customer_id` 引用客户，删单不删客户）。 |
| DPS 规则与参数配置 | `dps_pri_priority_factor_config`、`dps_net_netting_rule`、`dps_net_netting_rule_planning`、`dps_off_offset_rule`、`dps_sto_safety_stock_config`、`dps_sto_safety_coefficient`、`dps_sto_stock_point_demand_ratio`（配置类）等。 |
| AMS 排程交互配置 | `ams_rul_task_drag`、`ams_timeline_config`（拖拽规则、时间轴配置，与单无直接生命周期绑定）。 |

## 五、关联字段要点（节选，字段以 DDL 为准）

- **客户订单 — 客户**：`customer_id` → `dps_ord_customer.id`（删单**不**删客户）。  
- **客户订单 — 需求**：`demand_order_id` = 客户订单 `id`；`demand_type` = 客户订单需求类型。  
- **需求 — Pegging**：`demand_id`、`supply_id` 等。  
- **制造订单 — 子表**：`work_order_id`、`operation_id`、`routing_id` 等与 `sds_ord_*` 衔接。  
- **工序 — 投入 / 产出**：`sds_ord_operation_input` / `sds_ord_operation_output` 以 **`operation_id`** 归属工序；需求/供应侧可能通过 **`operation_input_id`、`operation_output_id`** 与 Pegging 衔接（以 DDL 为准）。  
- **工序 — 资源**：`sds_ord_operation_resource` 以 **`operation_id`**（等）归属工序。  
- **工序 — 下达**：`operation_id`、`order_id`（或版本 id）与下达头/版本表衔接。  

更细的「多跳」说明仍可参考历史版本调研笔记；**删单治理请以本文 P0/P1/P2 为准**。

## 附录：在 scp_ams 中自查表清单

```sql
-- 工序投入/产出/资源：基表与视图是否在库中存在（请在可连库环境执行）
SELECT TABLE_NAME, TABLE_TYPE
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'scp_ams'
  AND (
    TABLE_NAME IN (
      'sds_ord_operation_input',
      'sds_ord_operation_output',
      'sds_ord_operation_resource',
      'v_prs_ord_operation_input',
      'v_prs_ord_operation_output',
      'v_prs_ord_operation_resource'
    )
    OR LOWER(TABLE_NAME) LIKE '%operation%input%'
    OR LOWER(TABLE_NAME) LIKE '%operation%output%'
  )
ORDER BY TABLE_TYPE, TABLE_NAME;

SELECT TABLE_NAME, TABLE_TYPE
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'scp_ams'
  AND TABLE_TYPE IN ('BASE TABLE', 'VIEW')
ORDER BY TABLE_NAME;

SELECT TABLE_NAME
FROM information_schema.TABLES
WHERE TABLE_SCHEMA = 'scp_ams'
  AND (
    LOWER(TABLE_NAME) LIKE '%ope%'
    OR LOWER(TABLE_NAME) LIKE '%operation%'
    OR LOWER(TABLE_NAME) LIKE '%schedule%'
  )
ORDER BY TABLE_NAME;
```
