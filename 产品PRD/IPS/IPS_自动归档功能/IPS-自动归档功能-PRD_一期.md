# 1. 需求概述
## 1.1 背景及问题
### 1.1.1 背景
SCP 平台在各业务模块（如主计划、MRP、S&OP 等）运行过程中，会持续产生大量业务历史数据（订单、日志、运行记录等），长期在线保存全部明细会带来以下问题：
- 数据库体量持续膨胀：历史数据长期累积，占用大量存储空间，影响日常运维成本。
- 在线查询性能风险：部分核心表（如交易明细、操作日志等）数据量过大，可能拖慢在线查询、统计与报表性能。
### 1.1.2 问题
#### 1.1.2.1 缺少平台级归档管理能力
- 各业务模块若要做历史数据归档，通常由数据库层面单独处理，未形成统一的归档配置管理与监控入口。
- 影响：无法在平台内统一配置、查看和运维各类归档配置，业务方也难以理解和参与规则制定。
## 1.2 目标（一期）
- 建设数据库归档管理能力：提供一个统一的「归档管理」配置与监控入口，支持对选定**模块**、**场景（具体源库）**下的**归档对象**按规则进行归档。
- **场景**：在用户选择模块后，须再选择**归档场景**；场景与**一个具体的业务数据库（库）**一一对应，用于明确本次归档的源库上下文。可选场景列表由系统按环境与权限返回。
- **目标库命名**：触发归档时创建的独立归档库，命名规则为 **`{场景库库名}_archeive_{yyyymmddhhmmss}`**（`{场景库库名}` 为所选场景对应源库的库名，如 `scp_sch`；时间戳为 14 位 `yyyymmddhhmmss`）。示例：`scp_sch_archeive_20260320143025`。
- 数据库存储：归档结果写入上述新建归档库，便于追溯与隔离。
- 仅支持手动触发：归档由用户点击「触发归档」手动执行；触发前须经过**双层确认**（见 2.1.4），不支持定时或数据量自动触发。
- **配置侧归档范围**：用户**仅选择业务对象**（一期下拉中仅 **「客户订单」**）；该对象所对应的物理表由**系统自动关联**（平台按模块/环境预置对象—表映射），**配置界面不展示表清单、不支持勾选表或表名筛选**；后续迭代扩展多对象时仍沿用「选对象、系统关联表」模式。
- - 本 PRD 以体现业务与交互为主，不绑定具体技术栈。

# 2. **详细解决方案**
## 2.1 **数据库归档管理**
### 2.1.1 模块描述
本模块为数据库层面的**归档配置管理**入口，统一支持：
* 以**配置**的形式定义「对哪个模块、哪个场景（源库）、哪个**业务对象**、按何种时间规则进行归档」；业务对象所涉物理表由系统解析，不在配置表单中逐表指定；
* 管理已配置的归档配置列表，查看配置名称、模块、场景、业务对象及**系统解析得到的表数量**（展示用）、归档规则；
* **仅支持手动触发**：用户点击「触发归档」后，经双层确认，系统创建新归档库（命名 **`{场景库库名}_archeive_{yyyymmddhhmmss}`**）并执行归档；
* **手工触发归档日志**展示每次触发的完整运维视图：涉及**归档对象**、**对象包含表数**、**各表原行数与已归档行数**（信息较密，界面需分层/折叠/表格化以保证可读性，见 2.1.3）；
* 日志增加三类状态及**撤回**：**归档任务处理流程**、**目标数据处理状态**、**源数据处理状态**；当源数据处理状态为**「未删除」**时，允许点击**「撤回」**取消本次归档任务（具体以服务端任务状态机为准，撤回后在线库未删除已拷贝数据、归档侧按产品约定回滚或标记作废）。

主要面向用户：
* **系统管理员 / 运维人员**：负责全局归档策略配置与执行。

### 2.1.2 输入
#### 2.1.2.1 业务与配置层面
* **归档配置元数据**：
  * 配置名称（唯一标识、可读性强），如 `SCH_Archive_Config`；
  * **目标归档库命名规则**：`{场景库库名}_archeive_{yyyymmddhhmmss}`，在保存/展示配置时可预览当前所选场景下的示例库名（含占位符说明）。
* **模块、场景与业务对象**：
  * **选择模块**（如 SCH、DFP、SOP、MPS、MRP、MDS）；
  * **选择场景**：必选；场景对应**一个具体源库**（库名作为命名片段来源）；
  * **选择业务对象**：一期下拉仅 **「客户订单」**；保存后由服务端/规则引擎按对象解析关联物理表，**无需、也不支持**在界面勾选表。
* **归档规则（仅按时间条件）**：
  * 基准字段（时间字段），如 `created_at`、`update_time`；
  * 归档范围：归档 X（日/月/年）之前的数据，基于服务器时间。

#### 2.1.2.2 用户录入（运行前配置）
在新建或编辑归档配置时，用户需要录入：
* 配置任务名称；
* 模块、场景（源库）、业务对象（一期为「客户订单」）；
* 归档规则：基准字段、归档 X（日/月/年）之前的数据。

#### 2.1.2.3 系统与权限（后端对接时）
* 当前环境与连接信息，决定可选的模块、**场景（库）**与对象-表映射；
* 用户角色与权限：新建/编辑配置、触发归档、查看日志、**撤回**等需按权限控制。

### 2.1.3 输出
#### 2.1.3.1 用户可见页面与反馈
* **归档配置列表页**：
  * 按配置展示：配置名称、模块、场景（可展示库名或场景名称）、**业务对象**及**系统关联表数**（仅展示，非用户勾选结果）、归档规则摘要；
  * 行内操作：触发归档、编辑、日志。
* **归档配置弹窗**：
  * 区域：基础信息（配置任务名称）、**选择模块**、**选择场景**、目标库命名规则预览、**选择业务对象**（一期下拉仅「客户订单」）、归档规则（按时间条件）；
  * 在填写不完整或配置存在明显冲突时，给出校验提示。
* **手工触发归档日志**（通过配置列表行内**「日志」**打开弹窗；**首页不展示**归档任务摘要或最近任务列表，避免与弹窗信息重复）：
  * **每条任务**采用「主行摘要 + 可展开明细」：
    * **主行**：触发 ID、目标归档库名、**涉及归档对象**（如客户订单）、**对象内表数**、**归档任务处理流程**（成功 / 进行中 / 失败）、**目标数据处理状态**（建档成功 / 建档失败）、**源数据处理状态**（未删除 / 已删除 / 失败）、触发时间、**操作**（源数据为**未删除**时显示**「撤回」**，否则显示占位）；**操作列与同行其它列底边线对齐**，避免单列高度不一致导致分隔线错位。
    * **展开区（表级明细）**：展示各物理表的**原行数**与**被归档行数**；交互上采用**分区卡片 + 分表行内统计**（表名醒目、数值右对齐或标签化），避免简陋的大边框嵌套表；有备注时在展开区底部展示。
  * 支持弹窗内滚动、分页（若一期实现）、导出（若一期实现导出则与列表字段一致）。

#### 2.1.3.2 数据与数据库输出
* **归档数据库**：
  * 触发归档时创建新库，命名 **`{场景库库名}_archeive_{yyyymmddhhmmss}`**；
  * 将符合时间条件的数据迁移至新库。
* **手工触发记录**：
  * 每次触发的记录须可还原：**对象列表、表数、各表原行数与归档行数**、上述三类状态、撤回操作痕迹（如有）。

#### 2.1.3.3 下游使用
* 运维可基于归档数据库做长期存档、冷备份或离线分析；
* 管理层可通过手工触发记录了解各类历史数据的归档频率与规模。

### 2.1.4 业务流程
#### 2.1.4.1 新建归档配置
1. 用户在归档管理页面点击「新建归档配置」。
2. 系统弹出配置弹窗，用户依次完成：
   * **基础信息**：填写配置任务名称；
   * **选择模块**：选择模块（如 SCH、DFP、SOP、MPS、MRP、MDS）；
   * **选择场景**：在下拉中选择本次归档对应的**具体源库/场景**；
   * **目标数据库命名规则**：展示/预览 `{场景库库名}_archeive_{yyyymmddhhmmss}`；
   * **选择业务对象**：一期仅「客户订单」；物理表由系统按对象**自动关联**，界面不勾选表；
   * **归档规则**：配置基准字段、归档 X（日/月/年）之前的数据。
3. 用户点击「保存配置」，通过校验后写入系统并在列表中展示。

#### 2.1.4.2 触发归档（双层确认）
1. 用户在配置列表中点击「触发归档」。
2. **第一层弹窗（说明与风险提示）**：
   * 展示：**归档概念**（将符合规则的在线数据复制到独立归档库并按策略处理源数据）、**归档流程**（建档 → 拷贝/迁移 → 源数据处理等步骤概要）、**归档风险**（如误配时间范围、磁盘空间、业务只读窗口等）。
   * 用户点击「确定」后进入第二层；点击「取消」关闭，不创建任务。
3. **第二层弹窗（执行前汇总与加载）**：
   * 展示即将执行的归档任务：**归档对象**、**表数**、**各表原行数与预计/已统计的待归档行数**（数据由接口实时查询，可能较慢）。
   * 查询过程中展示**加载中动效**（如转圈）；失败时提示重试或关闭。
   * 用户核对后点击「确定」**正式创建并触发**归档任务；点击「取消」关闭。
4. 任务进入执行后，用户在对应配置的**「日志」弹窗**中查看最新记录；可根据三类状态与「撤回」规则进行操作（不在首页单独展示任务列表）。

#### 2.1.4.3 撤回
1. 用户在日志中找到源数据处理状态为**「未删除」**的任务，点击「撤回」。
2. 系统取消或中止该归档任务后续对源数据的删除等步骤（具体状态迁移由后端定义），界面刷新状态。

### 2.1.5 业务规则
#### 2.1.5.1 配置与触发规则
* **若**创建新的归档配置，**则**必须填写唯一的配置任务名称；名称在当前系统中不得重复。
* **若**保存配置，**则**必须选择模块、**场景**、**业务对象**（一期默认必选「客户订单」）、配置归档规则（基准字段 + X 日/月/年之前）。
* 归档**仅支持手动触发**，且必须经过**双层确认**后方可真正触发。

#### 2.1.5.2 归档对象与范围规则（一期）
* **若**未选择模块或场景，**则**不允许保存配置。
* **业务对象**一期仅开放「客户订单」；后续迭代可在下拉中扩展多对象。
* **对象—表映射**由平台预置：用户配置中**不包含**物理表多选；执行归档与日志中的「表数/表级明细」均以系统解析结果为准。
* **按时间归档**：
  * 必须配置基准字段（一般为时间字段），以及「X（日/月/年）之前」的数值；
  * 系统以当前服务器时间为基准，计算实际时间边界。

#### 2.1.5.3 清理策略
* 一期归档在流程上会处理源库数据（与 v3 一致方向：归档成功后按策略删除源端已归档数据等）；**在未完成源数据处理前**，状态为「未删除」，允许**撤回**。

#### 2.1.5.4 状态字段定义
| 字段 | 可选值 | 说明 |
| --- | --- | --- |
| 归档任务处理流程 | 成功 / 进行中 / 失败 | 端到端任务是否完成、是否异常中断 |
| 目标数据处理状态 | 建档成功 / 建档失败 | 归档目标库是否创建成功、结构/写入是否就绪 |
| 源数据处理状态 | 未删除 / 已删除 / 失败 | 源库侧删除或等价处理是否完成；**未删除**时可撤回 |

#### 2.1.5.5 校验与安全
* 归档执行过程中出现异常时，应在日志中记录明确的失败原因，便于运维排查（例如：数据库连接失败、磁盘空间不足、创建新库失败等）；**禁止**向前端暴露内部技术堆栈或 SQL 原文（遵循平台异常规范）。

#### 2.1.5.6 权限与操作限制
* 仅具有相应权限的用户才能新建、编辑归档配置、触发归档、撤回；普通业务用户默认仅可查看日志，不可修改配置或触发归档。
* 对于生产环境，建议在权限与审批流程上增加额外控制（如需通过变更单后方可触发归档）。

# 3. **未来扩展点**
## 3.1 扩展业务对象
在「选择业务对象」中增加更多选项；每个对象对应的物理表集合仍由**平台预置、系统自动关联**，配置侧不回到「逐表勾选」模式（除非未来产品另定）。
## 3.2 增加规则条件
可增加按数据量、按 SQL 条件等归档规则。
## 3.3 增加存储方式
可增加 SQL 文件等存储方式。
