# 1. **需求概述**

## 1.1 背景及问题

### 1.1.1 背景

悠桦林 SCP 平台在各业务模块（如主计划、MRP、S&OP 等）运行过程中，会持续产生大量业务历史数据（订单、日志、运行记录等），长期在线保存全部明细会带来以下情况：

- **数据库体量持续膨胀**：历史数据长期累积，占用大量存储空间，影响日常运维成本。
- **在线查询性能风险**：部分核心表（如交易明细、操作日志等）数据量过大，可能拖慢在线查询、统计与报表性能。
- **缺乏统一的数据归档入口与策略**：目前更多依赖 DBA 或脚本手工处理，缺少平台级、可配置、可视化的自动归档能力。

### 1.1.2 问题

#### 1.1.2.1 缺少平台级自动归档能力

- 各业务模块若要做历史数据归档，通常由数据库层面单独处理，未形成统一的**归档任务管理与监控入口**。
- **影响**：无法在平台内统一配置、查看和运维各类归档任务，业务方也难以理解和参与规则制定。

#### 1.1.2.2 归档规则分散且不可视

- 现有归档策略若由脚本实现，往往难以用**业务视角**描述“对哪些表、按照怎样的时间范围或数据范围进行归档”。
- **影响**：规则难以复盘与评审，换人接手或业务变更时，调整成本高且风险大。

#### 1.1.2.3 归档与清理关系不清晰

- 归档之后是否立即从在线库中删除、删除到什么程度、由谁控制，缺少统一规范，可能出现：
  - 只做导出不做清理，数据库空间问题仍然存在；
  - 直接删除历史数据，缺乏与归档结果的严格对应关系。
- **影响**：存在数据丢失风险或运维成本无法真正降低。

## 1.2 目标

- **建设数据库自动归档管理能力**：提供一个统一的「自动归档」配置与监控入口，支持对选定业务表按规则进行周期性归档。
- **支持多种归档范围规则**：支持按时间、按数据量、按业务条件（SQL 条件）等方式定义归档范围，满足不同表的差异化需求。
- **明确归档后清理策略**：支持“按归档结果自动清理”与“手动配置清理规则”两种方式，降低数据丢失风险。
- **与实现解耦**：本 PRD 以业务与交互为主，不绑定具体技术栈；现有原型在 `产品原型图/自动归档功能` 目录下，仅作为界面与交互参考。

**范围与边界**：

- 当前以**单库内数据表的归档配置与任务管理**为主，默认归档结果写入数据库外部存储（如文件）；具体落地位置与介质在后端技术方案中确定。
- 本次不做：跨库/跨集群迁移类归档、归档数据在线检索界面、多租户空间隔离设计。

# 2. **详细解决方案**

## 2.1 **数据库自动归档管理**

### 2.1.1 模块描述

本模块为数据库层面的**自动归档任务管理**入口，统一支持：

- 以**任务**的形式定义“对哪些表、在什么条件下、以何种方式进行归档”，并周期性执行；
- 管理已配置的归档任务列表，查看其归档对象、触发条件与启停状态；
- 配置归档完成后的清理策略，控制在线库数据的保留范围。

主要面向用户：

- **系统管理员 / 运维人员**：负责全局归档策略设计与运维；
- **模块负责人 / 业务配置人员**：参与确定本模块关键表的归档范围与周期。

### 2.1.2 输入

#### 2.1.2.1 业务与配置层面

- **归档任务元数据**：
  - 任务名称（唯一标识、可读性强），如 `Daily_Trade_Archive`；
  - 归档目标存储格式（如 SQL 文件、NB3 格式等）；
  - 归档文件存储路径（如 `/data/backup/`，需具备写权限）。

- **归档触发规则**：
  - **定时触发**：按固定时间周期执行，如每周、每月、每季度、每年，在指定星期几/时间点或“月初/月中/月末”等触发；
  - **数据量触发**：当目标表总行数超过某个阈值（如 10,000 行）时触发归档。

- **归档对象与范围配置**：
  - 供搜索与选择的业务数据表列表（表名、说明由系统提供）；
  - 每个被选中的表需配置：
    - 归档方式：按时间、按数据量、按业务条件（SQL 条件）；
    - 基准字段（适用于按时间、按数据量），如 `created_at`；
    - 具体范围规则：
      - 按时间：如“归档 30 天之前的数据”“归档 12 个月之前的数据”等；
      - 按数据量：如“按基准字段降序，归档第 N 行之后的数据”；
      - 按 SQL 条件：通过字段+比较符+值组合出业务条件。

- **清理策略配置**：
  - 是否在归档成功后对在线数据进行清理；
  - 清理方式：自动清理（归档多少清理多少）或手动配置规则（按指定表+字段+条件清理）。

#### 2.1.2.2 用户录入（运行前配置）

在新建或编辑归档任务时，用户需要录入：

- 任务名称、存储格式、存储路径；
- 触发模式及其具体参数（如周期类型、时间点、阈值等）；
- 至少一个归档对象表及其归档方式与范围；
- 是否启用“归档成功后清理”以及对应的清理规则。

#### 2.1.2.3 系统与权限（后端对接时）

- 当前连接的数据库实例信息（环境、库名等），决定可选的表与实际执行范围；
- 文件系统或对象存储的访问权限，用于验证存储路径可写；
- 用户角色与权限：新建/编辑/启停任务、查看日志等需按权限控制。

### 2.1.3 输出

#### 2.1.3.1 用户可见页面与反馈

- **归档任务列表页**：
  - 按任务展示：任务名称、归档对象概览（表名汇总）、触发条件摘要（如“每周一 02:00”）、当前启停状态；
  - 行内操作：运行一次（立即触发）、编辑、查看日志。

- **归档任务配置弹窗**：
  - 多步区域：基础信息（任务名称、存储格式与路径）、触发规则、归档对象与范围、归档后操作；
  - 在填写不完整或配置存在明显冲突时，给出校验提示。

- **归档任务执行日志弹窗**：
  - 展示每次执行的归档记录：归档文件名、执行时间、状态（成功/失败）、文件路径、失败原因等；
  - 支持滚动查看历史记录，便于运维排查。

#### 2.1.3.2 数据与文件输出

- **归档文件**：
  - 按任务配置生成归档文件（如 SQL 导出文件、NB3 等格式）；
  - 文件命名规则可配置，如 `{task_name}_{timestamp}.{ext}` 或 `{db_name}_{timestamp}.{ext}`；
  - 文件保存至配置的存储路径，路径需可追溯到归档任务与时间。

- **任务执行记录**：
  - 每次归档执行的记录，包含：任务名称、执行时间、状态、归档文件名、文件路径、归档数据范围摘要（如时间区间、行数）等。

#### 2.1.3.3 下游使用

- 运维可基于归档文件做长期存档、冷备份或离线分析；
- 管理层可通过任务执行记录了解各类历史数据的归档频率与规模。

### 2.1.4 业务流程

#### 2.1.4.1 新建归档任务

1. 用户在自动归档管理页面点击「新建归档任务」。
2. 系统弹出配置弹窗，默认进入**基础信息**区域：
   - 用户填写任务名称、选择存储格式、配置存储路径并可点击「校验权限」确认路径可写。
3. 用户切换到**触发规则**区域：
   - 选择「定时触发」或「数据量触发」；
   - 定时触发：选择时间颗粒度（每周/每月/每季度/每年），并配置具体触发时间点；
   - 数据量触发：设置总行数阈值。
4. 用户切换到**归档对象与范围**区域：
   - 在搜索框中按关键字搜索数据表，并从结果中选择需要归档的表；
   - 对每个已选表，选择归档方式（按时间/按数据量/按 SQL 条件），配置基准字段与范围规则；
   - 支持批量设置基准字段，应用到所有已选表。
5. 用户切换到**归档后操作**区域：
   - 选择是否勾选“归档成功后清除已归档数据”；
   - 若勾选，则进一步选择“自动清理”或“手动配置规则”，并在手动模式下配置清理规则。
6. 用户点击「保存配置」，通过校验后，任务写入系统配置并在列表中展示。

#### 2.1.4.2 手动运行与状态查看

1. 用户在任务列表中点击「运行」，系统立即按当前配置执行一次归档任务。
2. 执行过程中，系统在后台完成数据抽取、归档文件写入以及（如配置）归档后清理操作。
3. 执行完成后，执行结果出现在「归档任务执行日志」中，状态为成功或失败。

#### 2.1.4.3 定时/数据量触发执行

1. 到达定时触发条件（如每周一 02:00）或数据量触发阈值时，系统自动创建对应的归档执行任务。
2. 执行流程与手动运行一致，执行结果记录在日志中。
3. 若某次执行失败，系统在日志中记录失败原因，供运维排查。

#### 2.1.4.4 归档后清理

1. 若任务未开启“归档成功后清除已归档数据”，则执行完归档后不对在线数据做删除，仅生成归档文件。
2. 若启用“自动清理”，则：
   - 系统根据本次归档所涉及的表与数据范围，删除与归档结果严格对应的在线数据；
   - 保证“归档多少删除多少”的对应关系。
3. 若启用“手动配置规则”，则：
   - 系统按用户在清理规则中配置的表、时间字段与条件执行删除；
   - 规则可与归档范围不同，由运维根据业务风险控制更精细的清理策略。

### 2.1.5 业务规则

#### 2.1.5.1 任务与触发规则

- **若**创建新的归档任务，**则**必须填写唯一的任务名称；任务名称在当前系统中不得重复。
- **若**选择「定时触发」，**则**必须指定时间颗粒度与具体时间点：
  - 每周：需选择星期几 + 时间；
  - 每月：可选择「月初」「月中」「月末」等抽象时间点 + 时间；
  - 每季度/每年：需指定季度/月份及日期 + 时间。
- **若**选择「数据量触发」，**则**必须配置总行数阈值，且阈值为正整数。

#### 2.1.5.2 归档对象与范围规则

- **若**某任务未配置任何归档对象表，**则**不允许保存任务。
- **按时间归档**：
  - 必须配置基准字段（一般为时间字段），以及“X（日/月/年）之前”的数值；
  - 系统以当前服务器时间为基准，计算实际时间边界。
- **按数据量归档**：
  - 必须配置基准字段，并给出“第 N 行之后”的数据量范围；
  - 行序以指定基准字段降序排序后的顺序为准。
- **按 SQL 条件归档**：
  - 通过字段+比较运算+值组合形成条件行，支持多个条件；
  - 条件之间的组合关系（与/或）在实现时需统一约定，本 PRD 推荐以“同一任务下多个条件默认按 AND 关系组合”。

#### 2.1.5.3 清理策略

- **若**未勾选“归档成功后清除已归档数据”，**则**系统仅生成归档文件，不对在线数据执行删除。
- **若**选择“自动清理”：
  - **当且仅当**本次归档执行成功，且生成归档文件成功时，系统才执行对应数据删除；
  - 删除范围应与本次归档的数据范围严格一致，避免删除未归档数据。
- **若**选择“手动配置规则”：
  - 每条清理规则至少包含：目标表、时间字段、比较运算、比较值；
  - 规则生效前，应在界面上以醒目提示标明“危险操作”，并要求用户明确确认。

#### 2.1.5.4 校验与安全

- 存储路径在保存任务时，推荐提供「校验权限」动作，检查路径是否可写；校验失败时禁止保存或给出明显告警。
- 归档任务执行过程中出现异常时，应在执行日志中记录明确的失败原因，便于运维排查（例如：路径不可写、数据库连接失败、磁盘空间不足等）。
- 删除操作需遵循数据库安全规范，避免一次性删除过大数据量导致锁表或严重性能问题；实际执行策略由后端在技术方案中约定。

#### 2.1.5.5 权限与操作限制

- 仅具有相应权限的用户才能新建、编辑、启停归档任务；普通业务用户默认仅可查看日志，不可修改配置。
- 对于生产环境数据库的归档任务，建议在权限与审批流程上增加额外控制（如需通过变更单后方可启用）。

### 2.1.6 原型图

- **原型位置**：`产品原型图/自动归档功能/自动备份功能原型.html`（文件名暂保留历史命名，界面文案已统一使用“归档”术语）。
- **主页面（自动归档任务列表）**：
  - 顶部：标题「数据库自动归档管理」，支持按场景（如生产环境主库、测试环境等）切换；
  - 右侧工具区：任务搜索框、「导入」「导出」「新建归档任务」等按钮；
  - 列表：展示任务名称、归档对象（表）、触发条件、状态、操作（运行/编辑/日志）。
- **新建/编辑任务弹窗**：
  - 分为基础信息、触发规则、归档对象与范围、归档后操作四大块；
  - 归档对象与范围中，通过搜索表名添加多张表，对每张表分别配置归档方式与范围。
- **日志弹窗**：
  - 展示每次执行的归档文件名、执行时间、状态、文件路径及失败原因，状态以不同颜色标签区分成功/失败。
