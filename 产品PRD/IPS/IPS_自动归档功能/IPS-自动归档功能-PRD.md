# 1. 需求概述
## 1.1 背景及问题
### 1.1.1 背景
SCP 平台在各业务模块（如主计划、MRP、S&OP 等）运行过程中，会持续产生大量业务历史数据（订单、日志、运行记录等），长期在线保存全部明细会带来以下问题：
- 数据库体量持续膨胀：历史数据长期累积，占用大量存储空间，影响日常运维成本。
- 在线查询性能风险：部分核心表（如交易明细、操作日志等）数据量过大，可能拖慢在线查询、统计与报表性能。
### 1.1.2 问题
#### 1.1.2.1 缺少平台级归档管理能力
- 各业务模块若要做历史数据归档，通常由数据库层面单独处理，未形成统一的归档配置管理与监控入口。
- 影响：无法在平台内统一配置、查看和运维各类归档配置，业务方也难以理解和参与规则制定。
#### 1.1.2.2 归档规则分散且不可视
- 现有归档策略若由脚本实现，往往难以用业务视角描述“对哪些表、按照怎样的时间范围或数据范围进行归档”。
- 影响：规则难以复盘与评审，换人接手或业务变更时，调整成本高且风险大。
## 1.2 目标
- 建设数据库归档管理能力：提供一个统一的「归档管理」配置与监控入口，支持对选定模块下的业务表按规则进行归档。
- 数据库存储：归档结果写入新建的独立数据库，库名按 scp_{模块名}_{yyyymmddhhmmss} 自动分配，便于追溯与隔离。
- 仅支持手动触发：归档由用户点击「触发归档」手动执行，不支持定时或数据量自动触发。
- 简化配置：支持选择模块、勾选要归档的表、配置按时间条件的归档规则；暂不支持业务对象、按数据量、按 SQL 条件；不支持配置归档后是否删除，系统归档后一定会删除已归档数据。
- 与实现解耦：本 PRD 以业务与交互为主，不绑定具体技术栈；现有原型在 产品原型图/IPS/IPS_自动归档功能 目录下，v3 为当前推荐版本。

# 2. **详细解决方案**
## 2.1 **数据库归档管理**
### 2.1.1 模块描述
本模块为数据库层面的**归档配置管理**入口，统一支持：
* 以**配置**的形式定义“对哪个模块、勾选哪些表、按何种时间规则进行归档”；
* 管理已配置的归档配置列表，查看配置名称、模块、归档表数量、归档规则；
* **仅支持手动触发**：用户点击「触发归档」后，系统创建新数据库（命名格式 `scp_{模块名}_{yyyymmddhhmmss}`）并执行归档；
* 日志清晰记录每次手工触发的：触发 ID、目标数据库名、归档范围（归档表数量）、触发时间、状态、备注。
主要面向用户：
* **系统管理员 / 运维人员**：负责全局归档策略设计与运维；
* **模块负责人 / 业务配置人员**：参与确定本模块关键表的归档范围与周期。

### 2.1.2 输入
#### 2.1.2.1 业务与配置层面
* **归档配置元数据**：
  * 配置名称（唯一标识、可读性强），如 `SCH_Archive_Config`；
  * 目标数据库命名规则：`scp_{模块名}_{yyyymmddhhmmss}`，触发归档时自动创建新库。
* **模块与表选择**：
  * 选择模块（如 SCH、DFP、SOP、MPS、MRP、MDS）；
  * 勾选要归档的表，支持表名模糊筛选；
  * 至少勾选一张表。
* **归档规则（仅按时间条件）**：
  * 基准字段（时间字段），如 `created_at`、`update_time`；
  * 归档范围：归档 X（日/月/年）之前的数据，基于服务器时间。

#### 2.1.2.2 用户录入（运行前配置）
在新建或编辑归档配置时，用户需要录入：
* 配置名称；
* 模块、至少一张要归档的表；
* 归档规则：基准字段、归档 X（日/月/年）之前的数据。

#### 2.1.2.3 系统与权限（后端对接时）
* 当前连接的数据库实例信息（环境、库名等），决定可选的模块与表；
* 用户角色与权限：新建/编辑配置、触发归档、查看日志等需按权限控制。

### 2.1.3 输出
#### 2.1.3.1 用户可见页面与反馈
* **归档配置列表页**：
  * 按配置展示：配置名称、模块、归档表数量（共 X 张）、归档规则摘要；
  * 行内操作：触发归档、编辑、日志。
* **归档配置弹窗**：
  * 区域：基础信息（配置名称）、选择模块、勾选要归档的表（含表名筛选）、归档规则（按时间条件）；
  * 在填写不完整或配置存在明显冲突时，给出校验提示。
* **手工触发归档日志弹窗**：
  * 展示每次手工触发的记录：触发 ID、目标数据库名、归档范围（X 张表）、触发时间、状态、备注；
  * 支持滚动查看历史记录，便于运维排查。

#### 2.1.3.2 数据与数据库输出
* **归档数据库**：
  * 触发归档时，系统创建新数据库，命名格式 `scp_{模块名}_{yyyymmddhhmmss}`；
  * 将符合时间条件的数据迁移至新库。
* **手工触发记录**：
  * 每次手工触发的记录，包含：触发 ID、目标数据库名、归档范围（归档表数量）、触发时间、状态、备注。

#### 2.1.3.3 下游使用
* 运维可基于归档数据库做长期存档、冷备份或离线分析；
* 管理层可通过手工触发记录了解各类历史数据的归档频率与规模。

### 2.1.4 业务流程
#### 2.1.4.1 新建归档配置
1. 用户在归档管理页面点击「新建归档配置」。
2. 系统弹出配置弹窗，用户依次完成：
   * **基础信息**：填写配置名称；
   * **选择模块**：选择模块（如 SCH、DFP、SOP、MPS、MRP、MDS），系统展示目标数据库命名规则 `scp_{模块名}_{yyyymmddhhmmss}`；
   * **勾选要归档的表**：在表列表中勾选需归档的表，支持表名模糊筛选，至少勾选一张表；
   * **归档规则**：配置基准字段（如 created_at）、归档 X（日/月/年）之前的数据。
3. 用户点击「保存配置」，通过校验后，配置写入系统并在列表中展示。

#### 2.1.4.2 触发归档与状态查看
1. 用户在配置列表中点击「触发归档」，系统立即按当前配置执行一次归档。
2. 执行过程中，系统创建新数据库（命名 `scp_{模块名}_{yyyymmddhhmmss}`），将符合时间条件的数据迁移至新库。
3. 执行完成后，执行结果出现在「手工触发归档日志」中，记录触发 ID、目标数据库名、归档范围（表数量）、触发时间、状态、备注。

### 2.1.5 业务规则
#### 2.1.5.1 配置与触发规则
* **若**创建新的归档配置，**则**必须填写唯一的配置名称；配置名称在当前系统中不得重复。
* **若**保存配置，**则**必须选择模块、至少勾选一张表、配置归档规则（基准字段 + X 日/月/年之前）。
* 归档**仅支持手动触发**，用户点击「触发归档」后执行；不支持定时或数据量自动触发。

#### 2.1.5.2 归档对象与范围规则
* **若**某配置未选择模块或未勾选任何表，**则**不允许保存配置。
* **按时间归档**（v3 仅支持此方式）：
  * 必须配置基准字段（一般为时间字段，如 `created_at`、`update_time`），以及"X（日/月/年）之前"的数值；
  * 系统以当前服务器时间为基准，计算实际时间边界。

#### 2.1.5.3 清理策略
* v3 归档后**系统一定会删除**在线库中已归档的数据，**不可配置**；删除范围与归档范围严格一致。此前版本支持配置"是否归档后删除"，v3 统一为归档后必删。

#### 2.1.5.4 校验与安全
* 归档执行过程中出现异常时，应在手工触发日志中记录明确的失败原因，便于运维排查（例如：数据库连接失败、磁盘空间不足、创建新库失败等）。

#### 2.1.5.5 权限与操作限制
* 仅具有相应权限的用户才能新建、编辑归档配置、触发归档；普通业务用户默认仅可查看日志，不可修改配置或触发归档。
* 对于生产环境数据库的归档配置，建议在权限与审批流程上增加额外控制（如需通过变更单后方可触发归档）。

# 3. **未来扩展点**
## 3.1 按对象配置
可让各个模块要按数据对象/业务对象将表先做分类，并且每一类对象可以被配置不同的归档规则。
## 3.2 增加规则条件
可增加按数据量（适用于无时间列或不需按时间归档的情况）、按SQL条件（适用于复杂场景）等归档规则条件。
## 3.3 增加存储方式
可增加SQL文件等存储方式。
