# IPS_接口日志 · 用户用例/测试用例（20260304145800）

## 1. 用例说明

- **需求名称**：IPS_接口日志  
- **所属模块**：IPS → 接口管理 → 接口日志 / 场景接口数据  
- **适用版本/时间戳**：本轮设计时间戳为 `20260304145800`，适用于基于本时间戳 PRD 实现的版本（默认对应 `release17` 及之后版本，具体以实际投产版本为准）。  
- **用例目的**：  
  - 从测试视角验证接口日志页面在阶段模型（外部系统→中间表→场景库）下的展示、过滤、联动能力；  
  - 验证 `resolveCount`/`applyCount`、运行状态（成功/失败/部分成功）以及失败明细在典型场景下的展示逻辑；  
  - 验证与场景接口数据页面的联动、查询边界与性能提示；  
  - 在不影响当前实现的前提下，预留对错误码、traceId、行级失败视图等未来扩展能力的测试占位。

## 2. 用例覆盖范围与不做范围

### 2.1 覆盖范围

- **模块/页面范围**  
  - 接口管理 → **接口配置**（仅涉及「日志」入口跳转行为）；  
  - 接口管理 → **接口日志**页面（概览 / 从外部获取数据 / 向外部提供数据 / 中间表 / 场景库分发）；  
  - 接口管理 → **场景接口数据**页面（按 `api_log_id` + 场景维度查看数据、与接口日志联动）。  

- **功能与场景范围**  
  - 按接口名称、时间范围、状态等维度的基本查询与过滤；  
  - 按批次号 / 子批次号的查询与过滤（如后端接口已支持）；  
  - 阶段 A/B/C 对应字段（`responseStatus`、`resolveCount`、`applyCount`、状态枚举、错误信息）的展示与组合；  
  - 成功 / 失败 / 部分成功批次在列表与详情中的展示；  
  - 失败明细展示与大数据量场景下的分页/滚动/导出行为；  
  - 从接口日志分发明细跳转到场景接口数据，并在场景接口数据页面按 `api_log_id` + 场景查询；  
  - 查询时间范围与数据量边界下的提示行为（例如最近 7 天默认、最大 90 天、超出范围/数据量过大时的提示）；  
  - 角色权限在当前版本是否区分的基础验证：默认以“运维/实施角色”验证全部能力，若后端已区分权限，再补充分角色用例。  

### 2.2 本轮不做或仅做展示性验证的范围

- **错误码 / traceId 能力**  
  - PRD 要求接口日志 **预留 `errorCode`、`traceId` 展示位置**，并与平台报错规范对齐；  
  - 本轮用例默认按以下策略处理：  
    - 若后端已返回 `errorCode`、`traceId`：  
      - 执行对应用例验证页面展示组合「错误码 + 用户可见文案 + traceId」是否符合规范；  
    - 若后端尚未返回或仅部分接口返回：  
      - 本轮用例仅验证**字段展示位置/占位符**不影响页面布局、不出现异常空位或字段错乱；  
      - 不强制验证错误码体系的完整性与业务含义，标记为“未来扩展，依赖后端落地”。  

- **场景接口数据的行级失败视图 / 只看异常记录**  
  - PRD 在 4.4.5 中已明确「只看异常记录」「中间表 vs 场景库对比视图」为 **本期不做，未来扩展**；  
  - 本轮用例不验证相应业务能力，仅在相关用例中标注“未来扩展”，避免将未实现能力误判为缺陷。  

- **重试/补偿能力**  
  - PRD 在未来扩展中提到手动重试/补偿能力，本轮不作为必须实现内容；  
  - 若实际版本已实现手动重试按钮，则可在本文件基础上增补相应用例。

## 5. 疑问与待确认事项（供产品与研发评估）

> 以下疑问/建议在编写用例过程中暴露出来，建议由产品经理与研发评估后，在 PRD 或实现方案中给出明确结论。

1. **阶段 B/C 状态与 `resolveCount`/`applyCount` 的精确定义**  
   - 当前 PRD将阶段 B/C 与 `resolveCount`/`applyCount` 做了映射，但未完全明确：在多场景、多表或多次重试的情况下，这两个字段分别统计哪些维度（单表/多表、单场景/多场景、是否包含重试等）。  
   - 建议在 PRD 或技术方案中明确定义这两个字段的统计口径，以便测试在部分成功/全部失败等场景下能准确判断预期。

2. **「部分成功」在不同视图中的状态展示规则**  
   - PRD 提到当 `resolveCount > applyCount` 时应以“部分成功”高亮提示，但未明确：  
     - 接口日志主列表与「场景库分发」Tab 是否完全一致使用“部分成功”状态，还是主列表统一视为“失败”；  
     - 对 `resolveCount > 0 && applyCount == 0` 的场景，是按失败还是部分成功展示。  
   - 建议补充一张「状态判定表」，按 `resolveCount` 和 `applyCount` 的组合（含 0/空值）列出应展示的状态文案与颜色。

3. **错误码/traceId 在本期的落地程度**  
   - PRD 要求预留 `errorCode`、`traceId` 展示位置，但未明确本期是否强制后端返回对应字段，还是仅作为未来扩展占位。  
   - 建议在 PRD 中增加一节「本期实现 vs 预留」说明，明确：  
     - 本期是否需要至少 1–2 个典型接口完整打通错误码+traceId；  
     - 若后端暂不支持，测试应如何判定“通过”（仅验证布局与占位即可，还是需要验证部分接口样例）。

4. **批次号/子批次号在查询条件中的支持程度**  
   - PRD 中强调按批次号/子批次号查询与过滤的需求，release17 中 `/extApiLog/queryData` 已存在，但其入参中对批次号/子批次号维度的支持程度仍未在接口文档中明确。  
   - 建议由后端在 release17/DCP 环境中确认并在 PRD 中补充：  
     - 是否支持按批次号/子批次号精确过滤；  
     - 若不支持，是视为本期缺口（需补齐）还是后续优化点。

5. **权限与多租户/多场景隔离策略**  
   - 现有 PRD 对权限和可见性仅有概括性说明，未明确不同角色（业务用户/实施/运维）的可见范围与操作边界（尤其是接口日志/场景接口数据中的导出能力）。  
   - 建议在后续 PRD 版本中补充：  
     - 不同角色在接口日志和场景接口数据上的最小可见性要求；  
     - 日志与数据导出是否需要更高权限或脱敏处理；  
     - 多租户场景下是否允许跨租户查看日志/数据。
