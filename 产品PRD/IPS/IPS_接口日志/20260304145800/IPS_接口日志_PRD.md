# 1. **需求概述**

## 1.1 需求涉及的模块

IPS（Integration Platform Service，集成平台服务）

## 1.2 需求名称

IPS_接口日志

## 1.3 需求简述

本需求针对 IPS 模块下的「接口管理」功能，核心对象为新的「接口日志」页面。当前系统有实验功能「接口管理」，包括「接口配置」「接口同步控制」「接口日志」「接口数据」。现有实现无法清晰看到流程【外部系统 → 中间表 → 场景库】中「获取到中间表」和「分发到场景库」两步的成功/失败日志，不利于排查问题和界定责任。

**本次改造目标**：
- 将原「接口日志」与「接口数据」合并为新「接口日志」页面，采用 Tab 式多视图（概览、从外部获取数据、向外部提供数据、中间表、场景库分发）；
- 明确阶段模型（外部系统→中间表→场景库），补齐两阶段成功/失败日志展示能力；
- 与「IPS_平台报错规范化优化」中约定的错误码、错误文案、traceId 等规范对齐，在接口日志中预留相关字段与展示位置。

**本次范围**：仅做前端页面与日志展示/过滤能力的改造设计，不直接设计 ETL/引擎层，但需为后端日志采集/存储提供字段与行为约束。

**与主设计文档的对应关系**：本需求为 IPS 接口管理模块的增强，若 IPS 主设计文档存在「接口管理」章节，应在本 PRD 对应模块中引用；无则视为新增能力。

# 2. **需求范围和流程概览**

## 2.1 基本概念说明

- **IPS**：Integration Platform Service，集成平台服务，负责任务调度、接口日志、数据集成等平台级能力。
- **接口管理**：IPS 下二级模块，包含接口配置、接口同步控制、接口日志、场景接口数据四个三级模块。
- **本需求涉及**：对「接口日志」与「场景接口数据」进行增强；接口配置与接口同步控制仅作与本需求相关的补充说明。

## 2.2 阶段模型与数据流程概览

### 2.2.1 接口调用阶段模型（业务定义）

数据流【外部系统 → 中间表 → 场景库】划分为三个阶段：

| 阶段 | 名称 | 主要输入 | 主要输出 | 关键状态 | 常见错误类型 |
|------|------|----------|----------|----------|--------------|
| **A** | 外部系统调用与基础校验 | 外部系统请求（URL、参数、请求体） | 接口响应（状态码、响应体、响应时间） | 成功 / 失败 | 连接超时、HTTP 4xx/5xx、参数错误、响应格式异常 |
| **B** | 解析并写入中间表 | 接口响应数据 | 中间表记录（可映射到 `resolveCount` 等统计） | 成功 / 部分成功 / 失败 | 解析失败、主键重复、字段校验失败、唯一约束冲突 |
| **C** | 从中间表分发到场景库 | 中间表数据 | 场景库记录（可映射到 `applyCount` 等统计） | 成功 / 部分成功 / 失败 | 外键约束、唯一约束、业务规则校验失败 |

**阶段与现有字段的映射**：
- 阶段 A：对应 `responseStatus`、`responseTime`、`errorMessage` 等 HTTP 层结果；
- 阶段 B：对应 `resolveCount`（解析/写入中间表数量）、中间表写入失败明细；
- 阶段 C：对应 `applyCount`（应用/写入场景库数量）、场景库分发失败明细。

### 2.2.2 业务流程概览

- 用户从「接口配置」卡片进入「接口日志」或「接口同步控制」；
- 新「接口日志」页面采用 Tab 切换：概览 / 从外部获取数据 / 向外部提供数据 / 中间表 / 场景库分发；
- 用户可设置查询条件、查看列表和详情、从分发明细跳转到「场景接口数据」；
- 原「接口数据」菜单入口可合并至新「接口日志」页面，或保留为独立入口但需与接口日志关联（见 4.4）。

### 2.3 数据流程概览

- **输入**：接口调用记录、接口执行批次、中间表数据快照、场景库记录；
- **输出**：用户在页面上看到的统计、列表、详情、失败明细；支持导出失败数据；
- **关联**：通过 `api_log_id` 关联接口日志与场景接口数据。

# 4. **详细解决方案**

## 4.1 **接口配置**

### 4.1.1 模块描述与本次改造范围

接口配置用于维护「有哪些接口」的元数据（来源、名称、分类等），并以卡片列表形式提供新增/修改/删除操作。**本模块沿用现有实现，本次改造不对页面结构和后端逻辑做任何调整，仅要求继续保留卡片上的「日志」「控制」两个入口，便于从接口配置跳转到接口日志和接口同步控制页面。**

### 4.1.2 与本需求的关联

- 从接口配置卡片点击「日志」，跳转至接口日志页面，并默认带入当前接口作为过滤条件（如接口名称或接口 ID）；  
- 从接口配置卡片点击「控制」，跳转至接口同步控制页面；  
- 其余行为（字段、交互、原型）全部按现有实现执行，不在本次改造范围之内。

---

## 4.2 **接口同步控制**

### 4.2.1 模块描述与本次改造范围

接口同步控制用于配置接口调度/开关（什么时候、如何调用这些接口）。**本模块同样沿用现有实现，本次改造不对其页面结构和后端逻辑做变更，只依赖从接口配置跳转到该页面的入口保持可用。**

### 4.2.2 与本需求的关联

- 用户可通过接口配置卡片上的「控制」按钮进入接口同步控制页面，对接口启停、调整调度策略等；  
- 接口同步控制与接口日志之间的更深层“状态联动”（例如在日志中显式展示「暂停/未执行」状态）**不纳入本期范围，视为未来扩展能力**，本期开发无需实现该部分。

---

## 4.3 **接口日志**（本次核心模块）

### 4.3.1 模块描述

本模块为「面向问题排查与责任界定」的视图，将原「接口日志」与「接口数据」合并为新「接口日志」页面。采用 Tab 式多视图（概览、从外部获取数据、向外部提供数据、中间表、场景库分发），使用户能清晰看到【外部系统→中间表→场景库】两阶段的成功/失败日志。

**与主设计文档的对应关系**：若 IPS 主设计文档存在「接口日志」章节，本模块为对该章节的增强；若存在「接口数据」章节，本模块合并其能力并纳入本页面。

### 4.3.2 输入

- 接口调用记录（来自 DCP 接口日志表，对应 `ApiLogPO` / `ApiLogDao` 所在物理表，表名与字段类型以实际 DDL 为准）；  
- 接口执行批次、中间表数据快照、场景库记录；  
- 接口配置：接口来源、名称、分类等元数据，用于筛选与展示。

### 4.3.3 输出

- 用户可见：概览统计、各 Tab 下的列表、详情抽屉、失败明细；
- 支持导出：当前筛选条件下所有失败数据（Excel）；
- 下游：场景接口数据页面可依赖 `api_log_id` 与接口日志关联跳转。

### 4.3.4 业务流程

1. 用户从接口配置进入接口日志页面，或直接访问接口日志路由；
2. 默认展示「概览」Tab：4 个统计卡片（从外部获取数据、向外部提供数据、中间表、场景库分发）+ 最近调用记录表；
3. 用户点击统计卡片或 Tab 切换至对应视图；
4. 各视图设置筛选条件（接口名称、时间范围、状态、接口分类、场景、接口日志 ID、是否有失败等），点击查询；
5. 列表展示后，用户点击行或「详情」可打开右侧抽屉，查看请求头/体、响应体、解析/应用数量、错误信息、失败明细等；
6. 分发明细抽屉中可「跳转场景接口数据」，携带 `api_log_id` 与场景参数；
7. 支持导出当前筛选条件下所有失败数据。

### 4.3.5 业务规则

#### 4.3.5.1 阶段与字段映射

- 若日志记录处于**阶段 A**（外部系统调用），则 `responseStatus` 表示 HTTP 层成功/失败；`responseTime` 为响应时间；`errorMessage` 为 HTTP 层或解析前的错误信息。
- 若日志记录涉及**阶段 B**（中间表写入），则 `resolveCount` 表示解析/写入中间表的记录数；`applyCount` 表示应用/写入场景库的记录数；`resolveCount` 与 `applyCount` 的差异可暗示阶段 B 或 C 存在问题。
- 若日志记录涉及**阶段 C**（场景库分发），则 `applyCount` 表示成功写入场景库的记录数；失败明细需在「场景库分发」Tab 或「分发明细」抽屉中展示。

#### 4.3.5.2 状态与展示

- 若运行状态为**成功**：列表项展示绿色状态标识，错误信息为空或「—」。
- 若运行状态为**失败**：列表项展示红色状态标识，错误信息区域展示错误文案；若存在「部分成功」状态，则展示黄色/警示色标识。
- 若批次存在**部分成功**（如 `resolveCount > 0` 且 `applyCount < resolveCount`）：前端应标红或高亮显示部分成功批次，便于用户快速定位。

#### 4.3.5.3 与平台错误规范的对齐

- 若平台已实施「IPS_平台报错规范化优化」，接口日志中的错误信息应遵循统一文案与错误码规范；
- 接口日志页面应预留**错误码**、**traceId** 的展示位置（如详情抽屉中）；当存在平台级错误码和接口级错误信息时，前端组合展示：**错误码 + 用户可见文案 + traceId**；
- 若后端尚未返回 `errorCode`、`traceId`，前端可预留占位，待后端支持后展示；**本期实现时需与后端确认**。

#### 4.3.5.4 查询与过滤

- 默认可查时间范围：建议**最近 7 天**（可配置）；
- 最大可查历史区间：建议**90 天**（可配置）；若用户选择超过范围，则提示「请缩小时间范围」；
- 若数据量超出单页加载上限，则采用分页；失败明细超过 100 条时，仅展示前 50 条并支持分页；超过 1000 条时，建议引导用户导出 Excel 后离线分析。

#### 4.3.5.5 清理策略

- 接口日志与场景接口数据为高频/大数据量场景，引入更细粒度日志后，表大小、索引、归档策略可能受影响；
- 建议在 PRD 中约束：**默认日志保留周期**（如 30 天/90 天，**待决策**）；**归档策略**（如超期数据归档至历史表或冷存储）由技术实现时确定；
- 若查询结果超出系统限制，前端提示「数据量过大，请缩小查询范围或联系管理员」。

#### 4.3.5.6 各 Tab 视图规则

- **概览**：统计卡片展示今日（或当前筛选时间范围内）的调用次数/行数；最近调用记录按时间倒序，限制最近 N 条（如 20 条）；点击行可下钻至对应详情或 Tab。
- **从外部获取数据**：筛选条件：接口名称、时间范围、状态；列表字段：序号、接口名称、批次号、请求方式、响应状态、解析/应用、响应时间、错误信息、重试、创建时间、操作；失败记录支持手动重试（若后端支持）。
- **向外部提供数据**：筛选条件：接口名称、时间范围、状态；列表字段：序号、接口名称、来源系统、批次号、响应状态、下发行数、响应时间、错误信息、创建时间、操作；下发行数表示本系统下发的行数，不记录外部解析/应用情况。
- **中间表**：筛选条件：接口分类、接口日志 ID（级联，支持模糊搜索）；列表字段：序号、数据ID、数据内容、入库时间；仅展示原始数据凭证，全量接收无拒收。
- **场景库分发**：筛选条件：接口分类、场景、接口日志 ID、是否有失败；列表字段：序号、接口日志ID、接口分类、场景、总数量、成功、失败、失败原因摘要、分发时间、操作；失败明细在抽屉中展示（数据ID、错误类型、错误详情）并支持导出。

### 4.3.6 原型图

本次沿用 `产品设计/产品需求/IPS/IPS_接口日志/接口日志.html` 的设计稿作为页面原型基础，不再单独产出正式原型图文档。设计稿包含：顶部 Tab（概览、从外部获取数据、向外部提供数据、中间表、场景库分发）、统计卡片、筛选栏、表格、详情抽屉、失败明细展示与导出。

### 4.3.7 用例

- 查询成功批次日志；
- 查询失败批次并定位到场景数据；
- 多场景下对比数据一致性；
- 导出失败明细；
- 从接口配置跳转至接口日志并携带接口过滤条件；
- 从分发明细跳转至场景接口数据。

具体编号与明细由产品测试经理在用例文档中扩展。

## 4.4 **场景接口数据**

### 4.4.1 模块描述

从业务场景视角，查看与某接口日志（`api_log_id`）相关联的「落地到场景库中的接口数据」。本模块为既有模块的增强，与接口日志形成联动，实现「从日志到数据」的下钻。

### 4.4.2 输入

- 接口分类、接口日志 ID、场景；  
- 后端接口 `/itf/getItfTableData` 动态返回表头和数据（前端在 release17 中已通过该接口获取场景接口数据，具体可用的过滤维度与行数上限需在联调阶段结合接口文档确认，并据此细化测试口径）。

### 4.4.3 输出

- 用户可见：按 `api_log_id` + 场景维度的数据明细表格，表头与列由后端动态返回；
- 支持从接口日志分发明细中「跳转场景接口数据」，携带 `api_log_id` 与场景参数。

### 4.4.4 业务流程

1. 用户从接口日志「场景库分发」Tab 或分发明细抽屉中点击「跳转场景接口数据」；
2. 跳转至场景接口数据页面，并携带 `api_log_id`、场景等参数；
3. 场景接口数据页面根据参数自动填充查询条件并执行查询；
4. 展示该接口日志在对应场景下的数据明细。

### 4.4.5 业务规则

- 若用户从接口日志跳转，则携带 `api_log_id` 与场景参数；场景接口数据页面应自动填充并查询；
- 若后端支持「异常/失败记录」筛选，则可在场景接口数据中提供「只看异常记录」筛选（**本期不做**：若后端暂不支持行级失败标记，则 PRD 中写明「本期不做」，作为未来扩展）；
- 若暂不支持行级失败标记，则场景接口数据仅展示结果数据，不展示「中间表 vs 场景库对比」视图（**本期不做**，作为未来扩展）。

### 4.4.6 原型图

沿用现有场景接口数据页面设计；本需求主要增加「从接口日志跳转并携带参数」的联动逻辑。

### 4.4.7 用例

- 按 `api_log_id` + 场景查询数据；
- 从接口日志分发明细跳转至场景接口数据并自动填充查询条件。

具体编号与明细由产品测试经理在用例文档中扩展。

# 5. **与后端/DB 实现的适配与嵌入方案**

## 5.1 依赖的后端接口

| 接口 | 用途 | 预期补充能力 |
|------|------|--------------|
| 现有 `/extApiLog/queryData` | 接口日志分页查询（主日志 + 子日志，已在 release17 中落地，由 `ApiLogController` / `ApiLogServiceImpl` 实现） | 前端仍需与后端确认：是否计划在查询参数中显式支持阶段字段、错误类型枚举等附加过滤维度（当前实现仅基于通用查询条件与父子日志结构）。 |
| 现有 `/extApiLog/detail/{id}` | 接口日志详情（已在 release17 中落地，用于按 ID 查询单条日志记录） | 需确认是否在本期版本中返回错误码、traceId 等字段，或暂时仅用于补充请求/响应明细。 |
| 现有 `/itf/getItfTableData` | 场景接口数据动态表头与数据（已在 release17 中落地，由 DCP itf 公共服务根据 `ApiCategoryEnum.mappingValue` 与场景信息选择具体中间表/视图） | 需补充确认其查询参数中对 `api_log_id` + 场景的支持范围（目前前端已按该模式调用，但仍需在接口文档中固化约束，如是否支持其它过滤维度、是否有行数上限等）。 |

**技术前置**：release17 分支中已经可以看到 DCP 的主要接口实现与中间表 Mapper，但仍未在 DB 层逐字段核对 DDL 与索引。建议在实施前将「确认现有表结构与接口字段」列为技术前置：  
- 对接口日志表与中间表的核心字段（尤其是 `api_log_id`、`resolveCount`、`applyCount`、`valid`、`invalid_reason` 等）做一次 DDL 级确认；  
- 明确 `/extApiLog/queryData`、`/extApiLog/detail/{id}`、`/itf/getItfTableData` 的入参/出参字段及可用过滤维度，并在接口文档中固化，避免前后端对字段含义产生偏差。

## 5.2 后端/DB 建议调整

- **新增或调整字段**（建议，非强制 DDL）：
  - 接口日志表：建议增加阶段字段（如 `phase`：A/B/C）、错误类型枚举（如 `errorType`）、错误码（`errorCode`）、traceId（`traceId`）；若后端已支持，则无需新增；
  - 中间表/场景表：若需支持失败明细，建议新增失败明细表或扩展现有表结构；**待确认**。
- **索引**：建议对 `api_log_id`、`create_time`、`status` 等常用查询维度建立索引；**待确认**。

## 5.3 前端嵌入与菜单调整

- **路由**：新接口日志页面应替换现有 `/dc/system/apiLog` 路由，或新建路由并保留旧路由作为兼容（**待决策**）；
- **菜单**：是否下线原「接口数据」菜单入口（`/dc/system/sceneInterfaceData`）？
  - 若**下线**：需将场景接口数据能力合并至新接口日志的「场景库分发」Tab 或通过「跳转场景接口数据」链接访问；需明确迁移策略；
  - 若**保留**：则保留「场景接口数据」为独立入口，但需与接口日志形成跳转联动；
- **建议**：新接口日志页面替换现有 `/dc/system/apiLog`；原「场景接口数据」保留为独立入口，从接口日志分发明细中提供「跳转场景接口数据」链接。

# 附录：与平台报错规范的关系

本需求需对齐「IPS_平台报错规范化优化」中约定的错误码、错误文案、traceId 等规范。具体包括：

- 接口日志中的错误信息应遵循统一文案规范（不直接透传异常、避免技术堆栈暴露）；
- 用户页面展示的错误提示应采用「简洁文案 + 错误码 + traceId」的组合；
- 接口日志详情中应预留错误码、traceId 的展示位置；
- 若平台规范尚未覆盖接口日志场景，建议在接口日志改造时一并纳入规范。
