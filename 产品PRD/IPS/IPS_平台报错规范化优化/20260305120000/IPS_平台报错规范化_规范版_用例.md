# IPS 平台报错规范化优化（规范版）— 测试用例

> 本文档基于《IPS 平台报错规范化优化 — 规范版 PRD》（第 2 章异常处理技术规范、第 3 章报错语句编写规范、第 4 章用例概览）编制，用于验收：
> - 抛出层级是否符合规范  
> - 是否统一使用异常枚举 + 国际化  
> - 错误码与文案是否在枚举及 properties 中同步维护  
> - 是否不存在技术信息透传前端  
> - 占位符数量与参数是否匹配  
> - 报错文案是否符合编写规范

## 2. 详细用例

### 2.1 抛出层级相关

#### IPS-ERR-001 Service 层抛出业务异常

- **前置条件**
  - 存在一个典型业务场景（如「创建计划任务」）的 Service 方法：`PlanService.createPlan(...)`。
  - 该方法中包含参数/业务校验（如必填字段校验、业务规则校验）。
  - 系统已按规范接入全局异常处理（`GlobalExceptionHandler` 或等价实现）。

- **操作步骤**
  1. 在测试环境中调用「创建计划任务」接口（通过前端页面或 Postman/接口平台皆可），构造明显不满足业务规则的请求（如缺少必填参数或违反状态约束）。
  2. 在 IDE 或日志中查看该次请求的调用栈/日志输出。
  3. 检查异常抛出位置（类名、方法名、调用链）。

- **预期结果**
  - 调用栈中显示业务异常（如 `BusinessException`）由 Service 层方法抛出，而非 Controller/DAO 层。
  - Controller 层仅调用 Service，无显式 `throw new BusinessException(...)` 逻辑。
  - 日志中能够看到异常在 Service 层被记录，其上层通过统一异常处理转换为标准响应。

---

#### IPS-ERR-002 DAO 异常转换为业务异常

- **前置条件**
  - 存在一个依赖数据库操作的 Service 方法（如「保存计划明细」），其内部调用 DAO/Repository 层。
  - 可以通过构造非法数据（如违反唯一索引约束）或临时调整数据，触发典型数据库异常（如 `DuplicateKeyException`、SQL 约束异常）。

- **操作步骤**
  1. 通过接口或单元测试调用该 Service 方法，构造将触发数据库约束错误的请求。
  2. 观察系统返回的响应体（HTTP 响应 JSON）以及后台日志。
  3. 检查是否存在数据库异常直接向上传递的现象（如堆栈回到 Controller）。

- **预期结果**
  - Service 层捕获 DAO 的具体异常（可通过日志或代码查看），并转换为规范的业务异常或系统异常。
  - 返回给前端的响应体中仅包含业务可理解的错误码和错误消息，不包含数据库表名、SQL 片段等技术细节。
  - 日志中记录了原始数据库异常堆栈，便于排查，但这些信息不会出现在前端响应中。

---

#### IPS-ERR-003 禁止 Controller 直接抛出业务异常

- **前置条件**
  - 已完成对相关模块 Controller 与 Service 的代码走查。

- **操作步骤**
  1. 在代码中全局检索 Controller 层（如包路径 `*.controller.*`）是否存在以下模式：`throw new BusinessException` 或直接引用 `BusinessErrorEnum` 抛异常。
  2. 若存在疑似代码，定位到具体类与方法，结合调用链分析实际运行路径。

- **预期结果**
  - Controller 层代码中不存在直接 `throw new BusinessException(...)` 或类似业务异常抛出的实现。
  - 如果发现违规代码，应记录缺陷：标明类名、方法名、问题代码行，并提出整改建议（改由 Service 层统一抛出）。

---

### 2.2 枚举 + 国际化相关

#### IPS-ERR-004 使用 BusinessErrorEnum + i18n 获取文案

- **前置条件**
  - 系统中已存在 `BusinessErrorEnum` 或等价的业务异常枚举类，字段至少包含 `code`（异常编码，对应 i18n key）、`desc`（默认中文描述）。
  - 已配置 `messages_zh_CN.properties`、`messages_en_US.properties` 文件。

- **操作步骤**
  1. 在代码中新增一处报错场景（或选取已有场景），为其指定 `BusinessErrorEnum` 枚举常量，如 `MDS001`。
  2. 在 `messages_zh_CN.properties` 中新增 key：`MDS001=物料不存在，请确认后重试。`；在 `messages_en_US.properties` 中新增 key：`MDS001=Material does not exist, please check and try again.`。
  3. 在 Service 中通过规范方式抛出异常，例如：  
     `throw new BusinessException(BusinessErrorEnum.MDS_MATERIAL_NOT_EXIST.getI18nErrordesc());`  
     （必须使用 `getI18nErrordesc()`，禁止使用 `getDesc()`）
  4. 切换系统语言或在不同 locale 下调用该报错点（若系统支持语言切换）。

- **预期结果**
  - 代码未直接硬编码中文/英文文案，而是通过 `BusinessErrorEnum` + `I18nUtil.getString(code)` 方式获取。
  - 在简体中文环境下，前端看到的提示为「物料不存在，请确认后重试。」。
  - 在英文环境下，前端看到的提示为 `Material does not exist, please check and try again.`。
  - 如缺失某语言配置，应有明确的 fallback 策略（如退回默认中文或默认 desc），但不出现 key 本身或异常信息。

---

#### IPS-ERR-005 国际化多语言配置完整性检查

- **前置条件**
  - 系统中已经存在一批 `BusinessErrorEnum` 枚举常量（至少覆盖主流程错误码，例如 DP001、SOP001、SYS001 等）。
  - 已有 `messages_zh_CN.properties`、`messages_en_US.properties` 等配置文件。

- **操作步骤**
  1. 统计或导出当前所有 `BusinessErrorEnum` 的 `code` 列表。
  2. 分别在 `messages_zh_CN.properties`、`messages_en_US.properties` 中检索这些 code 是否全部存在。
  3. 对任意缺失 key 的情况进行记录。

- **预期结果**
  - 所有 `BusinessErrorEnum.code` 均在 `messages_zh_CN.properties`、`messages_en_US.properties` 中配置了对应文案。
  - 未出现仅有中文文案/英文文案而缺少另一语言配置的错误。
  - 如存在缺失项，应按缺陷登记，要求补齐多语言文案。

---

#### IPS-ERR-006 错误码与文案维护位置规范

- **前置条件**
  - 开发需新增一个业务错误码（如 `MDS002`）。

- **操作步骤**
  1. 在 `BusinessErrorEnum` 中新增枚举常量，如 `MDS_STOCK_POINT_NOT_FOUND("MDS002", "默认描述")`。
  2. 在 `messages_zh_CN.properties` 中新增 `MDS002=库存点不存在，请确认后重试。`。
  3. 在 `messages_en_US.properties` 中新增 `MDS002=Stock point does not exist, please check and try again.`。
  4. 执行代码评审或 CI 检查，确认新增错误码的完整性。

- **预期结果**
  - 枚举中已定义，`code` 与 properties 中 key 一致。
  - 中文、英文 properties 中均存在对应 key-value。
  - 新增错误码时，未出现仅改枚举或仅改 properties 的情况。

---

### 2.3 无技术透传相关

#### IPS-ERR-007 业务异常不透传 e.getMessage

- **前置条件**
  - 全局异常处理器（如 `GlobalExceptionHandler`）已生效，统一处理 `BusinessException`。
  - 日志组件已正确记录异常堆栈。

- **操作步骤**
  1. 构造一个会抛出 `BusinessException` 的业务场景（可复用 IPS-ERR-001 的场景）。
  2. 触发异常后，抓取 HTTP 响应报文（JSON）。
  3. 同时查看服务端日志，确认内部异常堆栈（如 `NullPointerException`、`SQLException` 等）内容。

- **预期结果**
  - HTTP 响应体中的 `msg`（或等价字段）为业务可理解的固定文案，不包含以下内容：  
    - 异常类名（如 `NullPointerException`、`SQLException` 等）  
    - 方法名、类全限定名  
    - 数据库表名、字段名、SQL 语句  
    - `e.getMessage()` 或 `e.toString()` 的原始输出  
  - 日志文件中保留完整堆栈信息，用于运维和研发排查。

---

#### IPS-ERR-008 系统异常仅写日志不返回堆栈

- **前置条件**
  - 全局异常处理器对未捕获异常（如 `RuntimeException`）有统一处理逻辑，返回系统错误提示。

- **操作步骤**
  1. 在测试环境中构造一个非预期系统异常场景（如在某个路径中临时抛出 `new RuntimeException("test stack")`）。  
  2. 触发该路径，获取 HTTP 响应报文。
  3. 检查服务端日志，确认记录了原始异常及堆栈。

- **预期结果**
  - 前端返回的错误消息符合 3.2 中「系统错误」的模板，例如：「系统繁忙，请稍后重试。如有问题请联系管理员，并提供错误码与追踪ID。」。
  - 响应中不包含 `"test stack"` 等异常原始信息，也不包含堆栈、类名、SQL 等技术细节。
  - 日志中可看到完整异常堆栈和上下文信息，便于定位。

---

### 2.4 占位符匹配相关

#### IPS-ERR-009 占位符数量与参数匹配（正向）

- **前置条件**
  - 在 `messages_zh_CN.properties` 中配置一条带占位符的文案，例如：  
    - `DP002=物料「{0}」在计划期间「{1}」不存在，请确认后重试。`
  - `BusinessErrorEnum` 中存在对应 code，如 `DP002`。

- **操作步骤**
  1. 在 Service 代码中通过以下方式抛出异常：  
     `throw new BusinessException(BusinessErrorEnum.DP002.getI18nErrordesc("M001", "2026Q1"));`
  2. 触发该报错场景（通过接口调用或单测执行）。
  3. 观察前端返回的错误消息。

- **预期结果**
  - 返回文案中的占位符被正确替换为对应参数，示例：  
    「物料「M001」在计划期间「2026Q1」不存在，请确认后重试。」。
  - 不出现 `{0}`、`{1}` 未替换的情况。
  - 无格式化异常（如 `IllegalArgumentException: wrong number of arguments`）抛出。

---

#### IPS-ERR-010 占位符数量与参数不匹配（负向检查）

- **前置条件**
  - 仍然使用 IPS-ERR-009 中的 i18n 配置（`DP002` 带两个占位符）。

- **操作步骤**
  1. 在代码中刻意构造以下两种错误调用方式，并通过单测或灰度环境验证：  
     - 仅传 1 个参数：`getI18nErrordesc("M001")`  
     - 传 3 个参数：`getI18nErrordesc("M001", "2026Q1", "extra")`  
  2. 运行相关用例，观察是否产生运行时格式化异常或错误文案。
  3. 检查代码审查规范/静态检查是否能识别此类问题。

- **预期结果**
  - 通过代码审查或静态检查，能发现占位符数量与参数数量不匹配的调用点，并视为不符合规范的实现，需要整改。
  - 在实际发布版本的代码中，不应保留上述错误调用；运行环境中不会因占位符与参数不匹配产生格式化异常或错误文案。

### 2.5 文案规范相关

#### IPS-ERR-011 参数缺失场景报错文案规范

- **前置条件**
  - 存在一个前端表单或接口需要必填参数（如「用户名」「密码」或「计划名称」）。
  - 对应 Service 层有参数校验逻辑，并通过 `BusinessErrorEnum` 抛出异常。

- **操作步骤**
  1. 在前端页面或接口调用中，故意不填某个必填字段（如「计划名称」为空）。
  2. 提交表单或调用接口，触发参数缺失校验。
  3. 观察前端显示的错误提示。

- **预期结果**
  - 错误提示符合 3.2 中「参数缺失」模板：  
    - 使用「{参数名}」明确指出具体缺失的参数，例如：「「计划名称」不能为空。」  
  - 不出现以下不规范表述：  
    - 仅提示「不能为空」而不指明参数名。  
    - 混用中英文标点或未使用「」包裹参数名。  
  - 句尾使用中文标点（。），整体表述简洁清晰。

---

#### IPS-ERR-012 配置缺失场景报错文案规范

- **前置条件**
  - 存在依赖系统配置的业务场景（如「计划期间」未配置时无法运行计划）。
  - 对应 Service 在检测到配置缺失时，会抛出异常。

- **操作步骤**
  1. 在测试环境中暂时移除或禁用某项关键配置（如「计划期间」配置），或使用专门的测试配置。
  2. 触发依赖该配置的业务操作（如运行计划、生成需求等）。
  3. 观察前端返回的错误消息。

- **预期结果**
  - 错误提示符合 3.2 中「配置缺失」模板，例如：  
    - 「尚未配置「计划期间」，请前往主数据完成配置后重试。」  
  - 文案中包含：  
    - 明确指出缺失的配置项名称（使用「」包裹，如「计划期间」）；  
    - 提供配置入口或菜单路径（如「主数据」）；  
    - 给出下一步指引（如「完成配置后重试」）。  
  - 不出现仅提示「未配置」但不指明配置项或路径的情况。

---

#### IPS-ERR-013 状态不允许场景报错文案规范

- **前置条件**
  - 存在对业务对象状态进行管控的操作（如「已生产的工序不允许调整」）。
  - 系统实现了状态检查逻辑。

- **操作步骤**
  1. 构造一个业务对象，使其处于不允许本次操作的状态（如工序已生产完成）。
  2. 尝试执行受限操作（如调整工序、变更计划等）。
  3. 观察前端显示的错误提示。

- **预期结果**
  - 提示文案符合 3.2 中「状态不允许」模板，例如：「该工序已生产，不得调整。」。
  - 文案中应包含：  
    - 当前状态（如「已生产」）；  
    - 不允许的操作（如「调整」）；  
  - 不应仅提示「状态不允许」等空泛句式。
  - 用词符合 3.4 术语统一要求（如使用「工序」「任务」等规范术语）。

---

#### IPS-ERR-014 系统错误场景文案与指引规范

- **前置条件**
  - 系统已按规范在系统级异常（非业务可预知错误）时返回统一的系统错误提示。

- **操作步骤**
  1. 通过模拟外部依赖不可用（如暂时关闭某个外部服务或中断网络）等方式，触发系统级错误。
  2. 捕获前端展示的错误提示和返回 JSON。
  3. 检查文案内容及是否包含错误码、追踪 ID 等信息。

- **预期结果**
  - 错误提示符合 3.2 中「系统错误」模板，类似：  
    - 「系统繁忙，请稍后重试。如有问题请联系管理员，并提供错误码与追踪ID。」  
  - 文案中：  
    - 不包含技术细节（堆栈、SQL、内部服务名称等）；  
    - 明确给出下一步操作指引（稍后重试、联系管理员）；  
    - 指明需要提供的关键信息（错误码、追踪 ID）。  
  - 错误码和追踪 ID 可在日志或监控系统中被检索到，与该次请求一一对应，便于排查。
