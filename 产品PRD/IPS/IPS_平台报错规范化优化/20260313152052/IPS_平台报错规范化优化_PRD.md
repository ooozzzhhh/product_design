# 1. **需求概述**

## 1.1 需求涉及的模块
IPS、MDS、SDS、SOP、MPS、MRP、AMS/SCH、DFP（平台级需求，覆盖所有供应链计划模块）

## 1.2 需求名称
IPS_平台报错规范化优化

## 1.3 需求简述
当前供应链计划系统的报错提示存在以下问题：
- 报错内容不统一，有的只说"操作失败"，用户不知道哪里出错；
- 技术信息透传给用户（如"NullPointerException"），用户看不懂；
- 报错提示固定在代码中，业务方无法根据实际场景调整文案；
- 缺少下一步操作指引，用户不知道该怎么办。

**问题**：报错提示不规范导致用户体验差，增加实施与运维成本；报错文案无法灵活调整，无法适应不同客户的业务术语和场景。

**需求价值**：
- 让用户在遇到错误时能看得懂、知道怎么办；
- 业务方可根据实际场景编辑报错文案，无需修改代码；
- 统一报错规范，降低实施与运维成本。

# 2. **需求范围和流程概览**

## 2.1 基本概念说明
**本需求涉及**：
- 新增：报错提示管理模块（值集配置界面）
- 修改：所有业务模块的报错提示展示逻辑

## 2.2 业务流程概览
**新增流程**：
1. 业务管理员进入"系统管理 > 值集管理 > 报错提示配置"
2. 查询需要编辑的错误码
3. 编辑中英文报错文案（系统校验占位符数量）
4. 保存后立即生效

**修改流程**：
- 原流程：业务操作 → 触发错误 → 显示固定报错文案
- 新流程：业务操作 → 触发错误 → 根据错误码从值集加载报错文案 → 填充占位符 → 显示报错

## 2.3 数据流程概览
**输入数据来源**：
- 错误码定义（系统预置）
- 业务管理员编辑的报错文案（存储在值集表）

**输出数据去向**：
- 前端界面：显示给用户的报错提示
- 日志系统：记录错误码、追踪ID、技术异常信息

# 3. **对标分析**

## 3.1 SAP ERP 报错提示机制
**实现方式**：
- 错误消息存储在消息类（Message Class）中，每条消息有唯一编号（如 M3001）
- 支持占位符（&1、&2），运行时动态填充
- 支持多语言，根据用户登录语言显示对应文案
- 业务顾问可通过 SE91 事务码编辑消息文本，无需修改代码

**报错内容特点**：
- 包含消息编号，便于用户反馈和技术支持定位
- 文案业务导向，如"物料 &1 在工厂 &2 中不存在"
- 提供消息类型（错误/警告/信息），用不同颜色区分

**可借鉴点**：
- 错误码与文案分离，支持配置化管理
- 占位符机制灵活，但数量由代码决定
- 多语言支持是标配

## 3.2 Oracle EBS 报错提示机制
**实现方式**：
- 错误消息存储在 FND_NEW_MESSAGES 表中
- 支持消息令牌（Token），类似占位符
- 通过"应用程序消息"功能配置和维护

**报错内容特点**：
- 消息编号 + 应用模块前缀（如 INV-12345）
- 提供详细说明和建议操作

**可借鉴点**：
- 错误码按模块分类管理
- 提供"建议操作"字段，指导用户下一步

# 4. **详细解决方案**

## 4.1 **报错提示维护**

### 4.1.1 模块描述
本PRD为技术改造，而非具体功能。

### 4.1.2 输入
**依赖数据**：
- 错误码定义（后端预置，包含错误码、模块、占位符数量、占位符说明）
- 默认报错文案（后端预置，中英文）

**表结构**（复用现有值集表）：
- `collection_management`：值集主表
- `collection_value`：值集明细表

**字段说明**：
仅用于解释逻辑，若开发过程中有变动需要，应以实际情况为准：
| 字段名 | 中文名 | 类型 | 说明 |
|--------|--------|------|------|
| collection_code | 值集编码 | VARCHAR | 固定为"ERROR_MESSAGE" |
| value_code | 错误码 | VARCHAR | 如 MDS001、SOP002 |
| value_name_cn | 中文文案 | TEXT | 用户可见的中文报错提示 |
| value_name_en | 英文文案 | TEXT | 用户可见的英文报错提示 |
| remark | 占位符说明 | TEXT | 如"{0}=物料编码, {1}=工厂编码" |
| attribute1 | 模块 | VARCHAR | 如 MDS、SOP、MPS |
| attribute2 | 占位符数量 | INT | 如 0、1、2 |

### 4.1.3 输出
**对外产出**：
- 报错文案配置数据（存储在 `collection_value` 表）
- 供各业务模块调用的报错文案（通过接口或缓存提供）

### 4.1.4 业务流程
1. 业务管理员进入"系统管理 > 值集管理 > 报错提示配置"
2. 在列表中查询错误码（支持按错误码、模块、关键词查询）
3. 点击"编辑"按钮，弹出编辑框
4. 系统显示当前错误码的占位符说明（如"{0}=物料编码"）
5. 管理员编辑中英文报错文案
6. 点击"保存"，系统校验占位符数量是否匹配
7. 校验通过后保存，立即生效；校验失败则提示错误

### 4.1.5 业务规则
**查询规则**：
- 支持按错误码精确查询（如输入"MDS001"）
- 支持按模块筛选（如选择"MDS"，显示所有 MDS 开头的错误码）
- 支持按关键词模糊查询（在中英文文案中搜索）

**编辑权限**：
- 仅"系统管理员"角色可编辑报错文案
- 错误码、模块、占位符数量、占位符说明为只读字段，不可编辑

**占位符校验规则**：
- 保存时系统自动检查文案中的占位符数量是否与系统定义一致
- 若不一致，提示："占位符数量不匹配，当前错误码支持 N 个占位符（{0}、{1}...），请调整后保存"
- 占位符必须连续编号（{0}、{1}、{2}...），不能跳号（如不能只有 {0} 和 {2}）
- 占位符格式必须为 `{数字}`，不支持其他格式

**占位符使用约束**：
- **核心约束**：占位符数量由后端代码决定，用户只能调整文案内容和占位符位置，不能随意增减占位符
- **已有占位符的报错**：用户可调整文案和占位符位置，但占位符数量必须保持一致
  - 示例：原文案"物料{0}不存在"，可改为"物料编码{0}在系统中不存在，请检查后重试"，但不可改为"物料不存在"（删除了{0}）
- **无占位符的报错**：用户只能调整文案内容，不能新增占位符
  - 示例：原文案"系统繁忙，请稍后重试"，可改为"系统正在处理中，请稍后重试"，但不可改为"系统繁忙{0}，请稍后重试"（新增了{0}）

**多语言规则**：
- 中英文文案必须同时维护，不能只填一个
- 保存时校验中英文文案是否都已填写

**生效规则**：
- 保存后立即生效，无需重启系统
- 建议使用缓存机制，避免每次报错都查询数据库

## 4.2 **报错提示展示规范**

### 4.2.1 模块描述
本模块为修改模块，规范各业务模块中报错提示的展示方式。所有业务模块在触发错误时，必须按统一规范展示报错信息。

### 4.2.2 输入
**依赖数据**：
- 错误码（由业务逻辑触发时指定）
- 占位符参数值（如物料编码、工厂编码等）
- 报错文案配置（从值集表或缓存中加载）
- 用户语言设置（中文/英文）

### 4.2.3 输出
**对外产出**：
- 前端界面显示的报错提示（包含错误码、业务对象、问题描述、下一步指引）

### 4.2.4 业务流程
1. 用户在业务模块中执行操作（如保存、提交、计算等）
2. 系统检测到错误条件（如参数缺失、数据不存在、状态不允许等）
3. 系统根据错误类型确定错误码（如 MDS001）
4. 系统从值集表或缓存中加载该错误码的报错文案（根据用户语言选择中文或英文）
5. 系统将占位符参数值填充到文案中（如将物料编码填入{0}）
6. 系统在前端界面显示完整的报错提示

### 4.2.5 业务规则
**报错信息必须包含的要素**：
- **错误码**：格式为 [错误码]，如 [MDS001]
- **业务对象**：明确指出哪个对象/参数出了问题（如"物料"、"计划版本"）
- **问题描述**：用业务语言说明问题（如"不存在"、"格式不正确"、"已锁定"）
- **下一步指引**：告诉用户该做什么（如"请前往…配置"、"请稍后重试"）

**按场景的报错文案规范**：
| 场景 | 文案结构 |
|------|----------|
| 参数缺失 | [错误码] 「{参数名}」不能为空 |
| 参数格式/范围 | [错误码] 「{参数名}」格式不正确或超出有效范围，请检查后重试 |
| 配置缺失 | [错误码] 尚未配置「{配置项}」，请前往「{菜单路径}」完成配置后重试 |
| 数据不存在 | [错误码] {业务对象}不存在，请确认后重试 |
| 数据已存在 | [错误码] 「{对象标识}」已存在，请勿重复添加 |
| 状态不允许 | [错误码] {业务对象}已{状态}，无法{操作} |
| 删除限制 | [错误码] {业务对象}存在关联数据，无法删除，请先解除关联 |
| 并发冲突 | [错误码] {业务对象}已被他人修改，请刷新后重试 |
| 权限不足 | [错误码] 无权限执行该操作，请联系管理员 |
| 系统错误 | [错误码] 系统繁忙，请稍后重试。如有问题请联系管理员，并提供错误码 |

**禁止的报错方式**：
- 只说"操作失败"、"运行失败"，不说明具体问题
- 透传技术信息（如"NullPointerException"、"SQLException"、SQL 语句）
- 参数缺失不指明参数名
- 配置缺失不指明配置项和路径

# 5. **未来扩展说明**
## 5.1 报错提示分级管理
当前版本所有报错提示均为"错误"级别，未来可扩展支持多级别报错：
- **错误**（Error）：阻断操作，必须解决才能继续
- **警告**（Warning）：不阻断操作，但提示用户注意
- **信息**（Info）：提示性信息，不影响操作
不同级别用不同颜色区分（如错误用红色、警告用黄色、信息用蓝色）。

## 5.2 报错提示历史记录
当前版本不记录报错提示的修改历史，未来可扩展支持：
- 记录每次修改的时间、修改人、修改前后内容
- 支持版本回滚
- 支持审计查询
此功能优先级较低，可在后续版本中实现。
