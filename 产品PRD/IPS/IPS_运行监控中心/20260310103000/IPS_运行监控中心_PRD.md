# 1. **需求概述**

## 1.1 背景及问题

### 1.1.1 背景

悠桦林 SCP 平台包含生产排程、主计划、产销协同、需求预测、物料需求计划、集成平台等业务模块。

* **业务上存在的大量“长时任务”**：历史需求清洗、供需匹配计算、规则式排程、产销计划报告生成、安全库存计算、接口同步等，往往一次运行就需要几分钟甚至更久。

* **共性的需求**：这类任务都需要清晰的**运行入口**、直观的**进度监控**以及统一的**结果查看**方式，方便业务和运维人员日常巡检与排错。

* **当前现状**：有部分日志功能，但尚未形成统一的平台级能力和交互规范。

### 1.1.2 问题

#### 1.1.2.1 运行入口与监控方式不统一

各模块的长时任务入口分散在不同菜单与页面，运维与业务人员无法在同一视图下查看「当前模块下有哪些可运行功能」「各功能最近运行情况如何」。

**影响**：跨模块巡检成本高，无法快速回答「今天跑了什么、哪些还在跑」。

#### 1.1.2.2 任务状态与结果展示缺乏统一语义



不同模块对运行结果的处理方式不一致：有的仅提供简单日志，有的仅有任务列表，缺乏统一的**任务状态**（成功 / 运行中 / 失败 / 已终止）与**结果视图**（输入快照、运行日志、计算结果、摘要说明）。

**影响**：排查失败任务、对比历史运行、审计追溯时体验不一致，难以形成统一运维规范。

#### 1.1.2.3 平台化能力缺失

部分模块已有较成熟的运行结果中心交互与视觉规范，但未抽象为平台级能力，其他模块无法复用。

**影响**：重复建设、设计与交互不统一，后续扩展成本高。

## 1.2 目标

* **建立平台级「运行监控中心」**：将各业务模块的一次性长时任务统一到同一交互范式与视觉规范下，通过**功能卡片 + 功能详情页 + 运行配置/进度弹窗 + 执行历史列表**的标准结构，支持用户快速定位功能、查看运行概览、发起运行、查看或终止进度、下载算法文件或报错日志。

* **支持高数据量场景**：执行历史支持**分页**（每页条数可配置）与**筛选**（任务 ID、操作人、日期范围、任务状态），保证大量历史记录下的体验与性能。

* **与后端解耦**：本 PRD 描述业务逻辑与用户交互，不绑定具体前端技术实现；正式产品可在 **scp-foundation** 等仓库中按现有技术架构（如 Vue、微前端）实现，并与中台/算法服务对接。

* 当前以**原型与交互闭环**为主，运行进度与结果数据可为模拟数据；与后端 API 的对接为后续迭代。

# 2. **详细解决方案**

## 2.1 **运行监控中心**

### 2.1.1 模块描述

本模块在 SCP 平台中提供一个统一的**运行监控中心**，让用户在同一入口下完成从“发现可运行功能”到“发起运行、看进度、看结果”的完整链路。

* **主要页面形态**：一个主页面（模块功能卡片列表）+ 若干弹窗（运行配置、运行进度、执行节点详情等）。

* **核心职责**：

  * 按业务模块展示可监控功能卡片，帮助用户快速知道“这个模块下能跑什么”。

  * 进入功能详情后，展示该功能的运行概览和执行历史。

  * 支持用户配置并发起运行、查看或终止运行进度、查看执行节点与结果详情。

* **主要用户**：运维人员、业务配置人员、实施与支持人员。

* **期望结果**：用户可以在一条清晰的业务路径中完成「选模块 → 选功能 → 看概览 → 发起运行 → 看进度/终止 → 查历史/下载文件」。

### 2.1.2 输入

#### 2.1.2.1 业务与配置层面

* **业务模块与可监控功能**：平台需维护「哪些业务模块接入运行监控中心」「每个模块下有哪些可监控功能」；每个功能有名称、简要描述、运行按钮文案（如「执行清洗」「计算安全库存」）及运行前需填写的参数（如数据源、日期范围、计划版本等），部分功能还需选择运行模式（增量/全量）、策略模式（规则/优化）、策略名称。

* **执行步骤定义**：每个功能对应一套执行步骤（用于进度展示与节点详情），例如「数据加载 → 校验 → 计算 → 写回」；每步有名称与默认说明文案。

#### 2.1.2.2 用户录入（运行前）

* **通用配置**（部分功能展示）：运行模式、策略模式、策略名称。

* **功能差异化参数**：如需求预测的数据源与数据范围、产销协同的计划版本与报告周期、排程的排程规则与订单范围等；必填项在提交前需校验。

#### 2.1.2.3 系统与权限（后端对接时）

* 运行历史数据：任务 ID、所属模块与功能、开始时间、结束时间、耗时、操作人、任务状态、进度信息、计划版本/需求版本（若模块支持）、输入信息快照、运行日志、计算结果、摘要等。

* 用户权限：运行、终止、下载算法文件、下载报错日志等操作需按权限控制。

### 2.1.3 输出

#### 2.1.3.1 用户可见的页面与反馈

* **模块功能卡片列表页**：当前模块下的功能卡片、每张卡片的统计信息（最近运行状态与时间、今日/本周运行次数、本周使用用户数、平均耗时）。

* **功能详情页**：运行概览（今日/本周次数、成功率、平均耗时）、执行历史列表（支持筛选与分页）、单条记录展开后的输入信息、运行日志、计算结果、摘要说明。

* **运行配置弹窗**：用户填写/选择后提交，系统创建运行任务并打开进度弹窗。

* **运行进度弹窗**：整体进度（以「当前步骤/总步骤」形式）、步骤时间线、运行中时可点击「终止」。

* **执行节点详情弹窗**：按步骤展示每步状态（未执行/执行中/已完成/失败/已终止）及说明。

#### 2.1.3.2 导出文件

* **算法文件**：成功任务可下载本次运行的配置/策略快照。

* **报错日志**：失败任务可下载错误明细（如 CSV），含错误级别、模块、错误代码、异常描述等；编码需支持中文。

#### 2.1.3.3 数据与下游（后端对接后）

* 运行记录持久化，供运维、审计与报表使用；中台/算法服务可基于任务 ID 反查详细结果。

### 2.1.4 业务流程

#### 2.1.4.1 模块切换与功能发现

1. 用户在页面顶部通过模块下拉选择业务模块（如物料需求计划、生产排程、产销协同等）。

2. 系统展示该模块下的**功能卡片**列表；每张卡片展示功能名称、描述、最近运行状态与时间、今日/本周次数、本周使用用户数、平均耗时；若该功能当前有正在运行的任务，功能名称右侧展示加载动画。

3) 用户点击某张卡片，进入该功能的**功能详情页**。

#### 2.1.4.2 发起运行

1. 用户在功能详情页点击**运行按钮**（按钮文案按功能配置，如「执行清洗」「计算安全库存」）。

2. 系统弹出**运行配置弹窗**：展示通用配置（若该功能需要）与功能差异化参数；必填项未填写时不允许提交。

3) 用户点击「确认并开始」后，系统创建一条**运行中**的任务记录，并刷新执行历史与概览；配置弹窗关闭，**运行进度弹窗**自动打开。

4) 进度弹窗展示整体进度（以「当前步骤/总步骤」形式）与步骤时间线；进度随任务推进更新（原型阶段可为模拟推进）。

#### 2.1.4.3 查看进度与终止

1. 运行中任务在功能详情页的**执行历史列表**中显示为「运行中」，进度以步骤分数展示（如 2/4）；用户可点击该任务行的「查看进度」再次打开进度弹窗。

2. 进度弹窗在「查看运行中任务」时，右上角展示\*\*「终止」按钮\*\*；用户点击后，该任务状态变为**已终止**，弹窗关闭，历史列表与卡片统计刷新。

3) 用户也可直接关闭进度弹窗，仅关闭视图，任务在后台继续；任务结束后状态由系统更新为成功或失败。

#### 2.1.4.4 查看结果与排查失败

1. **成功任务**：列表行以绿色主题与勾号展示；用户可点击「算法文件」下载配置快照，展开行可查看输入信息、运行日志、计算结果、摘要说明。

2. **失败任务**：列表行以红色主题与叉号展示；用户可点击「报错日志」下载错误明细；展开行可查看日志与输入信息以辅助排查。

3) **已终止任务**：列表行以黄色主题与减号展示；时长列显示「已终止」；仅提供「已终止」标识与「节点」按钮，不提供「算法文件」「报错日志」按钮，操作区宽度与成功/失败行保持一致，避免布局错位。

#### 2.1.4.5 执行节点查看

1. 用户在执行历史任意任务行点击「节点」按钮，打开**执行节点详情弹窗**。

2. 弹窗内按该功能的执行步骤顺序展示每步状态：未执行、执行中、已完成、失败、已终止。

3) 运行中任务高亮当前步骤（旋转图标 +「计算中…」）；已终止任务高亮终止时所在步骤（减号 +「已终止」）；失败任务高亮失败步骤。

#### 2.1.4.6 筛选与分页

1. 用户在功能详情页使用筛选区：输入任务 ID 或操作人关键字、选择开始日期起止、选择任务状态（全部/成功/失败/运行中/已终止），点击「重置」清空筛选。

2. 列表仅展示当前功能、且满足筛选条件的记录；默认按开始时间倒序。

3) 分页：每页条数可选（如 10/20/50/100），默认 20；提供首页、上一页、下一页、末页；底部展示「显示第 X–Y 条，共 Z 条」及当前页/总页数。切换每页条数或筛选条件时重置到第 1 页。

### 2.1.5 业务规则

#### 2.1.5.1 模块与功能

* **若**某业务模块需接入运行监控中心，**则**需在平台侧维护该模块及其下的可监控功能列表；每个功能需有名称、描述、运行按钮文案、运行前参数定义（含是否必填），以及是否展示运行模式/策略等通用配置。

* **若**某功能需要运行前填写通用配置（运行模式、策略模式、策略名称），**则**在运行配置弹窗中展示该区域；**否则**仅展示该功能差异化参数。

#### 2.1.5.2 任务标识与版本

* 任务 ID 需全局唯一，且能从 ID 上区分所属模块/功能（如带模块或功能前缀）；具体格式由实现方与后端约定。

* **若**业务模块具备计划版本/需求版本概念（如主计划、产销协同、需求预测、物料需求计划），**则**运行记录中记录并展示计划版本、需求版本；**若**模块无版本概念，**则**不展示版本相关列。

#### 2.1.5.3 任务状态与状态迁移

* **任务状态**及用户可见含义、操作约定：

| 状态  | 含义        | 列表展示与可操作项                                       |
| --- | --------- | ----------------------------------------------- |
| 运行中 | 任务正在执行    | 显示「运行中」、进度为步骤分数；可「查看进度」、可「终止」；卡片上该功能名称右侧显示加载动画。 |
| 成功  | 任务正常完成    | 绿色主题、勾号；可下载「算法文件」、展开查看结果与日志。                    |
| 失败  | 执行过程异常或失败 | 红色主题、叉号；可下载「报错日志」、展开查看日志与输入。                    |
| 已终止 | 用户主动终止    | 黄色主题、减号；时长列显示「已终止」；仅可查看节点与展开详情，不可下载算法文件/报错日志。   |

* **状态迁移**：

  * **当**任务正常执行完毕，**则**由运行中 → 成功；记录结束时间、耗时、摘要（若有）。

  * **当**执行过程发生异常或失败，**则**由运行中 → 失败；保留当前步骤与日志。

  * **当**用户在运行进度弹窗点击「终止」，**则**由运行中 → 已终止；弹窗关闭并刷新列表与概览。

#### 2.1.5.4 校验规则

* **运行配置弹窗**：功能定义的必填项未填写时，**禁止**提交；需提示用户补全必填项。

* **筛选**：日期范围若只填开始未填结束（或反之），按已填单边筛选；任务 ID/操作人关键字为空时，表示该维度不限制。

* **分页**：当前页号大于总页数时（如筛选后结果变少），自动修正到最后一页。

#### 2.1.5.5 统计口径

* **今日运行次数**：当前功能下、开始日期为系统当日日期的任务数量。

* **本周运行次数**：当前功能下、开始时间落在「本周一 00:00:00」至「当前时刻」之间的任务数量。

* **本周使用用户数**：上述「本周」任务中，操作人去重后的个数。

* **平均耗时**：本周内、状态为成功的任务，其耗时的算术平均；展示为「Xs」或「Xm Ys」；无有效记录时展示「-」或空。

* **最近运行状态/时间**：当前功能下按开始时间倒序第一条任务的状态与开始时间（展示到分或秒，由产品约定）。

#### 2.1.5.6 权限与安全（后端对接时落实）

* 运行、终止、下载算法文件、下载报错日志等操作需根据当前用户权限控制；无权限时按钮不展示或禁用。

* 报错日志可能包含敏感信息；导出前需做脱敏与权限校验，具体由后端与安全规范约定。

#### 2.1.5.7 性能与数据量

* 执行历史**必须**分页加载；默认每页条数与可选档位由产品约定（如 20 条，可选 10/20/50/100）；后端需支持按页码、每页条数、筛选条件返回当页数据与总条数。

* 历史数据量极大时，建议后端支持排序与分页游标；前端可考虑虚拟滚动（是否实现按项目约定）。


### 2.1.6 并发控制规范

#### 2.1.6.1 并发模式定义

**并发模式枚举**（`ConcurrencyModeEnum`）：

| 模式编码 | 模式名称 | 业务含义 | 技术实现方式 |
|---------|---------|---------|-------------|
| `PARALLEL` | 可并行 | 同一功能可同时运行多个任务，无限制 | 不使用分布式锁，直接提交任务 |
| `EXCLUSIVE` | 不可并行 | 同一功能同时只能运行1个任务，第二个请求直接拒绝 | Redis分布式锁 |
| `QUEUEABLE` | 可排队 | 同一功能同时只能运行1个任务，但第二个请求可排队等待 | Redis分布式锁 + Redis队列 |
| `LIMITED_PARALLEL` | 限制并发数 | 同一功能最多同时运行N个任务（N可配置），超出限制的请求被拒绝或排队 | Redis计数器 + 分布式锁 |

**配置方式**：
- 在功能配置中声明并发模式（`concurrencyMode`）与并发数限制（`maxConcurrency`，仅 `LIMITED_PARALLEL` 模式需要）
- 锁key格式：`concurrency:{functionCode}:{scenario}:lock`
- 队列key格式：`concurrency:{functionCode}:{scenario}:queue`
- 锁超时时间：15分钟（可配置）

#### 2.1.6.2 各并发模式的业务规则

**PARALLEL（可并行）**：
- 适用场景：数据查询、报表生成等无状态操作
- 前端交互：无限制，用户可随时发起运行
- 后端实现：直接提交任务到线程池执行

**EXCLUSIVE（不可并行）**：
- 适用场景：数据写入、状态修改等有状态操作（如MRP计算、排程计算）
- 前端交互：若检测到锁已被占用，后端返回错误提示（如"上一次MRP正在运行中，请稍后再试！"），前端在运行配置弹窗提交时展示该提示并阻止提交；功能卡片上该功能名称右侧展示加载动画
- 后端实现：使用Redis分布式锁（`setIfAbsent`），任务完成后在 `finally` 块中释放锁

**QUEUEABLE（可排队）**：
- 适用场景：用户可接受等待的功能，任务执行时间较短（如5-10分钟内）
- 前端交互：若锁已被占用，后端返回排队位置（如"当前排队位置：第3位"），前端在运行配置弹窗提交后展示排队提示并关闭弹窗；功能详情页执行历史列表中，该任务显示为"排队中"状态；轮询接口返回排队位置，前端实时更新
- 后端实现：使用Redis分布式锁 + Redis队列，任务完成后从队列中取出下一个任务并自动启动

**LIMITED_PARALLEL（限制并发数）**：
- 适用场景：需要控制系统资源占用的功能（如算法计算、大数据处理）
- 前端交互：若拒绝模式，返回错误提示（如"当前已有3个任务在运行，请稍后再试"）；若排队模式，同 `QUEUEABLE` 模式
- 后端实现：使用Redis计数器记录当前运行中的任务数，任务提交前检查计数器值是否小于 `maxConcurrency`

#### 2.1.6.3 死锁管理与异常处理

**死锁预防**：
- 锁超时时间：15分钟（可配置），防止异常情况下锁未释放
- 锁释放：任务执行完成（成功/失败/已终止）后，**必须**在 `finally` 块中释放锁

**死锁检测**：
- 后端定期检查锁的持有时间，若超过阈值（如20分钟）且任务状态异常，自动释放锁并记录告警日志

**死锁处理**：
- 若发生死锁导致任务失败，将任务状态置为"失败"
- 在运行日志与报错日志中记录死锁相关信息（涉及的资源、等待时长、异常码等）
- 前端在失败任务的"报错日志"下载文件中体现

#### 2.1.6.4 并发控制框架

**核心类**：`ConcurrencyControlManager`

**关键方法**：
- `tryAcquire(functionCode, scenario, taskId)`：尝试获取运行许可
- `release(functionCode, scenario, taskId)`：释放运行许可
- `enqueue(functionCode, scenario, taskId)`：将任务加入排队队列
- `dequeue(functionCode, scenario)`：从排队队列中取出下一个任务
- `getRunningCount(functionCode, scenario)`：获取当前运行中的任务数
- `getQueuedCount(functionCode, scenario)`：获取排队中的任务数

### 2.1.7 实时推送机制

#### 2.1.7.1 推送方式

**方式1：前端定时轮询**（当前实现）

- **轮询接口**：`GET /progressListener/process/getProgressByTaskId?taskId={任务ID}`
- **轮询间隔**：推荐 1～2 秒
- **轮询生命周期**：进度弹窗打开时启动轮询，弹窗关闭或任务结束时停止轮询；组件销毁时（`beforeDestroy`）必须清理定时器，避免内存泄漏
- **适用场景**：浏览器不支持WebSocket、网络限制、后端未启用WebSocket等情况

**方式2：WebSocket实时推送**（推荐实现）

- **WebSocket端点**：`/ws/progress/{taskId}`
- **连接建立**：前端通过 `new WebSocket('ws://host/ws/progress/{taskId}')` 建立连接，后端验证taskId有效性
- **消息推送**：任务进度更新时，后端主动推送消息到前端
- **适用场景**：需要更实时的推送能力（如秒级更新、多用户协同监控）

#### 2.1.7.2 WebSocket消息格式

**消息格式**（JSON）：

```json
{
  "type": "PROGRESS_UPDATE",
  "taskId": "xxx",
  "timestamp": 1678886400000,
  "data": {
    "status": "PROCESSING",
    "overallProgress": 75,
    "currentNodeName": "需求汇总与净需求计算",
    "nodeListByTaskId": [...]
  }
}
```

**消息类型枚举**：

| 类型 | 说明 | 触发时机 |
|------|------|---------|
| `PROGRESS_UPDATE` | 进度更新 | 节点进度更新时 |
| `STATUS_CHANGE` | 状态变更 | 任务状态变为成功/失败/已终止时 |
| `NODE_START` | 节点开始 | 某节点开始执行时 |
| `NODE_COMPLETE` | 节点完成 | 某节点执行完成时 |
| `ERROR` | 错误消息 | 任务执行失败时 |
| `HEARTBEAT` | 心跳消息 | 每30秒发送一次，保持连接活跃 |

#### 2.1.7.3 消息可靠性保障

**断线重连**：
- 前端检测到连接断开时，自动尝试重连（最多5次，间隔3秒）
- 重连失败后，降级到轮询模式

**消息补发**：
- 前端重连成功后，后端发送当前最新进度（避免重连期间的进度丢失）
- 前端对比本地缓存的进度，若有差异则更新

**心跳检测**：
- 后端每30秒发送心跳消息（`HEARTBEAT`）
- 前端收到心跳后响应 `PING`
- 若60秒内未收到心跳，前端主动断开并重连

#### 2.1.7.4 降级策略

**配置开关**：
- 后端提供配置项（如 `websocket.enabled=true`），控制是否启用WebSocket
- 前端根据后端配置或连接结果，决定使用WebSocket还是轮询

**降级逻辑**：
- 前端优先尝试WebSocket连接
- 若WebSocket不可用（浏览器不支持、网络限制、后端未启用等），自动降级到轮询模式
- 降级后的轮询逻辑与现有实现保持一致

### 2.1.8 终止与异常跳过

#### 2.1.8.1 终止接口规范

**接口路径**：`POST /progressListener/process/interruptTask`

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| taskId | String | 是 | 任务ID |

**错误码定义**：

| 错误码 | 错误信息 | 说明 |
|--------|---------|------|
| IPS010 | 任务不存在或已结束 | taskId无效或任务已完成/失败/已终止 |
| IPS011 | 当前节点不可中断 | 当前节点的 `isInterruptible()` 返回false |
| IPS012 | 任务状态不允许终止 | 任务状态不是"运行中" |
| IPS013 | 终止失败，请稍后重试 | Future.cancel()返回false或其他异常 |

**前端交互流程**：
1. 用户在进度弹窗点击"终止"按钮
2. 前端弹出二次确认对话框："确认终止当前运行？终止后已产生的中间结果可能丢失。"
3. 用户确认后，前端发送终止请求
4. 后端返回成功：前端关闭进度弹窗，刷新执行历史列表，该任务显示为"已终止"
5. 后端返回失败（如IPS011）：前端展示错误提示，保持任务为运行中状态

#### 2.1.8.2 节点级可中断控制

**节点枚举接口**（`ProgressNodeEnum`）：

每个功能的运行节点需定义 `isInterruptible()` 属性，标识该节点是否可中断。

**可中断节点**（`isInterruptible = true`）：
- 适用场景：数据加载、校验、计算类步骤（如"物料分层排序"、"基础数据获取与校验"、"执行供需平衡计算"）
- 终止逻辑：后端收到终止请求后应尽快停止执行（通过 `Future.cancel(true)` 中断线程），将任务状态置为"已终止"（`INTERRUPT`），并清理临时数据或释放资源

**不可中断节点**（`isInterruptible = false`）：
- 适用场景：结果写回、事务提交类步骤（如"建议计划生成与报告汇总"）
- 终止逻辑：若正在执行则不应立即终止，以免数据不一致；后端应返回错误提示（如"当前节点 '建议计划生成与报告汇总' 不可中断"），前端在进度弹窗中展示该提示并保持任务为运行中状态

#### 2.1.8.3 异常跳过框架

**节点枚举接口扩展**：

在 `ProgressNodeEnum` 中增加 `isSkippableOnError()` 属性，标识该节点失败时是否可跳过并继续执行下一节点。

```java
public interface ProgressNodeEnum {
    // 原有方法...

    /**
     * 是否可在异常时跳过
     * @return true表示该节点失败时可跳过并继续执行下一节点，false表示失败时终止整个任务
     */
    default boolean isSkippableOnError() {
        return false;  // 默认不可跳过
    }
}
```

**异常跳过逻辑**：
- 在 `ProcessProgressManager` 中捕获节点异常时，根据 `isSkippableOnError()` 属性决定是否继续执行下一节点
- 若可跳过：记录失败但继续执行下一节点，在运行日志中明确记录"节点 X 失败但已跳过"类信息
- 若不可跳过：终止整个任务，将任务状态置为"失败"

**日志记录规范**：

```
[2026-03-10 10:30:45] [INFO] 节点开始执行：物料分层排序
[2026-03-10 10:30:50] [SUCCESS] 节点执行成功：物料分层排序，耗时5秒
[2026-03-10 10:30:50] [INFO] 节点开始执行：基础数据获取与校验
[2026-03-10 10:31:00] [WARN] 节点执行失败但已跳过：基础数据获取与校验，原因：部分BOM数据缺失（10条），已跳过并继续执行
[2026-03-10 10:31:00] [INFO] 节点开始执行：执行供需平衡计算
[2026-03-10 10:32:00] [SUCCESS] 节点执行成功：执行供需平衡计算，耗时60秒
```

**前端展示**：
- 在执行节点详情弹窗中，失败但已跳过的节点显示为黄色警告图标（而非红色错误图标）
- 节点说明文案显示："节点执行失败但已跳过：部分BOM数据缺失（10条）"
- 在运行日志展开区，该节点的日志行显示为黄色背景

**配置方式**：

方式1：在节点枚举中硬编码（推荐用于固定规则）
方式2：通过配置表动态配置（推荐用于需要灵活调整的场景）

#### 2.1.9.1 节点级业务指标

**记录格式**：

```

```

**示例**：

**记录方式**：

#### 2.1.9.2 性能监控接口（按需触发）

**接口路径**：`GET /progressListener/process/getPerformanceMetrics`

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| taskId | String | 是 | 任务ID |

**响应数据**：

包含JVM内存使用情况（堆内存、非堆内存、GC次数与耗时）、线程情况（线程数、峰值线程数、守护线程数）、数据库连接池情况（活跃连接数、空闲连接数、最大连接数）、节点执行情况（各节点耗时、处理数量、进度百分比）。

#### 2.1.9.3 前端展示

**进度弹窗中增加"性能详情"按钮**：

- 位置：进度弹窗右上角，"终止"按钮旁边
- 样式：蓝色按钮，图标为仪表盘
- 点击后：弹出性能详情对话框，展示JVM、线程、数据库连接池等指标

**性能详情对话框**：

- 标题：性能监控 - 任务ID
- 内容：JVM内存使用情况、线程情况、数据库连接池情况、节点执行情况
- 底部：刷新按钮、关闭按钮

#### 2.1.9.4 性能问题的体现方式

基于现有业务模块的实现经验，若任务运行时间过长（如超过 2 小时），用户可通过以下方式感知与排查：

- **执行历史列表**：时长列实时更新（通过轮询刷新），用户可直观看到任务已运行多久；若超出预期，可点击"查看进度"或"节点"查看当前卡在哪个步骤
- **执行节点详情**：展示每个节点的执行状态、开始时间、结束时间、耗时；若某节点长时间处于"执行中"，用户可据此判断该节点可能存在性能瓶颈
- **节点日志内容**：后端可在节点进度信息中记录关键指标，如"当前处理 X 条 / 总 Y 条"、"数据加载耗时 X 秒"等
- **运行日志**：后端可在日志中记录关键节点的开始/结束时间、处理数量、异常信息等

#### 2.1.9.5 后续增强方向

**独立的性能监控模块**（后续迭代规划）：

- 实时性能监控：集成Prometheus + Grafana，实时采集与展示系统指标
- 慢查询分析：集成数据库慢查询日志，自动分析与告警
- 数据卡点定位：通过分布式追踪（如Zipkin、SkyWalking），定位性能瓶颈
- 性能告警：当CPU、内存、线程数等指标超过阈值时，自动发送告警通知

**当前 PRD 范围**：本 PRD 主要聚焦于业务层面信息，暂不包含系统级性能监控（如 CPU、内存、线程、数据库连接池等技术指标）的实时展示与告警。若需要更细粒度的性能监控与告警，应作为独立的性能监控模块在后续迭代中规划。

## 2.2 ⭐**新功能接入与运行节点规范**



* **新功能如何接入运行监控中心**（需要提供哪些信息、遵守哪些约定）；

* 在运行监控过程中，**如何划分“运行节点”**，以及每个节点里应该包含哪些成功信息与报错/警告信息。

运行节点是监控过程里最重要的信息载体。设计合理的节点，可以让运维与业务人员快速看懂：任务现在跑到哪一步、每一步大致做了什么、在哪一步出现了异常或告警，从而更快判断任务健康度并定位问题。



### 2.2.1 新功能接入运行监控中心的流程与要求



当某个新功能开发完成后希望接入运行监控中心进行统一管理时，需满足以下通用要求（不绑定具体实现形式）：



* **归属与展示**

  * 明确该功能所属的**业务模块**（如生产排程、主计划、需求预测等）；若该模块尚未接入运行监控中心，需先完成模块级接入。

  * 在运行监控中心内为该功能提供：**功能名称**、**简要描述**、**运行入口文案**（如「执行 XX」「计算 XX」），以便在功能卡片与详情页中展示。

* **运行前参数**

  * 定义该功能在「发起运行」前需要用户选择或填写的**业务参数**（如数据范围、版本、规则、范围等），并标明必填项；运行监控中心将统一以弹窗形式收集这些参数并在运行记录中留存快照。

* **运行节点定义（核心）**

  * 按 2.2.2、2.2.3、2.2.4 的规范，为该功能定义**运行节点**（即执行步骤/子步骤）及每个节点可输出的**运行成功信息**与**报错/警告信息**；以便在进度展示、执行节点详情及运行日志中一致呈现。

* **任务标识与状态**

  * 该功能产生的每次运行需生成**唯一任务标识**，并随执行过程更新**任务状态**（运行中 / 成功 / 失败 / 已终止）与**当前节点/进度**；运行监控中心依赖这些信息驱动进度条、步骤时间线与历史列表。

* **可选的通用配置**

  * 若该功能需要运行模式（如增量/全量）、策略模式（如规则/优化）、策略名称等通用配置，需声明并在运行配置弹窗中展示；否则仅展示功能专属参数。

接入完成后，该功能即可在运行监控中心对应模块下以卡片形式展示，用户可从卡片进入详情、发起运行、查看进度与节点、查看历史与下载算法文件/报错日志。具体配置方式（如配置表、配置文件、管理后台）由实现方在落地时决定，本 PRD 仅约定能力与信息要求。



### 2.2.2 运行节点在监控中的定位



运行监控中心认为：**运行节点是监控过程中最重要的信息**。



* **为什么节点最重要**

  * 任务级状态（成功/失败/运行中/已终止）只能回答「任务有没有跑完、结果如何」，无法回答「卡在哪一步、哪一步有警告、哪一步产生了多少量」。

  * 节点将一次运行拆解为**可观测的步骤**，每个步骤可携带**执行时间**、**成功类指标**（处理条数、生成数量、通过率等）以及**报错/警告信息**（警告数、异常条数、逾期率等）。这样运维与业务人员既能看整体进度，也能在问题发生时快速定位到具体步骤与原因。

* **节点的使用场景**

  * **运行中**：在进度弹窗中按节点展示当前执行到哪一步，当前节点可展示「计算中…」等状态及该节点已产生的部分信息。

  * **运行结束后**：在执行节点详情（及运行日志）中按节点顺序展示每步的**运行成功信息**与**报错/警告信息**，支持展开/折叠以兼顾总览与细查；失败或已终止时，可高亮异常所在节点。

因此，新功能接入运行监控中心时，**必须**定义其运行节点及节点内信息规范，而不是仅上报任务级状态。本能力框架在设计上参考了业界排程/计划类软件（如 Asprova）中的「消息窗口」与运行日志思路：按层级展示步骤、每步带执行时间与数量/警告统计，便于用户快速把握运行健康度与问题所在。



### 2.2.3 功能如何划分节点：规范与规则



一个功能应如何划分成多个运行节点，建议遵循以下**通用规范与规则**（不限定具体业务，便于各模块一致落地）：



* **按逻辑步骤划分，而非按技术实现**

  * 节点应对应**业务可理解的步骤**，避免按函数调用、线程等纯技术粒度划分，以便业务与运维能直接理解。

* **顺序与依赖**

  * 节点之间应有**明确的执行顺序**（串行或可描述的依赖）；运行监控中心按该顺序展示步骤时间线，并据此计算「当前步骤/总步骤」类进度。若存在并行分支，需约定在展示层如何归并为一条主时间线。

* **粒度建议**

  * **顶层节点**不宜过多，建议单次运行在 **5～15 个主步骤** 内，便于一屏内浏览；若某步骤本身很复杂，可再拆为**子节点**（见下条）。

  * 单个节点代表**一个有明确起止、可上报执行时间与结果**的单元；过细会导致信息噪音大，过粗则无法发挥「定位到步骤」的价值。

* **支持层级（父节点与子节点）**

  * 允许一个节点下包含**子节点**，形成层级结构。

  * 展示上可支持**展开/折叠**：折叠时仅显示父节点名称与汇总信息，展开时显示各子节点的运行成功信息与报错信息。层级深度建议不超过 2～3 层，以保持可读性。

* **每个节点应可上报的信息**

  * **执行时间**：该节点开始到结束的耗时，便于做性能排查。

  * **成功类指标**：与该节点业务含义一致的数量或比率，用业务术语命名，避免仅写技术指标。

  * **警告/错误类指标**：该节点产生的警告数、错误数或异常条数；若有，应在节点上明确标识并在节点详情或消息列表中展示具体内容。

* **节点命名与说明**

  * 节点名称应**简短且表意清晰**；必要时可为节点配置**默认说明文案**，用于在未产生具体消息时展示。运行过程中或结束后，可用**实际产生的消息内容**覆盖或补充该说明。

以上规则旨在形成**可复用的节点划分规范**：新功能在接入时，按「逻辑步骤、顺序、适度粒度、可选的层级、每节点可上报时间与成功/报错信息」来定义节点，即可与运行监控中心的进度展示、节点详情及日志查看能力对齐。



### 2.2.4 节点内应包含的信息：运行成功信息与报错信息



每个运行节点应能承载以下两类信息，以便运行监控中心统一展示并支持问题定位：



* **运行成功信息**

  * 指该节点**正常完成**时产生的、用于说明「做了什么、结果如何」的信息，包括：执行耗时、处理数量或生成数量、与该节点业务含义一致的业务指标（如比率、平均值等）。展示时可按「信息」类图标或中性样式呈现，用于确认该步骤已成功执行并了解关键结果。

* **报错信息与警告信息**

  * **报错信息**：该节点执行失败或发生异常时的信息（异常码、异常描述、失败条数等）；应导致任务状态变为「失败」或至少在该节点上标记为「失败」，并可在报错日志导出中体现。

  * **警告信息**：该节点执行完成但存在需要关注的问题，包括警告数量、业务异常统计（如逾期数量及占比、不足/异常条数等）。展示时应用**警告/错误**类图标或颜色区分，便于用户快速发现异常节点；在节点详情或消息列表中应能展开查看具体内容。

* **统一展示约定**

  * 运行监控中心对节点内信息的展示不做具体格式强制，但建议：每条信息可带时间戳（可选），便于与日志对齐；成功类与警告/错误类在视觉上可区分；支持按节点折叠/展开，避免单次运行信息过多时难以浏览。

新功能在定义运行节点时，应按上述约定为每个节点设计可输出的**运行成功信息**与**报错/警告信息**，并在实现侧在节点开始/结束或关键检查点上报这些信息，以便运行监控中心统一汇聚和展示，满足「节点是监控过程中最重要信息」的目标。

### 2.2.6 ⭐示例：MRP 运行（供需匹配计算）的节点划分与示例提示词



本小节以 MRP 模块中的「物料需求计算（MRP 运行 / 供需匹配计算）」为例，做两件事：



* 把一个真实功能如何从**数据输入 → 处理 → 数据输出**三大阶段拆解为运行节点，按照 2.2.2～2.2.4 的规范展示出来，作为节点划分的参考示例； &#x20;

* 基于该场景，给出一段**完整的实际提示词示例**，方便开发同事学习。



#### 2.2.6.1 功能与运行流程简述（业务视角）



* **功能名称**：物料需求计算（MRP 运行 / 供需匹配计算）。 &#x20;

* **业务目标**：在既定策略与计划周期下，综合客户订单、制造订单、预测、库存、在途等信息，分层计算物料净需求，生成计划订单/采购建议，并输出短缺、逾期等异常信息。 &#x20;

* **运行前参数**：场景/数据源、MRP 策略、订单类型与范围、计划起止日期等（由前端运行配置弹窗收集）。 &#x20;

* **典型入口与接口位置（示意）**： &#x20;

  * 前端入口：某「MRP 策略维护」页面的「运行」按钮； &#x20;

  * 前端调用：对应的 API 封装文件中某个 `mrpStrategyRun` 或 `run` 方法； &#x20;

  * 后端入口：某 `MrpController.run` 风格的接口（如 `POST /api/mrp/run`），内部调用相应的 Service 进行计算。

#### 2.2.6.2 节点划分示例（按「数据输入 → 处理 → 数据输出」三大步骤展开）



下表将一次 MRP 运行拆解为 8 个业务节点，并在「大类」一列中标明其属于数据输入、处理还是数据输出。各模块在设计自身节点时，可先确定三大步骤，再在每个大步骤下细化类似的业务节点。



| 顺序 | 大类   | 节点名称          | 节点说明（业务语言）                                            |
| -- | ---- | ------------- | ----------------------------------------------------- |
| 1  | 数据输入 | 低阶码校验与环境准备    | 检查是否已完成低阶码计算，校验运行场景、模块与当前系统状态是否允许执行 MRP。              |
| 2  | 数据输入 | 策略与计划周期确定     | 读取默认或指定的 MRP 策略，确定计划起止时间（历史回溯 + 计划结束），锁定本次运行范围。       |
| 3  | 数据输入 | MRP 数据准备      | 按策略与周期加载策略物料、BOM、工艺、库存、在制/在途、客户订单、制造订单等，形成 MRP 运行上下文。 |
| 4  | 处理   | 物料层级排序        | 按低阶码对物料排序，确定自上而下的计算顺序，保证上下层依赖关系被正确处理。                 |
| 5  | 处理   | 需求汇总与净需求计算    | 对每个库存点物料汇总独立需求与非独立需求，结合库存与已有供应，完成净需求计算。               |
| 6  | 处理   | 计划供应生成与下层需求推导 | 按策略生成/调整计划订单、采购建议等供应，并由当前层物料推导下层物料的非独立需求，同时处理冗余供应。    |
| 7  | 数据输出 | 结果持久化与计划生成    | 将净需求与计划供应落地为系统中的 MRP 结果（计划订单/采购建议、Pegging 关系、关键指标等）。  |
| 8  | 数据输出 | 分配关系重建与工单同步   | 清理旧分配关系，重建物料分配关系，并将新增/更新的工单同步到执行系统或下游模块。              |



**各节点的运行成功信息与报错/警告信息（对应 2.2.4）**



* **节点 1「低阶码校验与环境准备」** &#x20;

  * 成功信息示例：低阶码校验耗时、当前低阶码状态（已完成/无需计算）、允许执行的模块/场景列表。 &#x20;

  * 报错/警告信息示例：低阶码未执行或正在执行、模块与策略模式不匹配（如非 MRP 模块却使用优化算法策略）、系统处于不允许运行的冻结状态等。

* **节点 2「策略与计划周期确定」** &#x20;

  * 成功信息示例：选用的策略编码与名称、计划起止日期、回溯期时长、本次运行覆盖的时间跨度。 &#x20;

  * 报错/警告信息示例：找不到可用策略、策略物料范围为空、策略被停用或与当前场景不兼容。

* **节点 3「MRP 数据准备」** &#x20;

  * 成功信息示例：加载的策略物料数量、BOM 校验通过数量、成功加载的库存记录数/订单条数等。 &#x20;

  * 报错/警告信息示例：BOM 校验失败条数、策略物料范围为空、关键主数据缺失（如缺少物料主数据或工艺）的数量等。

* **节点 4「物料层级排序」** &#x20;

  * 成功信息示例：参与计算的物料数、低阶码层数、排序耗时。 &#x20;

  * 报错/警告信息示例：无法获取层级信息导致被跳过的物料数（一般仅在极端数据异常时出现）。

* **节点 5「需求汇总与净需求计算」** &#x20;

  * 成功信息示例：已处理物料数/总物料数、生成净需求条数，可滚动输出「当前已处理 X 条 / 总 Y 条」类进度信息。 &#x20;

  * 报错/警告信息示例：异常需求条数（如超出策略范围、日期不在计划周期内）、被忽略或截断的需求数等。

* **节点 6「计划供应生成与下层需求推导」** &#x20;

  * 成功信息示例：新增/调整的计划供应数量、补充或削减的供应条数、冗余供应被回收的条数。 &#x20;

  * 报错/警告信息示例：因约束（最小批量、容量上限等）无法完全满足的需求条数、冗余供应条数、被迫推迟或提前的需求条数。

* **节点 7「结果持久化与计划生成」** &#x20;

  * 成功信息示例：写入的计划订单/采购建议数量、生成的 Pegging 关系条数、结果表写入耗时、标记为当前版本的计划标识等。 &#x20;

  * 报错/警告信息示例：结果持久化失败、事务回滚、部分结果写入失败条数等。

* **节点 8「分配关系重建与工单同步」** &#x20;

  * 成功信息示例：删除的旧分配关系条数、重建的分配关系条数、成功同步到下游系统的工单数量。 &#x20;

  * 报错/警告信息示例：部分工单同步失败条数、下游系统不可用或响应超时等。

**与 2.2.3、2.2.4 规范的对应关系**



* 以上 8 个节点均为按**业务逻辑步骤**划分的可理解步骤（低阶码校验、策略确定、数据准备、净需求计算、结果落地等），且顺序清晰，符合 2.2.3 中关于「按逻辑步骤划分」「顺序与依赖」的要求。 &#x20;

* 顶层节点数控制在 5～15 之间，粒度适中；其中需求汇总/净需求计算、计划供应生成等节点在需要时可以继续拆分为子节点（例如按低阶码层级或物料分组），满足 2.2.3 中关于「粒度与层级」的建议。 &#x20;

* 每个节点都约定了至少三类信息：执行时间（可通过耗时统计体现）、成功类指标（数量、比率、覆盖范围等）、警告/错误信息（条数与典型原因），对应 2.2.4 中对「运行成功信息」与「报错/警告信息」的要求。 &#x20;

* 从「数据输入 → 处理 → 数据输出」三大阶段出发再向下细化节点的方式，可以作为所有新功能接入运行监控中心时的通用拆分思路：先明确三大阶段，再在每个阶段中定义若干业务可理解的节点，并为每个节点设计可上报的成功/异常信息。

#### 2.2.6.3 示例提示词（MRP 运行场景）



下面是一个可直接复制、用于与 AI 协作拆分「MRP 运行」节点的实际提示词示例，开发同事在使用时可结合自身项目中的实际类名、接口路径做少量调整：



> 「现在有一个长时任务功能，名称是：**物料需求计算（MRP 运行 / 供需匹配计算）**，属于 **物料需求计划（MRP）模块**。 &#x20;
>
> 业务目标：**在既定策略与计划周期下，综合客户订单、制造订单、预测、库存、在途等信息，分层计算物料净需求，生成计划订单/采购建议，并输出短缺、逾期等异常信息。** &#x20;

>
> 运行前需要选择/填写的关键参数包括：**场景/数据源、MRP 策略、计划组织、物料或物料系列范围、计划起止日期、需求版本、库存版本等**。 &#x20;
>
> 运行中需要支持：**分步骤进度展示（数据输入 / 处理 / 数据输出）、节点详情查看、任务中断、成功时的结果概览，失败时的报错日志下载等。** &#x20;

> 主要代码位置： &#x20;
>
> * 前端调用：某 API 封装文件中的 `mrpStrategyRun` 方法（POST 调用 MRP 运行接口）； &#x20;
>
> * 后端入口：某 `MrpController.run` 风格的接口（例如 `POST /api/mrp/run`），内部调用对应的 Service（如某 `MrpRunService`）执行计算逻辑。 &#x20;
>
> * 请你参考（此处放**运行监控中心 PRD 的 2.2.2～2.2.4 规范）**，结合上述信息，帮我拆分 **5～15 个运行节点**，并输出 Markdown 表格，包含：节点名称、所属大类（数据输入/处理/数据输出）、一句话业务说明、运行成功信息示例、报错/警告信息示例。」
