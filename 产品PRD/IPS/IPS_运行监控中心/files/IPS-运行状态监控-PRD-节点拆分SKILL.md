# 运行节点拆分 SKILL（给 AI 的指令）

> 使用时机：当用户希望你“设计/优化某个长时运行任务的执行节点（步骤）”、或“帮我切一切节点，方便挂到运行监控中心”时，加载本 SKILL 的规范来思考和回答。

## 1. 你的目标

- 帮助用户用**业务语言**拆分出 5～15 个顶层运行节点（必要时可建议父子节点结构），覆盖「数据输入 → 处理 → 数据输出」三大阶段。
- 为每个节点设计示例性的**运行成功信息**和**报错/警告信息**，方便用户后续在运行监控中心中展示。
- 在信息不完整时，主动通过少量高价值问题向用户补齐关键信息，而不是直接拒绝。

## 2. 你在回答前需要搞清楚的关键信息

在现有对话信息的基础上，检查以下 4 点是否已明确；如缺失，用 1～3 个简短问题向用户补齐。

1. **功能与业务目标**
   - 功能名称、所属业务模块/场景是什么？
   - 一句话业务目标：这次运行要解决什么问题/产出什么结果？
2. **运行入口与关键交互**
   - 用户从哪里触发运行？（页面/菜单/按钮的大致位置或名称）
   - 运行前，需要配置/选择的关键参数有哪些？（如场景、版本、时间范围、策略、对象范围等）
   - 运行中/运行后，用户期望看到什么？（进度条、分步骤状态、节点详情、是否可中断、是否导出报错文件等）
3. **大致业务步骤**
   - 请用户用业务语言描述 3～7 个大步骤（例如：数据准备 → 计算/匹配 → 结果写入/同步）。
   - 若用户只给了很模糊的信息，你应先给出一个“你理解的大致步骤草案”，并请用户确认或修正。
4. **是否已有实现代码/接口（可选）**
   - 若用户主动提到已有接口/服务/页面，你可以请求：接口路径、类名或页面名称，用于更精准地贴合真实逻辑。

> 重要：在拆分节点前，你**应优先尝试了解并阅读相关产品代码**，以避免纯想象的流程。  
> - 若用户的项目中有名为 `scp-foundation` 的代码仓库或文件夹，优先从这里查找相关模块的实现。  
> - 若你在当前环境中**找不到** `scp-foundation`（或用户未使用该命名），需要明确向用户询问「产品代码所在的仓库/根目录路径」，再继续做基于代码的节点拆分。

## 3. 节点拆分时应遵循的规则

1. **按「数据输入 → 处理 → 数据输出」组织思路**
   - 数据输入：环境检查、参数确认、数据准备（收集需求/供应/主数据/配置等）。
   - 处理：计算、匹配、排程、汇总、层级遍历等。
   - 数据输出：结果生成、写库、下游同步、异常汇总、收尾。

2. **顶层节点数量与粒度**
   - 顶层节点控制在 **5～15 个**。
   - 单个节点应是业务可理解的完整步骤，而不是代码级细节。
   - 若步骤过多，可通过父子节点方式收敛（例如“数据准备”下再细分子节点）。

3. **业务可读性优先**
   - 节点名称与说明使用业务术语（如「收集需求数据」「生成结果单据并落库」），避免纯技术词汇做标题。
   - 技术约束（事务、批次、异步等）可放在节点说明或报错信息中。

4. **每个节点必须具备的三类信息（示例级别即可）**
   - 运行成功信息示例：执行耗时、处理或生成的数量、关键业务指标（如覆盖率、成功率、异常占比等）。
   - 报错信息示例：导致失败的典型错误（主数据缺失、参数非法、写库异常等）。
   - 警告信息示例：任务可完成但存在风险/问题（如部分数据被跳过、越界/逾期、容量不足等）。

5. **顺序与依赖要清晰**
   - 节点顺序要能反映真实业务流程，必要时在说明中标明关键依赖（如“仅在校验通过后才执行后续步骤”）。

## 4. 推荐输出格式

在用户要求你「帮忙拆节点」时，你的标准输出应包含：

1. **简短总述**：1～3 段话，概括你理解的业务场景、三大阶段和整体思路。
2. **节点表格**：使用 Markdown 表格，推荐表头为：
   - 顺序
   - 大类（数据输入 / 处理 / 数据输出）
   - 节点名称
   - 节点说明（业务语言）
   - 运行成功信息示例
   - 报错/警告信息示例

如有必要，你可以额外补充小结（例如哪些节点适合支持中断、哪些节点建议有子节点），但要保持整体回答易扫读。

## 5. 信息不足时的应对方式

当用户给的信息非常少（例如只说“帮我拆一下 XX 的节点”），你需要：

1. 基于通用经验假设这是一个「数据准备 → 处理 → 输出」的长时任务。
2. 给出一个**明确标注为“初稿/草案”**的节点拆分方案，让用户看到你的理解方向。
3. 在同一条回答中提出 1～3 个关键澄清问题，请用户补充，便于后续迭代优化。

切记：不要因为信息不完整就拒绝回答，而应在说明假设前提的基础上，先给出一个合理初稿，再通过后续问答逐步收敛。
