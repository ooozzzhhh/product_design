# 1. **需求概述**

## 1.1 背景及问题

### 1.1.1 背景

悠桦林 SCP 平台包含生产排程、主计划、产销协同、需求预测、物料需求计划、集成平台等业务模块。

* **业务上存在的大量"长时任务"**：历史需求清洗、供需匹配计算、规则式排程、产销计划报告生成、安全库存计算、接口同步等,往往一次运行就需要几分钟甚至更久。

* **共性的需求**：这类任务都需要清晰的**运行入口**、直观的**进度监控**以及统一的**结果查看**方式,方便业务和运维人员日常巡检与排错。

* **当前现状**：有部分日志功能,但尚未形成统一的平台级能力和交互规范。

### 1.1.2 问题

#### 1.1.2.1 运行入口与监控方式不统一

各模块的长时任务入口分散在不同菜单与页面,运维与业务人员无法在同一视图下查看「当前模块下有哪些可运行功能」「各功能最近运行情况如何」。

**影响**：跨模块巡检成本高,无法快速回答「今天跑了什么、哪些还在跑」。

#### 1.1.2.2 任务状态与结果展示缺乏统一语义

不同模块对运行结果的处理方式不一致：有的仅提供简单日志,有的仅有任务列表,缺乏统一的**任务状态**(成功 / 运行中 / 失败 / 已终止)与**结果视图**(输入快照、运行日志、计算结果、摘要说明)。

**影响**：排查失败任务、对比历史运行、审计追溯时体验不一致,难以形成统一运维规范。

#### 1.1.2.3 平台化能力缺失

部分模块已有较成熟的运行结果中心交互与视觉规范,但未抽象为平台级能力,其他模块无法复用。

**影响**：重复建设、设计与交互不统一,后续扩展成本高。

## 1.2 目标

* **建立平台级「运行监控中心」**：将各业务模块的一次性长时任务统一到同一交互范式与视觉规范下,通过**功能切换下拉框 + 功能详情页 + 运行配置/进度弹窗 + 执行历史列表**的标准结构,支持用户快速定位功能、发起运行、查看或终止进度、下载算法文件或报错日志。

* **支持高数据量场景**：执行历史支持**分页**(每页条数可配置)与**筛选**(任务 ID、操作人、日期范围、任务状态),保证大量历史记录下的体验与性能。

* **与后端解耦**：本 PRD 描述业务逻辑与用户交互,不绑定具体前端技术实现；正式产品可在 **scp-foundation** 等仓库中按现有技术架构(如 Vue、微前端)实现,并与中台/算法服务对接。

* 当前以**原型与交互闭环**为主,运行进度与结果数据可为模拟数据；与后端 API 的对接为后续迭代。

# 2. **详细解决方案**

## 2.1 **运行监控中心**

### 2.1.1 模块描述

本模块在 SCP 平台中提供一个统一的**运行监控中心**,让用户在同一入口下完成从"选择功能"到"发起运行、看进度、看结果"的完整链路。

* **主要页面形态**：一个主页面(功能详情页,含功能切换下拉框)+ 若干弹窗(运行配置、运行进度、执行节点详情等)。

* **核心职责**：

  * 通过功能切换下拉框让用户快速切换不同功能查看对应日志。

  * 展示当前功能的执行历史列表。

  * 支持用户配置并发起运行、查看或终止运行进度、查看执行节点与结果详情。

* **主要用户**：运维人员、业务配置人员、实施与支持人员。

* **期望结果**：用户可以在一条清晰的业务路径中完成「选功能 → 发起运行 → 看进度/终止 → 查历史/下载文件」。

### 2.1.2 输入

#### 2.1.2.1 业务与配置层面

* **业务模块与可监控功能**：平台需维护「哪些业务模块接入运行监控中心」「每个模块下有哪些可监控功能」；每个功能有名称、简要描述、运行按钮文案(如「执行清洗」「计算安全库存」)及运行前需填写的参数(如数据源、日期范围、计划版本等),部分功能还需选择运行模式(增量/全量)、策略模式(规则/优化)、策略名称。

* **执行步骤定义**：每个功能对应一套执行步骤(用于进度展示与节点详情),例如「数据加载 → 校验 → 计算 → 写回」；每步有名称与默认说明文案。

* **并发控制模式**：每个功能需配置其并发控制模式,包括可并行、不可并行、可排队、限制并发数等,用于控制同一功能是否允许同时运行多个任务。

#### 2.1.2.2 用户录入(运行前)

* **通用配置**(部分功能展示)：运行模式、策略模式、策略名称。

* **功能差异化参数**：如需求预测的数据源与数据范围、产销协同的计划版本与报告周期、排程的排程规则与订单范围等；必填项在提交前需校验。

#### 2.1.2.3 系统与权限(后端对接时)

* 运行历史数据：任务 ID、所属模块与功能、开始时间、结束时间、耗时、操作人、任务状态、进度信息、计划版本/需求版本(若模块支持)、输入信息快照、运行日志、计算结果、摘要等。

* 用户权限：运行、终止、下载算法文件、下载报错日志等操作需按权限控制。

### 2.1.3 输出

#### 2.1.3.1 用户可见的页面与反馈

* **功能详情页**：页面顶部功能切换下拉框(用于切换不同功能查看对应日志)、执行历史列表(支持筛选与分页)、单条记录展开后的输入信息、运行日志、计算结果、摘要说明。

* **运行配置弹窗**：用户填写/选择后提交,系统创建运行任务并打开进度弹窗。

* **运行进度弹窗**：整体进度(以「当前步骤/总步骤」形式)、步骤时间线、运行中时可点击「终止」、可点击性能图标查看性能详情。

* **执行节点详情弹窗**：按步骤展示每步状态(未执行/执行中/已完成/失败/已终止)及说明。

* **性能详情弹窗**：展示当前任务的性能信息,包括各节点执行情况、业务指标等。

#### 2.1.3.2 导出文件

* **算法文件**：成功任务可下载本次运行的配置/策略快照。

* **报错日志**：失败任务可下载错误明细(如 CSV),含错误级别、模块、错误代码、异常描述等；编码需支持中文。

#### 2.1.3.3 数据与下游(后端对接后)

* 运行记录持久化,供运维、审计与报表使用；中台/算法服务可基于任务 ID 反查详细结果。

### 2.1.4 业务流程

#### 2.1.4.1 功能切换与日志查看

1. 用户进入运行监控中心页面,页面启动时直接显示功能详情页(日志列表)。

2. 页面顶部展示**功能切换下拉框**,用户可通过下拉框选择不同的业务功能(如「历史需求清洗」「安全库存计算」「MRP 运行」等)。

3. 切换功能后,页面展示该功能的**执行历史列表**；每条记录展示任务 ID、开始时间、时长、操作人、计划版本(若模块有版本)、进度(步骤分数或排队位置)、状态等信息。

#### 2.1.4.2 发起运行

1. 用户在功能详情页点击**运行按钮**(按钮文案按功能配置,如「执行清洗」「计算安全库存」)。

2. 系统弹出**运行配置弹窗**：展示通用配置(若该功能需要)与功能差异化参数；必填项未填写时不允许提交。

3. 用户点击「确认并开始」后：
   - **若该功能允许并行运行**：系统直接创建运行中任务,配置弹窗关闭,**运行进度弹窗**自动打开。
   - **若该功能不支持排队且已有任务运行中**：系统提示「上一次运行正在进行中,请稍后再试」,不允许提交。
   - **若该功能支持排队且已有任务运行中**：系统创建排队中任务,提示「当前排队位置：第X位」,配置弹窗关闭,执行历史列表中该任务显示为「排队中」状态。

4. 进度弹窗展示整体进度(以「当前步骤/总步骤」形式)与步骤时间线；进度随任务推进更新(原型阶段可为模拟推进)。

#### 2.1.4.3 查看进度与终止

1. 运行中任务在功能详情页的**执行历史列表**中显示为「运行中」,进度以步骤分数展示(如 2/4)；用户可点击该任务行的「查看详情」按钮再次打开进度弹窗。

2. 进度弹窗在「查看运行中任务」时,右上角展示「终止」按钮与蓝色性能图标按钮；用户点击终止后,系统弹出二次确认对话框,确认后该任务状态变为**已终止**,弹窗关闭,历史列表刷新。

3. 用户点击蓝色性能图标按钮,弹出**性能详情对话框**,展示该功能的性能信息(各功能需根据自身实际情况自定义要为用户展示怎样的性能信息，可以理解为"运行健康度"，以缓解用户对运行时间过长怀疑是否运行卡死的焦虑)。

4. 用户也可直接关闭进度弹窗,仅关闭视图,任务在后台继续；任务结束后状态由系统更新为成功或失败。

#### 2.1.4.4 查看结果与排查失败

1. **成功任务**：列表行以绿色主题与勾号展示；用户可点击「算法文件」下载配置快照,展开行可查看输入信息、运行日志、计算结果、摘要说明。

2. **失败任务**：列表行以红色主题与叉号展示；用户可点击「报错日志」下载错误明细；展开行可查看日志与输入信息以辅助排查。

3. **已终止任务**：列表行以黄色主题与减号展示；时长列显示「已终止」；仅提供「已终止」标识与「节点」按钮,不提供「算法文件」「报错日志」按钮,操作区宽度与成功/失败行保持一致,避免布局错位。

4. **排队中任务**：列表行显示「排队中」状态,进度列显示排队位置(如「第3位」)；用户可查看排队状态,等待前序任务完成后自动开始执行。

#### 2.1.4.5 页面信息实时刷新

1. **执行历史列表刷新**：功能详情页的执行历史列表需要定时自动刷新,以展示任务的最新状态(排队中 → 运行中 → 成功/失败/已终止)、进度更新(步骤分数变化)、排队位置变化等。

2. **运行进度实时更新**：当用户打开运行进度弹窗查看运行中任务时,弹窗内的整体进度(当前步骤/总步骤)、步骤时间线(当前执行节点高亮与状态)需要实时更新,让用户能够持续观察任务推进情况。

3. **刷新频率与用户体验**：页面自动刷新的频率应在保证信息时效性与系统性能之间取得平衡；用户在查看详情、筛选、分页等操作时,刷新不应打断当前操作或导致页面跳动。

#### 2.1.4.6 执行节点查看

1. 用户在执行历史任意任务行点击「节点」按钮,打开**执行节点详情弹窗**。

2. 弹窗内按该功能的执行步骤顺序展示每步状态：未执行、执行中、已完成、失败、已终止。

3. 运行中任务高亮当前步骤(旋转图标 +「计算中…」)；已终止任务高亮终止时所在步骤(减号 +「已终止」)；失败任务高亮失败步骤。

#### 2.1.4.7 筛选与分页

1. 用户在功能详情页使用筛选区：输入任务 ID、输入操作人关键字、选择开始日期起止、选择任务状态(全部/成功/失败/运行中/已终止/排队中),点击「查询」按钮手动触发查询,点击「重置」清空筛选。

2. 列表仅展示当前功能、且满足筛选条件的记录；默认按开始时间倒序。

3. 分页：每页条数可选(如 10/20/50/100),默认 20；提供首页、上一页、下一页、末页；底部展示「显示第 X–Y 条,共 Z 条」及当前页/总页数。切换每页条数或筛选条件时重置到第 1 页。

### 2.1.5 业务规则

#### 2.1.5.1 模块与功能

* **若**某业务模块需接入运行监控中心,**则**需在平台侧维护该模块及其下的可监控功能列表；每个功能需有名称、描述、运行按钮文案、运行前参数定义(含是否必填)、并发控制模式,以及是否展示运行模式/策略等通用配置。

* **若**某功能需要运行前填写通用配置(运行模式、策略模式、策略名称),**则**在运行配置弹窗中展示该区域；**否则**仅展示该功能差异化参数。

#### 2.1.5.2 任务标识与版本

* 任务 ID 需全局唯一,且能从 ID 上区分所属模块/功能(如带模块或功能前缀)；具体格式由实现方与后端约定。

* **若**业务模块具备计划版本/需求版本概念(如主计划、产销协同、需求预测、物料需求计划),**则**运行记录中记录并展示计划版本、需求版本；**若**模块无版本概念,**则**不展示版本相关列。

#### 2.1.5.3 任务状态与状态迁移

* **任务状态**及用户可见含义、操作约定：

| 状态  | 含义        | 列表展示与可操作项                                       |
| --- | --------- | ----------------------------------------------- |
| 排队中 | 任务等待前序任务完成 | 显示「排队中」、进度列显示排队位置(如「第3位」)；等待自动开始执行。 |
| 运行中 | 任务正在执行    | 显示「运行中」、进度为步骤分数；可「查看详情」、可「终止」。 |
| 成功  | 任务正常完成    | 绿色主题、勾号；可下载「算法文件」、展开查看结果与日志。                    |
| 失败  | 执行过程异常或失败 | 红色主题、叉号；可下载「报错日志」、展开查看日志与输入。                    |
| 已终止 | 用户主动终止    | 黄色主题、减号；时长列显示「已终止」；仅可查看节点与展开详情,不可下载算法文件/报错日志。   |

* **状态迁移**：

  * **当**任务提交时,根据并发控制模式决定初始状态：若允许立即执行则为运行中,若需排队则为排队中。

  * **当**排队中任务轮到执行,**则**由排队中 → 运行中。

  * **当**任务正常执行完毕,**则**由运行中 → 成功；记录结束时间、耗时、摘要(若有)。

  * **当**执行过程发生异常或失败,**则**由运行中 → 失败；保留当前步骤与日志。

  * **当**用户在运行进度弹窗点击「终止」并确认,**则**由运行中 → 已终止；弹窗关闭并刷新列表。

#### 2.1.5.4 并发控制模式

* **可并行**：同一功能可同时运行多个任务,无限制；用户可随时发起运行。

* **不可并行**：同一功能同时只能运行1个任务；若已有任务运行中,第二个请求被拒绝,系统提示「上一次运行正在进行中,请稍后再试」。

* **可排队**：同一功能同时只能运行1个任务,但第二个请求可排队等待；系统提示排队位置(如「当前排队位置：第3位」),任务显示为「排队中」状态,前序任务完成后自动开始执行。

* **限制并发数**：同一功能最多同时运行N个任务(N可配置);超出限制的请求被拒绝,系统提示「当前已有X个任务在运行,请稍后再试」。

#### 2.1.5.5 校验规则

* **运行配置弹窗**：功能定义的必填项未填写时,**禁止**提交；需提示用户补全必填项。

* **筛选**：日期范围若只填开始未填结束(或反之),按已填单边筛选；任务 ID/操作人关键字为空时,表示该维度不限制。

* **分页**：当前页号大于总页数时(如筛选后结果变少),自动修正到最后一页。

#### 2.1.5.6 权限与安全(后端对接时落实)

* 运行、终止、下载算法文件、下载报错日志等操作需根据当前用户权限控制；无权限时按钮不展示或禁用。

* 报错日志可能包含敏感信息；导出前需做脱敏与权限校验,具体由后端与安全规范约定。

#### 2.1.5.7 性能与数据量

* 执行历史**必须**分页加载；默认每页条数与可选档位由产品约定(如 20 条,可选 10/20/50/100)；后端需支持按页码、每页条数、筛选条件返回当页数据与总条数。

* 历史数据量极大时,建议后端支持排序与分页游标；前端可考虑虚拟滚动(是否实现按项目约定)。

#### 2.1.5.8 节点异常与任务终止

* 各功能在接入过程中，必须配置每个节点在发生异常时是否自动终止任务，需按自身实际情况判断。

* 异常需终止任务的场景举例：在MRP运算中校验出某个物品的BOM存在死循环。

* 异常无需终止任务的场景举例：在MRP运算中校验出某个物品没有任何可用供应数据。

  * 顶部：运行监控中心标题与图标；右侧为**功能切换下拉框**(用于切换不同功能查看对应日志,如「历史需求清洗」「安全库存计算」「MRP 运行」等)；右上角为运行按钮(文案按功能配置,如「计算安全库存」「执行清洗」)。

  * 点击行可展开输入信息、运行日志、计算结果、摘要。底部分页：每页条数下拉、显示范围与总条数、上一页/下一页等。

* **运行进度弹窗**

  * 标题：「正在执行计算」或「查看运行进度」；在查看运行中任务时,右上角展示：
    - **蓝色性能图标按钮**(仪表盘图标,点击弹出性能详情)
    - **「终止」按钮**(黄色)

  * 中部：整体进度条,以步骤分数(如 2/4)展示,运行中时有进度动画。

  * 下方：步骤时间线,每步为圆点 + 步骤名称 + 说明；当前步骤为旋转图标 +「计算中…」；已完成为勾号。

  * 底部：关闭/关闭窗口按钮。

* **性能详情弹窗**

  * 标题：性能监控 - 任务ID

  * 内容：展示该功能的性能信息,包括各节点执行情况、业务指标等(具体指标由各功能定义)。

  * 底部：刷新按钮、关闭按钮。

* **执行节点详情弹窗**

  * 内容：按步骤顺序展示每步状态(未执行/执行中/已完成/失败/已终止)及说明；失败步骤为红色叉号,已终止步骤为黄色减号。

## 2.2 **新功能接入与运行节点规范**

* **新功能如何接入运行监控中心**(需要提供哪些信息、遵守哪些约定)；

* 在运行监控过程中,**如何划分"运行节点"**,以及每个节点里应该包含哪些成功信息与报错/警告信息。

运行节点是监控过程里最重要的信息载体。设计合理的节点,可以让运维与业务人员快速看懂：任务现在跑到哪一步、每一步大致做了什么、在哪一步出现了异常或告警,从而更快判断任务健康度并定位问题。

### 2.2.1 新功能接入时的节点配置规范

当新功能接入运行监控中心时,除了需要定义节点的业务含义、顺序、成功信息与报错信息外,还需要为每个节点配置以下**通用参数**,以支持运行监控中心的统一管控能力：

#### 2.2.1.1 节点可终止性配置

* **配置目的**：定义哪些节点允许用户终止、哪些节点不可终止,确保关键操作(如数据写回、事务提交)不会被用户误操作中断。

* **配置规则**：
  - **可终止节点**：用户在运行进度弹窗点击「终止」按钮时,若当前执行节点被标记为「可终止」,则系统允许终止该任务,任务状态变为「已终止」。
  - **不可终止节点**：若当前执行节点被标记为「不可终止」,则即使用户点击「终止」按钮并确认,系统也不会立即终止任务,而是提示「当前节点正在执行关键操作,无法终止,请等待该节点完成后再终止」或直接禁用终止按钮。

* **典型场景**：
  - 可终止节点：数据加载、校验、计算等可中断且不影响数据一致性的节点。
  - 不可终止节点：结果持久化、事务提交、数据写回、分配关系重建等涉及数据库写操作或状态变更的节点。

* **开发提醒**：在定义节点时,需要为每个节点显式配置「可终止」或「不可终止」标识；该配置后续会做成通用参数,便于运行监控中心统一管控终止行为。

#### 2.2.1.2 节点异常处理策略配置

* **配置目的**：定义节点执行异常时的处理策略,区分「可跳过继续执行」与「必须终止整个流程」的节点,提升任务容错性与稳定性。

* **配置规则**：
  - **异常可跳过**：若某节点被标记为「异常可跳过」,则该节点执行失败时,系统记录异常信息(在节点详情与运行日志中展示),但不终止整个任务,而是继续执行下一节点。适用于非关键校验、可选的数据补充、辅助性统计等节点。
  - **异常必须终止**：若某节点被标记为「异常必须终止」,则该节点执行失败时,系统立即终止整个任务,任务状态变为「失败」,不再执行后续节点。适用于关键数据加载、核心计算、结果持久化等节点。

* **典型场景**：
  - 异常可跳过：辅助性数据加载(如历史统计信息)、可选的校验步骤(如警告级别的数据质量检查)、非关键的日志记录等。
  - 异常必须终止：策略物料加载失败、BOM 校验失败、净需求计算异常、结果持久化失败等。

* **开发提醒**：在定义节点时,需要为每个节点显式配置「异常可跳过」或「异常必须终止」策略；该配置后续会做成通用参数,便于运行监控中心统一管控异常处理行为。若节点被配置为「异常可跳过」,则该节点的异常信息应在节点详情中以警告形式展示,而非导致整个任务失败。

### 2.2.2 新功能接入运行监控中心的流程与要求

当某个新功能开发完成后希望接入运行监控中心进行统一管理时,需满足以下通用要求(不绑定具体实现形式)：

* **归属与展示**

  * 明确该功能所属的**业务模块**(如生产排程、主计划、需求预测等)；若该模块尚未接入运行监控中心,需先完成模块级接入。

  * 在运行监控中心内为该功能提供：**功能名称**、**简要描述**、**运行入口文案**(如「执行 XX」「计算 XX」),以便在功能切换下拉框与详情页中展示。

* **运行前参数**

  * 定义该功能在「发起运行」前需要用户选择或填写的**业务参数**(如数据范围、版本、规则、范围等),并标明必填项；运行监控中心将统一以弹窗形式收集这些参数并在运行记录中留存快照。

* **并发控制模式**

  * 声明该功能的并发控制模式：可并行、不可并行、可排队、限制并发数(需指定最大并发数)。

* **运行节点定义(核心)**

  * 按 2.2.2、2.2.3、2.2.4 的规范,为该功能定义**运行节点**(即执行步骤/子步骤)及每个节点可输出的**运行成功信息**与**报错/警告信息**；以便在进度展示、执行节点详情及运行日志中一致呈现。

* **任务标识与状态**

  * 该功能产生的每次运行需生成**唯一任务标识**,并随执行过程更新**任务状态**(排队中 / 运行中 / 成功 / 失败 / 已终止)与**当前节点/进度**；运行监控中心依赖这些信息驱动进度条、步骤时间线与历史列表。

* **可选的通用配置**

  * 若该功能需要运行模式(如增量/全量)、策略模式(如规则/优化)、策略名称等通用配置,需声明并在运行配置弹窗中展示；否则仅展示功能专属参数。

接入完成后,该功能即可在运行监控中心功能切换下拉框中展示,用户可选择该功能、发起运行、查看进度与节点、查看历史与下载算法文件/报错日志。具体配置方式(如配置表、配置文件、管理后台)由实现方在落地时决定,本 PRD 仅约定能力与信息要求。

### 2.2.3 运行节点在监控中的定位

运行监控中心认为：**运行节点是监控过程中最重要的信息**。

* **为什么节点最重要**

  * 任务级状态(成功/失败/运行中/已终止)只能回答「任务有没有跑完、结果如何」,无法回答「卡在哪一步、哪一步有警告、哪一步产生了多少量」。

  * 节点将一次运行拆解为**可观测的步骤**,每个步骤可携带**执行时间**、**成功类指标**(处理条数、生成数量、通过率等)以及**报错/警告信息**(警告数、异常条数、逾期率等)。这样运维与业务人员既能看整体进度,也能在问题发生时快速定位到具体步骤与原因。

* **节点的使用场景**

  * **运行中**：在进度弹窗中按节点展示当前执行到哪一步,当前节点可展示「计算中…」等状态及该节点已产生的部分信息。

  * **运行结束后**：在执行节点详情(及运行日志)中按节点顺序展示每步的**运行成功信息**与**报错/警告信息**,支持展开/折叠以兼顾总览与细查；失败或已终止时,可高亮异常所在节点。

因此,新功能接入运行监控中心时,**必须**定义其运行节点及节点内信息规范,而不是仅上报任务级状态。本能力框架在设计上参考了业界排程/计划类软件(如 Asprova)中的「消息窗口」与运行日志思路：按层级展示步骤、每步带执行时间与数量/警告统计,便于用户快速把握运行健康度与问题所在。

### 2.2.4 功能如何划分节点：规范与规则

一个功能应如何划分成多个运行节点,建议遵循以下**通用规范与规则**(不限定具体业务,便于各模块一致落地)：

* **按逻辑步骤划分,而非按技术实现**

  * 节点应对应**业务可理解的步骤**,避免按函数调用、线程等纯技术粒度划分,以便业务与运维能直接理解。

* **顺序与依赖**

  * 节点之间应有**明确的执行顺序**(串行或可描述的依赖)；运行监控中心按该顺序展示步骤时间线,并据此计算「当前步骤/总步骤」类进度。若存在并行分支,需约定在展示层如何归并为一条主时间线。

* **粒度建议**

  * **顶层节点**不宜过多,建议单次运行在 **5～15 个主步骤** 内,便于一屏内浏览；若某步骤本身很复杂,可再拆为**子节点**(见下条)。

  * 单个节点代表**一个有明确起止、可上报执行时间与结果**的单元；过细会导致信息噪音大,过粗则无法发挥「定位到步骤」的价值。

* **支持层级(父节点与子节点)**

  * 允许一个节点下包含**子节点**,形成层级结构。

  * 展示上可支持**展开/折叠**：折叠时仅显示父节点名称与汇总信息,展开时显示各子节点的运行成功信息与报错信息。层级深度建议不超过 2～3 层,以保持可读性。

* **每个节点应可上报的信息**

  * **执行时间**：该节点开始到结束的耗时,便于做性能排查。

  * **成功类指标**：与该节点业务含义一致的数量或比率,用业务术语命名,避免仅写技术指标。

  * **警告/错误类指标**：该节点产生的警告数、错误数或异常条数；若有,应在节点上明确标识并在节点详情或消息列表中展示具体内容。

* **节点命名与说明**

  * 节点名称应**简短且表意清晰**；必要时可为节点配置**默认说明文案**,用于在未产生具体消息时展示。运行过程中或结束后,可用**实际产生的消息内容**覆盖或补充该说明。

以上规则旨在形成**可复用的节点划分规范**：新功能在接入时,按「逻辑步骤、顺序、适度粒度、可选的层级、每节点可上报时间与成功/报错信息」来定义节点,即可与运行监控中心的进度展示、节点详情及日志查看能力对齐。

### 2.2.5 节点内应包含的信息：运行成功信息与报错信息

每个运行节点应能承载以下两类信息,以便运行监控中心统一展示并支持问题定位：

* **运行成功信息**

  * 指该节点**正常完成**时产生的、用于说明「做了什么、结果如何」的信息,包括：执行耗时、处理数量或生成数量、与该节点业务含义一致的业务指标(如比率、平均值等)。展示时可按「信息」类图标或中性样式呈现,用于确认该步骤已成功执行并了解关键结果。

* **报错信息与警告信息**

  * **报错信息**：该节点执行失败或发生异常时的信息(异常码、异常描述、失败条数等)；应导致任务状态变为「失败」或至少在该节点上标记为「失败」,并可在报错日志导出中体现。

  * **警告信息**：该节点执行完成但存在需要关注的问题,包括警告数量、业务异常统计(如逾期数量及占比、不足/异常条数等)。展示时应用**警告/错误**类图标或颜色区分,便于用户快速发现异常节点；在节点详情或消息列表中应能展开查看具体内容。

* **统一展示约定**

  * 运行监控中心对节点内信息的展示不做具体格式强制,但建议：每条信息可带时间戳(可选),便于与日志对齐；成功类与警告/错误类在视觉上可区分；支持按节点折叠/展开,避免单次运行信息过多时难以浏览。

新功能在定义运行节点时,应按上述约定为每个节点设计可输出的**运行成功信息**与**报错/警告信息**,并在实现侧在节点开始/结束或关键检查点上报这些信息,以便运行监控中心统一汇聚和展示,满足「节点是监控过程中最重要信息」的目标。

### 2.2.6 节点拆分 SKILL(给开发同事和 AI 使用)

为方便各模块在接入运行监控中心时快速拆分节点,沉淀了一份通用的「节点拆分 SKILL」,开发同事在设计新功能节点时可配合 AI 使用。

详见 `产品设计/产品PRD/IPS/IPS_运行监控中心/节点拆分SKILL.md`(若存在)

* **建议做法**：

1. **提示词**：先按本 PRD 2.2.2～2.2.4 的规范,用业务语言简要描述功能的目标、运行入口、关键参数、大致业务步骤；

2. **准备好与该功能相关的产品代码位置**：包括但不限于：后端接口路径(如 `POST /api/xx/run`)、Controller / Service 类名、前端页面或弹窗组件、前端 API 封装文件路径等,便于 AI 基于真实代码来校正节点拆分；

3. **调用节点拆分SKILL**：与 AI 协作生成节点拆分表格；

4. **检查**：最后结合本地代码检查AI输出的数据校准节点名称、顺序和成功/报错信息示例。

按照上述方式,开发同事可以在每个具体功能上复用同一套节点拆分方法,与 AI 协作完成节点方案。

### 2.2.7 示例：MRP 运行(供需匹配计算)的节点划分与示例提示词

本小节以 MRP 模块中的「物料需求计算(MRP 运行 / 供需匹配计算)」为例:

#### 2.2.7.1 功能与运行流程简述(业务视角)

* **功能名称**：物料需求计算(MRP 运行 / 供需匹配计算)。

* **业务目标**：在既定策略与计划周期下,综合客户订单、制造订单、预测、库存、在途等信息,分层计算物料净需求,生成计划订单/采购建议,并输出短缺、逾期等异常信息。

* **运行前参数**：场景/数据源、MRP 策略、订单类型与范围、计划起止日期等(由前端运行配置弹窗收集)。

* **典型入口与接口位置(示意)**：

  * 前端入口：某「MRP 策略维护」页面的「运行」按钮；

  * 前端调用：对应的 API 封装文件中某个 `mrpStrategyRun` 或 `run` 方法；

  * 后端入口：某 `MrpController.run` 风格的接口(如 `POST /api/mrp/run`),内部调用相应的 Service 进行计算。

#### 2.2.7.2 节点划分示例(按「数据输入 → 处理 → 数据输出」三大步骤展开)

下表将一次 MRP 运行拆解为 8 个业务节点,并在「大类」一列中标明其属于数据输入、处理还是数据输出。各模块在设计自身节点时,可先确定三大步骤,再在每个大步骤下细化类似的业务节点。

| 顺序 | 大类   | 节点名称          | 节点说明(业务语言)                                            |
| -- | ---- | ------------- | ----------------------------------------------------- |
| 1  | 数据输入 | 低阶码校验与环境准备    | 检查是否已完成低阶码计算,校验运行场景、模块与当前系统状态是否允许执行 MRP。              |
| 2  | 数据输入 | 策略与计划周期确定     | 读取默认或指定的 MRP 策略,确定计划起止时间(历史回溯 + 计划结束),锁定本次运行范围。       |
| 3  | 数据输入 | MRP 数据准备      | 按策略与周期加载策略物料、BOM、工艺、库存、在制/在途、客户订单、制造订单等,形成 MRP 运行上下文。 |
| 4  | 处理   | 物料层级排序        | 按低阶码对物料排序,确定自上而下的计算顺序,保证上下层依赖关系被正确处理。                 |
| 5  | 处理   | 需求汇总与净需求计算    | 对每个库存点物料汇总独立需求与非独立需求,结合库存与已有供应,完成净需求计算。               |
| 6  | 处理   | 计划供应生成与下层需求推导 | 按策略生成/调整计划订单、采购建议等供应,并由当前层物料推导下层物料的非独立需求,同时处理冗余供应。    |
| 7  | 数据输出 | 结果持久化与计划生成    | 将净需求与计划供应落地为系统中的 MRP 结果(计划订单/采购建议、Pegging 关系、关键指标等)。  |
| 8  | 数据输出 | 分配关系重建与工单同步   | 清理旧分配关系,重建物料分配关系,并将新增/更新的工单同步到执行系统或下游模块。              |

**各节点的运行成功信息与报错/警告信息(对应 2.2.4)**

* **节点 1「低阶码校验与环境准备」**

  * 成功信息示例：低阶码校验耗时、当前低阶码状态(已完成/无需计算)、允许执行的模块/场景列表。

  * 报错/警告信息示例：低阶码未执行或正在执行、模块与策略模式不匹配(如非 MRP 模块却使用优化算法策略)、系统处于不允许运行的冻结状态等。

* **节点 2「策略与计划周期确定」**

  * 成功信息示例：选用的策略编码与名称、计划起止日期、回溯期时长、本次运行覆盖的时间跨度。

  * 报错/警告信息示例：找不到可用策略、策略物料范围为空、策略被停用或与当前场景不兼容。

* **节点 3「MRP 数据准备」**

  * 成功信息示例：加载的策略物料数量、BOM 校验通过数量、成功加载的库存记录数/订单条数等。

  * 报错/警告信息示例：BOM 校验失败条数、策略物料范围为空、关键主数据缺失(如缺少物料主数据或工艺)的数量等。

* **节点 4「物料层级排序」**

  * 成功信息示例：参与计算的物料数、低阶码层数、排序耗时。

  * 报错/警告信息示例：无法获取层级信息导致被跳过的物料数(一般仅在极端数据异常时出现)。

* **节点 5「需求汇总与净需求计算」**

  * 成功信息示例：已处理物料数/总物料数、生成净需求条数,可滚动输出「当前已处理 X 条 / 总 Y 条」类进度信息。

  * 报错/警告信息示例：异常需求条数(如超出策略范围、日期不在计划周期内)、被忽略或截断的需求数等。

* **节点 6「计划供应生成与下层需求推导」**

  * 成功信息示例：新增/调整的计划供应数量、补充或削减的供应条数、冗余供应被回收的条数。

  * 报错/警告信息示例：因约束(最小批量、容量上限等)无法完全满足的需求条数、冗余供应条数、被迫推迟或提前的需求条数。

* **节点 7「结果持久化与计划生成」**

  * 成功信息示例：写入的计划订单/采购建议数量、生成的 Pegging 关系条数、结果表写入耗时、标记为当前版本的计划标识等。

  * 报错/警告信息示例：结果持久化失败、事务回滚、部分结果写入失败条数等。

* **节点 8「分配关系重建与工单同步」**

  * 成功信息示例：删除的旧分配关系条数、重建的分配关系条数、成功同步到下游系统的工单数量。

  * 报错/警告信息示例：部分工单同步失败条数、下游系统不可用或响应超时等。

**与 2.2.3、2.2.4 规范的对应关系**

* 以上 8 个节点均为按**业务逻辑步骤**划分的可理解步骤(低阶码校验、策略确定、数据准备、净需求计算、结果落地等),且顺序清晰,符合 2.2.3 中关于「按逻辑步骤划分」「顺序与依赖」的要求。

* 顶层节点数控制在 5～15 之间,粒度适中；其中需求汇总/净需求计算、计划供应生成等节点在需要时可以继续拆分为子节点(例如按低阶码层级或物料分组),满足 2.2.3 中关于「粒度与层级」的建议。

* 每个节点都约定了至少三类信息：执行时间(可通过耗时统计体现)、成功类指标(数量、比率、覆盖范围等)、警告/错误信息(条数与典型原因),对应 2.2.4 中对「运行成功信息」与「报错/警告信息」的要求。

* 从「数据输入 → 处理 → 数据输出」三大阶段出发再向下细化节点的方式,可以作为所有新功能接入运行监控中心时的通用拆分思路：先明确三大阶段,再在每个阶段中定义若干业务可理解的节点,并为每个节点设计可上报的成功/异常信息。
