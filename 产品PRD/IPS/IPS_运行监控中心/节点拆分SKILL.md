**适用范围**：为 SCP **运行监控中心**所纳管的长时运行任务，用业务语言拆分运行节点（步骤），并为每个节点设计可上报的成功信息、报错与警告示例；同时给出并发模式、性能展示、节点级异常策略与可终止性建议，供接入配置与实现对照。

**两类接入（输入不同，输出同一套规范）**：

| 类型 | 典型场景 | 主要依据 |
|------|----------|----------|
| **现有业务接入改造** | 功能已落地，需对齐真实逻辑与监控配置 | **代码**（后端服务、任务入口、事务与写库、前端调用链等） |
| **新功能接入** | 尚无实现代码，仅有需求与设计 | **PRD**（流程、规则、数据范围、非功能要求）+ **原型图**（页面步骤、操作顺序、状态与反馈文案） |

两类任务的**节点拆分粒度、业务语言表述、成功/报错/警告示例、以及第 4 节推荐输出格式完全一致**；差异仅在于：配置建议后的**证明/依据**写「代码位置与片段说明」还是「PRD 章节/段落 + 原型界面与交互说明」。

# 运行节点拆分 SKILL

## 1. 你的目标

- 帮助用户用**业务语言**拆分出 5～15 个顶层运行节点（必要时可建议父子节点结构），覆盖「数据输入 → 处理 → 数据输出」三大阶段。
- 为每个节点设计示例性的**运行成功信息**和**报错/警告信息**，方便用户后续在运行监控中心中展示。
- 为该功能提供以下**四类配置建议**（每类均需附带**可追溯依据**；依据来源随接入类型而定，见第 2、3、4 节）：
  - **并发模式建议**：可并行 / 不可并行 / 可排队 / 限制并发数
  - **性能信息建议**：如处理数量、耗时、通过率、异常条数等
  - **节点异常处理建议**：哪些节点异常可跳过、哪些必须终止整个流程
  - **节点可终止性建议**：哪些节点可终止、哪些不可终止
- **有代码**：依据以**实现代码**为主；**无代码**：依据以 **PRD + 原型图** 为主（从需求中的数据写入/状态变更、流程阶段、界面上的进度与中断设计等推导；须在输出中标注「待实现确认」若存在不确定处）。
- 在信息不完整时，主动通过少量高价值问题向用户补齐关键信息，而不是直接拒绝。

## 2. 你在回答前需要搞清楚的关键信息

在现有对话信息的基础上，检查以下 5 点是否已明确；如缺失，用 1～3 个简短问题向用户补齐。

### 2.1 通用（两类接入都要明确）

1. **功能与业务目标**
   - 功能名称、所属业务模块/场景是什么？
   - 一句话业务目标：这次运行要解决什么问题/产出什么结果？
2. **运行入口与关键交互**
   - 用户从哪里触发运行？（页面/菜单/按钮的大致位置或名称）
   - 运行前，需要配置/选择的关键参数有哪些？（如场景、版本、时间范围、策略、对象范围等）
   - 运行中/运行后，用户期望看到什么？（进度条、分步骤状态、节点详情、是否可中断、是否导出报错文件等）
3. **大致业务步骤**
   - 请用户用业务语言描述 3～7 个大步骤（例如：数据准备 → 计算/匹配 → 结果写入/同步）。
   - 若用户只给了很模糊的信息，你应结合**已有材料**（代码或 PRD/原型）给出一个“你理解的大致步骤草案”，并请用户确认或修正。

### 2.2 接入类型与主要依据（必须先判定）

4. **接入类型：有代码 vs 无代码**
   - **有代码（现有业务改造）**：用户能提供或可检索到实现该功能的代码路径。
   - **无代码（新功能）**：尚无合并实现，但有 **PRD** 与 **原型图**（或其一；缺一则应索要）。

### 2.3 按类型补齐材料

**A. 有代码时**

- 可请求：接口路径、类名、页面名称、模块名，用于贴合真实逻辑。
- **在拆分节点与写配置建议前，必须阅读与该功能相关的已实现代码**（后端服务、领域逻辑、任务入口、持久化/事务、以及如有则前端调用链），避免凭想象写流程。
- 优先在**当前对话可访问的工作区**内检索；若工作区中不存在或未包含该功能实现，须向用户索要**仓库根路径、模块或包名、或关键类/接口/HTTP 路径**，再继续。
- 不要假定任何固定仓库名或目录名一定存在；以用户确认或可检索到的路径为准。
- **阅读代码时需重点关注**：
  - 是否涉及数据写入、状态修改（判断并发模式）
  - 关键计算逻辑、处理数量、耗时统计（判断性能信息）
  - 哪些步骤是数据加载/校验/计算（可终止、异常可跳过），哪些步骤是数据写回/事务提交（不可终止、异常必须终止）

**B. 无代码时（新功能：以 PRD + 原型为主）**

- **必须阅读**用户提供的 **PRD** 与 **原型图**（HTML/完整前端项目/图片/Figma 说明/文档内嵌图均可）；若仅其一，另一项须说明「依据不足」并请用户补充或接受带前提的草案。
- **从 PRD 中重点提取**：
  - 业务流程阶段、先后依赖、是否写库/同步外部系统/变更业务状态（推导并发与终止策略）
  - 批量范围、统计类验收指标、性能或 SLA 描述（推导性能展示项）
  - 异常与补偿、部分成功、跳过规则（推导节点级异常可跳过 vs 必须终止）
  - 非功能需求：是否允许并行触发、队列、资源限制等表述
- **从原型图中重点提取**：
  - 触发运行的页面、向导步骤、进度展示方式、错误/警告的展示位置与文案倾向
  - 是否存在「取消/中断」类控件及对应步骤（推导可终止性）
  - 多分支、子任务是否在 UI 上拆步展示（与节点粒度对齐）
- 推导处若 PRD/原型未写清，须在输出中**显式标注「待产品/研发确认」**，并列出需确认的最小问题集；不得伪装成已有代码事实。

> **统一原则**：无论 A 还是 B，**节点顺序与业务含义不得与主要依据矛盾**；无代码时允许存在「实现后需复核」的条目，但表格与四类建议的结构仍须写满，便于后续对照代码一次性修订。

## 3. 节点拆分时应遵循的规则

1. **按「数据输入 → 处理 → 数据输出」组织思路**
   - 数据输入：环境检查、参数确认、数据准备（收集需求/供应/主数据/配置等）。
   - 处理：计算、匹配、排程、汇总、层级遍历等。
   - 数据输出：结果生成、写库、下游同步、异常汇总、收尾。

2. **顶层节点数量与粒度**
   - 顶层节点控制在 **5～15 个**。
   - 单个节点应是业务可理解的完整步骤，而不是代码级细节。
   - 若步骤过多，可通过父子节点方式收敛（例如“数据准备”下再细分子节点）。

3. **业务可读性优先**
   - 节点名称与说明使用业务术语（如「收集需求数据」「生成结果单据并落库」），避免纯技术词汇做标题。
   - 技术约束（事务、批次、异步等）可放在节点说明或报错信息中。

4. **每个节点必须具备的三类信息（示例级别即可）**
   - 运行成功信息示例：执行耗时、处理或生成的数量、关键业务指标（如覆盖率、成功率、异常占比等）。
   - 报错信息示例：导致失败的典型错误（主数据缺失、参数非法、写库异常等）。
   - 警告信息示例：任务可完成但存在风险/问题（如部分数据被跳过、越界/逾期、容量不足等）。

5. **顺序与依赖要清晰**
   - 节点顺序要能反映真实业务流程，必要时在说明中标明关键依赖（如”仅在校验通过后才执行后续步骤”）。
   - 若存在**并行分支**，须约定在**展示层**如何归并为**一条主时间线**，以便与运行监控中心的「当前步骤 / 总步骤」及步骤时间线一致（避免用户看到与整体进度矛盾的展示）。

6. **给出四类配置建议（必须附带证明和依据；证明形式随接入类型二选一）**

   **有代码时**：证明以**代码**为准（方法名、类、事务/SQL、调用链等）。

   **无代码时**：证明以 **PRD（章节/小节/需求编号）+ 原型（页面名/步骤名/控件与说明）** 为准；若文档未覆盖，写「文档未约定」+ **合理推断**并标注 **待确认**。

   - **并发模式建议**
     - 有代码：写入/状态修改多 → 不可并行或可排队；只读无状态 → 可并行；重计算/批处理 → 限制并发数；用代码中的写路径、锁、队列消费等佐证。
     - 无代码：从 PRD 是否描述写库、版本切换、互斥场景、资源瓶颈；原型是否暗示单用户任务或全局任务；未写明则给出默认假设并待确认。
   - **性能信息建议**
     - 有代码：从循环、批量、日志与统计字段提炼指标。
     - 无代码：从 PRD 的验收指标、对象范围、耗时/条数类描述；原型的列表规模、进度文案；缺失则列出「建议采集」项并待实现选型确认。
   - **节点异常处理建议**
     - 有代码：事务边界、关键写步骤 vs 可降级读/算步骤，用代码佐证可跳过/必须终止。
     - 无代码：从 PRD 的异常流程、部分成功、跳过规则、一致性要求推导；未写清则分列「建议策略」与「待研发确认」。
   - **节点可终止性建议**
     - 有代码：事务提交段、不可逆写操作 → 不可终止；其前步骤结合取消 API 若存在则标可终止。
     - 无代码：从原型「取消/关闭」出现阶段与 PRD 是否允许中途落库推导；未约定则按行业惯例标注待确认。

## 4. 推荐输出格式（两类接入共用，不得删减结构）

在用户要求你「帮忙拆节点」时，你的标准输出应包含以下 **1～6**（顺序固定）；**无代码**时须在文末增加第 **7** 项。**有代码**时第 4～6 项内的「依据」小节标题用 **代码证明**；**无代码**时用 **PRD/原型依据**（写法：PRD 章节或需求编号 + 原型页面/步骤/控件说明）。

1. **接入类型声明**：一行写明本次为「有代码 / 无代码」；无代码时列出已阅读的 PRD、原型文件路径或名称。

2. **简短总述**：1～3 段话，概括业务场景、三大阶段和整体思路。

3. **节点表格**：使用 Markdown 表格，推荐表头为：
   - 顺序
   - 大类（数据输入 / 处理 / 数据输出）
   - 节点名称
   - 节点说明（业务语言）
   - 执行耗时（该节点起止耗时；可与成功信息合并为一列，但须显式体现）
   - 运行成功信息示例
   - 报错/警告信息示例

4. **并发模式建议**：
   - 建议的并发模式：可并行 / 不可并行 / 可排队 / 限制并发数（如限 3 并发）
   - 建议理由：（有代码）实现侧证据；（无代码）需求与交互侧证据
   - **代码证明** 或 **PRD/原型依据**（与接入类型一致）

5. **性能信息建议**：
   - 建议展示的性能指标：如处理物料数、生成订单数、计算耗时、通过率、异常条数等
   - 建议理由与 **代码证明** 或 **PRD/原型依据**

6. **节点配置建议**：
   - **异常处理策略**：
     - 异常可跳过的节点：节点名称 + 理由 + **代码证明** 或 **PRD/原型依据**
     - 异常必须终止的节点：节点名称 + 理由 + **代码证明** 或 **PRD/原型依据**
   - **可终止性配置**：
     - 可终止的节点：节点名称 + 理由 + **代码证明** 或 **PRD/原型依据**
     - 不可终止的节点：节点名称 + 理由 + **代码证明** 或 **PRD/原型依据**

**无代码时必须增加第 7 项**（与第 1～6 项同属交付物，不得省略）：

7. **实现后复核清单**：列出待研发/产品确认项及「有代码后应回填的证明位置」（便于从本输出过渡到「有代码」再跑一轮修订）。

如有必要，可额外补充小结，但要保持整体易扫读。

## 5. 信息不足时的应对方式

当用户给的信息非常少（例如只说“帮我拆一下 XX 的节点”），你需要：

1. 先判断更可能走 **有代码** 还是 **无代码**；若无法判断，问一个选择题（「是否已有该功能代码可检索？」）。
2. 基于通用经验假设这是一个「数据准备 → 处理 → 输出」的长时任务。
3. 给出一个**明确标注为“初稿/草案”**的节点拆分方案（**仍尽量按第 4 节完整结构输出**，依据处写「待补充材料」）。
4. 在同一条回答中提出 1～3 个关键澄清问题：有代码则索要路径/类名；无代码则索要 PRD 与原型路径或附件。

切记：不要因为信息不完整就拒绝回答，而应在说明假设前提的基础上，先给出一个合理初稿，再通过后续问答逐步收敛。
