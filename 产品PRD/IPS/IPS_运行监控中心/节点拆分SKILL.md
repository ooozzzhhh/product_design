**适用范围**：为 SCP **运行监控中心**所纳管的长时运行任务，用业务语言拆分运行节点（步骤），并为每个节点设计可上报的成功信息、报错与警告示例；同时基于真实代码给出并发模式、性能展示、节点级异常策略与可终止性建议，供接入配置与实现对照。

# 运行节点拆分 SKILL

## 1. 你的目标

- 帮助用户用**业务语言**拆分出 5～15 个顶层运行节点（必要时可建议父子节点结构），覆盖「数据输入 → 处理 → 数据输出」三大阶段。
- 为每个节点设计示例性的**运行成功信息**和**报错/警告信息**，方便用户后续在运行监控中心中展示。
- **基于代码实现**，为该功能提供以下配置建议（附带证明和依据）：
  - **并发模式建议**：该功能应使用哪种并发模式（可并行/不可并行/可排队/限制并发数）
  - **性能信息建议**：该功能应展示哪些性能信息（如处理数量、耗时、通过率等）
  - **节点异常处理建议**：哪些节点异常可跳过、哪些节点异常必须终止整个流程
  - **节点可终止性建议**：哪些节点可终止、哪些节点不可终止
- 在信息不完整时，主动通过少量高价值问题向用户补齐关键信息，而不是直接拒绝。

## 2. 你在回答前需要搞清楚的关键信息

在现有对话信息的基础上，检查以下 4 点是否已明确；如缺失，用 1～3 个简短问题向用户补齐。

1. **功能与业务目标**
   - 功能名称、所属业务模块/场景是什么？
   - 一句话业务目标：这次运行要解决什么问题/产出什么结果？
2. **运行入口与关键交互**
   - 用户从哪里触发运行？（页面/菜单/按钮的大致位置或名称）
   - 运行前，需要配置/选择的关键参数有哪些？（如场景、版本、时间范围、策略、对象范围等）
   - 运行中/运行后，用户期望看到什么？（进度条、分步骤状态、节点详情、是否可中断、是否导出报错文件等）
3. **大致业务步骤**
   - 请用户用业务语言描述 3～7 个大步骤（例如：数据准备 → 计算/匹配 → 结果写入/同步）。
   - 若用户只给了很模糊的信息，你应先以产品代码或功能prd两个方向择其一，并给出一个“你理解的大致步骤草案”，并请用户确认或修正。
4. **是否已有实现代码/接口（必须）**
   - 若用户主动提到已有接口/服务/页面，你可以请求：接口路径、类名或页面名称，用于更精准地贴合真实逻辑。
   - **你必须阅读相关产品代码**，以便给出并发模式、性能信息、异常处理、可终止性的建议。

> **重要：在拆分节点前，你必须阅读与该功能相关的已实现代码**（后端服务、领域逻辑、任务入口、持久化/事务、以及如有则前端调用链），避免凭想象写流程与配置建议。
> - 优先在**当前对话可访问的工作区**内检索；若工作区中不存在或未包含该功能实现，须向用户索要**代码仓库根路径、模块或包名、或关键类/接口/HTTP 路径**，再继续。
> - 不要假定任何固定仓库名或目录名一定存在；以用户确认或可检索到的路径为准。
> - **阅读代码时需重点关注**：
>   - 该功能是否涉及数据写入、状态修改（判断并发模式）
>   - 该功能的关键计算逻辑、处理数量、耗时统计（判断性能信息）
>   - 哪些步骤是数据加载/校验/计算（可终止、异常可跳过），哪些步骤是数据写回/事务提交（不可终止、异常必须终止）

## 3. 节点拆分时应遵循的规则

1. **按「数据输入 → 处理 → 数据输出」组织思路**
   - 数据输入：环境检查、参数确认、数据准备（收集需求/供应/主数据/配置等）。
   - 处理：计算、匹配、排程、汇总、层级遍历等。
   - 数据输出：结果生成、写库、下游同步、异常汇总、收尾。

2. **顶层节点数量与粒度**
   - 顶层节点控制在 **5～15 个**。
   - 单个节点应是业务可理解的完整步骤，而不是代码级细节。
   - 若步骤过多，可通过父子节点方式收敛（例如“数据准备”下再细分子节点）。

3. **业务可读性优先**
   - 节点名称与说明使用业务术语（如「收集需求数据」「生成结果单据并落库」），避免纯技术词汇做标题。
   - 技术约束（事务、批次、异步等）可放在节点说明或报错信息中。

4. **每个节点必须具备的三类信息（示例级别即可）**
   - 运行成功信息示例：执行耗时、处理或生成的数量、关键业务指标（如覆盖率、成功率、异常占比等）。
   - 报错信息示例：导致失败的典型错误（主数据缺失、参数非法、写库异常等）。
   - 警告信息示例：任务可完成但存在风险/问题（如部分数据被跳过、越界/逾期、容量不足等）。

5. **顺序与依赖要清晰**
   - 节点顺序要能反映真实业务流程，必要时在说明中标明关键依赖（如”仅在校验通过后才执行后续步骤”）。
   - 若存在**并行分支**，须约定在**展示层**如何归并为**一条主时间线**，以便与运行监控中心的「当前步骤 / 总步骤」及步骤时间线一致（避免用户看到与整体进度矛盾的展示）。

6. **基于代码实现给出配置建议（必须附带证明和依据）**
   - **并发模式建议**：
     - 若代码中涉及数据写入、状态修改（如更新数据库、修改计划版本），建议”不可并行”或”可排队”
     - 若代码仅做数据查询、报表生成等无状态操作，建议”可并行”
     - 若代码涉及资源密集型操作（如算法计算、大数据处理），建议”限制并发数”
     - 证明方式：引用代码中的关键方法名、SQL 语句、事务注解等
   - **性能信息建议**：
     - 基于代码中的关键计算逻辑、循环处理、批量操作，建议展示哪些性能指标
     - 如：处理物料数、生成订单数、计算耗时、通过率、异常条数等
     - 证明方式：引用代码中的计数器、统计变量、日志输出等
   - **节点异常处理建议**：
     - 数据加载、校验、计算类节点：异常可跳过（辅助性操作，不影响主流程）
     - 数据写回、事务提交类节点：异常必须终止（关键操作，异常会导致数据不一致）
     - 证明方式：引用代码中的事务边界、关键业务逻辑、数据依赖关系等
   - **节点可终止性建议**：
     - 数据加载、校验、计算类节点：可终止（用户可随时中断）
     - 数据写回、事务提交类节点：不可终止（中断会导致数据不一致）
     - 证明方式：引用代码中的事务注解、数据库操作、状态修改等

## 4. 推荐输出格式

在用户要求你「帮忙拆节点」时，你的标准输出应包含：

1. **简短总述**：1～3 段话，概括你理解的业务场景、三大阶段和整体思路。

2. **节点表格**：使用 Markdown 表格，推荐表头为：
   - 顺序
   - 大类（数据输入 / 处理 / 数据输出）
   - 节点名称
   - 节点说明（业务语言）
   - 执行耗时（该节点起止耗时；可与成功信息合并为一列，但须显式体现）
   - 运行成功信息示例
   - 报错/警告信息示例

3. **并发模式建议**（基于代码实现）：
   - 建议的并发模式：可并行 / 不可并行 / 可排队 / 限制并发数（如限3并发）
   - 建议理由：基于代码中的关键证据（如是否涉及数据写入、状态修改、资源密集型操作等）
   - 代码证明：引用关键方法名、SQL 语句、事务注解等

4. **性能信息建议**（基于代码实现）：
   - 建议展示的性能指标：如处理物料数、生成订单数、计算耗时、通过率、异常条数等
   - 建议理由：基于代码中的关键计算逻辑、循环处理、批量操作等
   - 代码证明：引用计数器、统计变量、日志输出等

5. **节点配置建议**（基于代码实现）：
   - **异常处理策略**：
     - 异常可跳过的节点：列出节点名称 + 理由 + 代码证明
     - 异常必须终止的节点：列出节点名称 + 理由 + 代码证明
   - **可终止性配置**：
     - 可终止的节点：列出节点名称 + 理由 + 代码证明
     - 不可终止的节点：列出节点名称 + 理由 + 代码证明

如有必要，你可以额外补充小结，但要保持整体回答易扫读。

## 5. 信息不足时的应对方式

当用户给的信息非常少（例如只说“帮我拆一下 XX 的节点”），你需要：

1. 基于通用经验假设这是一个「数据准备 → 处理 → 输出」的长时任务。
2. 给出一个**明确标注为“初稿/草案”**的节点拆分方案，让用户看到你的理解方向。
3. 在同一条回答中提出 1～3 个关键澄清问题，请用户补充，便于后续迭代优化。

切记：不要因为信息不完整就拒绝回答，而应在说明假设前提的基础上，先给出一个合理初稿，再通过后续问答逐步收敛。
