# 1. 背景分析

## 1.1 业务背景

本 PRD 源于《供应链计划系统（Java）异常处理规范与开发指导》的落地需求。该规范定义了系统级异常处理的技术规范，但尚未以产品需求文档形式呈现，开发与 AI 编码时缺乏统一可执行的产品级输入。本规范版 PRD 旨在将规范转化为 PRD 格式，并补充规范中未涉及的**报错提示语句编写规范**，供开发与 AI 编码时直接遵循。

## 1.2 目标用户

- **开发人员**：编写或改造 Java 异常代码时需遵循本 PRD
- **AI 编码助手**：在生成代码时需按本 PRD 的规范约束输出
- **产品经理**：作为验收与规范维护的参考依据

## 1.3 典型使用场景

- 新增业务异常时：按枚举规范、抛出层级、错误码格式编写
- 改造现有报错点时：对照不合规清单，替换为规范写法
- 编写报错文案时：对照第 3 章场景模板与禁用表述

## 1.4 需求来源

- 规范来源：`产品设计/产品需求/IPS/IPS_平台报错规范化优化/供应链计划系统（Java）异常处理规范与开发指导.md`
- 不合规清单：`产品设计/产品知识库/IPS/代码/IPS_不符合规范报错提示清单_对照规范.md`
- 与已有 PRD（20250304143000）的迭代关系：本规范版 PRD 侧重「规范转 PRD + 报错语句编写规范补充」，已有 PRD 侧重「全量改造示范清单、模块级施工图」

# 2. 设计目标

## 2.1 目标提炼

| 目标类型 | 具体目标 |
|----------|----------|
| **技术规范落地** | 异常定义、编码、抛出、处理逻辑统一；异常信息支持国际化；异常描述支持占位符 |
| **文案规范补充** | 报错提示语句符合编写规范（用户可理解、可操作性、一致性） |
| **可执行性** | 供开发与 AI 在编写代码时直接遵循，无需反复查阅规范原文 |

## 2.2 预期业务价值

- 降低技术信息泄露风险，提升用户安全与体验
- 统一错误码与错误文案，便于问题定位与客服沟通
- 减少开发与 AI 在编写报错时的随意性，提高产出质量

# 3. 设计现状

## 3.1 PRD 概述

本规范版 PRD 共 6 章，结构如下：

| 章节 | 内容 | 来源 |
|------|------|------|
| 1 | 需求概述 | 本 PRD |
| 2 | 规范转 PRD：异常处理技术规范 | 与《供应链计划系统（Java）异常处理规范与开发指导》一致 |
| 3 | 报错提示语句编写规范 | **本 PRD 补充**（规范中未提及） |
| 4 | 与现有 PRD 的衔接 | 本 PRD |
| 5 | 用例（规范符合性验收） | 本 PRD |
| 6 | 未来扩展 | 本 PRD |

**核心设计要点**：

- **第 2 章**：异常分类与业务大类编码（DP/SOP/MPS/MRP/MDS/SCH/SYS）、枚举设计规范（`BusinessErrorEnum`、code/desc、国际化、占位符）、抛出规范（Service 层抛出、DAO 异常转换）、禁止行为
- **第 3 章**：总体原则（用户可理解、可操作性、一致性）、按场景文案模板（参数缺失、配置缺失、数据不存在、状态不允许、导入/导出失败、任务运行中、系统错误）、禁用表述、术语统一、标点与格式
- **第 4 章**：明确与已有 PRD 的协调关系：技术规范以本 PRD 为准；错误码命名两套并存（枚举 code 与 errorCode）；文案模板与改造清单以已有 PRD 为准，本 PRD 第 3 章为补充

## 3.2 原型设计

本 PRD 无前端设计，无原型图。符合任务背景说明「无需前端设计」。

# 4. 设计问题

## 4.1 功能完整性

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| **规范与 PRD 的衔接存在缺口** | 中 | 规范来源文档「抛出规则」示例使用 `getDesc()`（如 3.2.1 `throw new BusinessException(BusinessErrorEnum.SOP_DATA_EMPTY.getDesc())`），而本 PRD 2.3 已正确使用 `getI18nErrordesc()`。规范要求「禁止硬编码」且支持国际化，本 PRD 的写法符合要求。建议在本 PRD 第 2 章或第 4 章显式说明：「规范来源文档的抛出示例使用 getDesc()，与本 PRD 不符；开发与 AI 应以本 PRD 的 getI18nErrordesc() 为准」，或建议在规范来源文档中同步修正示例，避免开发查阅规范时产生混淆。 |
| **枚举 code 与 errorCode 的映射关系未细化** | 中 | 第 4 章说明「两套可并存：枚举 code 用于 i18n key，errorCode 用于 API 响应与日志，需在实现时建立映射」。但未给出映射规则（如 DP001→IPS_AUTH_* 的对应关系）、示例或实现指引，开发与 AI 易产生歧义。 |
| **用例覆盖不足** | 低 | 第 5 章用例仅 5 条，覆盖抛出层级、枚举+国际化、无技术透传、占位符匹配、文案规范。缺少：DAO 异常转换、循环内禁止抛异常、禁止模式等验收用例。 |
| **与不合规清单的对应关系未显式建立** | 低 | 不合规清单按 7 个合规维度分类，本 PRD 第 3 章报错语句编写规范主要对应「文案来源」「技术信息」维度，但未在 PRD 中显式引用不合规清单的维度与条目，便于改造时对号入座。 |

## 4.2 用户体验（开发/AI 使用体验）

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| **报错语句编写规范的实用性较好** | — | 第 3 章按场景模板（参数缺失、配置缺失、数据不存在等）设计清晰，禁用表述与术语统一表可直接指导编写，具有较高实用性。 |
| **场景模板与已有 PRD 存在重叠** | 中 | 本 PRD 3.2 按场景的文案模板与已有 PRD 4.3.5.2 按场景的文案模板存在重叠（如参数缺失、配置缺失、系统错误等）。第 4 章说明「文案模板与改造清单以现有 PRD 第 4.3、第 5 章为准」，但未明确本 PRD 3.2 与 4.3.5.2 的优先级关系：是「本 PRD 为补充、冲突时以已有 PRD 为准」，还是「两者合并使用」。建议在 PRD 中显式说明「本 PRD 3.2 与已有 PRD 4.3.5.2 互补，新增场景时优先参考本 PRD 3.2 的简洁模板，已有 PRD 4.3.5.2 提供更细场景与典型落地模块」。 |
| **文档跳转路径较多** | 低 | 开发需同时查阅规范、本 PRD、已有 PRD、不合规清单。建议在 PRD 开篇增加「使用指引」：先看本 PRD 第 2 章技术规范 + 第 3 章文案规范；改造时再查已有 PRD 第 5 章改造示范清单；验收时对照不合规清单维度。 |

## 4.3 业务流程支持

| 问题 | 严重程度 | 说明 |
|------|----------|------|
| **与已有 PRD 的协调一致性总体良好** | — | 第 4 章明确技术规范以本 PRD 为准、文案模板与改造清单以已有 PRD 为准，本 PRD 第 3 章为补充，避免了规范冲突。 |
| **错误码双轨制的落地风险** | 中 | 枚举 code（DP001、SOP001）与 errorCode（IPS_AUTH_USER_NOT_FOUND_01）两套并存，若实现时未建立映射或映射规则不清，会导致 i18n 与 API 响应不一致、日志检索困难。建议在 PRD 中补充「实现时 mapping 表或示例」，或明确「本期改造以已有 PRD 的 errorCode 为准，枚举 code 仅用于 i18n key，两者需在实现时建立一一对应」。 |

# 5. 竞对对标（SAP）

## 5.1 功能对比

| 维度 | SAP IBP / 典型企业级做法 | 本规范版 PRD |
|------|--------------------------|--------------|
| 错误码体系 | 统一错误码（如 BAPI 错误码），结构化返回 | 支持业务大类前缀 + 3 位序号（DP001/SOP001）、以及模块_场景域_问题类型_序号（IPS_AUTH_*）两套 |
| 国际化 | 错误信息支持多语言 | 支持 messages_zh_CN.properties、messages_en_US.properties |
| 可追溯性 | traceId/requestId 支持问题定位 | 已有 PRD 与未来扩展中涉及 traceId；本规范版 PRD 未单独展开 |
| 用户可见文案 | 技术信息与用户文案分离 | 本 PRD 明确禁止 e.getMessage() 透传，强调用户可理解、可操作性 |

## 5.2 设计差异

- **SAP**：错误码与文案通常由平台统一管理，业务模块按规范注册；本 PRD 强调各业务模块（DP/SOP/MPS/MRP 等）按业务大类前缀编码，与 SAP 的集中式管理略有差异，但更符合多模块分布式架构。
- **SAP**：错误码与 i18n key 多为同一标识；本 PRD 存在枚举 code 与 errorCode 双轨，需在实现时建立映射，避免歧义。

## 5.3 最佳实践参考

- 错误码统一、用户可见文案与技术信息分离、可追溯 ID（traceId）— 本 PRD 及已有 PRD 已对齐。
- 建议：在实现时明确 traceId 的注入位置（网关/链路/本仓），确保与 SAP 级可观测性一致。

# 8. 审核结论

## 8.1 设计成熟度评估

| 维度 | 评估 | 说明 |
|------|------|------|
| 规范与 PRD 衔接 | 良好 | 第 2 章与规范基本一致，存在少量抛出方法命名不一致需修正 |
| 报错语句编写规范 | 良好 | 第 3 章按场景模板、禁用表述、术语统一设计清晰，实用性高 |
| 与已有 PRD 协调 | 良好 | 第 4 章明确了分工关系，但双轨错误码的映射与两套文案模板的优先级需进一步细化 |

**综合评估**：设计成熟度**中等偏上**，可进入开发落地；建议在实施前完成上述高优先级修正与补充。

## 8.2 优先级建议

| 优先级 | 建议 | 说明 |
|--------|------|------|
| P0 | 明确抛出方法以本 PRD 为准 | 避免开发查阅规范来源时误用 getDesc()，应以 getI18nErrordesc() 为准 |
| P1 | 补充枚举 code 与 errorCode 映射指引 | 降低实现歧义，确保 i18n 与 API 响应一致 |
| P2 | 明确本 PRD 3.2 与已有 PRD 4.3.5.2 的关系 | 避免开发在改造时两套模板冲突 |
| P3 | 扩展用例、增加使用指引 | 提升验收完整性与使用体验 |

## 8.3 后续行动计划

1. **产品经理**：根据本审核报告修正 PRD 中高优先级问题（抛出方法、映射指引、模板关系说明）。
2. **开发**：改造时以本 PRD 第 2 章技术规范 + 第 3 章文案规范为约束，以已有 PRD 第 5 章改造示范清单为对号入座依据；验收时对照不合规清单的 7 个合规维度。
3. **AI 编码**：在生成异常相关代码时，优先引用本 PRD 第 2、3 章，避免使用 getDesc()、e.getMessage() 透传等禁止模式。
