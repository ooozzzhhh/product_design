# SDS 供应汇总功能优化 — 产品设计审核报告

---

## 1. 背景分析

### 1.1 业务背景

SDS（供需管理）模块中的「物料供应能力汇总表」（supplySummary）是供需协同的核心查询视图，负责将来自 5 类供应来源（采购订单、调拨订单、生产订单、期初库存、采购预测库存）的数据聚合为「维度×时段」矩阵表格，支持按供应分类筛选与多维度分组展示。

本次需求源自问题分析报告 `SDS_BUG_2026-03-06_01`，属于「问题驱动型」代码优化需求，解决三类已知缺陷：架构可扩展性瓶颈、接口协议缺陷导致的列排序问题，以及安全合规问题（SQL 信息外泄 + 两个附带 Bug）。

### 1.2 目标用户

- **直接用户**：研发与测试团队（本需求产出为代码改造交付物）
- **间接受益方**：使用供应汇总表的计划员、物料管理员（感知到列顺序修正、错误提示改善）
- **运维/实施团队**：未来扩展供应来源时受益于「只写数据不改代码」的架构改造

### 1.3 需求来源与问题背景

| 问题 | 根因 | 影响 |
|------|------|------|
| 可扩展性差 | 每新增供应来源需同步修改 `getTableNames()`、6 条 DAO SQL、`SupplyTypeEnum` 三处代码 | 遗漏任何一处将导致数据静默缺失，且需发版部署 |
| 列顺序错乱 | 接口响应未含列元数据，前端无法区分维度列与时间列的顺序 | 用户无法得到预期的「维度列在左、时间列在右」视图 |
| SQL 信息外泄 | `groupByParam` 直接字符串插值进 SQL 的 GROUP BY 子句，无白名单校验 | `BadSqlGrammarException` 携带 SQL 内容透传前端，存在安全隐患 |
| 两个附带 Bug | `orderFlag` 去重判断格式不一致（camelCase vs 下划线）；`translate()` 在空列表时直接 `get(0)` 越界崩溃 | 维度下拉重复项、特定数据场景下服务崩溃 |

## 2. 设计目标

从 PRD 提炼的设计目标如下：

| 目标 | 预期业务价值 | 量化指标 |
|------|--------------|----------|
| 降低供应来源扩展成本 | 未来新增供应类型只向通用表写数据，不改代码、不发版 | 从「改 3 处代码 + 回归测试 + 部署」降为「写入数据库记录」 |
| 修复列顺序混乱 | 维度列始终在左、时间列在右，表格可读性提升 | `columns` 字段中 `DIMENSION` 列 `order` 值均小于所有 `TIME` 列 |
| 消除 SQL 信息外泄 | 前端响应体中不再包含任何 SQL 片段或技术堆栈信息 | 安全扫描：任意非法 `groupByParam` 均返回业务提示，响应体无 SQL 关键词 |
| 规范异常处理 | 对齐项目 `BusinessErrorEnum` + `getI18nErrordesc()` 规范，错误提示国际化 | 3 处硬编码文案替换为 SDS001～SDS003 标准错误码 |
| 修复两个附带 Bug | 维度下拉不再出现重复「订单标识」；枚举匹配失败时服务不崩溃 | Bug A/B 的相关用例全部通过 |

## 3. 设计现状

### 3.1 PRD 概述

PRD 将本次优化拆分为三个子模块，各自独立，改造清晰：

**子模块 4.1 — 供应来源可扩展机制（架构改造 + 新建数据库表）**

新建通用供应大表 `sds_supply_universal`，核心字段为 `supply_version_id`、`supply_type`（VARCHAR(64) 供应子类型标识）、`item_code`、`stock_point_code`、`supply_date`、`quantity`、`order_flag`。同时对现有 6 条 DAO SQL 各追加一段固定的 UNION ALL 分支，`getTableNames()` 和 `SupplyTypeEnum` 同步更新。改造完成后，新增供应类型只需向通用表写数据，不再改 SQL 或代码。

**子模块 4.2 — 表格列元数据接口（接口协议改造）**

在 `/supplySummary/page` 接口响应中新增 `columns` 字段（`List<ColumnDefinition>`），与 `list`、`total` 并列。`ColumnDefinition` 包含 `key`、`label`、`type`（DIMENSION/TIME）、`order` 四个字段。维度列从 `groupByParam` 解析，时间列从 MDS feign 返回的时段列表生成，前端按 `order` 升序渲染。原有 `list` 和 `total` 字段保持不变，向后兼容。

**子模块 4.3 — 输入校验与异常处理规范化（安全加固 + 规范整改）**

新增 `groupByParam` 白名单校验（调用 `dimensionDropdown()` 获取合法字段集）；在 DAO 调用处包裹 try-catch 防止 `BadSqlGrammarException` 透传；3 处硬编码 `BusinessException` 替换为 `BusinessErrorEnum`（SDS001～SDS003）；修复 Bug A（`orderFlag` 去重判断改为下划线格式）；修复 Bug B（`translate()` 的 `get(0)` 前增加 `isEmpty()` 判断）。

### 3.2 原型设计

本需求为后端代码优化类，经产品经理与用户确认后跳过原型设计。PRD 中各子模块均注明「本子模块为纯后端架构改造，不涉及前端页面变更，无需原型图」。前端感知到的变化仅为：
- 响应结构中新增 `columns` 字段（向后兼容）
- 错误提示文案从技术字符串改为业务友好文案
- 维度下拉列表中「订单标识」不再重复出现

## 4. 设计问题

### 4.1 功能完整性

#### 4.1.1 遗漏点：`queryCriteriaParam` 的 WHERE 条件参数缺乏校验

**问题描述**：PRD 4.3 中明确提到「对 `queryCriteriaParam` 仅在文档中提及建议校验，未在本期实现」（第 5.3 节）。`queryCriteriaParam` 作为 WHERE 条件参数，若也存在字符串插值进 SQL 的情况，同样存在 SQL 注入风险，但本期未对其做白名单或格式校验。

**风险等级**：中（本期明确暂缓，属有意识的 defer，已在第 5.3 节标记为后续迭代）

**建议**：在本期 PRD 中明确记录「`queryCriteriaParam` 的 SQL 注入风险已识别，后续迭代补齐校验」，同时在代码注释中标注 TODO，防止遗忘。

#### 4.1.2 遗漏点：`sds_supply_universal` 缺少数据写入规范

**问题描述**：PRD 4.1 详细描述了通用表的 DDL 结构和查询接入方式，但未说明「谁负责向 `sds_supply_universal` 写数据」、「通过何种接入方式写入」（API 接口？数据导入？ETL 作业？）。若没有明确的数据写入机制，通用表实际上是一张永远为空的表，架构改造的价值无法落地。

**风险等级**：高（影响 4.1 改造目标的实际业务价值）

**建议**：在 PRD 中补充「通用表数据写入机制」章节，至少说明：第一批使用通用表的供应来源是什么、由哪个系统/服务写入、写入时机与频率。若本次改造仅为「预留架构能力」，需在 PRD 中显式声明并得到业务方确认。

#### 4.1.3 遗漏点：`SupplySummaryPageResult` 包装类的创建位置与影响范围

**问题描述**：PRD 4.2.2 提及「建议新增一个 `SupplySummaryPageResult` 包装类（若现有包装类不便扩展）」，但未明确：
- 是否一定要新建包装类，还是可以扩展现有 VO？
- 若新建包装类，Controller 返回类型是否需要同步变更？
- 影响的接口 Swagger/API 文档是否需要更新？

**风险等级**：低（实现细节模糊，可能导致研发自行决策）

**建议**：明确「`SupplySummaryPageResult` 是否为新建类还是扩展现有类」，并在改动文件清单（4.3.5.7）的格式下补充 4.2 的改动文件清单。

#### 4.1.4 遗漏点：`dimensionDropdown()` 作为白名单来源的性能影响

**问题描述**：PRD 4.3.5.3 的白名单校验伪代码调用 `getAllowedFieldNames()`，其来源是 `dimensionDropdown()` 的结果。若 `dimensionDropdown()` 内部涉及 `information_schema` 动态查询（PRD 中提到「动态查询出的公共字段名」），则每次 `page()` 请求均会额外触发一次 `information_schema` 查询，高并发场景下可能带来性能问题。

**风险等级**：中（`information_schema` 查询在高并发下是已知的 MySQL 性能瓶颈）

**建议**：明确 `getAllowedFieldNames()` 的实现方式：优先方案为在应用启动时缓存合法字段集（字段集为相对静态数据），避免每次请求触发 `information_schema` 查询。在 PRD 业务规则中补充此缓存策略。

### 4.2 用户体验

由于本需求为纯后端改造，不涉及前端页面新增，用户体验维度主要体现在错误提示文案质量和列排序修复效果。

#### 4.2.1 错误提示文案质量：总体合格，有一处细节待确认

PRD 中的三条错误文案：
- SDS001：「尚未配置『时段』，请前往『时段设置』完成配置后重试。」—— 符合规范，指出配置项和菜单路径。
- SDS002：「『供应类型』参数不合法，请刷新页面后重新选择。」—— 符合规范，指出操作建议。
- SDS003：「『聚合维度』包含不合法字段，请重新选择聚合维度后查询。」—— 符合规范。

**待确认**：SDS001 中「时段设置」是否为前端实际存在的菜单路径名称？若菜单名不一致，用户按提示找不到对应入口，文案将失去指引价值。建议研发在实现前与前端确认菜单路径名称。

#### 4.2.2 `translate()` 降级行为的用户体验

PRD 规定 Bug B 修复后，`translate()` 在枚举匹配失败时降级返回原始 code 值（如 `UNKNOWN_ENUM_CODE`）。该 code 值对业务用户无语义，将直接呈现在列表数据中。虽然此场景（数据库中存在未知 `supply_type`）属于异常情况，但降级展示原始 code 可能引起业务用户疑问。

**建议**：在降级时返回「未知供应类型」（中文）而非原始 code 字符串，对业务用户更友好。或在 warn 日志基础上同步触发告警，便于运维及时修复数据。

### 4.3 业务流程支持

#### 4.3.1 计划/非计划分类下的 `order_flag` 过滤逻辑与现有表一致性

PRD 规定通用表的计划/非计划归属由 `order_flag` 字段决定（`NO`=计划，`YES`=非计划），与现有 5 张供应表一致。此设计合理，无问题。

#### 4.3.2 前端切换维度后的 `columns` 缓存清理

PRD 4.2.6 指出「切换聚合维度重新查询时，`columns` 随响应动态更新，前端需重新渲染列头（不缓存旧列定义）」。这是对前端的行为约束，但 PRD 中未明确「前端如何感知需要清除缓存」——是否需要新增一个版本号字段或时间戳字段帮助前端判断列定义是否变化？

**建议**：若前端存在列定义缓存机制，建议在 `columns` 中增加一个 `version` 或 `groupByParamHash` 字段，使前端能精确判断是否需要重建列。若前端每次请求后均无条件重渲染列，可忽略此建议。

## 5. 竞对对标（SAP IBP + Blue Yonder Luminate Planning）

PRD 第 3 章已提供了较为完整的初步对标分析，本节在此基础上深化。

### 5.1 功能对比

| 对比维度 | 本次方案 | SAP IBP | Blue Yonder Luminate Planning |
|----------|----------|---------|-------------------------------|
| 供应来源扩展机制 | 通用大表 `sds_supply_universal` + 固定 UNION ALL 分支；新增供应类型只写数据 | 「Supply Source Category」配置化机制；通过 HANA 视图层 UNION 合并，应用层无感知 | 统一 Supply Fact 表 + `supply_type` 字段区分；查询层对具体供应来源表无感知 |
| 列元数据下发方式 | 服务端在 `/page` 响应中附带 `columns` 列表（key/label/type/order） | 「Key Figure」+「Planning Level」二维结构；列定义完全由服务端下发，支持固定列/滚动列区分 | 「View Definition」接口独立下发列定义；含每列数据类型、排列顺序，并记忆上次用户 Group By 选择 |
| 错误码与国际化 | `BusinessErrorEnum` + `messages_zh_CN/en_US.properties`；支持中英文 | 统一 Error Code + 国际化机制；Error Code 与 HTTP 状态码解耦 | 统一 Error Code + 国际化；客户端通过 Error Code 决定是否重试/跳转配置页 |
| 前端供应子类型下拉 | 目前前端下拉只支持 3 种分类（计划/非计划/全部），不区分通用表内子类型；后续扩展计划中 | 前端支持按 Supply Source Category 细粒度筛选 | 前端支持按 Supply Type 动态加载筛选项 |

### 5.2 设计差异

**差异 1：列元数据是否独立接口**

IBP 和 Blue Yonder 均将列定义从数据查询接口中解耦，通过独立接口（View Definition）下发。本次方案将 `columns` 与 `list`/`total` 合并在同一接口响应中。

- 优点：实现简单，减少一次前端 HTTP 请求。
- 潜在问题：每次翻页查询都会重传列定义（列定义通常是静态的，不随分页变化），存在轻微带宽浪费；更重要的是，若未来列定义需要在不查询数据的情况下获取（如「切换维度时先渲染列头，再异步加载数据」），当前设计无法支持。

**差异 2：供应来源扩展的底层实现**

IBP 使用数据库视图（HANA View）合并所有供应来源，应用层 SQL 完全屏蔽底层多表结构。Blue Yonder 使用统一 Fact 表。本次方案使用固定 UNION ALL 分支（虽有通用大表，但现有 5 张供应表的分支未迁移到通用表，仍保留在 SQL 中）。

PRD 第 5.1 节已识别此问题，将「DAO SQL 视图化」列为长期演化方向，属于有意识的分阶段实现，合理。

**差异 3：列元数据字段的丰富程度**

本次 `ColumnDefinition` 仅含 4 个字段（key/label/type/order），而 IBP/BY 的列定义通常还包含 `width`（列宽）、`frozen`（是否固定列）、`sortable`（是否可排序）。PRD 第 5.2 节已将这三个字段列为未来扩展方向，属于合理的 MVP 设计。

### 5.3 最佳实践参考

**参考 1（来自 Blue Yonder）：列定义记忆用户上次 Group By 选择**

Blue Yonder 的 View Definition 接口会记忆用户最近一次的 Group By 选择，在响应中回传。本次方案未实现此特性（前端每次需重新选择维度）。对于高频使用用户，此特性可提升操作效率，建议作为后续迭代点。

**参考 2（来自 IBP）：Error Code 与 HTTP 状态码解耦**

IBP 中 Error Code 与 HTTP 状态码独立设计，客户端可根据 Error Code 决定是否重试、是否跳转配置页。本次方案的 SDS001～SDS003 均归属于同一 HTTP 200（业务错误码）模式，与现有项目规范一致，无差异。

**参考 3（来自两款产品）：供应来源配置化**

两款产品均通过配置化（而非枚举）管理供应来源类型。本次方案仍使用 `SupplyTypeEnum` 硬编码枚举，是 MVP 阶段的合理取舍，但 PRD 第 5.1 节的「供应类型配置化」演化路径应被纳入产品 Roadmap，并告知业务方当前是「枚举驱动」阶段。

## 6. 设计修正与补充

### 6.1 功能修正

#### 6.1.1 【必要】补充通用大表数据写入机制说明

**背景**：4.1.2 中已识别，`sds_supply_universal` 建好后需要有数据写入机制，否则架构改造无实际业务价值。

**修正内容**：在 PRD 4.1 中新增「4.1.8 数据写入机制」章节，明确说明：
- 第一批接入通用表的供应来源是什么（即使本次暂无，也需显式说明「本次暂无供应来源接入，通用表为预置架构」）

#### 6.1.2 【建议】`queryCriteriaParam` 安全风险的 TODO 标记

**背景**：4.1.1 中已识别，`queryCriteriaParam` 的 WHERE 条件参数未做白名单校验，属于已知风险的有意暂缓。

**修正内容**：在 PRD 5.3 节的「输入校验的后续加固」中，将「建议校验」改为「计划在下一个迭代补齐校验」，并在研发实现代码中对 `queryCriteriaParam` 相关处理位置加 TODO 注释，注明「SQL 注入风险已知，待下一迭代补齐白名单校验，参考 SDS-BUG-2026-03-06-01 问题分析报告」。

#### 6.1.3 【建议】`translate()` 降级文案改为业务友好提示

**背景**：4.2.2 中已识别，原始 code 值（如 `UNKNOWN_ENUM_CODE`）对业务用户无语义。

**修正内容**：将规则 4.3.5.6 的 fallback 逻辑修改为返回「未知供应类型」（中文）或 `unknown supply type`（英文，通过 i18n 机制）而非原始 code 字符串。新增 i18n key：

```properties
# 中文
SDS_TRANSLATE_UNKNOWN=未知供应类型
# 英文
SDS_TRANSLATE_UNKNOWN=Unknown supply type
```

### 6.2 界面优化

本需求为纯后端改造，无界面新增或结构性调整。以下建议针对前端配合改动的说明文档。

#### 6.2.1 【建议】补充 `columns` 列定义的前端渲染指引

PRD 4.2.6 已有前端改动说明，但建议将以下边界场景明确写入文档供前端参考：
- `columns` 为空数组时（MDS 未配置时段且 `groupByParam` 为空），前端表格的降级展示方式
- `columns` 中 `DIMENSION` 列的 `key` 值未在 `list[0]` 中找到对应字段时（异常场景），前端的容错处理

### 6.3 交互改进

#### 6.3.1 【建议】考虑将列定义接口独立化（后续迭代）

**背景**：参考 Blue Yonder 的「View Definition 独立接口」模式（见 5.2 差异 1）。

**本期无需实施**，但建议在产品 Roadmap 中记录：若未来需要支持「切换维度时先渲染列头、再异步加载数据」的交互模式，需将列定义独立为 `/supplySummary/columns` 接口，接受 `groupByParam` 作为入参，返回 `List<ColumnDefinition>`。

#### 6.3.2 【建议】白名单校验的 `getAllowedFieldNames()` 增加应用级缓存

**背景**：见 4.1.4 分析，`information_schema` 动态查询在高并发下有性能风险。

**建议实现**：在 `SupplySummaryServiceImpl` 中，将 `getAllowedFieldNames()` 的结果使用 `@Cacheable` 或 static 变量在应用级缓存，缓存生命周期与应用实例生命周期一致（字段集为静态数据，不需要动态刷新）。

## 8. 审核结论

### 8.1 设计成熟度评估

| 子模块 | 成熟度 | 说明 |
|--------|--------|------|
| 4.1 供应来源可扩展机制 | **中等偏高** | 架构方案清晰，DDL 和 SQL 改造细节完整，可直接交付研发；主要缺口是通用表数据写入机制未说明，影响实际业务价值的落地 |
